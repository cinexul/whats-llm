〔本章で Claude Code のインストール・コマンド・画面・能力に触れる箇所は、いずれもバージョン更新とともに変わります。一律 **公式ドキュメントを参照** してください。本章では「何か・いつ使うか・どう判断するか・何に気をつけるか」だけを扱います。一字一句のコマンドや深い機能は **付録 C** と公式ドキュメントを参照してください。〕

第24章で、私たちは「調査 → 計画 → 実行 → 検収」という共通ワークフローを徹底的に学びました。本章と次章は、それを **具体的なツールの上に落とし込む** 回です――本章は **Claude Code**、次章は **Codex** を扱います。二つの章はわざと「別々に」書きます。両者は同じ仕事のやり方を共有してはいるものの、**どうインストールし、どう操作し、それぞれどんな癖を持つか** は同じではないからです。

まず安心できる一言を。**本章も、あなたがコードを書けることを求めません。** ここで作るのは、「初めて Claude Code に触れるとき、頭の中に持っておくべき一枚の地図」です――どんな姿をしているのか、CLAUDE.md というものは結局何をしているのか、よく使う操作の勘どころは何か。「どのコマンドを打つ、どのキーを押す」の具体は、本章では一切書き込まず、すべて公式ドキュメントと付録 C に委ねます。理由は第24章で述べたとおりです。こうしたものは更新が速すぎて、本に書き込めば古くなり、かえってあなたを誤らせるだけだからです。

## 一、最小の入門マインド：まず「いくつもの顔を持つ」と認める

多くの人が最初の一歩でつまずくのは、頭の中で Claude Code を「ある固定の Web ページ」や「ある固定のソフトのウィンドウ」だと思い込むからです。**そうではありません。**

Claude Code は **Anthropic** 発で、コアは **Claude**――この二点は安定した事実で、覚えておいて構いません。しかしその **形態は一種類ではありません**。**ターミナル（コマンドライン）、コードエディタ(IDE)プラグイン、デスクトップや Web の画面、さらにはクラウドのタスク** として目にすることもあります。**自分のマシンで動くものもあれば、クラウドで動くものもあり、具体的な形態はそれぞれ異なり、しかも更新され続けています。公式ドキュメント／製品の公式な形態を参照してください。**

ですから、正しい入門マインドは次の三つです。

- **「それが結局どんなソフトなのか」にこだわらない。** まず公式ドキュメントで「いま推奨されるのはどの形態か、どうインストールするか」を見て、そのとおりに一度なぞればよいのです（**公式ドキュメントを参照**）。
- **コアが変わらないことを見定める。** どの形態であっても、背後にいるのは「自分でプロジェクトを読み、ファイルを変更し、コマンドを実行する」Claude です――第24章の「連れて仕事をする方法」は、どの形態でも通用します。
- **最初は欲張らない。** インストールしたら、まず **やり直しのきく小さなプロジェクト** で、最小のリクエストを一つ通してみましょう（詳しくは第25章）。いきなり大事なものの上で手を動かさせてはいけません。

> **ポイント：** 「Claude Code がどんな形態か」という問いは、**公式ドキュメントに外注** しましょう。あなたが持ち込むべきは「Web ページなのかコマンドラインなのか」ではなく、第24章で身につけた判断力――どんな仕事を振り、できているか・いないかをどう見るか――です。形態は変わりますが、判断力は変わりません。

## 二、CLAUDE.md ＝ プロジェクトへの「入社のしおり」

これは Claude Code の設計の中で、専門知識がなくても最も理解しておく価値のあるものであり、本章の山場でもあります。

第24章のたとえを思い出してください。Claude Code は **とても優秀だが、入社初日の新しい同僚** のようなものでした。新しい同僚の最大の問題は何でしょう。**プロジェクトの流儀を知らない** のです。どんなスタイルを好むか、どこが触ってはいけない地雷か、テストを走らせるには何をするか、命名にどんなこだわりがあるか――これらを彼はまったく知らず、あなたが言わなければ自分の思い込みで進め、結果つまずきます。

**CLAUDE.md は、あなたがこの新しい同僚に書いて渡す「入社のしおり」です。** 通常はプロジェクトの中に置くテキストファイルで（**具体的なファイル名・置き場所・有効になり方は公式ドキュメントを参照**）、その中にこのプロジェクトの「流儀」を平易な言葉で書きます。たとえば――

- このプロジェクトはだいたい何をするものか、主なファイルはどこにあるか。
- 私たちのコードのスタイル、命名の習慣。
- どこが触ってはいけないか（「この設定は動かさないで」「このディレクトリは自動生成なので手で直さないで」）。
- テストやチェックを走らせるにはどうするか。

このしおりがあれば、Claude Code は毎回の開始時に「まず流儀を一読してから」手を動かせるので、つまずきがぐっと減ります。あなたも、毎回の対話でこれらを一から言い直す必要がなくなります――一度書けば、長く効きます。

### 大事な明確化：これは「製品レイヤーの記憶」であって、モデル本体が覚えたのではない

ここで **最も誤解しやすい一点を、必ず明確にしておかねばなりません**。それは第13章（そして第3章・第9章も忘れずに）に直接呼応します。

あなたはこう思うかもしれません。「ああ、これでこのプロジェクトを **覚えてくれた** んだ」と。――この言い方は、半分正しく、半分間違いです。

> **モデル本体は確かに対話をまたいだ記憶を持たず**、毎回まっさらから始まります（第3章・第9章）。しかし **Claude Code はモデルの外側に、もう一層の「製品レイヤーの記憶」を足しています**。CLAUDE.md というしおりは、**毎回の開始時に、製品が自動でそれを今回のコンテキストに入れ直して** モデルに見せているのです。ですからモデルが「記憶力を生やした」のではなく、**製品が毎回あなたの代わりにプロジェクトの流儀を「机」の上に並べ直している** のです（第3章の机に戻ります）。蓋を開けてみれば、下にあるのは依然として「記憶を持たないモデル＋毎回入れ直すコンテキスト」です。

なぜここで、わざわざ種明かしをしておくのか。ここが腑に落ちれば、盲信するのではなく、**AI の挙動を先読みできる** ようになるからです。

- **しおりに書いていない流儀は、知りようがありません。** あなたが「分かっているはず」と思う暗黙のルールも、CLAUDE.md（か今回の対話）に落ちていない限り、やはり知りません――「忘れた」と責めないでください。あなたが「入れて」いないのです。
- **しおりが長いほど、「机」を多く占めます。** 毎回コンテキストに入れ直される（第3章）ので、だらだら長く書くと場所を圧迫し、肝心の点が埋もれかねません。**良いしおりは「精選された流儀の一覧」であって、プロジェクト説明書の全文ではありません。**
- **AI が「見落とす」こともあります。** しおりを入れ直したからといって、一つひとつ守るとは限りません――やはり確認は必要です（第24章・第27章）。

> **ポイント：** 「プロジェクト記憶」「プロジェクトコンテキストファイル」といった言い方（CLAUDE.md はその一種）を見ても、第3章の「モデルに記憶はない」と矛盾するとは思わないでください。どちらも **製品レイヤー** の気の利いた仕掛けで、**毎回、覚えておくべきものをコンテキストに置き直している** のです。これはまさに第3章の「机」の延長であって、モデル本体の記憶ではありません。具体的にどう書くか・どこに置くか・優先順位はどうかは、**公式ドキュメントを参照** してください。

## 三、よく使う操作の勘どころ：「何をするか」だけ、「どのキーを押すか」は書かない

以下は、あなたがほぼ確実に使うことになる数種類の操作です。**注意：ここでは「それが何のためのもので、いつ使うか」だけを述べ、具体的にどう起動するか（コマンド・ショートカット・ボタン）は一律 公式ドキュメントを参照してください。** これこそ本書の安全なやり方です――「概念」をあなたに教え、「変わりやすい具体的なキー操作」は公式に委ねます。

| あなたがやりたいこと | だいたい何のための操作か | 心に留めておくべきこと |
| --- | --- | --- |
| **まず手を動かさず、調査だけさせる** | 一言で「まず見るだけで変えないで」と指示する（第24章の「調査」の一歩） | 手戻りを省く最初の一手。まず見当違いの場所を探していないか確認する |
| **まず計画を出させてから手を動かさせる** | 多くの形態は「先に計画を出し、承認してから変える」モードを内蔵している | 第24章の「関門1」。数十秒で数十分の手戻りを防ぐ（詳しくは第27章） |
| **許可の確認** | 「結果を伴うこと」（ファイル変更、コマンド実行）をするたびに、たいてい立ち止まって尋ねてくる | 何も考えず「同意」を連打しない――第26章で詳しく述べる安全弁 |
| **diff（変更前後の対比）を見る** | 「どの行を足し、どの行を削ったか」を示してくれる | 検収の核心動作。読めないものは通さない（第25章） |
| **@ のような書き方でファイルを指定する** | 「私が言っているのはこのファイル」と正確に指す | プロジェクト中を当てずっぽうに探させるより、ずっと効率的。**具体的な記号は公式を参照** |
| **中断／停止させる** | 進む方向がずれてきたと感じたら、いつでも中断し、はっきり伝え直してから続けさせる | 「いつでも止められる」のはあなたの基本的な権利であって、邪魔ではない |

これらを合わせれば、日々 Claude Code を連れて仕事をするときの「ハンドルとブレーキ」になります。気づくでしょう――これらは **すべて第24章のワークフローに対応** しています。調査、計画の承認、diff を見る、いつでも止める――それを Claude Code の中で「具体的にどうするか」に置き換えただけです。**そして「具体的にどうするか」の層は、つねに公式ドキュメントを最新の拠り所とします。**

> **よくある場面：** 初心者が最も陥りやすい落とし穴は二つです。一つは **許可の確認で何も考えず「同意」を連打する** こと――ブレーキを外したのと同じです（第26章・第30章）。もう一つは **diff を見るのを面倒がってマージする** こと――「やり終えた」を「やり遂げた」と取り違えています（第25章）。この二つには Claude Code が関門を用意してくれています。落とし穴はツールにではなく、あなたが本当に見たかどうかにあります。

## 四、もっと深く知りたい？これらは「付録 C」にあります、ここで探さないで

Claude Code には、もっと忠実に、もっと強力に動かすための「上級の使い方」がまだ山ほどあります。しかしこれらは **まさにバージョンとともに最も変わりやすい部分** なので、本書の扱いは明確です。**ここでは存在を名指しするだけにとどめ、具体的な使い方は一律 付録 C と公式ドキュメントへ向けます。**

将来あなたはこうした言葉を耳にするかもしれません。まずは顔だけ覚えて、「これらは Claude Code を拡張するためのものだ」と分かれば十分です。

- **スラッシュコマンド(slash commands)**：対話の中でスラッシュで始める一群の近道指令で、よく使う機能をすばやく呼び出します。**全表と使い方は付録 C ／公式ドキュメントを参照。**
- **settings.json のような設定**：一つの設定ファイルで挙動をまとめて定めます（たとえば既定の許可の粒度）。**具体的なフィールドは公式を参照。**
- **サブエージェント(subagents)**：「分身」を派生させ、サブタスクを並行で処理させます。
- **MCP（外部ツール接続）**：外部のツールやデータソースにつなぐ標準的なインターフェース（第22章で概念を扱いました）。
- **Hooks（フック）／ Skills（スキル）**：特定のタイミングであなたの定めた動作を自動で起動させたり、再利用できる一そろいの能力を持たせたりします。

> **ポイント：** これらの上級機能はとても強力ですが、**入門したての非プログラマーにとって、どれも第一歩ではありません**。あなたの第一歩はつねに、やり直しのきく環境で（第25章・第27章）、第24章のワークフローを使い、小さなタスクを **安全に** 通して検収しきることです。「連れて仕事をする」手応えがついてから、**付録 C** をめくってこれらの上級の使い方を解錠しても遅くありません。**これらすべての具体的なコマンド・フィールド・スイッチは、公式ドキュメントを最新の拠り所とします。**

## 五、安全と費用、初日から心に留めておく

これらは第24章でも触れましたが、Claude Code に当てはめて、もう一度念を押します――本当にあなたのものを動かし、本当にトークン単位でお金がかかるからです。

- **まず「やり直しのきく」環境を整える。** AI が **あなたのローカルのプロジェクト** で手を動かすときは、必ず **バージョン管理(Git)** で管理し、しかも **新しいブランチを切った** 状態で動かさせてください（手取り足取りは第26章）。ぐちゃぐちゃになってもワンタッチできれいな状態に戻せる――これがあなたの最大の安全網です。（Web／クラウドの形態には、たいていプラットフォーム側の隔離・巻き戻しの仕組みがあります。**公式ドキュメントを参照。**）
- **鍵(API key)・パスワード・本物の個人情報は絶対に外に貼らない。** 対話に貼ったりコードに書き込んだりしないでください――外部に送信されます（第37章）。
- **トークン単位で課金されます。** 読ませるファイルが多いほど、会話が長いほど、減りが速くなります。タスクを絞り、会話をすっきり保てば、お金も節約でき、品質も上がります（第38章）。
- **生成したコードはあなたの責任です。** マージ前に検証し、それに責任を持ってください（第39章）。

## まとめ

- Claude Code は Anthropic 発でコアは Claude（安定した事実）。だが **形態は多様で、しかも更新され続けている**――どうインストールし、どう操作するかは **公式ドキュメントを参照**。
- **CLAUDE.md ＝ プロジェクトへの「入社のしおり」**：プロジェクトの流儀を一度書けば、毎回の開始時に自動で読まれる。ただしこれは **製品レイヤーの記憶**――毎回しおりをコンテキストに入れ直しているのであって、**モデル本体が覚えたのではない**（第3章・第13章）。
- よく使う操作（調査、計画を出す、許可の確認、diff を見る、@ で指定、いつでも止める）はすべて第24章のワークフローに対応する。**具体的にどう起動するかは公式ドキュメントを参照**。
- スラッシュコマンド、settings.json、サブエージェント、MCP、Hooks、Skills などの **深い機能は一律 付録 C を参照**。入門の第一歩ではない。
- 初日から心に留める：Git のブランチを切ってから手を動かす、鍵は絶対に外に貼らない、トークンは節約して使う、生成物は自分で責任を持つ。

次章は **Codex** に移ります――同じ一そろいのワークフローが、OpenAI 側のツールに落ちるとき、どこが同じで、どこが違うのかを見ていきます。

   - A. まったく正確。モデルはこれから永久に覚えている

   - C. CLAUDE.md はモデルのパラメータを変える
   - D. しおりは有料ユーザーにしか効かない
