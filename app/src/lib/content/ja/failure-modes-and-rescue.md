〔本章はいかなる具体的ツールのコマンドや画面にも依存しません――ここで述べるのは「つまずきの共通点」と「応急処置の汎用動作」です。あるツールで具体的にどう操作するかに触れる箇所は、一律 **公式ドキュメントを参照**（付録 C を参照）。〕

ここまでの数章で、「プログラミングエージェントをうまく連れて仕事をする方法」を徹底的に述べました――共通ワークフロー（第24章）、Claude Code への落とし込み（第28章）、Codex への落とし込み（第29章）。しかし、正直に言っておかねばならないことが一つあります。**どれだけ熟達しても、つまずきは起こります。** AIは間違え、的を外し、自信たっぷりに勝手をやります――これは第24章で述べたとおり、プログラミングエージェントの「間違えうる + 手を動かす」という本性が決めることです。

ですから本章は「あらゆるつまずきを避ける方法」（それは非現実的です）を教えるのではなく、もっと実のある二つを教えます。**一つ、最もよくある八つの「つまずきの現場」を見分けられること――病根がどこにあり、どう対処するか。二つ、本当に行き詰まった・ぐちゃぐちゃに直してしまったとき、あなたをきれいな状態へ引き戻す「応急処置リスト」を持つこと。** 本章はツールを問わず、Claude Code にも Codex にも当てはまります。

## 一、八つのつまずきの現場：症状 → 原因 → 対策

以下の一つひとつは、初心者が繰り返し踏む落とし穴です。**自分に当てはめて** みてください――おそらくあなたは、すでにいくつか踏んでいます。

### 1. タスクが曖昧で、抜け漏れだらけ

- **症状**：「このページをちょっと最適化して」と言ったら、出てきたものはまったく望みと違うか、半分しかできていない。
- **原因**：指示が大ざっぱすぎて、何が欲しいかを **推測** するしかなく――そしてしばしば外します（第11章「欲しいものをはっきり言う」に呼応）。
- **対策**：目標を **具体的に** 言う。何を達成するか、どこを触るか、何をもって「やり遂げた」とするか。まず **AIの理解を一度復唱させて** から手を動かさせるほうが、後から手戻りするよりずっと省けます。

### 2. 一度に大きく変えすぎて、確認のしようがない

- **症状**：「この機能を丸ごと作り終えて」と言ったら、十数ファイルを変更し、画面いっぱいの diff を前に茫然。問題が起きてもどこから調べればよいか分からない。
- **原因**：変更が大きいほど **死角が大きい**。大きすぎて一行ずつ確認できず、巻き戻しにくい（第24章で繰り返し強調しました）。
- **対策**：**タスクを小さく分ける**――小さなまとまりを作り、その都度確認し、一度コミットして、次のまとまりへ。小さく刻んでゆっくり進むことが、かえって最も速い道です（第25章・第29章）。

### 3. diff を見ず、「同意」を連打する

- **症状**：許可を求められるたびに即「同意」を押し、気づけば触ってはいけないファイルを動かし、大事なものまで削っていた。
- **原因**：「許可の確認」を面倒な慣例と取り違える――**ブレーキを外したのと同じ**（第26章）。
- **対策**：**許可の前に、何をしようとしているか・何を変えたか（diff）をまず見る**。この一手間は数秒ですが、防げるのは数時間の災難かもしれません。読めなければまず「この操作は何をするの?」と聞き、**読めないものは通さない**（第25章）。

### 4. 一つの対話を最初から最後まで使い、切り替えない

- **症状**：同じ対話で数十往復するうち、AIはどんどん「頑固」になる――新しい要望を無視し、すでに却下した案を蒸し返し、自己矛盾する。
- **原因**：これこそ第3章・第13章で述べた「机」です――対話が長いほど **ノイズが信号を覆い**、肝心な点が埋もれ、矛盾が積み重なります。鈍くなったのではなく、机が埋まったのです。
- **対策**：**一つの話題に一つの対話**。「頑固」になり始めたと感じたら、思い切って **新しい対話を開き**、「まだ必要な情報」を改めてはっきり伝え直す（第13章）。きれいな机に替えるほうが、乱れた机で探し続けるよりずっと速い。

### 5. ハルシネーションを鵜呑みにし、裏取りしない

- **症状**：AIは自信たっぷりに、まったく存在しない書き方を使い、調べても出てこない根拠を引く。それを丸ごと受け取って、結果として間違える。
- **原因**：AIが追うのは「本物らしさ」であって「本物であること」ではありません（第9章「ハルシネーション」）。**流暢で自信たっぷりなほど、信頼できるとは限りません。**
- **対策**：**肝心なことほど、すらすらと即答してきたことほど、自分の手で確かめる**。根拠を出させ、テストで検証させる。肝心の事実は自分でもう一度調べる（第9章）。

### 6. 失敗した方針に固執して直し続け、直すほど歪む

- **症状**：ある方向が明らかに行き詰まったのに、なお元の案にしがみついて一言ずつ直させ、結果として直すほど乱れ、迷うほど的を外す。
- **原因**：誤ったコンテキストがすでに「机」を埋め尽くし、AIは自分の以前の誤りに **引きずられて** います（第13章）。乱れたコンテキストの中で無理に直すのは、たいてい逆効果です。
- **対策**：**この道は行き止まりだと認め、巻き戻して + やり直す**。一つ前のきれいな状態に戻り、新しい対話を開いて、「さっき踏んだ落とし穴」を前提として伝え直す（第13章・第24章）。これはほぼいつも、無理に直すより速い。

### 7. テストなしで大改修する

- **症状**：直し終えたものをそのままマージ・公開し、数日後にようやく、元は正常だった機能が壊れていたと気づく。
- **原因**：「やり終えたと言った」を「やり遂げた」と取り違える――**検証がなければ、確信もない**（第24章・第25章）。
- **対策**：直し終えたら **テストを走らせ**、元の機能を壊していないか確かめる。自分でも「この部分は何をしているか」を一言で言ってみる。**言えない・テストしていないものは、マージしない**（第25章）。

### 8. 鍵／機密情報を貼ってしまった

- **症状**：手間を惜しんで、API キー（API key）・パスワード・本物の個人情報を対話に貼り、あるいはコードに書いて公開リポジトリにプッシュした。
- **原因**：あの鉄則を忘れた――**貼ったものは差し出したのと同じ**（第3章）。鍵は「財布の鍵」で、手にした者は誰でもあなたのお金を使える（第29章）。
- **対策**：**鍵／パスワード／機密情報は絶対に対話に入れない、公開リポジトリに入れない**（第37章）。万一貼ってしまったら、**ただちに該当プラットフォームで古い鍵を失効させ、新しいものに換える**（**具体的な操作は公式ドキュメントを参照**）。たかをくくってはいけません。

> **ポイント:** この八つをつなげて見れば、実は **三つの原則** にしか反していないと分かります。①話をはっきりさせていない（1・2）、②「手を動かす前後の二つの関門」を守っていない（3・5・7）、③コンテキスト／データを管理できていない（4・6・8）。八つの現場を丸暗記するより、この三つを覚えるほうが役立ちます――これらはまさに、第24章のワークフローと、本書全体の安全規律を、裏返しにしたものなのです。

## 二、行き詰まったらどうするか：五歩の応急処置リスト

ここまでは「予防」、本節は「救助」です。AIがすでに的を外し、自分も頭が真っ白で、ぐちゃぐちゃに直してしまったとき――**慌てず、乱れた局面の中で粘り続けないでください**。以下の五歩のとおりにすれば、ほぼいつも引き戻せます。

```text
行き詰まった/直しすぎた、順番どおりに：
①まず中断 ──→ ②現状を整理 ──→ ③一つ前の良い状態へ巻き戻す ──→ ④新しい対話でやり直す ──→ ⑤診断
（すぐ止める）   （急いで続けさせない）   （きれいな所へ戻る）          （きれいな机に替える）   （なぜ的を外したか考える）
```

**① まず中断――すぐに止める**
おかしいと感じたら、**まず止めること** が第一です。落とし穴へ走り続けるのを見ていてはいけません。「いつでも止められる」のはあなたの基本的な権利です（第26章）。早く止めるほど損失は小さい。止まって、深呼吸して、慌てて「同意」を押し続けないでください。

**② 現状を整理――まず棚卸し、急いで続けさせない**
止まったら、まず自問します。**いま何を変えたか? どれが望んだもので、どれが的を外して作ったものか? 一つ前の「きれいで、使えた」状態はどこにあるか?** 現状を整理しきって初めて、どこへ退くか分かります。この一歩は後の「巻き戻し」の土台です――整理せずに巻き戻すと、退きすぎたり、足りなかったりします。

**③ 一つ前の良い状態へ巻き戻す――これがブランチを切っておいた意味**
これが応急処置の核心動作です。もしあなたが **前の数章の言うことを聞いて**、最初から **Git** で管理し、しかも **新しいブランチを切って** いたなら（第26章）、いまあなたは **ワンタッチでそのきれいな状態に戻り**、AIが的を外して作ったものをまとめて帳消しにできます。

> **ポイント:** 第25章・第28章・第29章で繰り返し「まず Git のブランチを切ってから手を動かす」と唱えたのは、**まさにこの第③歩のためです**。巻き戻しは失敗ではなく、あなたの **最大の安全網** です。いわゆる「やり直しのきく環境」が現金化されるのが、この能力です。（Web／クラウドの形態には、たいていプラットフォーム側の巻き戻し／版管理の仕組みがあります。**公式ドキュメントを参照。**）具体的にどう巻き戻すかのコマンドは、**付録 C と公式ドキュメントを参照**――本章は「一つ前の良い状態へ戻る」という動作そのものだけを述べます。

**④ 新しい対話でやり直す――きれいな机に替える**
きれいな状態に戻ったら、**もう乱れてしまったあの対話で続けないでください**。**新しい対話** を開き、「さっきの道は行き止まりで、どんな落とし穴を踏んだか」を **前提** として伝え、その教訓を携えて改めて計画させます。なぜ新しく開くのか。古い対話の「机」には誤った試行が積み重なっており（第13章）、その上で続ければ、AIはやはり古い誤りに引きずられるからです。

**⑤ 診断――「なぜ的を外したか」を考え、同じ所で二度こけない**
最後に一分かけて振り返ります。今回的を外した根は、どこにあったか? **タスクが曖昧すぎた**（第一節の1へ戻る）? **一度に変えさせすぎた**（2）? それとも **早く巻き戻すべきだったのに、無理に直した**（6）? 病根を見つけ、次にやり直すときは **源で塞ぐ**――たとえば今度はまず計画を出させて一度確認する（第27章）、あるいはタスクをもっと小さく分ける。

> **実際にはこうなりがちです:** 初心者が最もよくやってしまうのは「つまずき」そのものではなく、**つまずいた後にどうしても巻き戻したがらない** ことです――「もう少し調整すれば直る」と思い込み、結果として乱れた局面に深くはまり、三十分後には最初よりひどくなります。**熟練者の本能はちょうど逆です。方向が違うと感じたら、乱れた山に継ぎを当て続けるより、巻き戻してやり直すほうを選びます。** 巻き戻し + 新しい対話は、たいてい最も速い道であって、降参ではありません。

## 三、一言の総綱

本章を一言にまとめ、身につけて持ち歩ける言葉にします。

> **予防は、第24章のワークフローを守ること（はっきり言う・小さく分ける・diff を見る・テストを走らせる）。救助は、「中断→棚卸し→巻き戻し→やり直し→診断」を覚えること。** つまずきは怖くなく、怖いのは乱れた局面の中で踏ん張り続けることです。あなたの最大の心強さは、あの「やり直しのきく環境」――だから「まずブランチを切ってから手を動かす」この一歩は、絶対に省かないでください。

## まとめ

- プログラミングエージェントは **必ずつまずきます**（間違えうる + 手を動かす）。目標は「絶対につまずかない」ではなく、**現場を見分け、救い戻せる** こと。
- **八つのつまずきの現場**：曖昧でタスクを取りこぼす、一度に変えすぎ、diff を見ず許可を連打、対話を切り替えない、ハルシネーションを鵜呑み、失敗した方針に固執、テストなしで大改修、鍵を貼ってしまう。これらは実は三つにしか反していない――はっきり言っていない、二つの関門を守っていない、コンテキスト／データを管理できていない。
- **五歩の応急処置リスト**：①まず中断 ②現状を整理 ③一つ前の良い状態へ巻き戻す ④新しい対話でやり直す ⑤診断。核心は第③歩の巻き戻し――これこそ「まず Git のブランチを切ってから手を動かす」が現金化される価値です。
- 最大の誤りは **つまずいた後に巻き戻さず**、乱れた局面で踏ん張ること。熟練者の本能は **むしろ巻き戻してやり直す** こと（第13章・第24章・第25章に呼応）。
- 具体的な巻き戻しコマンド・ツール操作は **公式ドキュメントを参照**（付録 C を参照）。本章が述べる「動作」そのものはツールを問わず、これからも古くなりません。

ここまでで、第4部「AIコーディング実践」は一区切りです。あなたはすでに、共通ワークフロー、二大ツールの入門の地図、そしてつまずいたときの応急処置の力を手にしました。次の部では、いよいよ **本当に手を動かしてプロジェクトを作り**、これらの力を目に見える成果に変えていきます。

## 小テスト

> 全 6 問。概念・誤解・場面・操作の 4 種類をカバーしています。まず自分で考えてから、引用ブロックの答えと解説を見てください。

1. **[基礎・概念問題]** 本章の「つまずき」に対する基本的な態度は？
   - A. 十分に熟達しさえすれば、永遠につまずかずに済む
   - B. プログラミングエージェントは「間違えうる + 手を動かす」。つまずきは常態。目標はつまずきを根絶することではなく、見分けられ、救い戻せること
   - C. つまずいたら、そのツールがだめだということ、替えるべき
   - D. つまずきはすべてユーザーが愚かなせい
   > **答え B。** 第24章で述べたとおり、つまずきはプログラミングエージェントの「間違えうる + 手を動かす」本性に由来します。成熟した使い方は **救えること**――八つの現場を見分け、五歩の応急処置を使えること――であって、永遠に間違えないと夢想することではありません。

2. **[基礎・概念問題]** 五歩の応急処置リストの正しい順番は？
   - A. まず続けさせる → それから考える
   - B. まず中断 → 現状を整理 → 一つ前の良い状態へ巻き戻す → 新しい対話でやり直す → 診断
   - C. まずプロジェクトを削除 → ツールを再インストール
   - D. まず新しい対話を開く → それから中断するか考える
   > **答え B。** 順番が大事です。**まず損切り（中断）、それから棚卸し（現状を整理）、きれいな所へ戻り（巻き戻し）、きれいな机に替え（新しい対話）、最後になぜかを考える（診断）。** 棚卸しを飛ばしていきなり巻き戻すと退く位置を誤りやすく、中断せずに続ければ深みにはまります。

3. **[応用・誤解問題]** 「AIが的を外した、元の案にしがみついて一言ずつ直せばよい」――なぜしばしば、直すほど悪化するのか？
   - A. 問題ない、何度か直せばよい
   - B. 誤った試行がすでにコンテキストという「机」を埋め尽くし、AIは自分の以前の誤りに引きずられている。すべきは巻き戻し + 新しい対話でやり直すこと
   - C. 直すとき、AIは速くなる
   - D. 直すのにトークンはかからない
   > **答え B。** これは八つ目の現場「失敗した方針に固執」で、根は第13章のあの埋まった「机」にあります。**きれいな状態に戻り、教訓を携えた新しい対話を開く** ことは、ほぼいつも乱れた局面で無理に直すより速い――これが応急処置リストの③④歩です。

4. **[応用・誤解問題]** 「巻き戻しは失敗、降参に等しい」――この考えはどこが間違い？
   - A. 間違っていない、巻き戻しは恥ずかしい
   - B. 巻き戻しはあなたの **最大の安全網** で、つまずいた後の熟練者の本能。たいてい最も速く軌道に戻る道であって、降参ではない
   - C. 巻き戻すとプロジェクト全体が削除される
   - D. 有料版でしか巻き戻せない
   > **答え B。** 「まず Git のブランチを切ってから手を動かす」（第26章）が下敷きにしていたのが、この瞬間です。初心者が最もよくやらかすのは、まさに **巻き戻さず、乱れた山に踏ん張り続ける** こと。結果はどんどん悪化します。「むしろ巻き戻してやり直す」を本能にすることが、本章で最も価値のある一言です。

5. **[基礎・場面問題]** AIが仕事をするのを見ていたら、突然、あなたが大事だと思うファイルを削除しようとしている。この瞬間に最もすべきことは？
   - A. どうせ AI だから理由があるはず、「同意」を押す
   - B. ただちにこの操作を **中断／不承認** にし、何をしようとしているのか見極める。分からなければまず「この操作は何をするの?」と聞く
   - C. パソコンの電源を切る
   - D. プロジェクト全体を作り直す
   > **答え B。** これは「diff を見ず同意を連打」（第3の現場）と、応急処置の第①歩「まず中断」を同時に踏みます。**許可の前にまず見極め、読めないものは通さない**（第26章・第25章）――「いつでも止められる」のはあなたの基本的な権利で、早く止めるほど損失は小さい。

6. **[基礎・操作／観察問題]** ツールを実際にインストールする必要はありません。本章の八つのつまずきの現場を見返し、**自分に当てはめて**――過去に AI を使ったとき（たとえチャットだけでも）、最もよく踏んだのはどの二、三条でしたか? その対応する「対策」を書き写し、これから目に入る場所に貼っておきましょう。
   > **観察できるはず:** たいていの人は、自分が同じ一、二条で繰り返しこけていると気づきます（とてもよくあるのは「対話を切り替えない」と「ハルシネーションを鵜呑み」です）。八つの現場は実は三つの原則にしか反していません――**はっきり言っていない、二つの関門を守っていない、コンテキスト／データを管理できていない**。自分の高頻度の項目を見つめ、源で塞ぐほうが、八条を全部覚えるより役立ちます。具体的なツールでどう巻き戻すか・どう操作するかは、**公式ドキュメントを参照**（付録 C を参照）。
