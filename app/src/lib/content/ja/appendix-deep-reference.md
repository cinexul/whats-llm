本文の中で、私たちは「そのコマンドを具体的にどう打つのか、あのスイッチは何という名前なのか」を繰り返し後回しにして、「細かいことは付録 C で」と言ってきました。これがその付録 C です。Claude Code と Codex という二つのツールに **いったいどんな能力があり、それぞれの能力がどんな問題を解決し、いつ使うべきで、どこでつまずきやすいのか** を、ここにまとめて分かりやすく説明します――ただし **正確なコマンドをあなたの代わりに暗記することはしません**。

なぜコマンドを暗記しないのでしょうか。それはまさに、本書が繰り返し強調してきたあの一線だからです。

> **ポイント（まずこの段落を読んでください）：** この二つのツールはどちらも猛烈な速さで更新を重ねています。**コマンドの名前、パラメータ、設定項目のフィールド、ファイルが何という名前で、どこに置かれ、イベントがどう命名されるか――これらはすべてバージョンによって変わります。** ですから本付録では、「どんな能力があり、それぞれどんな問題を解決し、使うべきかどうかをどう見極め、どんな落とし穴を踏みやすいか」という **比較的安定した** ことだけを扱います。**正確なコマンド、パラメータ、設定の書き方は、一律に公式ドキュメントを参照してください。** この付録を「地図」として、公式ドキュメントを「リアルタイムの道路状況」として使ってください。

## 一、まずどこで調べるかを覚える（公式ドキュメント）

地図がどんなに良くても、最新の道路状況と組み合わせる必要があります。「このコマンドは今いったいどう書くのか」に出くわしたら、この二か所で調べれば間違いありません。

| ツール | 提供元 | 公式ドキュメントはどこで探すか |
| --- | --- | --- |
| Claude Code | Anthropic | 「Claude Code 公式ドキュメント」で検索するか、Anthropic のドキュメントサイト（たとえば docs.claude.com 配下の Claude Code 部分）にアクセス〔公式情報を優先〕 |
| Codex | OpenAI | 「OpenAI Codex 公式ドキュメント」で検索するか、OpenAI の公式ヘルプセンター / 開発者向けドキュメントで Codex を調べる〔公式情報を優先〕 |

> **ポイント：** 一つの良い習慣――**使う前に、公式ドキュメントの「クイックスタート」や「更新履歴」にざっと目を通すこと**。バージョンが変わったとき、いちばん手間がかからないのは、記憶を頼りに無理やり打つのではなく、公式がどう言っているかを直接見ることです。

## 二、それらはどこに「住んでいる」のか：ターミナル、IDE、Web、クラウド

どんなコマンドを暗記するよりも先に、まず一つのよくある誤解を解いておきましょう。

多くのチュートリアルは「Claude Code / Codex は、あなたのパソコンにインストールするコマンドラインツールだ」と言います。これは **一種類の形態についてだけ正しい** にすぎません。実際には、あなたは次のような場所でそれらに出くわす可能性があります。

- **ターミナル（コマンドライン）：** あなた自身のパソコンの黒い枠の中で動きます。
- **IDE プラグイン：** VS Code や JetBrains のようなエディタに組み込み、コードを書きながら使います。
- **Web / クラウドのタスク：** ブラウザの中で用件を任せると、それは **遠隔のクラウド環境** の中で動き、終わったら結果をあなたに渡します。
- そして、バージョンごとに次々と現れる新しい形態。

> **ポイント（本書を貫く第三の一線）：** **「それが必ずあなたのこのパソコンの上で動いている」と決めてかからないでください。** それは **ローカルで動く可能性もあれば、クラウドで動く可能性もあり**、具体的な形態は製品の更新とともに変わります。このことはとても重要です。なぜなら、それが「あなたのローカルのファイルを読めるかどうか」「あなたのコードがこのマシンから一度でも外に出たかどうか」を直接左右するからです――プライバシーとセキュリティに関わります（第37章を参照）。**あなたが使っている形態についての具体的な説明は、公式ドキュメントを参照してください。**

## 三、会話内コマンド（スラッシュコマンドなど）

**何か：** ツールと対話している途中で、ある特殊な接頭辞（よくあるのはスラッシュ `/`）を付けて打ち込む「ショートカット指令」のことです。「コードを書く」のとは関係のない操作――たとえばヘルプを見る、今のコンテキストを空にする、モデルやモードを切り替える、プロジェクト規約を初期化する、外部の接続を管理するなど――をさせます。

**メンタルモデル：** 普通の言葉は「私があなたにやってほしいこと」、スラッシュコマンドは「私があなたというツールそのものをちょっと操作したい」というものです。前者は内容、後者は制御スイッチです。

**どんなものがあるか、どう書くかをどう知るか：** いちばん確実な方法は、会話の中でそれが備えているコマンド一覧を見ることです（多くのツールは `/` を入力するか「ヘルプ」と打つと一覧を出してくれます）。**正確にどんなコマンドがあり、それぞれのパラメータがどうなっているかは、公式ドキュメントとツール内の表示を参照してください。**

**よくある落とし穴：** 「AI に言う言葉」と「スラッシュコマンド」を取り違えること。記憶を空にさせたいなら、その専用のコマンドを打つのであって、「前のことは忘れてください」と文章で言うのではありません――後者では、それは本当に空にするとは限りません。

## 四、許可の仕組みと設定ファイル

**何か：** この種のツールはファイルを変更したりコマンドを実行したりと手を動かせるので、どれも「**許可（permission）**」の仕組みを内蔵しています。危険な操作の前にまず立ち止まってあなたに尋ねるか、あなたが事前に決めたルールに従って自動で通す / 止めるのです。これらのルールは通常、ある **設定ファイル** の中に保存されます（Claude Code は `settings.json` のたぐいの名前を使う習慣があり、Codex には独自の設定があります）。

**メンタルモデル：** これを、よく働く実習生に権限を設定することだと考えてください――どんなことは自分でやってよく、どんなことはまず手を挙げてあなたに尋ねなければならないか。設定ファイルはこの「権限リスト」です。

**典型的な許可の強さ、いくつか（直感であって、正確な名称ではありません）：**

| おおよその段階 | 意味 | 向いている場面 |
| --- | --- | --- |
| 毎ステップ尋ねる | どんな変更も、どんなコマンドも、まず同意を求める | 使い始め、または重要なプロジェクトの中で |
| 一部を自動で通す | ファイルを読む、情報を調べるなどの「読み取り専用」操作は自動でやり、変更はなお確認を取る | 慣れてから速度を上げたいとき |
| 計画だけ立てて手は動かさない | 案を出すだけで、実際には何も変更しない | それが何をしようとしているか先にはっきり見たいとき |

> **ポイント：** **正確に何種類のモードがあり、それぞれ何という名前で、設定ファイルの中でフィールドをどう書くかは、すべて公式ドキュメントを参照してください。** 初心者にとっての安全な既定は「変更系の操作はすべて、それにまず尋ねさせる」ことです。

**よくある落とし穴：** 速さを求めて、いきなり許可を「全自動」に上げること。それが動かしてはいけない場所で手を動かしてしまってから、防御を設けていなかったことに気づくのです。**重要なプロジェクトであるほど、人による確認というあの関門を残しておきましょう。**

## 五、プロジェクト規約ファイル：ツールに「このプロジェクトの説明書」を一つ渡す

**何か：** プロジェクトの中に **規約ファイル** を一つ置いて、「このプロジェクトはどんな技術を使い、ディレクトリをどう分け、どんな必ず守るべきルールがあり、どんなことが禁止か」を書き留めておくことができます。ツールは毎回作業を始めるときにそれを読み、その通りにやります。Claude Code は `CLAUDE.md` という名前のファイルを使う習慣があり、Codex には独自の同等のやり方があります。**正確なファイル名、置き場所、効き方は公式ドキュメントを参照してください。**

**これは「製品レイヤーの記憶」であって、モデルが記憶力を身につけたわけではありません。** この点は、第3、13章のあの本書を貫く一線に呼応します。

> **ポイント：** モデル本体には **会話をまたいだ記憶がありません**。それが「このプロジェクトのルールを覚えている」ように見えるのは、**製品が毎回作業を始めるたびに、この規約ファイルをコンテキストに入れ直して** モデルに渡しているからです――製品レイヤーが「入れ直して」いるのであって、モデルが自分で覚えているのではありません。この一条を理解すれば、こう分かります。**規約は分かりやすく、具体的に書けば書くほど、その振る舞いは安定します**。規約に書かれていないことは、それは「覚えていない」のです。

**どう書けば役に立つか（ポイント）：**

- **固有名詞を使い、形容詞を使わないこと。** 「コードはきれいに」では役に立ちません。「HTTP リクエストは一律 `services/apiClient` を通す、直接 `fetch` するのは禁止」なら役に立ちます。
- **「必ずやること」と「禁止すること」をはっきり書くこと。** たとえば「サードパーティの依存を新しく追加する前には必ず尋ねる」「既存の共通モジュールを迂回して自分で車輪を再発明しない」など。
- **踏んだ落とし穴をその都度書き足すこと。** それがどこかで本筋からそれたら、対応するルールを加える――このファイルは **使えば使うほど使い勝手がよくなる** 生きたドキュメントです。

詳しい「規約ファイルの雛形」と、ワークフロー一式をどう組むかは、**付録 D（ハーネス方法論）** を参照してください。

## 六、計画モードと「推論の強さ」

**計画モード（plan mode のたぐい）：** それに **まず案を出させ、直接は手を動かさせない** ようにし、あなたが見て頷いてから、それが実装する、というものです。これは本書が最も重視する習慣の一つで（第27章）、多くのツールがこのスイッチを内蔵しているので、毎回手作業で念を押す必要はありません。

**推論の強さ / 思考の度合い：** ツールによっては、「もう少し考えさせるか、それとも素早く答えさせるか」を調整できます。難しい問題やトレードオフの必要な設計には高めに、些細な小事には低めにして、時間とお金を節約しましょう。

> **ポイント：** **この二つのスイッチが正確に何という名前で、どう調整するかは、公式ドキュメントを参照してください。** しかしその背後にある道理は安定しています。**重要なことは、それにまずよく考えさせ、まず計画を出させてから、手を動かさせる。**

## 七、サブエージェント（subagents）

**何か：** あるたぐいの仕事を、**専用に設定された「分身」** に任せてやらせること――たとえばコードレビュー専門のもの、資料調査専門のものなど。それらはそれぞれ独自の説明（一段の設定で、よく名前、職責の説明、使えるツールなどを含みます）を持っています。

**メンタルモデル：** チームに役割分担をするようなものです。メインの会話は「プロジェクトマネージャー」、サブエージェントはそれぞれ得意分野を持つ「メンバー」で、対応する仕事を振り出し、互いに干渉せず、それぞれ集中します。

**よくある落とし穴：** いきなりサブエージェントを大量に組むこと。実は **たいていの日常タスクは、メインの会話一つで十分** です。「このたぐいの仕事はいつも繰り返されていて、決まった型が必要だ」とはっきり感じたら、そのとき専用に一つ設定することを考えましょう。**正確な設定フィールドや定義の仕方は公式ドキュメントを参照してください。**

## 八、フック（hooks）

**何か：** ツールが **ある瞬間に、あなたが指定した動作を自動で起動する** ようにすること。たとえば「ファイルを変更し終えるたびに、自動でフォーマット / テストを一通り走らせる」「あるたぐいの操作を始める前に、自動でチェックを一つ行う」など。

**メンタルモデル：** 家に取り付ける自動化のようなものです――「ドアを開けたら明かりが点く」。「どんなイベントが起きたら、自動で何をするか」をあなたが定義すれば、あとはそれが代わりに見張ってくれます。

**よくある落とし穴：** これはやや上級者向けの機能で、普通の使い方なら **まったく手を出さなくて構いません**。必要になったときに理解すればよいのです。**どんなイベントがあり、どう設定するかは、公式ドキュメントを参照してください。**

## 九、Skills（技能）

**何か：** 「あるたぐいのことをやるための標準的な手順 + 必要な資料」を一式まとめて、再利用できる「**技能パック**」にして、必要なときにツールに呼び出させること。本文第22章で扱ったものと同じことで、ここでは一点だけ注意を促します。それは **能力を再利用できるようにする** 仕組みだということです。

**よくある落とし穴：** 使うために使うのはやめましょう。まず仕事を平易な言葉ではっきり説明し、**繰り返し現れる決まった型** だけが技能パックにする価値があります。**パッケージの形式や呼び出し方は公式ドキュメントを参照してください。**

## 十、MCP：AI に外部のツールとデータをつなぐ

**何か：** MCP（第22章で詳述）は、AI ツールが **外部のツールやデータソース** ――たとえばあるデータベース、ある社内システム、あるたぐいのファイルサービス――につなげるようにする「汎用インターフェース」です。接続は通常、「MCP サービスを追加する」という一つのコマンドか、一段の設定によって行います（Claude Code の `claude mcp add` のたぐいの使い方など）。

**メンタルモデル：** AI に「周辺機器」をつなぐようなものです。モデル自体はおしゃべりができるだけですが、MCP をつなげば、それらの外部リソースを調べに行き、取りに行き、使いに行けるようになります。

> **ポイント（セキュリティ）：** 外部データをつなぐ ＝ **外部のコンテンツ** を会話に引き込むことです。**外部のコンテンツの中には「偽の指令」（プロンプトインジェクション（prompt injection）、第22、40章）が潜んでいる可能性があります。** 信頼できるソースだけをつなぎ、過大な権限を与えないでください。**正確な追加コマンドや設定の書き方は公式ドキュメントを参照してください。**

## 十一、インストール、起動、トラブルシューティング

**最小の道筋（考え方であって、一字一句のコマンドではありません）：** ツールをインストールする → ログイン / アカウントや鍵を設定する → あるプロジェクトのディレクトリの中で起動する → まずは安全な小さなタスク（たとえば「このプロジェクトが何をするものか、平易な言葉で説明して」）を一つ通してみて、自信をつける。

**最もよくある、いくつかの行き詰まり：**

- **インストールできない / 起動できない：** 多くは実行環境（バージョン、依存）かネットワークの問題です――エラーの原文をコピーして AI に渡し、読んでもらうのが、たいてい最速です。
- **ログイン / 認証に失敗する：** アカウント、プラン、鍵の問題です。公式の「クイックスタート」と照らし合わせて一項目ずつ確認しましょう。
- **あなたのファイルを読めない：** まず自分が使っているのがどの形態か（ローカルかクラウドか、本付録の第二節を参照）を確認しましょう――クラウドの形態はそもそもあなたの手元のマシンのファイルを読めません。

> **ポイント：** **正確なインストールコマンド、システム要件、ログイン方法は公式ドキュメントを参照してください**。しかもこれらは最もバージョンによって変わりやすいものです。インストールの前に公式の「クイックスタート」を一目見るのが、いちばん手間がかかりません。

## 十二、Git / GitHub / PR などの操作

**何か：** ローカルのコードプロジェクトの中で、この種のツールは通常、バージョン管理に関わることを手伝えます――変更を確認する、コミットする、ブランチを切る、さらには PR（マージリクエスト）を一つ起草するのを手伝うことまで。これはまさに、第25章の「diff を見る、巻き戻す、検収する」がツールの中に落とし込まれたものです。

**メンタルモデル：** それは変更を **記録し、巻き戻す** のを手伝ってくれます。だからこそあなたは安心して、それに手を動かさせられるのです――気に入らなければ巻き戻せばよいのですから（第25、26章）。

**よくある落とし穴：** あなたが diff を見ないまま、それに「ついでにコミット / プッシュ」させること。**何をコミットし、どこにプッシュし、PR を出すかどうかは、あなたが決めるべきです**。とりわけ「プッシュ / マージ」のような外向きで取り消しにくい操作は、必ず自分で確認してください。**正確なコマンドは公式ドキュメントを参照してください。**

## まとめ

- この付録は Claude Code / Codex の **能力の地図** です：どんな能力があり、それぞれどんな問題を解決し、いつ使い、どんな落とし穴を踏みやすいかを分かりやすく説明します。
- **わざと、正確なコマンドをあなたの代わりに暗記しません** ――コマンド、パラメータ、設定、ファイル名、イベント名はどれもバージョンによって変わるので、**一律に公式ドキュメントを参照してください**。使う前に公式の「クイックスタート / 更新履歴」にざっと目を通しましょう。
- 本書を貫くいくつかの一線を、ここでもう一度強調します。**それがあなたの手元のマシンで動くと決めてかからない**（ターミナル / IDE / Web / クラウドのいずれもあり得ます）。**プロジェクト規約ファイルは「製品レイヤーの記憶」** であり、毎回入れ直すことで効きます。**重要なプロジェクトであるほど、「まず計画を見る、変更はまず尋ねさせる、コミットは自分で決める」といういくつかの関門を残しておく**。
- これらの能力を再利用できるワークフロー一式につなぎたければ、続けて **付録 D** を読んでください。
