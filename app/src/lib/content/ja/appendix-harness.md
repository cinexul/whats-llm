付録 C が「ツールにはどんなボタンがあるか」だとすれば、この付録 D が扱うのは「**それらのボタンを、どうつないで信頼できる仕事のやり方一式にするか**」です。これは、現場のチームが何度も磨き上げてきた方法論で、その名を **Harness（手綱を付ける）** といいます――核心はただ一言です。**AI を野放しにして自由に振る舞わせるのではなく、「まず理解し、次にルールを立て、まず計画を出し、手を動かし、最後に検証する」という枠をはめること。** あなたが Claude Code を使おうと Codex を使おうと、このやり方は同じように役立ちます。

> **ポイント：** AI がコードを書くときの最大のリスクは「書けない」ことではなく、「**勝手に判断して、別のやり方を一から始める**」ことです――プロジェクトにある既存の書き方を無視し、自分の流儀を一式むりやり押し込んで、全体の設計を台無しにしかねません。Harness はまさにこれを防ぐためのものです。AI を **既存のやり方に沿って** 仕事させ、ひっくり返して作り直させないのです。この方法論は第26、27、28、29章と表裏一体なので、つなげて読むことをお勧めします。

## 一、Harness とは何か：四つの動作

AI を、**とても有能だが、来たのは初日で、まだあなたのプロジェクトに不慣れ** な協力者だと考えてください。こういう人に、いきなり本筋のコードを書かせたりはしませんよね。あなたなら、その人にこうさせるはずです。

1. **まず理解する** ――既存のコード、既存のルールを把握する。
2. **ルールを立てる** ――「このプロジェクトはどう物事を進めるか」を白黒つけて書き留める。
3. **まず計画を出す** ――手を動かす前に、どのファイルを変更するか、どの既存のものを再利用するかをまず言わせ、承認してから着手させる。
4. **そして検証する** ――やり終えたらルールと照らし合わせて自己点検し、あなた自身も diff を見る（第27章）。

この四つの動作が Harness です。これはどのツールの特定のボタンにも依存しない、一式の「心法（こころえ）」です。

## 二、五ステップのワークフロー（この通りにやる）

上の四つの動作を具体的な会話に落とすと、この五ステップになります。各ステップに「AI に言う言葉」の例を一つずつ示します――これらは **平易な言葉のプロンプトであって、製品のコマンドではありません**。この通りに直せばそのまま使えます。

| ステップ | このステップで何をさせるか | プロンプト例（平易な言葉） |
| --- | --- | --- |
| ① 探索 | まず現状を把握し、手は動かさない | 「手を動かす前に、まずこのプロジェクトを調査して：全体の構造、今回の要件に近い既存のやり方、再利用できる共通モジュール（ネットワークリクエスト / エラー処理 / ログ / フォームなど）、命名とコードのスタイル。**ファイルパスを付けて** 一覧で挙げて、まだコードは書かないで。」 |
| ② 規約化 | 調査結果をルールとして固める | 「上の調査結果を、AI が見るためのプロジェクト規約に整理して：どの仕組みを必ず再利用すべきか、どんなことが禁止か。」 |
| ③ 計画の関門 | 案を出し、あなたの承認を待つ | 「タスクは〈……〉。手を動かす前にまず計画を出して：どのファイルを変更 / 追加するか、どの既存の仕組みを再利用するか、新しい依存を加える必要があるか（必要なら理由を説明する）。**私が承認するまでコードを書かないで。**」 |
| ④ 実装 | 既存のものに沿って | 「承認。既存のディレクトリ構造、命名、スタイルに厳密に従い、さっき見つけた共通モジュールを再利用して実装して。**自分で新しい抽象を持ち込まないで。**」 |
| ⑤ 検証 | 自己点検 + あなたが見張る | 「さっき参考にした既存の書き方と照らし合わせて自己点検して：車輪の再発明をしていないか、全体の設計を壊していないか、共通モジュールを迂回していないか。」 そのうえで **あなた自身が diff を見て、テストを走らせる**（第27章）。 |

> **ポイント：** ワークフロー全体で最も肝心なのは、**第 ③ ステップの「計画の関門」** です――手を動かす前に、AI が何をしようとしているかをまず見るのです。多くのツールは「計画モード」（第27章、付録 C）を内蔵していて、このステップを強制するのを手伝ってくれます。**正確なスイッチは公式ドキュメントを参照してください。** 「まず計画を見てから通す」習慣を身につけることは、どんなコマンドを覚えるよりも重要です。

## 三、プロジェクト規約ファイル：コピーできる雛形

第 ② ステップで生まれた「規約」は、プロジェクトの中の **規約ファイル** 一つに落とし込み、ツールが毎回作業を始めるときに自動で読み取れるようにするのが最善です（Claude Code は `CLAUDE.md` を使う習慣があり、Codex には独自の同等の仕組みがあります。**正確なファイル名と置き場所は公式ドキュメントを参照してください**、付録 C を参照）。

> **ポイント（製品レイヤーの記憶、第3、13章に呼応）：** このファイルで「プロジェクトのルールを覚えさせる」ことができるのは、**モデル本体が記憶力を身につけたからではなく**、**製品が毎回作業を始めるたびに、ファイルの中身をコンテキストに入れ直して** モデルに渡しているからです。だから、**ファイルに書き込んだことだけを AI は「覚えている」のであり、書いていないことは「覚えていない」のです。** 規約が具体的であるほど、挙動は安定します。

以下は汎用の雛形です。`<...>` をあなたのプロジェクトの実際の状況に置き換えてください。

```markdown
# プロジェクト規約 / AI 協働ルール

## 最も重要な原則（必ず守る）
- 既存の実装方法と設計に沿って進め、自分の流儀を持ち込まない。
- 新しい仕組み（ライブラリ / 抽象 / レイヤー）を導入する前に、まず既存に同種のものがないか調べる。
  - あれば再利用する。なければ、または不十分なら、手を動かす前に理由と代替案を説明し、私の承認を待つ。
- 全体のアーキテクチャ（レイヤー構造、依存の向き、命名規則）を壊す変更は禁止。

## 実装の流れ（必須）
1. まず関連する既存の実装、共通モジュール、類似機能を調査し、参考の場所（パス）を挙げる。
2. 実装計画（どのファイルを変更 / 追加するか、何を再利用するか、新しい依存を加えるかとその理由）を出す。
3. 私が承認してから実装する。承認前にコードを書かない。
4. 既存のチェック / フォーマット / テストを一通り通す。

## 技術スタックとバージョン
- 言語 / フレームワーク / 主なライブラリ：<例：TypeScript 5 / React 18 / Vite>

## ディレクトリ構造と職責
- <例> components/：画面　hooks/：ロジック　services/：インターフェース通信　utils/：共通関数
- 依存の向き：<例> 画面 → hooks → services の一方向、逆方向やレイヤーをまたぐのは禁止。

## 必ず再利用する共通の仕組み（再作成は禁止）
- ネットワークリクエスト：<例 services/apiClient> を使う（あちこちで直接 fetch を書くのは禁止）
- エラー処理：<例 AppError / errorHandler> を使う
- ログ：<例 logger> を使う（直接 console.log は禁止）

## 禁止事項
- 勝手にサードパーティの依存を追加する
- 既存の共通の仕組みを迂回して自分で車輪を再発明する
- レイヤーをまたぐ違反の依存（画面層が直接データベース / インターフェースの詳細に触れるなど）
```

## 四、Harness を本当に効かせるためのいくつかのコツ

| コツ | どうするか |
| --- | --- |
| 規約は固有名詞で | 「きれいに書いて」では役に立たない。**再利用すべきモジュール名、ファイルパス、禁止事項** を固有名詞で書き切ってこそ役に立つ。 |
| 探索・計画・実装を分ける | 一言で全部を片付けようとしない。**調査と「計画の承認」という関門** を単独で一巡させるのが、最大のコツ。 |
| 小さく速く進む | 一度に大量の変更をさせると、最も本筋からそれやすい。**小さく分けて、小さなステップごとに review する**。 |
| 規約は育てる | AI が本筋からそれたら、対応するルールを規約ファイルに書き足し、チームで共有して継続的に更新する――**使えば使うほど使い勝手がよくなる**。 |
| 最後は必ず人が見る | Harness があっても本筋からそれることはある。**必ず「既存の書き方に合っているか」という観点で自分が diff を審査すること**（第27章）。 |
| 過度に抽象化しない | 「再利用」がやり過ぎると、必要のないカプセル化を生む。**既存のコードの粒度に沿って進む** だけでよい。 |

## 五、公開前チェックリスト（最も照らし合わせて一巡すべき部分）

この二つのリストを手元に貼っておきましょう。**AI に渡す前に** 一つ目を、**その成果物を本当に使う / コミットする前に** 二つ目を、一巡してください。

### 内容を AI に渡す前

- 渡す内容の中に、**機密コード、本番データ、個人のプライバシー** はないか？（あればまず匿名化するか、いっそ渡さない――第37章を参照）
- **API 鍵、パスワード、接続情報** が紛れ込んでいないか？（鍵は絶対に渡してはいけない――第40章を参照）
- あなたが使っている **サービスとプラン** は、会社 / 場面の規定で許されているものか？
- **前提**（言語、フレームワーク、バージョン）をはっきり伝えたか？
- 十分な **コンテキスト**（関連するコード、ログ、期待する振る舞い）を与えたか？

### その成果物をコミット / 採用する前

- **diff を一行ずつ読んで理解してから review** したか？（第27章）――**読んで分からないコードは使わない**。
- 自動 / 手動の **テストは通るか**、もともと正常だった機能を壊していないか（回帰）？
- **セキュリティの観点** でひと目見たか（入力検証、権限、インジェクションのリスク）？
- 新しく追加した **依存ライブラリ** は、ライセンスと出所が信頼できるか？
- AI が出した **関数名 / API / 設定項目は、本当に存在するか**？（公式ドキュメントと照合する――もっともらしくでっち上げることがある、第9章を参照）
- **重要な事実、数字、引用** を、自分で再確認したか？（第9、15、39章）

## 六、チームでルールを立てるとき（率いる人へ）

一人で使うなら習慣に頼ればよいですが、大勢で使うなら **明文のルール** に頼る必要があります。チームを率いるなら、少なくとも次の六つを定めておきましょう。

1. **線を引く**：どのデータを AI に渡してよく、どれは絶対にだめか、白黒つけて書く。
2. **ツールを限定する**：どのサービス、どのプランを使ってよいか、どの外部ソースにつなげてよいか（付録 C の MCP セキュリティに呼応）。
3. **承認フローを定める**：AI が生成したコードの、review → マージ → デプロイの承認の流れは何か。
4. **鍵を管理する**：鍵をどう保管し、誰が触れてよく、**漏れたらどう更新・失効させるか**（第40章）。
5. **共有資産を作る**：プロジェクト規約ファイル、よく使うプロンプトのテンプレート（付録 B）を整理して、チームで共用する。
6. **仕組みを作る**：定期的に共有会を開き、質問の窓口を設け、ルールを定期的に見直す――AI は変わるので、ルールもそれに合わせて更新する。

## まとめ

- **Harness ＝ AI に「理解 → 規約 → 計画 → 実装 → 検証」という手綱を付ける** こと。AI を既存のやり方に沿って仕事させ、別のやり方を一から始めさせないのです。Claude Code も Codex も適用できます。
- 落とし込みは **五ステップのワークフロー** で、そのうち **「計画の関門」（手を動かす前にまず計画を見る）が命門（いちばんの急所）** です。
- 規約は **プロジェクト規約ファイル** に書き込む――これは **製品レイヤーの記憶**（毎回入れ直す）であることを忘れずに。具体的に書くほど安定します。
- 二つの **チェックリスト** が両端を守ります。渡す前に情報を漏らさない、使う前に読んで理解する。**読んで分からないコードは使わない、重要な事実は自分で再確認する。**
- 製品の仕様、価格、データ条項はどれも変わります――**最新情報は、公式ドキュメントと、あなた自身（会社）の AI 利用規程を優先してください。**
