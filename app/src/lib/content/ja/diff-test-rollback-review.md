〔本章で具体的な製品のコマンド・画面・ボタンに触れる箇所は、いずれもバージョン更新とともに変わります。一律 **公式ドキュメントを参照** してください。本章では **共通する、安定した** 関門の動作だけを扱います――この部分は古くなりません。〕

第4部のうち、もし一章だけしか学ぶ余裕がないなら、それは本章です。

第23章で、私たちは一つの鉄則を確かめました：**AI は手を動かし、しかも間違える。だから「分からないもの、検証していないものを、通すな。」** けれども問題が起きます――**コードを書けない人が、どうやって AI の書いたコードを「検証」するのか？**

朗報は：**最終チェックは、一行一行のコードを読み解くことと同じではありません。** 必要なのは、これ以上ないほど素朴な四つのことです――**変更を読む(diff)、テストを一通り走らせる、ワンタッチで巻き戻す（ロールバック）やり方を知る、三つの問いで最終チェックする**。この四つは、プログラミング未経験でも身につけられ、しかも身につけたあとには、本当に「AI に自分の代わりに仕事をさせる」ハンドルを握ったことになります。

一つずつ見ていきましょう。

## 一、diff を読む：一種の「間違い探し」ゲーム

AI がコードを変更し終えたら、最初にすべきは「できた？」と尋ねることではなく、**変更を diff（差分/差異）の形で見せてもらう** ことです。

**diff とは「変更前後の対比」のことです。** この言葉に身構えないでください。本質はとても単純です――ほとんど同じ二枚の絵で「間違い探し」をするように、diff は「違う箇所」を **直接囲み出して** くれます。原稿を一行ずつ突き合わせる必要はありません。

たとえばこんな見た目です（これは **共通の例示** で、具体的な色や記号はツールによって少し異なります。**公式ドキュメントを参照**）：

```diff
  function greet(name) {
-   return "こんにちは、" + name;
+   return "こんにちは、" + name + "！お帰りなさい";
  }
```

読み方は三つだけ、覚えれば十分です：

- **緑色で、先頭に `+` が付いた行 ＝ 新しく足した行**（上の例の行が「歓迎の言葉を足した」）。
- **赤色で、先頭に `-` が付いた行 ＝ 削った行**（上の例の行が「元の書き方が置き換えられた」）。
- **色も記号もない行 ＝ 動いていない**、文脈を見せるためにそこに置いてあるだけです。

これだけです。一度の変更は、いくつものファイル、いくつもの追加・削除に及ぶことがありますが、diff は **一箇所ずつはっきり並べてくれます**：「どのファイルで、どの行を足し、どの行を削ったか」。あなたの仕事は、各行のコードの文法を読み解くことではなく、プログラミング未経験でも答えられる二つの問いに答えることです：

1. **触った場所は、合っているか？** 「トップページのタイトル色を変えて」と頼んだのに、diff の中で「ユーザーログイン」のコードが触られていたら――これは **明らかに違います**。たとえそのコードが読めなくても、本筋からそれたと分かります。
2. **触った範囲は、桁外れに大きくないか？** 小見出しを一つだけ変えてほしいと頼んだのに、diff には十数個のファイル・数百行の変更が表示されたら――**警鐘が鳴るべきです**。タスクを誤解したか、「ついでに」触ってはいけない箇所をたくさん動かしたかのどちらかです。

> **ポイント：** **diff を見るとは、「何を、どれだけ動かしたか」を見ることで、「各行のコードがどういう意味か」を見ることではありません。** これは、料理人でなくても「頼んだのはサラダ一皿なのに、出てきたのはどうして牛肉の角煮の鍋なのか」が分かるのと同じです――**合っているか、量は合っているか、はプログラミング未経験でも完全に判断できます。** これこそ diff という道具の最大の価値です：「AI がいったい何を変更したか」を広げて見せ、その一方的な言い分を鵜呑みにせずに済むようにしてくれるのです。

## 二、テストを走らせる：「もともと正常だったものを壊していない」と確かめる

「AI が何を変更したか」を読み取ったら、次の命取りになりうる問いはこうです：**新しい機能を足すときに、もともと正常だった機能を壊していないか？**

これはプログラミングではあまりにありふれていて、専門の言い回しがあるほどです――「あちらを立てればこちらが立たず」。A を直してと頼むと、A は直ったのに、うっかり B を壊してしまう。**しかも B が壊れた箇所は、あなたが見ている diff の中にすらないかもしれず**、diff を見るだけでは気づけません。

ここで出番なのが **テスト** です。

**「テスト」とは、あらかじめ書かれた一群の小さなチェックで、「これらの機能はまだ正常か」を自動で確かめる専用のものです。** 製品が工場から出荷される前の **品質検査ライン** のようなものだと思ってください：各製品がコンベアを一周し、機械が「点くべきランプは点くか、回るべき車輪は回るか」を自動で点検し、一つでも不合格があれば **赤いランプが即座に点きます**。

「率いる側の人」であるあなたにとって、やることは拍子抜けするほど簡単です：

- 変更が終わったら、**「テストを一通り走らせて」と頼む**。
- 結果を見る：**全部緑（すべて合格）＝ 高い確率で旧機能を壊していない**。**赤が出る（失敗したテストがある）＝ 何かを壊している。マージせず、直させる。**

テストを書けなくても、テストのコードを読み解けなくても、まったく構いません――**「全部通ったか/通らなかったか」という結果だけを見ればよい** のです。品質検査機の原理が分からなくても「緑ランプか赤ランプか」は分かるのと同じです。

> **誤解 vs 実際：** 多くの初心者は「AI が直したと言うなら、直ったのだ」と思います。**実は、「AI がそう思っている」ことと「テストに耐える」ことは別物です。** AI は本心から「直した」と思い込んでいるかもしれませんが（だましているのではなく、もっともらしく間違えているだけです――第9章のハルシネーション）、テストというラインは **情け容赦がありません**――一通り走らせれば、本物か偽物かは即座に明らかになります。だから習慣にしてください：**変更したら必ずテストを走らせ、赤ランプが点いていればマージしない。**

一つ補足：すべてのプロジェクトに出来合いのテストがあるわけではありません。なければ、せめて **最も素朴な人手での検証** をしてください――変更が影響した機能を **自分の手で一度使ってみて**、まだ正常か見るのです。たとえ「ページを開いてみて崩れていないか見る」だけでも、何も検証しないよりはずっとましです。

## 三、巻き戻し（ロールバック）：ワンタッチでやり直せる。これがあなたの最大の心のよりどころ

前の二つは「事後のチェック」でした。けれども内心こう危ぶんでいるかもしれません：**万一、本当にプロジェクトをぐちゃぐちゃに変更されてしまったら、プログラミング未経験の自分は、この惨状をどう収拾するのか？**

答えを聞けば完全に安心できます：**巻き戻し（ロールバック）――ワンタッチでやり直し、プロジェクトを AI が手を動かす前の、きれいな姿に戻すのです。**

なぜこれが可能なのでしょうか。ここで二つの場合を区別する必要があります：

**場合一：AI があなたのローカルのプロジェクトで手を動かす。** このとき巻き戻しは **バージョン管理(Git)** というものに頼ります。Git は **プロジェクトの写真を絶えず撮るカメラ** のようなものだと思ってください：手を動かす前、プロジェクトは「きれいな一枚の写真」です。ひとしきり変更したあと、万一気に入らなければ、**一言でプロジェクトを「あのきれいな写真に戻す」** ことができ、さきほどの変更はすべて取り消され、まるで何も起きなかったかのように、すっきりきれいになります。

この「写真カメラ」が後ろで支えているからこそ、安心して試させられるのです――**いつでもやり直せる** のですから。だから、先にはっきりさせておくべき前提があります：

> **ポイント：** **AI があなたのローカルのプロジェクトで手を動かすときは、必ず先に Git でプロジェクトを管理し、しかも先に「新しいブランチ」を切ってから手を動かさせてください。** 「新しいブランチを切る」は、こう理解してかまいません：**AI のために別の下書き用スペースを一区画用意し、下書きの上でこね回させ**、あなたの「正式稿」には触れさせない、ということです。こうすれば万一変更に失敗しても、下書きスペースを捨てればよく、正式稿は無傷です。このステップはあなたの **最大の安全網** で、具体的な操作は第26章で手取り足取り教えます。

**場合二：AI が Web/クラウドの形態で手を動かす。** このとき AI は通常 **あなたのローカルマシンで直接変更するのではなく**、プラットフォームが備える隔離環境の中で作業し、**プラットフォームには普通、独自の取り消し/破棄/リセットのやり方があります**。仕組みは Git とまったく同じではありませんが、目的は同じです：**あなたがやり直せるようにすること。** 具体的にどう取り消すか、どこまで戻せるかは、**公式ドキュメントを参照** してください。

このことの精髄は一言で覚えてください：**AI を「やり直しのきく」環境で手を動かさせること。** やり直しの具体的なボタンが何という名前か――ローカルなら Git、クラウドならプラットフォーム次第――は変わっても、「やり直せねばならない」というこの一点は、どこへ行っても変わりません。

## 四、最終チェックの三つの問い：最後のこの関門、分からなければマージするな

diff を見て、テストを走らせ、巻き戻せることを確かめた――最後に、これらすべてを、**いつでも使えて、いかなる道具も要らずにできる** 一つの動作にまとめます：**最終チェックの三つの問い**。

AI が「できました」と言うたびに、この三つの問いで一通りさらいます。**どれか一つでも答えられなければ、まだマージしないでください。**

1. **「何を変更したか？」**――diff を見ながら、「どこを何か所動かし、だいたい何をしているか」を言えるか。言えなければ、AI に説明させるか、あるいは **変更しすぎ** です（小さく分ける。第27章を参照）。
2. **「別のものを壊さないか？」**――テストは走らせたか？ 全部緑か？ テストがないなら、影響を受けた機能を自分の手で試したか？
3. **「この部分が何をしているか、一言で言えるか？」**――これが最も肝心な問いです。自分に（またはプログラミング未経験の人に）向かって、**一言のやさしい言葉** で「この変更は何をするものか」を言ってみてください。**もし言えないなら、それはあなたが分かっていないということです――分からないものは、絶対に通すな。**

三つ目の問いは、いちばん横着して飛ばされがちですが、それこそが **関門全体の魂** です。言えなくても恥ではありません。正しいやり方は **戻って AI に尋ねる** ことです：「この部分は何をしているの？ なぜこう変更したの？」 あなたが一言で復唱できるようになるまで説明させましょう。説明しきれない、あるいは説明すればするほど怪しくなるなら、その変更そのものに問題があるのかもしれません。

> **ポイント：** **「分からなければマージしない」――これが本章、そして第4部全体の最低ラインです。** やり終えた ≠ やり遂げた（第23章）。diff は何を変更したかを見せ、テストは何も壊していないと確かめる手伝いをし、巻き戻しは「やり直せる」よりどころを与え、そして最終チェックの三つの問いは、この三つを一本の、手軽に引ける安全ロープに撚り合わせます。**覚えてください：関門はあなたにあり、AI にはありません。**

## 五、四つの事を一連の動作につなぐ

これらを四つの孤立した知識点として扱わないでください。実はこれは一筋に流れる一本のラインで、第24章の「検収」の関門に対応します：

```text
AI が「できました」と言う
     │
     ▼
①diff を見る ──→ 何を変更した? 場所は合ってる? 範囲は正常?
     │
     ▼
②テストを走らせる ──→ 全部緑? 赤ランプがあればマージせず、突き返して直させる
     │
     ▼
③最終チェックの三つの問い ──→ 何を変更したか / 別のものを壊さないか / 一言で言えるか
     │
     ├── 三問とも通る ──→ マージし、次の小さな一歩へ
     │
     └── 一問でも詰まる ──→ 戻って聞きただす / 直させる / だめなら【巻き戻し】てやり直す
```

最後の「巻き戻してやり直す」が特に重要です：**直すほど乱れていく** とき、最もすっきりするのは惨状の中で無理に正そうとすることではなく、**きれいな状態に巻き戻し、新しい会話を開き、さきほど踏んだ落とし穴を前提として伝え直す** ことです（第13・30章でこの損切りの考え方を扱いました）。**巻き戻しは失敗ではなく、あなたの手にある最も使い勝手のよい切り札です。**

## 六、よくある誤解、まとめて訂正

| 誤解 | 実際 |
| --- | --- |
| diff を見るには各行のコードを読み解く必要があり、自分にはできない | diff を見るのは「何を、どれだけ動かしたか、場所は合っているか」を見ること。プログラミング未経験でも判断できる。緑＝足した、赤＝削った |
| AI が「直した」と言うなら、直ったのだ | 「AI がそう思っている」ことと「テストに耐える」ことは別物。変更したら必ずテストを走らせ、赤ランプが点いていればマージしない |
| テストは読み解けないので、使いようがない | テストのコードを読み解く必要はなく、「全部緑/赤がある」という結果だけ見ればよい |
| 万一壊したら、自分はおしまいだ | 「やり直しのきく」環境で手を動かさせれば怖くない：ローカルは Git に頼る（先に新しいブランチを切る）、Web/クラウドはプラットフォーム備え付けの取り消しに頼る（**公式ドキュメントを参照**） |

| AI が本筋からそれたら、その場で一行ずつ正してやらねば | 直すほど乱れるときは、**巻き戻し + 新しい会話でやり直す** ほうがたいてい最速（第13・30章） |

## まとめ

- **diff（差異）＝ 変更前後の対比**、「間違い探し」のように：**緑＝足した、赤＝削った**。見るのは「何を、どれだけ動かしたか、場所は合っているか」で、**各行のコードを読み解く必要はない**。
- **テストを走らせる ＝ もともと正常だった機能を壊していないと確かめる**、工場出荷時の品質検査ラインのように：あなたは「全部緑か、赤ランプがあるか」だけ見て、赤ランプが点いていればマージしない。テストがなければ、影響を受けた機能を自分の手で一度試す。
- **巻き戻し ＝ ワンタッチでやり直す**:AI を「やり直しのきく」環境で手を動かさせる――**ローカルは Git に頼る（先に新しいブランチを切る）**、Web/クラウドは **プラットフォーム備え付けの取り消しのやり方（公式ドキュメントを参照）** に頼る。
- **最終チェックの三つの問い**:①何を変更したか？ ②別のものを壊さないか？ ③一言で言えるか？――**どれか一つでも答えられなければ、まだマージしない。**
- 最低ラインを一言で：**「分からなければマージしない。」** 関門はいつも人の側にある（第23・39章）。直すほど乱れるときは、その場で無理に正すより巻き戻してやり直すほうがよい（第13・30章）。

次章（第26章）では、「やり直しのきく環境で手を動かさせる」ことを徹底的に扱います――許可の仕組みがどうあなたを守るのか、危険な操作はなぜ永遠に人の承認に委ねるのか。

## 小テスト

> 全 6 問。概念・誤解・場面・操作の 4 種類をカバーしています。まず自分で考えてから、引用ブロックの答えと解説を見てください。

1. **[基礎・概念問題]** 一つの diff の中で、「先頭に `+` が付いた緑色の行」は何を表す？
   - A. 削られた行
   - B. 新しく足された行
   - C. エラーが出た行
   - D. コメントの行
   > **答え B。** diff は「変更前後の対比」です：**緑＝足した(`+`)、赤＝削った(`-`)**、色のない行は動いていない、文脈を見せるためのもの。A を選ぶのは足すと削るを逆に覚えています――初学者が最も混同しやすい点で、「プラス記号は足す」と覚えれば間違えません。

2. **[基礎・概念問題]** 「テストを走らせる」のは主に何を確かめるため？
   - A. コードの見た目が美しいか
   - B. 今回の変更がもともと正常だった機能を壊していないか
   - C. AI の態度が良いか
   - D. プロジェクトが公開して儲かるか
   > **答え B。** テストは工場出荷時の品質検査ラインのように、「既存の機能がまだ正常か」を自動で点検する専用のものです。**あなたの目の前の diff の中にはなく**、変更を見るだけでは気づけない「あちらを立てればこちらが立たず」を捕まえられます。あなたは「全部緑か、赤ランプがあるか」だけ見ればよいのです。

3. **[基礎・誤解問題]** 「AI が『直した』と言うなら、確かに直ったのだ。そのままマージする」――どこが間違い？
   - A. 間違っていない、AI が良いと言えば良い
   - B. 「AI がそう思っている」ことと「テストに耐える」ことは別物。AI はもっともらしく間違えているかもしれず、diff とテストで検証すべき
   - C. AI は「直した」とは決して言わない
   - D. マージには先に課金が必要
   > **答え B。** AI はあなたをだましているのではなく、もっともらしく間違えている（第9章のハルシネーション）のです。テストというラインは情け容赦なく、一通り走らせれば本物か偽物かが即座に分かります。その口ぶりを鵜呑みにすることこそ、初心者が最もよくやる失敗です。

4. **[応用・誤解問題]** 「diff を見るには一行一行のコードが読み解ける必要があり、プログラミング未経験の自分にはそもそも無理だ」――この懸念はどこが余計か？
   - A. 余計ではない、コードが読めなければ diff は見られない
   - B. diff を見るのは「何を、どれだけ動かしたか、場所は合っているか」を見ること。「頼んだのはサラダなのに牛肉の角煮が出てきた」と見抜くのと同じで、プログラミング未経験でも完全に判断できる
   - C. diff はプログラマーしか開けない
   - D. プログラミング未経験者が diff を見るとプロジェクトを壊す
   > **答え B。** 最終チェックは各行の文法を読み解くことと同じではありません。「正しい場所を触ったか、範囲は正常か」、こうした判断にプログラミングの素養は要りません。「diff を見る」を「全コードを読み解く」と取り違えると、この最も役立つ関門の道具をみすみす手放すことになります。

5. **[基礎・場面問題]** AI があなたの **ローカルのプロジェクト** でコードを変更し、直すほど乱れてきた。「何も起きなかったことにしたい」。それができるためには、あなたは事前に何をしておく必要があった？
   - A. スクリーンショットを撮った
   - B. Git でプロジェクトを管理し、先に新しいブランチを切って AI を「下書きスペース」で手を動かさせた。これでワンタッチできれいな状態に巻き戻せる
   - C. コードを暗記した
   - D. パソコンの電源を切った
   > **答え B。** 「巻き戻し（ワンタッチでやり直す）」は Git という「写真カメラ」に頼り、**先に新しいブランチを切る** ＝ AI のために下書きスペースを一区画用意してこね回させ、正式稿には触れさせないことです。このステップはあなたの最大の安全網です（第26章で詳述）。Web/クラウドの形態を使うなら、プラットフォーム備え付けの取り消しのやり方に頼り、**公式ドキュメントを参照**。

6. **[応用・操作/観察問題]** 出来合いのコード変更を一つ（または本章の diff の例を）用意し、コードを読み解かなくてよいので、「最終チェックの三つの問い」だけを練習しましょう：**何を変更したか？ 別のものを壊さないか？ この部分が何をしているか、一言で言えるか？** 三つ目に答えられないとき、あなたはどうするつもり？
   > **観察できるはず：** 最終チェックの三つの問いは神秘的なものではなく、プログラミング未経験でもできます。三つ目に答えられないときの正しい動作は、**戻って AI に尋ねる** ことです：「この部分は何をしているの、なぜこう変更したの」と、あなたが一言で復唱できるようになるまで説明させる――AI の口ぶりが確信に満ちているからといって **通すのではなく**。「分からなければマージしない」という筋肉の記憶を身につけることは、どんなコマンドを覚えるよりも価値があります。具体的なツールのコマンド・画面は **公式ドキュメントを参照**。
