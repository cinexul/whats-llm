## 基礎概念

**人工知能**（人工智能 ・ Artificial Intelligence (AI)）  
機械に「本来は人間の知能が必要なこと」をやらせる技術全般を指す、とても大きな総称です。将棋を指す、画像を見分ける、迷惑メールをはじく、なども全部AIに含まれます。  
*よくある誤解：*「なんでもできるコンピュータの頭脳」と思われがちですが、実際には数多くの技術の総称であり、大規模言語モデルはその一分野にすぎません。  

**機械学習**（机器学习 ・ Machine Learning (ML)）  
人間が一つひとつルールを書くのではなく、大量のデータからプログラム自身に「規則性を見つけさせる」AIの作り方の一つです。深層学習や大規模言語モデルの土台になっています。  
*よくある誤解：*機械が「自分で理解した」と思われがちですが、実際にはデータから統計的に規則性を割り出しているだけで、意味を理解しているわけではありません。  

**深層学習**（深度学习 ・ Deep Learning (DL)）  
機械学習の一種で、何層も重ねた「計算ネットワーク」（ニューラルネットワーク）を使って学習します。今の大規模言語モデル、画像生成、音声合成の多くはこれに基づいています。  

**ニューラルネットワーク**（神经网络 ・ Neural Network）  
脳の神経細胞（ニューロン）から着想を得た計算の仕組みで、たくさんの単純なユニットが層状につながってできています。深層学習は、この層を何層も重ねたニューラルネットワークを使います。  
*よくある誤解：*「人間の脳のように考える」と思われがちですが、実際には単なる比喩を借りているだけで、中身はすべて数値計算であり、本物の神経とは関係ありません。  

**大規模言語モデル**（大语言模型 ・ Large Language Model (LLM)）  
膨大なテキストから「単語と単語がどうつながるか」を学んだAIモデルで、ClaudeやChatGPTといった製品の中核です。  
*よくある誤解：*「なんでも知っているコンピュータの頭脳」と思われがちですが、実際には言語の処理が得意なモデルの一種にすぎず、もっともらしく間違えることもあります。  

**モデル**（模型 ・ Model）  
学習後に得られる膨大な数値（パラメータ）の集まりと、それを動かす仕組みを合わせたものです。「ルールを一つひとつ書いた」ふつうのプログラムとは違います。  
*よくある誤解：*モデルは大きなデータベースやif文の塊だと思われがちですが、実際には学習で得たパラメータであり、答えはその場で計算しているのであって、どこかから検索しているわけではありません。  

**生成AI**（生成式AI ・ Generative AI）  
判断や分類をするだけでなく、新しいコンテンツ（文章、画像、コード、音声など）を「作り出せる」AIです。大規模言語モデルはその中でもっとも代表的なものの一つです。  

**自然言語処理**（自然语言处理 ・ Natural Language Processing (NLP)）  
コンピュータに人間の言語を扱わせる（理解・翻訳・要約・質問応答など）技術分野です。大規模言語モデルは、この分野で近年もっとも強力なツールになっています。  

**汎用人工知能**（人工通用智能 ・ Artificial General Intelligence (AGI)）  
人間のようにあらゆる課題をこなせるAIという、構想上の目標です。今の大規模言語モデルはまだAGIではなく、この言葉は主に議論や将来像の中で登場します。  
*よくある誤解：*今あるチャットAIをAGIだと思われがちですが、実際には言語が得意な専用モデルにすぎず、「なんでもできる」状態にはほど遠い段階です。  

**チャットボット**（聊天机器人 ・ Chatbot）  
人間と対話形式でやり取りできるプログラムです。今の優秀なチャットボットの中核は多くが大規模言語モデルですが、「チャットボット」という言葉自体は大規模言語モデルより古く、もっと広い意味を持ちます。  

**アルゴリズム**（算法 ・ Algorithm）  
ある問題を解くための、決まった手順やルールの集まりです。大規模言語モデルは一つひとつ書き込まれたアルゴリズムではなく、データから学習したものであり、ここが従来のプログラムとの違いです。  
*よくある誤解：*大規模言語モデルを「あらかじめ書かれたアルゴリズム／ルール」だと思われがちですが、実際にはその振る舞いは学習で得たパラメータから生まれており、「なぜそう答えたか」を一つひとつ書き出せる人はいません。  

**コンピュータビジョン**（计算机视觉 ・ Computer Vision）  
コンピュータに画像や動画を「見て理解させる」AIの分野です（顔の認識や標識の読み取りなど）。言語を扱う大規模言語モデルとは、AIの中の別の分野であり、「AIは大規模言語モデルだけではない」と理解するための対比に役立ちます。  

## モデルと学習

**パラメータ**（参数 ・ Parameter）  
モデル内部にある膨大な「調整可能な数値」のことで、学習とはこれを少しずつ調整し続けることです。言語の規則性は、この数値の中に「圧縮」されて入っています。よく言われる「数十億パラメータ」とは、この数のことです。  
*よくある誤解：*パラメータが多いほど必ず賢いと思われがちですが、実際にはパラメータ数は規模の一側面にすぎず、データの質や学習方法も同じくらい重要です。  

**重み**（权重 ・ Weight）  
パラメータの中でもっとも主要な種類で、ニューラルネットワークの中で「あるつながりがどれだけ重要か」を表します。学習の過程は、基本的にこの重みを繰り返し調整することです。  

**学習（トレーニング）**（训练 ・ Training）  
モデルに大量のデータを見せ、内部のパラメータを調整し続ける過程で、出力を期待に近づけることが目的です。学習が終わって初めて、使えるモデルが得られます。  

**事前学習**（预训练 ・ Pre-training）  
大規模言語モデルを作る第一段階です。膨大なテキストを読ませ、「次の単語を当てる」練習を繰り返させることで、言語の規則性をパラメータに学び込ませます。この工程がもっとも時間とお金がかかります。  

**ファインチューニング**（微调 ・ Fine-tuning）  
事前学習済みのモデルに、比較的小さなデータセットでさらに「補習」をさせ、特定の種類のタスクが得意になったり、指示どおりに答えやすくなったりさせることです。アライメントの代表的な手段の一つです。  
*よくある誤解：*ファインチューニング＝新しい知識を事実としてモデルに「教えて」記憶させること、と思われがちですが、実際には「振る舞いやスタイル」を調整する面が大きく、自社の資料を読ませたいならRAGのほうが向いています。  

**指示チューニング**（指令微调 ・ Instruction Tuning）  
専用のファインチューニングの一種で、「指示とよい回答」の例を大量に使ってモデルに人の指示を聞き取って実行させ、「続きを書く機械」から「指示どおりに動くアシスタント」へと変えます。  

**アライメント**（对齐 ・ Alignment）  
さまざまな方法でモデルの振る舞いを「役に立ち、誠実で、無害」になるよう調整し、回答を人の意図や価値観に合わせることです。アライメントされていないモデルは、でたらめを言いやすくなります。  
*よくある誤解：*アライメントすればモデルは絶対に安全で信頼できると思われがちですが、実際にはリスクを下げられるだけで、「過度に慎重になり、すぐに回答を拒む」という副作用も生みます。  

**人間のフィードバックによる強化学習（RLHF）**（人类反馈强化学习 ・ Reinforcement Learning from Human Feedback (RLHF)）  
アライメントの主流の手法の一つです。モデルの複数の回答に人が点数をつけて順位を決め、その好みを使ってモデルを学習させることで、人が好む回答を出しやすくします。  

**学習データ**（训练数据 ・ Training Data）  
モデルの学習に使うテキストやコードなどの材料です。データの質・範囲・偏りは、モデルが最終的に持つ能力やクセに直接影響します。  
*よくある誤解：*モデルが学習データの原文をそのまま保存していると思われがちですが、実際に学んでいるのはその中の規則性であり、原文が丸ごとモデルに保存されているわけではありません。  

**知識カットオフ**（知识截止 ・ Knowledge Cutoff）  
モデルの学習データの「締め切り」です。この時点より後に起きたことや、後から出た新バージョンを、モデル自身は知りません。  
*よくある誤解：*AIは常に最新情報を知っていると思われがちですが、実際には別途インターネットにつなぐか資料を読ませない限り、カットオフ以降のことは何も知らず、でたらめを作ることもあります。  

**基盤モデル**（基础模型 ・ Foundation Model / Base Model）  
膨大な汎用データで学習された「土台」となる大規模言語モデルで、それ自体が幅広い能力を持ち、さらにファインチューニングやアライメントを経てさまざまな製品になります。  
*よくある誤解：*基盤モデルと完成したチャット製品が混同されがちですが、実際には基盤モデルはまだ「対話の作法」が整っておらず、そのまま使うと指示に従わないことがよくあります。  

**蒸留**（蒸馏 ・ Distillation）  
大きくて強いモデルを「先生」にして、より小さな「生徒」モデルを学習させ、小さなモデルが低いコストで大きなモデルの性能に近づけるようにすることです。  

**モデル規模**（模型规模 ・ Model Scale / Model Size）  
モデルがどれだけ「大きい」かを測る尺度で、主にパラメータの数と学習データの量で見ます。規模が大きいほどたいてい強力ですが、その分遅く、高くもなります。  

**モデルのバージョン・世代**（模型版本/代际 ・ Model Version / Generation）  
同じシリーズのモデルは新しいバージョンを次々と出していき（世代交代など）、それに伴って能力や価格も変わります。  

**学習時の文脈長**（上下文长度（训练时） ・ Context Length）  
モデルを設計・学習する際に定められた「一度に最大どれだけのトークンを処理できるか」という能力の上限で、使うときにはコンテキストウィンドウの大きさに対応します。  

**スケーリング則**（规模法则 ・ Scaling Laws）  
モデル・データ・計算資源が大きいほど効果も良くなりやすい、という経験則です。各社がこぞってモデルを大きくする理由を説明していますが、無限にタダで効くわけではありません。  
*よくある誤解：*「大きくしさえすれば必ず良くなる」と思われがちですが、実際には効果は逓減し、データの質や学習方法も上限を左右します。  

## 動作原理

**トークン**（token ・ Token）  
大規模言語モデルから見た文字の最小単位で、一つの単語、単語の一部、あるいは一文字のこともあります。モデルは「文字」単位ではなく、トークン単位で読み、計算します。  
*よくある誤解：*1トークン＝1文字や1単語だと思われがちですが、実際にはモデル自身が切り出した断片であり、日本語・英語、長い単語・短い単語で切り方が異なります。  

**トークン化（分かち書き）**（分词 ・ Tokenization）  
一続きの文章を一つひとつのトークンに分割する処理です。モデルによって切り方が違い、これが同じ一文でもモデルによってトークン数が変わる理由です。  

**コンテキストウィンドウ**（上下文窗口 ・ Context Window）  
モデルが一度に「同時に見られる」トークンの上限で、会話全体をこの「一時的な作業台」に収める必要があります。これを超えると、古い内容から押し出されて消えていきます。  
*よくある誤解：*AIの長期記憶だと思われがちですが、実際にはその回の会話のための一時的な容量にすぎず、閉じたり満杯になったりすると忘れ、しかも満杯に近いほど遅く高くなります。  

**次トークン予測**（下一个词预测 ・ Next-Token Prediction）  
大規模言語モデルがやっている中核はこれ一つだけです。前の内容をもとに「次にもっとも出てきそうなトークン」を当て、一つずつつないでいって、まとまった文章にします。  
*よくある誤解：*AIは先に答えを考えてから話すと思われがちですが、実際には1トークンずつ吐き出しながら、書きつつ当てているのであって、あらかじめ用意した「答え」はありません。  

**確率分布**（概率分布 ・ Probability Distribution）  
モデルは次のトークンを選ぶとき、候補となる単語ごとに確率を計算し（例：「東京 92%、江戸 0.8%」）、その中から一つを選びます。  

**サンプリング**（采样 ・ Sampling）  
あの「候補単語の確率表」から、実際に出力する単語を選び出す処理です。選び方は安定寄り（常に高確率を選ぶ）にも、ランダム寄りにもでき、温度などのパラメータで制御します。  

**温度（temperature）**（温度 ・ Temperature）  
「出力のランダムさ」を制御するつまみです。下げると安定し、ほぼ毎回同じになります。上げると発散して創造的になりますが、その分外しやすくもなります。  
*よくある誤解：*温度を上げる＝賢くなると思われがちですが、実際には「マイナーな単語を選ぶ思い切り」を変えるだけで、正誤とは無関係であり、上げすぎるとかえって暴走しやすくなります。  

**Top-p（核サンプリング）**（Top-p采样 ・ Top-p (Nucleus) Sampling）  
ランダムさを制御するもう一つのつまみです。「確率を足し合わせて十分高くなる、ひと握りの候補単語」の中だけから選び、マイナーすぎる選択肢を除外します。温度とよく併用されます。  

**拡張思考（推論の深さ）**（推理强度/扩展思考 ・ Reasoning Effort / Extended Thinking）  
答えを出す前に、モデルに内部で「数手余分に考えさせ」（思考の下書きを生成させ）、より多くの計算資源と引き換えに、より確かな結果を得る仕組みです。難問に向いており、簡単な問いには不要です。  
*よくある誤解：*「長く考えるほど答えは必ず正しくなる」と思われがちですが、実際には複雑な推論は向上させられるものの、その分遅く高くなり、なお間違えることもあり、簡単なタスクではむしろ無駄になります。  

**ベクトル**（向量 ・ Vector）  
数字を並べてできた「座標」です。単語や文章をベクトルに変えると、機械はそれらの遠近、つまり意味の近さを数学で比べられるようになります。  

**埋め込み（エンベディング）**（嵌入 ・ Embedding）  
一続きの文章を「意味」を表す一連の数字（ベクトル）に変換するやり方です。意味の近い文章はベクトルも近くなり、検索やRAGの土台になります。  
*よくある誤解：*エンベディングは原文を保存していると思われがちですが、実際に保存しているのは「意味の座標」であり、元の文は読み取れませんが、誰と誰の意味が近いかは計算できます。  

**ベクトル空間**（向量空间 ・ Vector Space）  
すべての単語や文章のベクトルを並べて作った「多次元の空間」です。意味の近いものは近くに集まり、有名な「王様 − 男 ＋ 女 ≒ 女王」もここで起きています。  

**類似度**（相似度 ・ Similarity）  
二つの文章の「意味がどれだけ近いか」を測る数値で、それぞれのベクトルを比べて算出します。検索やRAGは、これを使って「もっとも関連する」資料を見つけ出します。  

**アテンション機構**（注意力机制 ・ Attention Mechanism）  
モデルが一文を読むとき、「各単語が他のどの単語にどれだけ注目すべきか」を決める能力です。要は重要な部分に印をつけられるということで、これによって長い文の中の重要なつながりをつかめます。  
*よくある誤解：*アテンション＝モデルが「本当に理解した」と思われがちですが、実際に計算しているのは単語と単語の統計的な関連であり、数学的な重みづけであって、理解ではありません。  

**自己注意（セルフアテンション）**（自注意力 ・ Self-Attention）  
アテンション機構の中心となる形で、一文の中の各単語が同じ文の中の他の単語を見ることで、文脈の関係を織り込んでいきます。Transformerの重要な部品です。  

**トランスフォーマー**（Transformer ・ Transformer）  
今のほぼすべての主流の大規模言語モデルの土台となるアーキテクチャで、中核はアテンション機構です。大規模言語モデルの「エンジンの型式」と理解すればよく、こういうものがあると知っておけば十分です。  
*よくある誤解：*Transformerを何か具体的な製品だと思われがちですが、実際には技術的なアーキテクチャであり、Claude、GPT、Geminiなどはどれもこの上に建てられています。  

**エンコーダ・デコーダ**（编码器/解码器 ・ Encoder / Decoder）  
Transformerの中の二種類の部品です。エンコーダは「入力を読み解く」役割を、デコーダは「出力を生成する」役割を担います。チャット系の大規模言語モデルは、主にデコーダのほうを使います。  

**知識**（知识 ・ Knowledge (parametric)）  
学習時にパラメータへ焼き込まれた「常識や事実」のことで、その回に与えた資料に頼らず、モデルがすらすらと答えられるものです。ただし、カットオフ以降のことはこの中に含まれません。  
*よくある誤解：*「モデルが知っている」を「モデルが原文を保存していて調べられる」と思われがちですが、実際にはそれは学習で得た統計的な印象であり、記憶を取り違えたり間違えたりすることもあります。  

**記憶（文脈上の記憶）**（记忆 ・ Memory (in-context)）  
「その回の会話」の中だけに存在する一時的な記憶で、すべてコンテキストウィンドウで支えられています。会話を閉じたりウィンドウから押し出されたりすると、きれいさっぱり忘れてしまいます。  
*よくある誤解：*AIは人間のようにあなたを長く覚えていると思われがちですが、実際には初期設定では会話をまたいだ記憶はなく、次に会うときは「記憶喪失」の状態です。  

**ハルシネーション**（幻觉 ・ Hallucination）  
モデルが、存在しない内容や誤った内容を、いかにもそれらしく自信たっぷりに語ることです。モデルが目指しているのは「本当らしいこと」であって、「本当であること」ではないからです。  
*よくある誤解：*流暢で自信があるほど信頼できると思われがちですが、実際にはすらすら語るほど警戒すべきで、重要な事実は必ず自分で再確認しなければなりません。  

**創発的能力**（涌现能力 ・ Emergent Abilities）  
モデルの規模が一定以上になると、小さなモデルには無かった新しい能力（多段階の推論など）が「突然」現れることです。水が一定の温度になって初めて沸騰するのに似ています。  
*よくある誤解：*創発をAIの「覚醒」や意識の芽生えのように言われがちですが、実際には規模がもたらした能力の飛躍にすぎず、中身は依然として数値計算です。  

**マルチモーダル**（多模态 ・ Multimodal）  
モデルが文字だけでなく、画像、音声、さらには動画など、複数の「モダリティ」の情報を扱えることです。たとえば、画像を見て説明したり、スクリーンショットを読んだりできます。  

**決定論的・確率的な出力**（确定性vs随机性 ・ Determinism vs Randomness）  
生成にはランダム性があるため、同じ問いを何度も尋ねると、答えが毎回少しずつ変わることがあります。「前回は正しかった」は「今回も正しい」を意味しません。  
*よくある誤解：*同じ入力には必ず同じ出力が返ると思われがちですが、実際には初期状態でランダム性があり結果は変わるので、重要な出力は毎回検証する必要があります。  

## 使い方

**プロンプト**（提示词 ・ Prompt）  
あなたがAIに送る、ひとまとまりの文章や指示です。与える背景が明確なほど、たいてい回答も良くなります——「入力の質が出力の質を決める」のです。  
*よくある誤解：*言わなかった意図まで察してくれると思われがちですが、実際にはあなたの心は読めないので、背景・要件・欲しい形式は、すべてはっきり言葉にする必要があります。  

**プロンプトエンジニアリング**（提示工程 ・ Prompt Engineering）  
「どう言えば話が明確になり、要件がはっきり伝わるか」を工夫して、AIに望む結果をより良く出させるための一連の手法です。神秘的なものではなく、本質は要望を上手に伝えることです。  

**システムプロンプト**（系统提示 ・ System Prompt）  
会話のいちばん最初で、AIの「役割と全体のルール」を設定する指示です。たとえば「あなたは厳密な法律顧問です」のように。これは会話全体のスタイルや境界線に影響します。  

**ゼロショット**（零样本 ・ Zero-shot）  
例を一切与えず、いきなりAIにタスクをやらせることです。簡単なタスクならたいてい十分ですが、複雑なものや形式が特殊なタスクには、いくつか例を示すほうが向いています。  

**フューショット（少数例）**（少样本/示例 ・ Few-shot）  
プロンプトの中に「入力と欲しい出力」の例をいくつか示し、AIにそれをまねさせることです。決まった形式や特定のスタイルが欲しいときに特に役立ちます。  

**思考の連鎖（Chain of Thought）**（思维链 ・ Chain of Thought (CoT)）  
AIに「推論の過程を一歩ずつ書き出させて」から結論を出させることで、数学やロジックなど複雑なタスクではたいてい精度が上がります。  

**文脈（コンテキスト）**（上下文 ・ Context）  
あなたがその回にAIへ与えるすべての情報（問い、背景、貼り付けた資料、これまでの会話）です。AIは今この瞬間、これらだけに基づいて答えるしかありません。  

**セッション（会話）**（会话/对话 ・ Session / Conversation）  
チャットを開いてから終えるまでの、ひとまとまりのやり取りです。同じ会話の中ではAIは前の内容を覚えていますが、新しい会話に切り替えると、ゼロから始まります。  

**マルチターン会話**（多轮对话 ・ Multi-turn Conversation）  
一つの話題をめぐって何度もやり取りすることです。長く続けるほど文脈が場所を取り、話がそれたり「話すほど鈍く」なったりしやすくなるので、必要なら新しい会話を始めるべきです。  
*よくある誤解：*長く話すほどAIがあなたを分かってくれると思われがちですが、実際には話題が混ざって文脈が満杯になるとかえって間違えやすく、一つの話題に一つの会話のほうが安定します。  

**要約**（摘要/总结 ・ Summarization）  
AIに長い文章を要点に圧縮させることです。コツは「私が与えた資料に基づいて要約し、勝手に足さない」ことで、そうしないと作り話が混ざりやすくなります。  

**構造化出力**（结构化输出 ・ Structured Output）  
AIに決まった形式で出力させることです。たとえば表、リスト、JSONなど。フィールドや形式をはっきり指定すると、結果がそろって使いやすくなります。  

**ロール設定（役割付与）**（角色设定 ・ Role / Persona Setting）  
AIにある身分や視点（「ベテラン編集者」「面接官」など）を指定し、口調や重点をあなたのニーズに合わせさせることです。  

**反復的な修正**（迭代修正 ・ Iterative Refinement）  
一度で完璧を期待せず、「下書きを出す → どこが違うか指摘する → 直させる」を何度も繰り返して、望む結果に近づけていくことです。AIを使いこなすうえでの常道です。  

**プロンプトテンプレート**（提示词模板 ・ Prompt Template）  
よく使う良い指示を「穴埋めテンプレート」として固定し、次回は中身を入れ替えるだけで使い回せるようにすることです。毎回ゼロから書く手間が省けます。  

**検証・裏取り**（验证/复核 ・ Verification / Fact-checking）  
AIが出した事実・数字・引用を、自分でもう一度調べて確かめることです。AIはハルシネーションを起こすので、重要な内容は決してそのまま使ってはいけません。  
*よくある誤解：*裏取りは面倒だから省ける、と思われがちですが、実際にはAIを使ううえでもっとも重要な一手であり、これを省くのはAIの誤りを自分の誤りにするのと同じです。  

**ウェブ検索・ツール連携**（联网搜索/工具增强 ・ Web Search / Tool-augmented）  
AIが回答時にインターネットで最新情報を調べたり、ツールを呼び出したりして、「知識カットオフ」や覚えていられない問題を補うことです。最新の内容を調べられますが、調べた結果もやはり再確認が必要です。  
*よくある誤解：*ネットにつなげば答えは必ず正確だと思われがちですが、実際には誤った情報源を選んだり読み違えたりすることもあり、最終的には引用をあなたが照合する必要があります。  

**再生成・言い換え**（重新生成/换个说法 ・ Regenerate / Rephrase）  
満足できない回答に対して、もう一度答えさせたり、別の聞き方で尋ねたりすることです。出力にはランダム性があり言い回しにも敏感なので、表現を変えると良い結果が得られることがよくあります。  

**たたき台**（草稿/初稿思维 ・ First Draft / Starting Point）  
AIの成果物を「完成品」ではなく「下書き」として扱う考え方です。AIが手早く取っかかりを作ってくれたら、あなたが判断し、修正し、仕上げます。これがもっとも安全な協働の心構えです。  
*よくある誤解：*AIに一発で完成させてそのまま提出しようと考えがちですが、実際には「下書きをするインターン」として使い、あなたが仕上げるほうが、出来も安全性も良くなります。  

**プロンプト漏洩**（提示词泄露 ・ Prompt Leaking）  
話術を使ってAIを誘導し、あなたが設定したシステムプロンプトや隠れた指示を「吐き出させる」ことです。ですから、本当の機密はプロンプトに書き込まないでください。  

## ツールのエコシステム

**モデルと製品の違い**（模型vs产品 ・ Model vs Product）  
モデルは土台となる中核（GPT、Claudeシリーズなど）で、製品は人が使えるよう包装されたもの（ChatGPTのウェブ版、Claudeアプリなど）です。一つの製品の裏で、異なるモデルに切り替えて使われることもあります。  
*よくある誤解：*製品名とモデル名を同じものだと思われがちですが、実際にはChatGPTは製品、GPTはモデルで、両者は「車」と「エンジン」の関係です。  

**API**（API ・ Application Programming Interface (API)）  
あなた自身のプログラムからモデルを直接「呼び出す」入口で、チャットのウェブ画面を介しません。「プログラムにAIをつなぐ」とは、たいていAPIを呼ぶことを指します。  
*よくある誤解：*APIを別のもっと強力なAIだと思われがちですが、実際には同じモデルにアクセスするもう一つの方法（プログラム対プログラム）にすぎず、使った分だけ課金されます。  

**APIキー**（API密钥 ・ API Key）  
APIを呼び出すときに「あなた本人である」ことを証明する機密の文字列で、アカウントの鍵に相当します。漏れると、他人があなたのお金を使えてしまいます。  
*よくある誤解：*キーをチャット欄やコードに貼っても大丈夫だと思われがちですが、実際には一度漏れると不正利用されかねず、決して公開してはならず、漏れたらすぐに無効化してリセットすべきです。  

**トークン課金**（Token计费 ・ Token-based Pricing）  
APIはたいていトークン数で課金され、入力も出力もお金がかかります。長く話すほど、資料を多く読ませるほど、費用もかさみます。  
*よくある誤解：*自分が打った文字だけが課金対象だと思われがちですが、実際にはAIの返信や、毎回持ち越される履歴の会話もトークンに数えられ、長い会話はとても費用がかさみます。  

**SDK**（SDK ・ Software Development Kit (SDK)）  
メーカーが提供する出来合いのコードライブラリ群で、開発者がより手軽にAPIを呼び出せるようにし、リクエストをゼロから組み立てる必要をなくします。  

**推論（インファレンス）**（推理(运行模型) ・ Inference）  
「すでに学習済みのモデルを動かして結果を出すこと」を指し、つまりあなたが質問するたびにAIが回答を生成する、その計算過程のことです。「学習」とは別物です。  
*よくある誤解：*ここでの「推論」を「論理的な推理」と理解されがちですが、実際には「モデルを動かして結果を出す」ことを指す工学用語です。  

**エージェント**（智能体 ・ Agent）  
質問に答えるだけでなく「手を動かせる」AIです。目標を与えると、自分でツールを呼び出し、「調査 → 実行 → 確認」のループに沿って前へ進めていきます。  
*よくある誤解：*エージェントを「より賢いチャットボット」だと思われがちですが、実際の要点は「本当に手を動かして物事を進められる」ことであり、能力もリスクもこの点から生まれます。  

**ツール呼び出し（ファンクションコーリング）**（工具调用 ・ Tool Use / Function Calling）  
モデルが回答時に「外部ツールを呼び出す」（データベースを調べる、計算する、リクエストを送るなど）仕組みです。これがエージェントが手を動かせる技術的な土台です。  
*よくある誤解：*モデルは自力でネットにつないだり計算したりできると思われがちですが、実際には与えられたツールを通してやる必要があり、ツールがつながっていなければ口で言うことしかできません。  

**Claude**（Claude ・ Claude）  
Anthropic社のAI製品／モデルシリーズの名称です（Opus、Sonnet、Haikuなどを含みます）。大規模言語モデルの一つの代表であり、「AIのすべて」ではありません。  

**Claude Code**（Claude Code ・ Claude Code）  
ターミナル（コマンドライン）上で動くプログラミングエージェントで、プロジェクトのファイルを読み、コードを直し、コマンドを実行できます。「手足のついたClaude」と理解すればよいでしょう。  

**ChatGPT**（ChatGPT ・ ChatGPT）  
OpenAI社のチャット製品で、中核はGPTシリーズのモデルです。多くの人が最初に触れる大規模言語モデルの製品です。  

**Codex**（Codex ・ Codex）  
OpenAIのこの系列における、プログラミング／コーディングエージェント方向の製品です。Claude Codeと同様に、どちらも「手を動かしてコードを書く」ツールに属します。  

**Gemini**（Gemini ・ Gemini）  
Google社のAI製品／モデルシリーズの名称です。これも主流の大規模言語モデルの一つで、ClaudeやChatGPTとよく並べて比較されます。  

**Copilot**（Copilot ・ Copilot）  
エディタやオフィスソフトに組み込まれたAIアシスタント（コード補完、文書アシスタントなど）を広く指します。ここでは「補助の副操縦士」という概念としてのみ扱います。  

**検索拡張生成（RAG）**（检索增强生成 ・ Retrieval-Augmented Generation (RAG)）  
まずあなたの資料ライブラリから関連する内容を「検索」し、それをモデルに渡して「この資料に基づいて」答えさせることです。知識カットオフを回避し、ハルシネーションを減らし、自社の資料を活用するために使います。  
*よくある誤解：*RAG＝資料をモデルに「学習」させること、と思われがちですが、実際には一時的に検索して文脈に流し込むだけで、モデル本体は変えておらず、資料を更新しても再学習は不要です。  

**ベクトルデータベース**（向量数据库 ・ Vector Database）  
「テキストのベクトル（エンベディング）」を専門に保存し、「もっとも近い内容」を素早く見つけ出せるデータベースです。RAGの中で「検索」の工程を担う代表的な部品です。  

**ナレッジベース**（知识库 ・ Knowledge Base）  
AIが検索して使うために、あなたが専用に整理した資料のまとまり（文書、ウェブページ、マニュアルなど）です。RAGは、ここから材料を取ってきます。  

**ローカルモデル**（本地模型 ・ Local Model）  
あなた自身のPC（または自前のサーバー）上で動き、クラウドを介さない大規模言語モデルです。利点はプライバシーとオフライン動作ですが、その代わりにハードウェアが必要で、たいていクラウドのモデルほど強力ではありません。  
*よくある誤解：*ローカルモデルは必ずクラウドに劣る、あるいは必ずより安全だと思われがちですが、実際にはより秘匿性は高いものの、能力や速度はあなたのハードウェアに左右され、強弱は具体的な機種によります。  

**オープンソースモデル・オープンウェイト**（开源模型/开放权重 ・ Open-source / Open-weight Model）  
モデルの重みを公開し、誰もがダウンロードして自分で動かせるようにしたモデルです。これによってローカル運用が可能になり、オフラインやプライバシーに敏感な場面でよく使われます。  
*よくある誤解：*「オープンウェイト」を「完全にオープンソースで商用可」と同一視されがちですが、実際にはモデルごとにライセンス条項の差が大きく、使う前にライセンスをよく確認する必要があります。  

**Ollama**（Ollama ・ Ollama）  
オープンソースの大規模言語モデルを、自分のPC上で手軽にダウンロードして動かせるツールです。コマンドが簡単で、初心者がローカルモデルの入門によく使います。  

**LM Studio**（LM Studio ・ LM Studio）  
グラフィカルな画面を備え、オープンソースの大規模言語モデルをローカルで手軽にダウンロードして動かせるデスクトップアプリです。コマンドを打ちたくない人がローカルモデルを試すのに向いています。  

**量子化**（量化 ・ Quantization）  
モデル内の数字をより「粗い」精度で保存することで、モデルを小さくし、ふつうのPCでも動くようにする手法です。代わりに品質はわずかに落ちます。ローカルで大規模言語モデルを動かす際によく使われます。  
*よくある誤解：*量子化するとモデルが「かなり鈍くなる」と思われがちですが、実際には適度な量子化ならたいてい品質はわずかしか下がらず、ハードウェアへの要求を大幅に下げられます。  

**スキル**（Skills/技能 ・ Skills）  
「ある物事をやるための手順と関連ファイル一式」を、再利用できる「スキル」としてまとめ、必要なときにだけ読み込んで、ふだんは文脈を占有しないようにしたものです。AIに与えるプラグアンドプレイのプラグインパックと理解すればよいでしょう。  

**モデルコンテキストプロトコル（MCP）**（模型上下文协议 ・ Model Context Protocol (MCP)）  
AIアプリが外部のツールやデータソースに統一的につなぐための開放標準で、よく「AIの世界のUSB端子」にたとえられます。挿せばさまざまな周辺機器が使えるようになります。  
*よくある誤解：*MCPを何か一つの製品やサーバーだと思われがちですが、実際には「インターフェースの規格」一式であり、具体的な機能は個々のMCPサーバーが提供します。  

**MCPサーバー**（MCP服务器 ・ MCP Server）  
MCP標準に従って、あるツールやデータのインターフェースを提供するプログラムです（あなたのカレンダーやデータベースにつなぐなど）。AIはクライアントとして、これに接続して能力を取り込みます。  

**プラグイン**（插件 ・ Plugin）  
AI製品に能力を追加で取り付ける小さなモジュールで、ある外部サービスにつないだり、ある機能を増やしたりできるようにします。「能力を拡張する」ことのくだけた言い方です。  

**マルチモーダル入力**（多模态输入 ・ Multimodal Input）  
文字を打つだけでなく、AIに画像・ファイル・音声などをアップロードして一緒に処理させられることです。たとえば、スクリーンショットを送って説明させるなど。  

**クラウドモデル**（云端模型 ・ Cloud Model / Hosted Model）  
メーカーのサーバー上で動き、ネットワーク越しにアクセスする大規模言語モデルです（ほとんどのチャット製品がこの形です）。利点は手間がかからず能力が高いことで、その代わりにネット接続が必要で、データが手元の機械の外に出ます。  
*よくある誤解：*「クラウド＝危険、ローカル＝万全」と思われがちですが、実際には両者にそれぞれ一長一短があり、あなたのデータがどれだけ機密か、ニーズが何かが肝心です。  

**コンテキストエンジニアリング**（上下文工程 ・ Context Engineering）  
「どの情報を、どんな順序で文脈に入れるか」を入念に組み立てる工夫です。多く与えるより的確に与えるほうが重要で、RAGや長いタスク、プロジェクトの文脈を使いこなすうえでの核心となる考え方です。  

**埋め込みモデル**（嵌入模型 ・ Embedding Model）  
文字をベクトル（エンベディング）に変換することに特化したモデルで、対話を生成する大規模言語モデルとはたいてい別のモデルです。RAGはこれを使って資料に「座標」を付けます。  

**プロンプトキャッシュ**（提示词缓存 ・ Prompt Caching）  
繰り返し使う同じ文脈を「キャッシュ」しておき、次に使い回すときに速く、お金も節約できるようにすることです。同じ長い前置きを何度も持ち回る場面でとても役立ちます。  

**アプリ・ラッパー層**（应用/封装层 ・ Application / Wrapper Layer）  
モデルやAPIの上に作られた具体的なアプリ（ウェブページ、プラグイン、業界向けツール）です。同じ土台のモデルの上に無数のアプリが育ち、この層があなたの実際の使い方を決めます。  
*よくある誤解：*「ガワをかぶせただけのアプリ」を一律に見下されがちですが、実際には良いラッパーはプロンプト・流れ・データに多くの工夫を凝らしており、体験の差は非常に大きくなり得ます。  

## AIプログラミングとエージェント

**コーディングエージェント**（编程智能体 ・ Coding Agent）  
コードを書き、ファイルを直し、コマンドを実行することに特化したエージェントです（Claude Code、Codexなど）。「とても有能だが入社したばかりの同僚」として扱い、最終的な責任はやはりあなたにあります。  
*よくある誤解：*全自動で納品でき、放っておいてよいと思われがちですが、実際には間違えたり理解がずれたりするので、人がdiffを見て、テストを走らせ、受け入れ確認をしなければなりません。  

**ターミナル（コマンドライン）**（终端/命令行 ・ Terminal / Command Line (CLI)）  
「文字でコマンドを打って」PCを操作する、黒い背景の窓です。多くのプログラミングエージェントはここで動きます。いかにも硬派に見えますが、手順どおりにやれば大丈夫です。  
*よくある誤解：*コマンドラインは「ハッカー専用で、PCを壊しやすい」と思われがちですが、実際には単なる別の操作方法であり、案内に従って一歩ずつ進めれば怖くありません。  

**差分（diff）**（diff(差异) ・ Diff）  
「変更前」と「変更後」を並べて示し、AIがどの行を動かしたのかを一目で分かるようにしたものです。変更を受け入れ確認するうえでの基本です。  
*よくある誤解：*diffを見るのはプログラマーだけに必要だと思われがちですが、実際には「何を変えたか」の対照表であり、コードが分からなくても、まずdiffを見てから許可する習慣をつけるべきです。  

**バージョン管理（Git）**（版本控制/Git ・ Version Control / Git）  
ファイルの変更履歴を毎回記録するシステムです。これがあれば、AIが壊してしまっても、ワンクリックで前のバージョンに戻せます。「AIに手を動かさせる勇気」を支える安全網です。  
*よくある誤解：*Gitはプログラマーのツールにすぎないと思われがちですが、実際には「いつでもやり直せる」力を与えてくれるもので、AIに思い切りやらせつつ暴走を防ぐための重要な保証です。  

**コミット**（提交/commit ・ Commit）  
今回のひとまとまりの変更を「一つ保存」して、Gitの履歴に記録することです。コミットのたびに、戻れるセーブポイントができ、後で調べたり戻したりするのに便利です。  

**ブランチ**（分支 ・ Branch）  
本線から「平行な作業線を一本分岐させる」ことで、その上では自由に変更でき、壊しても本線に影響しません。AIに大胆に手を入れさせるときは、先にブランチを切るほうが安全です。  

**巻き戻し（ロールバック）**（回滚 ・ Rollback / Revert）  
ファイルを以前のあるセーブポイントの状態に戻すことで、いわば「ワンクリックでやり直し」です。AIが直して問題が出たとき、ロールバックはもっとも速い応急手当の一つです。  

**テスト**（测试 ・ Test）  
「プログラムが期待どおりに動いているか」を自動でチェックするためのコードです。変更後に一度テストを走らせると、AIが他の場所を壊していないかを素早く見つけられます。  

**コードレビュー**（代码评审 ・ Code Review）  
変更を取り込む前に、人（または別のAI）がコードを一か所ずつ、正しく直っているか・良くなっているかをチェックすることです。AIがどれだけ速く書いても、レビューと許可は人が行わなければなりません。  

**デグレード（リグレッション）**（退化 ・ Regression）  
新しい機能を変更したことで、もともと正常だった古い機能を壊してしまうことです。既存のテストを一通り走らせることで、いち早く見つけます。  

**CLAUDE.md（プロジェクト文脈）**（CLAUDE.md/项目上下文 ・ CLAUDE.md / Project Context）  
プロジェクトに置いておき、AIが作業を始めるたびに最初に読む「プロジェクトの説明書」です。技術スタック、ルール、禁止事項などを書いておき、AIに常駐する記憶のような役割を果たします。  
*よくある誤解：*多く書くほど良いと思われがちですが、実際には書きすぎるとかえって文脈を占有して重点をぼかすので、「本当に作業に影響するルール」だけを書くべきです。  

**許可（パーミッション）の仕組み**（许可机制 ・ Permission System）  
プログラミングエージェントが「ファイルを直す、コマンドを実行する」前に、まずあなたに確認を取り、あなたがうなずいて初めて実行する仕組みです。危険な操作を人の確認に委ねることが、「手を動かしても暴走しない」ための要です。  

**プランモード**（计划模式 ・ Plan Mode）  
AIにまず「どう直すつもりか」の計画を挙げさせ、あなたが承認してから手を動かさせるモードです。大きな変更ほど、先に計画を見ることで、やり直しをもっとも多く減らせます。  

**サブエージェント**（子代理 ・ Subagent）  
メインのAIが、専門の作業のために一時的に送り出す「別のAI」です。独自の文脈を持ち、作業が終わると結果の要約だけを持ち帰り、本線の文章量を占有しません。  

**タスク分割**（任务拆分 ・ Task Decomposition）  
一つの大きなタスクを「調査 → 計画 → 実行 → 確認」などの小さなステップに切り分け、一歩ずつAIに渡すことです。こうするとAIが脱線しにくくなり、あなたも管理しやすくなります。  

**受け入れ確認**（验收 ・ Acceptance / Sign-off）  
最後にあなたが「今回の変更は確かに要件を満たし、他を壊していない」と確認してから取り込むことです。分からない、自信がないなら、マージしないこと——責任は人にあります。  
*よくある誤解：*テストが通れば万事OKと思われがちですが、実際にはテストはあくまで最低ラインであり、本当に要件を満たしているか、リリースできるかは、なお人が判断しなければなりません。  

**ワークフロー**（工作流 ・ Workflow）  
「一問一答」を、決まった一連のステップ（調査 → 計画 → 実行 → 受け入れ確認など）へと格上げし、AIに流れに沿ってタスクを進め続けさせつつ、暴走しないようにすることです。  

**統合開発環境（IDE）**（集成开发环境 ・ Integrated Development Environment (IDE)）  
プログラマーがコードを書くときの主力ソフトです（VS Codeなど）。多くのAIプログラミングアシスタントは、この中に組み込んで使えます。  

**リポジトリ**（仓库 ・ Repository (repo)）  
一つのプロジェクトのすべてのファイルと、その変更履歴をまとめた「総倉庫」で、たいていGitで管理されます。AIの作業・コミット・ロールバックは、これを中心に行われます。  

**プルリクエスト**（拉取请求 ・ Pull Request (PR)）  
協働プラットフォーム上で「私のこのひとまとまりの変更を本線に取り込むよう申請する」流れで、他の人がまずレビューしてからマージできるようにします。AIに説明文を用意させ、PRを起こさせることもできます。  

**リファクタリング**（重构 ・ Refactoring）  
プログラムの「外から見た振る舞い」を変えないまま、内部の構造を整理して、コードをより分かりやすく保守しやすくすることです。「既存のテストが全部通るか」で壊していないことを確認するのがよく使われます。  

**デバッグ**（调试 ・ Debugging）  
プログラムの中の誤り（バグ）を見つけて直すことです。プログラミングエージェントはエラーを読み、原因を調べ、試しに直して再度走らせる、という「試行錯誤のループ」を自分で回せます。  

**サンドボックス（隔離環境）**（沙箱/隔离环境 ・ Sandbox / Isolated Environment）  
「何か起きても本物のシステムには波及しない」隔離された実行環境です。この中でAIに思い切り試させるほうが、本番環境でやるよりずっと安全です。  

**トラブル時の応急処置**（急救/卡住怎么办 ・ Recovery / Troubleshooting）  
AIが行き詰まったり、直すほど混乱したりしたときの定番の動きです。今のものを中断し、新しい会話を始め、変更をロールバックし、問題を説明し直します。まず損失を止めてから、原因を調べます。  

## リスクと安全

**プライバシー**（隐私 ・ Privacy）  
あなたがAIに入力した内容がどう扱われるか、保存されたり学習に使われたりしないか、という問題です。個人情報や機密情報が関わるときは、特に注意が必要です。  
*よくある誤解：*入力したものは「使い終わったら消える」と思われがちですが、実際には保存するか、学習に使うかは製品の設定や条項次第で、しかも変わり得るので、まず確認すべきです。  

**データ分類（機密度レベル）**（数据分级 ・ Data Classification / Tiering）  
情報を機密度に応じていくつかの段階（公開、社内、機密、極秘など）に分け、それに基づいてどれをAIに与えてよいか、どれは絶対にだめかを決めることです。シンプルで実用的な、自衛のための枠組みです。  

**プロンプトインジェクション**（提示注入 ・ Prompt Injection）  
攻撃者が、AIが読むことになる外部のコンテンツ（ウェブページ、メール、文書）の中に「偽の指示」を仕込み、あなたがやらせるつもりのなかったことをAIにさせるようだます手口です。  
*よくある誤解：*リスクは自分が打った文字からしか来ないと思われがちですが、実際にはAIが読むどんな外部コンテンツの中にも地雷が埋まっている可能性があり、外部データをつなぐときは特に警戒が必要です。  

**ジェイルブレイク**（越狱 ・ Jailbreak）  
特殊な話術でAIを誘導し、安全上の制限を回避させて、本来言うべきでない内容を言わせることです。悪用行為であり、メーカーが継続的に防いでいる攻撃でもあります。  

**機密情報・シークレット**（敏感信息/密钥 ・ Sensitive Information / Secrets）  
パスワード、APIキー、データベース接続文字列、秘密鍵など、決して漏らしてはならないものです。チャット欄やコードには絶対に貼らないこと。万一貼ってしまったら、すぐに無効化してリセットします。  
*よくある誤解：*AIのチャット欄に貼るのは「内輪のこと」だと思われがちですが、実際には保存されたり漏れたりしかねず、家の鍵をグループチャットに送るようなものです。  

**個人情報（PII）**（个人信息(PII) ・ Personally Identifiable Information (PII)）  
氏名、住所、電話番号、本人確認番号、メールアドレスなど、特定の個人を識別できる情報です。AIに与える前に、混入していないか確認し、マスキングしておく必要があります。  
*よくある誤解：*本人確認番号だけがPIIだと思われがちですが、実際には氏名と所属、メールアドレス、位置情報などの組み合わせでも個人を特定できるので、どれも慎重に扱う必要があります。  

**コンテンツモデレーション**（内容审核 ・ Content Moderation）  
AIの入力と出力に安全フィルタをかけ、違法・有害・不適切な内容をはじくことです。メーカーが内蔵するガードレールも、あなた自身が生成内容を点検することも、どちらも指します。  

**著作権**（版权 ・ Copyright）  
AIが生成した内容を「そのまま使ってよいか、誰のものになるか」という著作権の問題で、ルールはまだ発展途上です。商用利用の前は特に慎重に確認する必要があります。  
*よくある誤解：*AIが生成したものは必ず自由に商用利用できると思われがちですが、実際には権利の帰属や利用可否は具体的な状況や現地のルール次第なので、決めつけてはいけません。  

**AIの利用開示（出所表示）**（署名/AI披露 ・ AI Disclosure / Attribution）  
ふさわしい場面で「この部分はAIが生成または補助した」と明記し、誠実さを示し、他の人が判断しやすくすることです。明記が必須かどうかは、場面や規定によります。  

**OSSライセンス**（开源许可证 ・ Open-source License）  
オープンソースのコードの利用条項です。AIがあなたに導入する第三者ライブラリについて、そのライセンスがその使い方を許しているか確認する必要があり、そうしないと違反になりかねません。  
*よくある誤解：*オープンソースは「自由に使える」と思われがちですが、実際にはライセンスごとに制限の差が大きく、中には自分のコードの公開や、表示の保持を求めるものもあります。  

**コンテキストコスト**（上下文成本 ・ Context Cost）  
文脈が長いほど、回答のたびに遅く、高くなり、脱線もしやすくなります。ですから「文脈を節約する」ことは、たいてい同時にお金を節約し、速度を上げ、品質を上げる、一石三鳥になります。  
*よくある誤解：*資料を全部まとめて詰め込むのがもっとも確実だと思われがちですが、実際には詰め込みすぎるとお金がかさみ動きも遅くなり、さらに重点をぼかして、かえって間違えやすくなります。  

**利用上限・レート制限**（额度/限流 ・ Quota / Rate Limit）  
メーカーが、あなたが単位時間あたりにどれだけ使えるかに設けた制限で、超えると待つか、より上の段階が必要になります。利用量が多いときに実際に直面します。  

**ゼロデータ保持**（零数据保留 ・ Zero Data Retention (ZDR)）  
サービス提供側があなたの入力と出力を保存しない（使い終わったら捨てる）という取り決めで、プライバシー要件の高い場面でよく使われます。利用できるか、どう有効化するかは製品によります。  

**学習への利用**（数据用于训练 ・ Use of Data for Training）  
あなたの入力が、サービス提供側に後続のモデルの学習へ使われるかどうかを指します。これは機密が「流出」しないかに関わるので、使う前に確認し、必要に応じてオフにすべきです。  
*よくある誤解：*すべての製品が初期設定であなたのデータを学習に使う（またはどれも使わない）と思われがちですが、実際には各社で方針が異なり、設定でき、しかも変わるので、一つずつ確認しなければなりません。  

**バイアス**（偏见 ・ Bias）  
モデルが学習データから学んでしまう、不公平な傾向（特定の集団への固定観念など）です。無意識のうちに回答に持ち込まれるので、警戒と再確認が必要です。  
*よくある誤解：*機械は「客観中立」だと思われがちですが、実際にはデータの中の偏りをそのまま学び取るので、出力が公平とは限らず、重要な場面では人が点検すべきです。  

**過度の依存**（过度依赖 ・ Over-reliance / Automation Bias）  
何も考えずにAIの答えを信じ、それを権威として扱うことです。AIは自信たっぷりに間違えますし、楽をしようとするときほど落とし穴にはまりやすく、肝心な判断はやはり人が下す必要があります。  
*よくある誤解：*「AIが言っているのだから間違いないだろう」と思われがちですが、実際にはAIに正誤を判断する能力はなく、それらしく言っているだけなので、重要な決定をそのまま丸投げしてはいけません。  

**マスキング（匿名化）**（数据脱敏 ・ Data Masking / Anonymization）  
資料をAIに与える前に、氏名・電話番号・キーなどの機密部分を消すか、ダミーの値に置き換えておくことです。「AIを使いたいがプライバシーも守りたい」ときによく使われる折衷策です。  

**チーム利用ガイドライン**（团队使用规范 ・ Team Usage Policy）  
チームであらかじめ「何を与えてよいか、何はだめか、誰が審査するか、キーをどう管理するか」を取り決めたルールです。個人の習慣を共通認識へと格上げし、誰かが無意識にやらかすのを防ぎます。  

**架空の出典**（幻觉引用 ・ Fabricated Citation）  
ハルシネーションの典型的な一形態で、AIが見た目は正規なのに実際には存在しない文献・リンク・条文を作り出すことです。引用は必ず一つずつ真偽を確かめてください。  
*よくある誤解：*「出典があるのだから信頼できる」と思われがちですが、実際には出典そのものが作り話の可能性があり、リンクや文献名はどれも開いて確かめる必要があります。  

**ガードレール**（安全护栏 ・ Guardrails）  
AIシステムに加える、さまざまな「越えてはならない」制限（違法な要求を拒む、危険な出力をフィルタするなど）です。メーカーの内蔵によるものも、利用側が自分で加えることもできます。  

## 略語

**AI（略語）**（AI（缩写） ・ AI = Artificial Intelligence）  
人工知能の英語の略語です。機械に知能の必要な作業をこなさせる、さまざまな技術を広く指します。  

**ML（略語）**（ML（缩写） ・ ML = Machine Learning）  
機械学習の英語の略語です。プログラムにデータから自分で規則性を学ばせる、一群の手法です。  

**DL（略語）**（DL（缩写） ・ DL = Deep Learning）  
深層学習の英語の略語です。多層のニューラルネットワークを使って学習する機械学習の手法です。  

**LLM（略語）**（LLM（缩写） ・ LLM = Large Language Model）  
大規模言語モデルの英語の略語で、ClaudeやChatGPTなどの製品の中核です。  

**NLP（略語）**（NLP（缩写） ・ NLP = Natural Language Processing）  
自然言語処理の英語の略語で、コンピュータに人間の言語を扱わせる技術分野を指します。  

**API（略語）**（API（缩写） ・ API = Application Programming Interface）  
アプリケーションプログラミングインターフェースの英語の略語です。あなたのプログラムからモデルを直接呼び出す入口です。  

**RAG（略語）**（RAG（缩写） ・ RAG = Retrieval-Augmented Generation）  
検索拡張生成の英語の略語です。まず資料を調べてから、モデルに資料に基づいて答えさせる手法です。  

**MCP（略語）**（MCP（缩写） ・ MCP = Model Context Protocol）  
モデルコンテキストプロトコルの英語の略語です。AIが外部のツールやデータに統一的につなぐための開放標準です。  

**SDK（略語）**（SDK（缩写） ・ SDK = Software Development Kit）  
ソフトウェア開発キットの英語の略語です。開発者がより手軽にAPIを呼び出せる、出来合いのコードライブラリです。  

**PII（略語）**（PII（缩写） ・ PII = Personally Identifiable Information）  
個人を識別できる情報の英語の略語で、氏名、住所、電話番号、メールアドレスなどを指します。  

**RLHF（略語）**（RLHF（缩写） ・ RLHF = Reinforcement Learning from Human Feedback）  
人間のフィードバックによる強化学習の英語の略語で、モデルのアライメントの主流の手法の一つです。  

**GPU（略語）**（GPU（缩写） ・ GPU = Graphics Processing Unit）  
画像処理装置の英語の略語です。もともとは描画用でしたが、大規模な並列計算が得意なことから、大規模言語モデルの学習と実行の主力ハードウェアになりました。  
*よくある誤解：*「ゲーム用のグラフィックボード」にすぎないと思われがちですが、実際には大規模言語モデルを動かす肝心の計算力であり、ローカルでモデルが速く動くかどうかは、これに大きく左右されます。  

**CLI（略語）**（CLI（缩写） ・ CLI = Command Line Interface）  
コマンドラインインターフェースの英語の略語です。文字でコマンドを打ってソフトやPCを操作する方式で、多くのプログラミングエージェントはここで動きます。  

**GPT（略語）**（GPT（缩写） ・ GPT = Generative Pre-trained Transformer）  
ある一群の大規模言語モデルの名称の略語（生成事前学習Transformer）で、ChatGPTの中核となるモデルシリーズです。  

**JSON（略語）**（JSON（缩写） ・ JSON = JavaScript Object Notation）  
よく使われる「構造化データ」のテキスト形式で、キーと値が対になっていて、機械が解析しやすいものです。AIにJSONで出力させると、プログラムが続けて処理しやすくなります。  

**ZDR（略語）**（ZDR（缩写） ・ ZDR = Zero Data Retention）  
ゼロデータ保持の英語の略語です。サービス提供側があなたの入力と出力を、使い終わったら保存しないという取り決めです。
