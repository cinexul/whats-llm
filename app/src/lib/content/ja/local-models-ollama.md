普段あなたが使う AI は、ほとんどが「よその家」に住んでいます――打ち込んだ文字はネットワークを飛んで、どこかの会社のサーバーで計算され、答えが戻ってきます。この仕組みはとても手軽ですが、一部の人にとっては気になる点も三つあります。内容は処理のために **自分のパソコンを離れ** なければならないこと、**ネットが切れると使えない** こと、そしてよく使うと **使った分だけお金がかかる** ことです。

そこで、別の道を考え始める人が出てきました。モデルを **直接自分のパソコンに入れて**、ローカルで動かし、ネットにつながず、外に出さず、課金されないようにできないか、と。答えはできます――これが本章で扱う **ローカルモデル（local model）**、そして一般の人がモデルを動かすのを助ける二種類のツール、Ollama と LM Studio です。先に一つはっきりさせておきます。ローカルモデルは「より高級な版」ではなく、「動かす場所を変えた版」です。そこには恩恵もあれば、れっきとした代償もあります。

## 一、なぜ自分のパソコンでモデルを動かしたい人がいるのか

モデルを自宅に持ち帰るのは、三つのものを狙ってのことです。まず、それぞれがいったい何を解決するのかをはっきりさせましょう。

| 欲しいもの | 何を解決してくれるか | 誰が特に気にするか |
| --- | --- | --- |
| **秘匿性** | 内容が終始ローカルを出ず、どの会社のサーバーにもアップロードされない | 機微な資料を扱う、データを「外に出す」のが嫌な人 |
| **オフライン** | ネットがなくても使える――飛行機の中、社内ネット、ネットが切れても動く | ネットが制限される、または隔離環境で作業する必要がある人 |
| **回数で課金されない** | 入れた後は使い放題で、一回ごとに課金されない | 使用量が多い、または課金の心理的負担に引きずられたくない人 |

この三つに共通するのは、どれも同じ一つのことから来ているという点です――**モデルが自分自身のマシン上で動き、入力も出力もローカルを離れない**。クラウドの仕組みが手軽なのは、まさに重い仕事を「よその家」でやるからです。そしてローカルの仕組みが秘匿・オフライン・無課金なのも、まさに重い仕事を「自分の家」に持ち帰ったからです。**利点と代償は、同じ一つの原因の裏表** なのです。

> **ポイント:** 「ローカル」を一種の技自慢だと思わないでください。これは本質的に **トレードオフの問題** です。「自分のマシンの計算力と、少しの設定の手間」と引き換えに、「秘匿・オフライン・無課金」を手に入れる。自分がどちらを最も気にするかをよく考えてから、この道を行くか決めましょう――「自分のパソコンに入れられる」というだけで人より上だと思うのではなく。

## 二、ローカル vs クラウド：一枚の表でトレードオフを整理する

これは本章で最も覚えるべき一枚の表です。ローカルとクラウドは「どちらが優れているか」ではなく、「異なる人、異なる場面のために用意されたもの」です。

| | クラウド(普段使っているもの) | ローカル(自分のパソコンに入れる) |
| --- | --- | --- |
| データはどこへ | 会社のサーバーに送って計算 | **ローカルを出ない** |
| ネット接続が要るか | 要る | **不要** |
| お金の払い方 | 従量課金 / プランの利用枠 | 入れた後は **回数で課金されない**(ただし相応のパソコンが要る) |
| 能力の強弱 | 通常 **より強い**(背後は巨大なマシン) | **自分のマシンに制限され**、同条件ではクラウドの最上位に **及ばないことが多い** |
| 始めやすさ | 開けば使えて手軽 | ダウンロード・設定が要り、少し敷居がある |
| 保守 | 会社が代わりに背負う | 自分で管理する |

一言で両者の性格をまとめると：

- **クラウド ＝ 強力・手軽だが、データが外に出る・従量でお金がかかる。**
- **ローカル ＝ 秘匿・オフライン・無課金だが、マシンに制限される・設定の敷居がある・通常クラウドほど強くない。**

最も見落とされやすい一行――**能力の強弱** に注意してください。クラウドで動くのはたいてい非常に大きなモデルで、背後には一般の人には買えないほどの計算力があります。自分のパソコンで動かせるのは、たいていその「スリム版」です。ですから「ローカルモデルを入れれば、効果が最強のクラウドと同じ」と期待すると、たいてい失望します。これは Ollama や LM Studio がダメなのではなく、手元のパソコンの「体格」が、どれだけ大きなモデルを動かせるかを決めているのです。

> **ポイント:** ローカルかクラウドかを選ぶには、まず自分にこう問いましょう。**「この作業でいちばん大事なのは、データを外に出さないことか、それとも効果が最高であることか?」** データを外に出したくない・オフラインで使いたい・課金されたくない――ローカルには代えがたい価値があります。最上位の効果がほしい・手軽がいい――素直にクラウドを使いましょう。これは第17章の「モデル/製品/API/エージェント」の地図と同じ道理です。まず需要がどの層にあるかを見極めてから、何を使うかを論じるのです。

## 三、ダウンロード前にひと目：手元のパソコンは耐えられるか

これはローカルモデルで最も実際的で、しかも初心者が最も見落としやすい一つの関門です。**すべてのモデルを手元のパソコンが動かせるわけではありません。**

たとえるなら、モデルは「荷物」、パソコンは「車」のようなものです。荷物には大小があり、車のトランクにも大小があります――**箱に収まらなければ、車は走り出せません**。モデルというこの「荷物」を収められるかどうかは、主に次の二つで決まります。

- **メモリ(RAM)**：パソコンが一時的に作業する空間。モデルは動くときに丸ごとメモリに「広げ」られ、大きいモデルほど使うメモリも多くなります。
- **VRAM(ビデオメモリ)**：グラフィックボード(GPU)上の専用メモリ。多くのモデルはグラボで動かすとずっと速くなりますが、グラボの「専用空間」は通常メモリよりさらに逼迫しています。

これが実際に何を意味するか。ダウンロードできる各ローカルモデルには、配布元がたいてい **おおよそどれだけのメモリ/ビデオメモリが必要か** を記しています。ダウンロード前に、その要件を自分のパソコンの構成と照らし合わせます――**収まってこそ、滑らかに動きます。構成を超える大きなモデルを無理に押し込めば、軽い場合でもひどく遅くなり(スライドショーのようにカクつく)、ひどい場合はそもそも動き出しません。**

```text
ダウンロード前の「ひと目合わせ」の動作：

  モデルが記す要件：      手元のパソコン：     結論
  ─────────────         ─────────           ──────
  16 GB メモリが必要  vs  手元は 8 GB     →  動かない / 極めて遅い
  8 GB メモリが必要   vs  手元は 16 GB    →  問題なし
  （具体的な数値はモデル配布元/公式ドキュメントに従う、本書は書き切らない）
```

あるモデルがどれだけの容量を要するか、自分のパソコンにどれだけ余裕があるかは、どう知ればよいでしょうか。**具体的な数値と確認方法は各社で異なり、しかもよく更新されるので、公式ドキュメントを確認してください**。まずはこの意識を持つだけでかまいません。**ローカルモデルには「ハードウェアの敷居」があり、ダウンロード前にまず構成をひと目合わせる** こと。ダウンロードし終えてから動かないと気づいて、長い時間を無駄にすることのないように。

> **ポイント:** 「ローカルモデルを動かせるか」は、動かしたいかどうかではなく、**マシンが耐えられるか** で決まります。こんな習慣を身につけましょう。ローカルモデルを見たら、まずそこに記されたメモリ/ビデオメモリの要件を探し、それから自分のパソコンの構成を見る――この手順は一分しかかからず、「数 GB をダウンロードしたのに、動かすとフリーズする」という骨折りを省けます。

## 四、量子化：大きなモデルを少し「やせさせて」載せやすくする

こう聞きたくなるかもしれません。では大きなモデルは普通のパソコンとは完全に縁がないのか、と。そうとも限りません。ここでよく耳にする言葉――**量子化（quantization）** があり、ひと言で理解しておく価値があります。

先の「荷物」のたとえを続けます。量子化は、荷物の中身を **真空圧縮袋で押し縮める** ようなもので、体積が小さくなり、小さなトランクに収めやすくなります。代償は――押しすぎると、服が少ししわになるかもしれない、ということです。

これが実際に何かに立ち返ると、量子化とは **モデルをより小さく、よりメモリを節約する形に「圧縮」する** 技術で、本来動かせなかったモデルを普通のパソコンでも動かせる可能性を与えてくれます。代償は **強く圧縮するほど、効果がたいてい少し落ちる**(回答の質がやや下がるかもしれない)ことです。ですから同じモデルにいくつもの「圧縮の段階」が用意されているのをよく目にします――軽く圧縮すれば効果はよいが構成をより多く使い、強く圧縮すれば資源を節約できるが質が少し割り引かれます。

> **ポイント:** 量子化はローカルモデルの世界でよくあるトレードオフです。**少しの「質」と引き換えに「自分のパソコンで動かせること」を手に入れる** のです。仕組みまで理解する必要はなく、ただこう知っていればかまいません――同じモデルにいくつもの版が用意されているとき、それはたいてい異なる量子化の段階で、**「動かせる」と「うまく答える」の間でバランス点を一つ選ぶ** ものなのです。具体的にどう選ぶか、各段階でどれだけ差があるかは、公式ドキュメントを参照してください。

## 五、Ollama と LM Studio：一般の人がモデルを動かすのを助けるツール

ここまで「ローカルモデル」を語ってきましたが、モデル自体はただのコアにすぎません(第17章を覚えていますか――モデルは「思考する頭」で、何かで包んで初めて使いやすくなります)。**裸のモデルと直接付き合うのはとても骨が折れます**。ダウンロードし、設定し、動かし、しかも対話する場所まで用意しなければなりません。

**Ollama と LM Studio は、こうした面倒事をまとめて肩代わりしてくれるツール** です――一般の人でも比較的手軽に、ローカルでモデルを **ダウンロード** し、**実行** し、**対話** できるようにしてくれます。

ここでついでに、よく出てくる言葉を一つ補っておきます。こうして自由にダウンロードし、自分のパソコンに入れて動かせるモデルは、たいてい **オープンウェイトモデル（open-weight model）** に属します――簡単に言えば、配布元がモデルを「公開」し、誰もがローカルにダウンロードして使うことを許したものです。こうしたダウンロードできるモデルがあるからこそ、ローカルというこの道が成り立ちます。一方、最も最先端のクラウドモデルの多くはダウンロードが公開されておらず、その製品や API を通じてしか使えません(第17章)。どのモデルがオープンウェイトか、商用できるかは、各社の条項が異なるので、**公式ドキュメントを確認してください**。

- おおよその印象としては(**具体的な機能・インターフェース・対応モデルは公式ドキュメントを参照**)、利用可能なローカルモデルの閲覧とダウンロードを助け、ダウンロードした複数のモデルを管理し、直接チャットできる入口を提供してくれます。
- 両者は立ち位置が少し異なり、それぞれ重点がありますが、一般の人にとって目的は同じです。**「ローカルでモデルを動かす」という本来かなり技術的なことを、数回クリックすれば始められるようにする** ことです。

具体的にどう入れ、どうモデルを落とし、コマンドやボタンがどんな見た目かは――こうした点は **バージョンによって頻繁に変わるので、本書では書き切らず、それぞれの公式ドキュメントを参照してください**。実際に手を動かす細かな操作は、付録 C で改めて詳しく扱います。

> **ポイント:** Ollama / LM Studio と「モデル」自体を混同しないでください。これらは **モデルではなく**、モデルを **ダウンロードし、動かし、使えるように** してくれる「執事」です――レストランとシェフの関係と同じです(第17章)。執事を変えても、動かすのは同じローカルモデルです。本当に効果を決めるのは、動かすのが **どのモデルか**、**パソコンがどれだけ強いか** であって、どの執事を使うかではありません。

## 六、よくある誤解、まとめて訂正

| 誤解 | 実際 |
| --- | --- |
| ローカルモデルは必ずクラウドより強い | むしろ逆――ローカルは自分のマシンの計算力に制限され、同条件ではクラウドの最上位に **及ばないことが多い**。強弱は **具体的な型番＋手元のハードウェア** で決まり、「ローカル」という言葉だけでは決まらない |
| ローカルモデル ＝ 絶対安全 | より **秘匿** である(データがローカルを出ない)のは確かだが、「秘匿」は「安全」とは違う――モデルは同じくハルシネーションし、間違える(第9章)。素性の知れないモデルをダウンロードするのもリスクがある |
| Ollama を入れれば「より良い AI」が手に入る | Ollama/LM Studio は **実行ツール** にすぎず、モデル本体ではない。本当に働くのは、ダウンロードしたそのモデルのほう |
| 適当にモデルを一つ落とせば動く | まずそのモデルの **メモリ/ビデオメモリ要件** が手元のパソコンと合うかを確認する。合わなければ極めて遅く、あるいは動き出さない |
| ローカルはギーク向けで、自分には関係ない | **データを外に出さないことを特に気にする、オフラインで使いたい、使用量が多い** なら、ローカルは現実的な価値のある一つの道――ただし少し設定の敷居がある |

## まとめ

- **ローカルモデル** を選ぶ人が狙うのは三つ：**秘匿**(データがローカルを出ない)、**オフライン**(ネットがなくても使える)、**回数で課金されない**――どれも「モデルが自分のマシンで動く」ことに由来する。
- ローカル vs クラウドは **トレードオフの問題**：クラウドは **強力・手軽だがデータが外に出る・従量課金**。ローカルは **秘匿・オフライン・無課金だが、マシンに制限され、敷居があり、通常クラウドほど強くない**。
- ダウンロード前に **まずハードウェア要件を見る**：モデルは **メモリ/ビデオメモリ** に「広がる」必要があり、収まらなければ動かない、あるいは動き出さない(具体的な数値は公式ドキュメントを参照)。
- **量子化** は、少しの質と引き換えに「普通のパソコンで動かせること」を手に入れる、よくあるトレードオフ。同じモデルの複数の版は、たいてい異なる量子化の段階。
- **Ollama / LM Studio** は、一般の人がローカルでダウンロード・実行・対話するのを助ける **ツール** であって、モデル本体ではない。具体的な使い方は公式ドキュメントを参照し、細かな操作は付録 C を参照。

次章では視点を変えます。モデルがいくら強くても、「言う」だけでは足りません――どうやってモデルに **外部のツールとデータ** をつなぎ、指定した資料を読ませ、決まった流れに沿って作業をさせるか? そのためには Skills と MCP の話をすることになります。

## 小テスト

> 全 6 問。概念・誤解・場面・操作の 4 種類をカバーしています。答えと解説は各問の下の引用ブロックにあります――まず自分で考えてから、照らし合わせてください。

1. **[基礎・概念問題]** 人々が自分のパソコンでローカルモデルを動かしたいと思う、主な狙いは次のどの三つか？
   - A. 秘匿、オフライン、回数で課金されない
   - B. 必ずより強い、必ずより速い、必ずパソコンが安く買える
   - C. 自動でネット接続、自動更新、自動バックアップ
   - D. 永遠に間違えない、永遠にハルシネーションしない
   > **答え A。** ローカルモデルの三つの恩恵――データがローカルを出ない(秘匿)、ネットがなくても使える(オフライン)、入れた後は課金されない――は、どれも「モデルが自分のマシンで動く」というこの一事から来ます。B はローカルを「より強い・より速い」と取り違え(実際はクラウドの最上位に及ばないことが多い)、D は「秘匿」を「間違えない」と誤解しています。モデルは同じくハルシネーションします(第9章)。

2. **[基礎・概念問題]** 「ローカル」と「クラウド」について、最も正確なのはどれ？
   - A. ローカルは永遠にクラウドより良い
   - B. クラウドは強力・手軽だがデータが外に出る・従量課金。ローカルは秘匿・オフライン・無課金だがマシンに制限され、通常クラウドほど強くない
   - C. クラウドはネット接続が不要で、ローカルはネット接続が必要
   - D. 両者は完全に同じで、名前が違うだけ
   > **答え B。** これはトレードオフの問題で、「どちらが永遠に良い」はありません。A は最もよくある誤解です。C は両者のネット接続の関係を **逆に言って** います(クラウドこそネット接続が要り、ローカルは不要)。D はデータの行き先、課金、能力といった一連の実際の違いを無視しています。

3. **[基礎・誤解問題]** 「ローカルモデルは必ずクラウドのものより強く、より賢い。」――どこが間違い？
   - A. 間違いではない、ローカルこそより強い
   - B. ローカルは自分のパソコンの計算力に制限され、同条件ではクラウドの最上位に及ばないことが多い。強弱は具体的な型番と手元のハードウェアで決まり、「ローカル」という言葉だけでは決まらない
   - C. ローカルは実はより弱く、永遠に使えないのが間違い
   - D. ローカルとクラウドは完全に異なるものを動かしているのが間違い
   > **答え B。** クラウドの背後には一般の人には買えないほど巨大な計算力があり、自分のパソコンで動かせるのはたいてい「スリム版」です。強弱を決めるのは **どのモデルか＋手元のハードウェア** であり、「ローカル」という言葉自体は強さを意味しません。この誤りを犯す人は、たいてい「自分のパソコンに入れる」を「より高級」と読み違えています。C はもう一方の極端に走っています(ローカルにも代えがたい価値があります)。

4. **[応用・誤解問題]** 「ローカルモデルはデータが外に出ないから、絶対に安全で、間違えない。」――どこが正しくない？
   - A. 完全に正しい
   - B. 「より秘匿」≠「絶対安全」：モデルは同じくハルシネーションし、間違える。素性の知れないモデルをダウンロードすること自体にもリスクがある
   - C. ローカルモデルは実はこっそりデータをアップロードする
   - D. ローカルモデルにはいかなる秘匿の利点もない
   > **答え B。** 「データがローカルを出ない」ことは、確かにモデルをより **秘匿** にしますが、「秘匿」と「安全/正確」は別のことです――回答が間違っている可能性はなお残りますし(第9章のハルシネーションは同じく存在します)、信頼できない出所からモデルを落とすこと自体にリスクがあります。「秘匿」を「安全」と同一視するのが、この問いが正したい混同です。D は実在する秘匿の利点を否定しており、こちらも正しくありません。

5. **[基礎・場面問題]** あるローカルモデルが気に入りました。ダウンロードする前に **まず最もすべき** 一つのことは？
   - A. そのままダウンロードを押し、動かなかったらそのとき考える
   - B. そのモデルに記されたメモリ/ビデオメモリの要件をひと目見て、自分のパソコンの構成で耐えられるか照らし合わせる
   - C. まずクラウドの会員を解約する
   - D. パソコンの他のソフトを全部アンインストールする
   > **答え B。** モデルはメモリ/ビデオメモリに丸ごと「広がる」必要があり、**収まらなければ極めて遅くなるか、あるいは動き出しません**。一分かけて構成をひと目合わせれば、「数 GB を落としたのにスライドショーになる」骨折りを省けます(具体的な数値は公式ドキュメントを参照)。A は最も踏みやすい落とし穴です。C・D はどちらも「ハードウェアが足りるか」という本当の関門に的を当てていません。

6. **[基礎・操作/観察問題]** まず自分のパソコンのメモリ(RAM)がおよそ何 GB かを調べます。それから任意のローカルモデルツール(Ollama / LM Studio など、**具体的には公式ドキュメントを参照**)を見て、同じモデルにしばしば大きさの異なる複数の版があることを確かめます。観察してみてください。小さい版ほど資源を節約できるが、効果は少し劣るのではないでしょうか。
   > **観察できるはず:** 同じモデルの複数の版は、たいてい異なる **量子化（quantization）** の段階です――強く圧縮するほど、ファイルが小さくメモリを節約できますが、回答の質はやや下がるかもしれません。逆に軽く圧縮するほど、構成は多く使いますが効果はよくなります。これこそローカルモデルの核心にあるトレードオフです。**「動かせる」と「うまく答える」の間でバランス点を選ぶ** ことです。一度自分のメモリとモデルの大きさを照らし合わせてみれば、「このパソコンはどれだけ大きなモデルを動かせるか」という直感が身につきます(各段階でどれだけ差があるかは公式ドキュメントを参照)。
