AI に触れ始めた人を最初に挫折させるのは、たいてい技術ではなく **名詞** です。Claude、ChatGPT、GPT、Gemini、Copilot、API、Agent、エージェント……これらはいったい誰が誰で、誰が誰を含んでいるのか。なぜ同じ会社にモデルも製品もあって、しかも名前がそっくりなのか。

本章は、あなたに一枚の **地図** を渡します。世の中のほぼすべての AI の名詞が、**四つの層** に収まることが分かるはずです。この四層の関係が見えれば、名詞の霧は大半が晴れます――この先の章で扱う製品比較（第18章）、Agent（第19章）、既存のプログラムに AI をつなぐ話（第35章）も、すべてこの地図の上で展開されます。

## 一、まず地図：四つの層

本章を通して使うたとえは、**とても腕のいい料理人** です。

| 層 | 何か | 料理人のたとえ | 例（**具体的な能力は公式ドキュメントを参照**） |
| --- | --- | --- | --- |
| ① モデル(model) | 本当に「考える」コア | 料理人本人の頭脳と腕前 | Claude、GPT、Gemini などの「モデルシリーズ」 |
| ② 製品(product) | モデルを、人がそのまま使えるものに包んだもの | 一軒のレストラン：メニュー、給仕、テーブルがある | Claude（アプリ/Web）、ChatGPT、Gemini（アプリ） |
| ③ API（インターフェース） | モデルの能力を **プログラム** から呼び出せるようにした窓口 | 料理人の「注文受付窓口」。よその店も注文できる | 各社の API / 開発者向けインターフェース |
| ④ Agent（エージェント） | モデルに「手」と一つの目標を持たせ、タスク全体を自分で走り切らせる | 自分で仕入れ・仕込み・配膳・片付けまでする料理人 | Claude Code、Codex などの「エージェント」 |

> **ポイント：** この四層は、互いに排他的な四つの別物ではなく、**内から外へ、層ごとに包み込んでいく** 関係です。同じ一つのモデルを、製品に詰めて人に使わせることも、API を通じてプログラムに使わせることも、自分で動く Agent に仕立てることもできます。まずこの一文を覚えてから、以下で一層ずつ解きほぐしていきましょう。

## 二、第①層 モデル：考える「コア」

**モデル** とは、第1部で説明したものです――膨大なテキストから「どう続きを返すか」を学んだ大規模言語モデル（第1章・第2章）です。本当に「計算する」コアですが、それ **自体は、あなたが直接開いて使えるソフトウェアではありません**。

- 「GPT を一つダウンロードしてすぐ話す」ことはできません。これは、料理人の「腕前」だけを単体で手に取れないのと同じです――腕前は人に宿り、厨房でこそ料理になります。
- 各社はたいてい **一つのモデルシリーズ** を持ち、その中で大小の段階（より強いが遅く高い / より速く安い）に分かれています。名前にはよくバージョン番号が付きます。**具体的な型番や能力は頻繁に更新されるので、本書では書き切らず、公式ドキュメントを参照してください。**

> **誤解 vs 実際：** 「Claude と ChatGPT は同類のもので、二択でいい」――厳密に言うと、**Claude はモデルシリーズでもあり、その製品を指すのにもよく使われます。一方 ChatGPT は製品で、そのコアのモデルは GPT と呼ばれます**。つまり「Claude vs ChatGPT」は、実は「コア+外殻」を「外殻」と比べていて、少しずれています。日常的にはこう言っても大きな問題はありませんが、その関係は頭の片隅に入れておきましょう。

## 三、第②層 製品：「人」が使うために包んだ層

料理人（モデル）がいるだけでは、まだ食事にありつけません。一軒の **レストラン** が要ります――メニュー、給仕、テーブル、レジ。**製品** とは、この包装の層です。モデルにチャット画面・ログイン・履歴・ファイルアップロードなどをかぶせ、一般の人が **開けばすぐ使える** ようにします。

- ChatGPT、Claude のアプリと Web、Gemini のアプリは、すべて製品です。その裏側には、それぞれのモデルがあります。
- 同じコアでも、包装はまったく違ってきます。Web 版、スマホアプリ、ブラウザ拡張、何かのオフィスソフトに組み込まれたもの……**人にやさしくすること** が、製品という層の目的のすべてです。

> **なぜこの層を区別するのか：** あなたが日常的に使い、直接タップして話せるものは、**ほとんどが「製品」** だからです。誰かが「どの AI が使いやすい？」と聞くとき、本当に比べているのは製品の体験（選び方は第18章で扱います）であって、裸のモデルではありません。「製品」と「モデル」を分けて考えれば、名前に振り回されずに済みます。

## 四、第③層 API：「プログラム」が使うための窓口

レストランは、入ってきた客のためのものです。でも、もし **別のタピオカ店** が自分のメニューに「看板カレー」を一品加えたいと思ったら、料理人を新たに雇い直す必要はありません――料理人の **注文受付窓口** から注文できます。注文を送れば料理人が作って渡し、タピオカ店はそれを自分で盛り付けて、自分の客に売るのです。

**API（アプリケーション・プログラミング・インターフェース）** とは、この「注文受付窓口」のことです。ただし注文するのは人ではなく、**プログラム** です。

- あなた自身の Web サイト・アプリ・スクリプトは、API を通じてモデルに質問を送り、答えを受け取って、**自分の製品に組み込む** ことができます。
- API を使うにはふつう、「注文しているのはあなた本人だ」と証明するための **鍵（API キー）** が必要で、それに基づいて **トークン単位で課金** されます（第3章・第38章）。
- この鍵はとても機密性が高いものです――**漏れれば、他人があなたのアカウントで注文し、あなたに請求が回ってくる** ことになります。だから「API キーを絶対にチャット欄に貼らない、公開リポジトリにコミットしない」は鉄則です（第37章）。

> **実際にどう出くわすか：** あなたや同僚が「自分たちのシステムに AI を組み込みたい」と言ったとき――たとえば会社の公式サイトのカスタマーサポートに自動で返答させたいとき――やるべきは **API を呼ぶこと** であって、ChatGPT の会員になることではありません。これがまさに、第35章「既存のプログラムに AI をつなぐ」という実習で手を動かす内容です。

## 五、第④層 Agent：「自分で動く」層

ここまでの三層では、料理人はまだ「あなたが注文したものを作る」だけでした。**Agent（エージェント）** はさらに一歩進みます。あなたは **目標** だけを伝えます――「今夜、一汁四菜のコースを用意して」――すると料理人は、自分で **メニューを立て、在庫を確認し、足りないものを仕入れ、一品ずつ作り、盛り付け、最後に厨房を片付ける** まで、すべてやります。途中で問題が起きれば（青菜が新鮮でない）、自分で何とかします（別の一品に替える）。

これを AI に当てはめると、Agent とはこういうものです。**モデルに「道具」（ファイルの読み書き、コマンドの実行、ネット接続、ほかのプログラムの呼び出しなど）を持たせ、一つの目標を中心に、自分で「調査→実行→確認→再調整」のループを回させる** ものです。

| | チャット製品（第②層） | Agent（第④層） |
| --- | --- | --- |
| できること | 文字で答える | 本当に **手を動かして操作する**（ファイル変更、コマンド実行など） |
| 誰が実行するか | 答えを見ながら自分でやる | Agent 自身が実行し、横で見張る |
| 進め方 | 一言聞けば、一言答える | 目標に向かって、自分で何歩も先まで進む |

本書の第4部で学ぶ **Claude Code、Codex は、まさに「プログラミング」という仕事をする Agent** です――「このページに検索機能を追加して」と言えば、自分でコードを読み、ファイルを変更し、テストを走らせます（第23章・第24章）。

> **誤解 vs 実際：** 「Agent って、ただの賢いチャットボットでしょ？」――肝心な違いは **「賢さ」ではなく「手を動かせること」** にあります。そして、本当にあなたのものを操作できるからこそ、リスクも一段上がります。Agent はあなたが予想していなかったことを勝手にやってしまうかもしれません。だから Agent は必ず「許可/確認」の仕組みと、「巻き戻せる」環境とセットで使います（第26章・第27章・第40章）。能力とリスクは、同じコインの裏表です。

## 六、四層はどう積み重なっているか

四層をつなげて、一つの全体像で見てみましょう。

```text
        ┌─────────────────────── Agent(自分で手順を走らせる)
        │   ┌─────────────────── 製品(人が使う画面)
        │   │   ┌─────────────── API(プログラムが使う窓口)
        │   │   │   ┌─────────── モデル(考えるコア)
  あなた → 目標   注文   呼び出し       次の語を予測
```

- 一番内側は、つねに **モデル** です。
- その外へ、**製品** に包んで人に使わせるか、**API** として開いてプログラムに使わせるかします。
- さらに外へ、道具と目標を与えれば **Agent** になります――そして Agent の内部では、しばしば API を通じてモデルを呼び出しています。

つまり「モデル / 製品 / API / Agent」は、あなたに四択を迫るものではなく、**今、自分が向き合っているのはどの層か** をはっきりさせるためのものなのです。

## 七、実用的な見極め：自分が向き合っているのはどの層か

今度 AI の名詞や要望に出くわしたら、自分に三つ問えば位置づけられます。

1. **人が画面で直接使っているのか、それともプログラムが裏で呼び出しているのか。** ―― 人が使う → たいてい「製品」。プログラムが呼ぶ → 「API」。
2. **「言う」だけなのか、それとも「手を動かす」のか。** ―― 答えるだけ → 製品/モデルの層。ファイルを変更し、コマンドを走らせ、ネットにつなぐ → 「Agent」の性質を帯びている。
3. **この名前は、会社のコア技術なのか、それとも開いて使えるアプリなのか。** ―― コア → 「モデル」。開けば使える → 「製品」。

> **ポイント：** 同僚が「うちも AI を導入しよう」と言ったとき、まず聞くべきは「どの会員を買う？」ではなく、**「人がチャット欄で使うのか、それともうちのシステムから自動で呼び出すのか、それとも自分で走る Agent を組むのか？」** です。この三つは、コストも・敷居も・リスクもまったく違います。まずどの層かをはっきりさせてから選定の話をする――これで、無駄な行き来を大きく減らせます。

## 八、よくある誤解、まとめて訂正

| 誤解 | 実際 |
| --- | --- |
| Claude と ChatGPT は同じ種類のものを比べている | Claude はモデル/製品を指すことが多く、ChatGPT は製品でコアは GPT。比べているのは同じ層ではない |
| モデルは「ダウンロードしてそのまま話せる」 | モデルはコアで、製品に詰めるか API を通さないと使えない（ローカルモデルは別のケース。第21章参照） |
| AI を導入する = 会員になる | **プログラム** に AI を使わせるなら API を通す。会員になるのは **人** が製品を使うため |
| Agent はチャットボットのアップグレード版 | Agent の本質は「手を動かす + 自分でループを回す」こと。リスクもそれに伴って上がる |
| AI を選ぶ = 最も強いモデルを選ぶ | 多くの人が選んでいるのは「製品の体験が手になじむか」であって、強弱はその一要素にすぎない（第18章） |

## まとめ

- AI の名詞は、たいてい **四つの層** に収まる：**モデル**（考えるコア）→ **製品**（人が使う画面）→ **API**（プログラムが使う窓口）→ **Agent**（自分で手順を走らせる）。
- これらは **層ごとに包み込む** 関係であって、四択ではない。同じモデルを、製品に包み、API として開き、Agent に仕立てられる。
- 日常的に開いて話すものは、ほとんどが **製品**。自分のシステムに AI をつなぐなら **API**。あなたの代わりに手を動かすのが **Agent**。
- 「どの層か」を見極める鍵となる問い：**人が使うのか、プログラムが使うのか。言うだけか、手を動かすか。**
- Agent の能力とリスクは一体――手を動かせるからこそ、許可と巻き戻しの仕組みが必須（第4部・第6部）。

次章では、この地図を使って、主要な AI 製品それぞれの位置づけと、「一般の人はどう選び、どう組み合わせて使うべきか」を見ていきます。

## 小テスト

> 全 6 問。概念・誤解・場面・操作の 4 種類をカバーしています。まず自分で考えてから、引用ブロックの答えと解説を見てください。

1. **[基礎・概念問題]** 「Claude はモデル、ChatGPT は製品」――この一文の意味に最も近いのは？
   - A. 両者に違いはない
   - B. モデルはコア（計算するもの）、製品はそのコアを包んで、あなたが使えるようにした画面の層
   - C. Claude のほうが高い
   - D. ChatGPT はモデルを使っていない
   > **答え B。** ChatGPT は OpenAI が GPT モデルの外側に包んだ製品。Claude はモデルシリーズの名前でもあり、その製品を指すのにもよく使われます。「コア vs 外殻」というこの関係を覚えれば、名詞は混乱しません。

2. **[基礎・概念問題]** API とは、おおまかに何か？
   - A. 一つの Web ページ
   - B. 自分のプログラムからモデルの能力を「呼び出す」ための窓口
   - C. 一種の有料会員
   - D. あるチャットボットの名前
   > **答え B。** Web/アプリは人が使うためのもの。API は **プログラム** が使うための、同じ能力の入口です。第35章「プログラムに AI をつなぐ」で使うのがこれです。

3. **[基礎・誤解問題]** 「Agent は、より賢いチャットボットだ」――どこが間違い？
   - A. 間違っていない
   - B. Agent の肝心な点は「より賢い」ことではなく、**本当に手を動かせる**（道具を呼び、ファイルを読み書きし、複数ステップを走る）こと。リスクもそこから来る
   - C. Agent はモデルを必要としない
   - D. Agent はチャットしかできない
   > **答え B。** チャットボットは「言う」だけ。Agent は「やる」。能力が上がると同時に、「予想していなかったことを勝手にやられる」リスクも付いてきます（第19章・第40章に呼応）。

4. **[応用・場面問題]** 同僚が「うちも AI を導入しよう」と言う。彼が本当にどの層を求めているか見極めるには、まず何を聞くべき？
   - A. 「どの会員を買う？」
   - B. 「人がチャット欄で使うのか、それともうちのシステムから自動で呼び出すのか？」
   - C. 「パソコンを買い替える必要はある？」
   - D. 「AI に自分たちの仕事が奪われる？」
   > **答え B。** 「人が製品を使う」「プログラムが API を呼ぶ」「自分で走る Agent を組む」は、それぞれ別物で、コストもリスクも違います。まずどの層かをはっきりさせてから選定の話をしましょう。

5. **[基礎・場面問題]** ときどき AI に文章を直してもらいたいだけ。次のうち、最も手軽でコスパがよいのは？
   - A. 既存のチャット製品をそのまま使う
   - B. 自分でプログラムを書いて API を呼ぶ
   - C. ローカルにモデルを立てる
   - D. エンジニアを雇う
   > **答え A。** 個人・低頻度・対話的なニーズなら、既存の製品はまさにあなたのために用意されています。API/ローカルモデル/Agent は「製品では満たせない」ときに初めて下りる段で、最初から重いものを持ち出す必要はありません。

6. **[応用・操作/観察問題]** 自分が使っている AI ツールを一つ選び、それがどの層にあるか言ってみましょう。裏側のモデルはおそらく何か？ 使っているのは **製品** の画面か、**API** か？ **手を動かせる**（ネット接続、ファイル読み込み、道具の呼び出し）機能があって、いくらか **Agent** の味があるか？
   > **観察できるはず：** 多くの人が日常的に使っているのは「製品」です。ネットにつなげたり、アップロードしたファイルを読めたり、道具を呼べたりするようになると、Agent の性質を帯び始めます。一つひとつ照らし合わせられれば、この地図は頭の中に収まります。
