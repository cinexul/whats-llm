ここまで、あなたが AI を使うやり方は、ずっと「画面を開いて、その中で自分でやり取りする」というものでした。この章でやることはそれとは違います:**あなた自身のプログラム** に AI を使わせるのです――あなたが対話ボックスの前で文字を打つのではなく、あなたのウェブサイト、アプリ、スクリプトが、裏側で **自動的に** 質問をモデルに送り、答えを受け取り、それをあなた自身のアプリに埋め込みます。

これが第17章の「四層の地図」の第③層――**API(インターフェース)** です。あの比喩を覚えていますか:レストランは中に入ってきた客に食事を出すもので、別の店がこの料理人の腕を使いたいなら「注文受付の窓口」から注文する、というものでした。API はこの窓口です。ただ、注文しに来るのが人ではなく、プログラムだというだけです。

先にこの章の位置づけをはっきりさせておきます:**この章は「考え方とデータの流れ」だけを扱い、どの言語のコードも書き込みません。** 各社の API の具体的な書き方、パラメータ、SDK の使い方は違いますし、しかもよく更新されるからです――それらは一律 **公式ドキュメントを参照**。あなたが持ち帰るべきは、「API をつなぐとはいったいどういうことか」という、エンジニアと話を合わせられて、自分でも判断できる枠組みです。

## 一、準備

たとえあなた自身がコードを書かないとしても、次のいくつかは心得ておく必要があります――とりわけ鍵と費用は、**最初の一行を書くときから管理する** ことです。

- **まず、本当に API が必要なのか、それとも会員になればよいのかを確認する。** これは最もよくある誤解なので、独立して述べます:もしあなた一人で、たまに AI に文章を直してもらう程度なら、出来合いのチャット **製品** で十分です(第17・18章)。**自分のシステムに自動で AI を呼び出させる** 必要があるとき――たとえば会社の公式サイトのカスタマーサポートに自動で返答させる、自分のプログラムにデータを一括処理させる――に限って、API を使う必要があります。**API をつなぐこと ≠ 会員になること、これは別の二つのことです。**
- **「鍵(API key)」を一つ手に入れる。** API を使うには、通常まず対応するプラットフォームで鍵を一つ申請します。これは「注文しているのはあなただ」と証明するためのもので、これに基づいて **トークン単位で課金** されます(第3・38章)。具体的な申請方法は **公式ドキュメントを参照**。
- **初日から鍵と費用を本気で扱う。** この鍵は極めて慎重に扱うべきものです――**漏れたら、他人があなたのアカウントで注文し、あなたに請求を付けられる** に等しいのです。だから次の第二節のステップ 0 は、その管理方法を専門に扱います。事故が起きてから慌てて補わないでください。
- **「データの流れ」がどんな形かを考え抜く。** API をつなぐことの本質は、「質問が出ていき、答えが返ってきて、また使われる」という一本の流れ作業を設計することです。まずこの線を頭の中ではっきり描き、コードを書くのはそれを実現するだけです。

> **ポイント:** API をつなぐことは、技術的な敷居は思ったほど高くありません(最小限で動くものはとても短い)。本当の敷居は二つの判断にあります:**一つは「API を使うべきか会員になるべきか」を区別すること、もう一つは最初から鍵と費用をきちんと管理すること** です。この二つさえはっきりさせれば、あとは公式ドキュメントに従って進めればよいのです。

## 二、一歩ずつやる

以下、「API をつなぐ」をいくつかのステップに分けますが、すべて考え方とデータの流れだけを扱います。どの言語を使い、どう書き、どのインターフェースを呼ぶかは **公式ドキュメントを参照**。

### ステップ 0:まず鍵を落ち着かせる(どんな機能を書くよりも前に)

このステップをすべての機能の前に置くのは、それが最も見落とされやすく、しかも最も大きな災いを招きやすいからです。三つの鉄則です(第37章のプライバシー、第38章の費用に呼応)。

| どうすべきか | なぜか |
| --- | --- |
| **鍵は「環境変数」に入れ、コードの中に書かない** | コードに直書きすると、鍵がコードと一緒にあちこち持ち運ばれ、いつでも漏れうる |
| **鍵を公開リポジトリ(公開された GitHub など)に絶対に提出しない** | 公開リポジトリの中の鍵は、自動スキャンプログラムに秒単位で拾われ、盗用される |
| **プラットフォームの管理画面で、アカウントに「利用上限/予算アラート」を設定する** | 万一鍵が漏れたり、プログラムが無限ループで叩きまくったりしても、上限が損失を食い止めてくれる |

「環境変数」とは平たく言えば、鍵のような機微な情報を **プログラムの外に置き**、実行時に読み込むことで、コードファイルの中にむき出しで書かないようにすることです。具体的な環境変数の設定方法、利用上限の設定方法は **公式ドキュメントを参照**――ただし「外に置く、提出しない、上限を設ける」というこの三原則は汎用で、古くなりません。

> このステップを「API のシートベルト」だと思ってください:車を出す前に、まずベルトを締める。

### ステップ 1:プログラムにあなたの代わりに「一言聞かせる」

最小限で動く核心は、たった一つです:**あなたのプログラムが、一つの質問を、API を通してモデルに送る。** このリクエストを普通の言葉で書くとこうなります。

```text
(あなたのプログラムが、裏側で自動的にこれをする)
モデルにこう言う:「これはあるユーザーの書き込みです。これが高評価か低評価か判断して、
          『高評価』か『低評価』だけで答えて。書き込みの内容:……」
```

これはあなたがチャットボックスで文字を打つのと本質的に同じです――どちらも「一段落の文章をモデルに送る」ことです。違いは一点だけ:**今はプログラムがあなたの代わりに送り、しかも一日に一万回送っても、あなたが手で打つ必要がない** ことです。何を送るか(つまり「プロンプト」)は、やはり第11章のあのこだわりです:要求をはっきり言い、形式をきっちり定める。

### ステップ 2:モデルの答えを受け取って戻す

モデルは処理し終えると、答えを **あなたのプログラムに送り返します**。あなたのプログラムが受け取るのは、たいてい一段落の文章です(あるいは少し構造のあるデータ。具体的な形式は **公式ドキュメントを参照**)。

ここに、初心者がよくはまる認識の落とし穴があります:**モデルが返す内容は、依然として間違うことがあり、あなたの求めた形式に従わないことがあります**(ハルシネーションは相変わらず起きます、第9章)。「高評価か低評価だけで答えて」と言っても、たまに一文の説明を返すかもしれません。だからプログラムの中で **返ってきた結果に対して検査とフォールバックをする** 必要があります――この部分は工学的な細部ですが、理念はあなたが持つべきです:**モデルが毎回おとなしく言うことを聞くと思い込まない。**

### ステップ 3:答えを「使って」、あなたのアプリに埋め込む

答えを受け取った後は、もう AI とは関係ありません――それはあなた **自身のアプリのロジック** です:「高評価/低評価」をデータベースに保存する、ページに表示する、通知メールを一通トリガーする……この部分は完全にあなたのプログラムが決めます。

データの流れ全体をつないで見ると、こんな一本の線になります。

```text
 ユーザー/あなたのデータ
      │
      ▼
 あなたのプログラム  ──(API を通して、鍵を携えて)──→  モデル
      ▲                                │
      │                                ▼
      └────────  モデルが答えを送り返す  ───────┘
      │
      ▼
 あなたのプログラムが答えを使う(保存/表示/次のステップをトリガー)
```

この線が分かれば、「プログラムに AI をつなぐ」精髄をすべてつかんだことになります:**あなたのプログラムが質問を送り出し、答えを受け取り、自分の流れに埋め込む。** 真ん中のあの「モデル」は借りてきた能力で、あなたが払うのはトークンの請求です。

### 結果を見る/検収する

最小版が動き出したら、次のいくつかで検収します。

1. **安定して「送り出して、受け取れる」か?** 本物のデータをいくつか通してみて、この流れ作業が通ったことを確認します。
2. **鍵は安全か?** 振り返って検査します:鍵は環境変数に入っているか、コードに書き込んでいないか、公開リポジトリに提出していないか、利用上限を設定したか。**これは繰り返し確認すべきです。**
3. **返ってきた結果にフォールバックはあるか?** わざと意地悪な入力を一つ食わせ、モデルが形式どおりに返さないとき、あなたのプログラムが落ちないか、処理しているかを見ます。
4. **いくらかかったか?** プラットフォームの管理画面で、今回のテストの利用量と費用を一目見て、「一回呼ぶとだいたいいくらか」の直感をつかみます(第38章)。

## 三、よくある落とし穴

| 落とし穴 | どういうことか/どう避けるか |
| --- | --- |
| **「API をつなぐ = 会員になる」と思い込む** | これが最大の誤解です。会員になるのは **人** に製品を使わせること、API をつなぐのは **プログラム** にインターフェースを使わせることで、鍵を申請し、トークン単位で課金されます(第17章)。まず自分がどちらを必要としているか区別を |
| **鍵をコードに直書きする/公開リポジトリに提出する** | 家の鍵をドアに貼り付けるようなものです。鍵は環境変数に置き、提出せず、上限を設ける――**最初の一行のコードより前に** やる(第37・38章) |
| **利用上限を設けず、叩きまくられたり無限ループで請求が爆発する** | プログラムのバグ一つで、一秒に数百回呼ぶこともあります。管理画面で予算アラートと利用上限を設定し、請求を食い止めましょう(第38章) |
| **モデルが毎回形式どおりおとなしく返すと想定する** | それは間違えるし、ずれます(第9章)。プログラムの中で返ってきた結果に検査とフォールバックを。むき出しで走らせない |
| **ユーザーのプライバシーデータをそのまま API に送る** | 第3章のあの一言を思い出してください:**送り込めば、すなわち外部に差し出したことになる**。機微な情報はまず匿名化を。学習に使われるかどうかは規約と設定によります(第37章、**公式ドキュメントを参照**) |
| **ネットのどこかの古いコードをそのまま丸写しする** | 各社の API は書き方が違い、しかもよく更新されます。**公式ドキュメントを参照** し、古いコード断片を盲信しないこと |

## 四、もう一歩進める

最小の流れ作業が通ったら、次のように広げられます(各ステップで鍵と費用という芯を忘れずに)。

- **「プロンプト」を磨く**:モデルに送るあの一段落の文章は、その質が答えの質を直接決めます。第11・12章のプロンプトの技を使い、必要ならテンプレートとして固定化します。
- **より構造化された出力を受け取る**:モデルにもっと整った構造を返させます(プログラムが直接使いやすいように)。具体的な要求の仕方は **公式ドキュメントを参照**。
- **トークンを節約して使う**:一括で呼ぶとき、トークンはお金そのものです。関係のない長文を毎回のリクエストにごっそり詰め込まず、リクエストを簡潔に保ちます(第38章の「節約して使う」一式が、ここではそのままお金の節約になります)。
- **もしプログラムに「自分で何ステップもやらせ、ツールを呼ばせる」ようになったら**:あなたは実は「API を一回呼ぶ」から **Agent** を作ることへ進んでいます(第19・34章)――このとき第34章の「ブレーキを取り付ける」規律を守る必要があります:お金を使う、データを削除する、外部へ送信する、これらは依然として人の確認を要するよう設定します。
- **会社の内部資料/外部ツールをつなぎたいなら**:それは第22・36章の Skills / MCP の話題です――新しい源を一つつなぐたびに、それが新しいリスクの入口だということを忘れずに。

> **ポイント:** 「プログラムに AI をつなぐ」とは **API** を使うことです(第17章):あなたのプログラムが質問をモデルに送り、答えを受け取り、自分のアプリに埋め込む――**考え方はシンプルで、データの流れは明快** です。ただし二つのことは、最初の一行のコードから見張る必要があります:**① API をつなぐこと ≠ 会員になること**(人が製品を使うのか、プログラムがインターフェースを呼ぶのかを区別);**② 鍵と費用を最初から管理する**――環境変数に置き、コードに書かず、公開リポジトリに提出せず、利用上限を設ける(第37・38章)。具体的な書き方は **公式ドキュメントを参照**。

## まとめ

- 「既存のプログラムに AI をつなぐ」= **API** を使うこと(第17章第③層):あなた自身のプログラムが裏側で自動的に質問をモデルに送り、答えを受け取り、アプリに埋め込む。
- 最小限で動くデータの流れはたった一本の線です:**あなたのプログラム →(鍵を携えて)→ モデル → 答えが返る → プログラムが使う**。この章はこの考え方だけを扱い、ある言語の SDK コードは書き込みません(**公式ドキュメントを参照**)。
- **最大の誤解は「API をつなぐ = 会員になる」** です:会員は人に製品を使わせること、API はプログラムにインターフェースを使わせ、トークン単位で課金されます。
- **鍵と費用は最初の一行を書くときから管理する**:環境変数に置き、コードに書かず、公開リポジトリに提出せず、利用上限を設ける(第37・38章に呼応)。これはステップ 0 で、すべての機能の前に置きます。
- モデルが返す内容は依然として間違うし、形式に従わないことがあるので(第9章)、プログラムは検査とフォールバックをする。送り込んだデータは差し出したことになるので、機微な情報はまず匿名化を(第37章)。

次章では、第22章のあの二つの概念――**Skills と MCP**――を本当に地に落とし、AI にデータを読み、ツールを使う新しい能力を広げます。

## 小テスト

> 全 6 問。概念・誤解・シナリオ・操作の 4 種類をカバーしています。答えと解説は各問の下の引用ブロックにあります――まず自分で考えてから、照らし合わせてください。

1. **[基礎・概念問題]** 「既存のプログラムに AI をつなぐ」とは、本質的に四層の地図(第17章)のどの層を使っているか?
   - A. 製品層(人が使う画面)
   - B. **API 層**(プログラムが使うインターフェース)
   - C. モデル層(むき出しのコア)
   - D. Agent 層
   > **答え B。** **プログラム** に裏側で自動的にモデルを呼ばせるのは、まさに API です(第17章第③層のあの「注文受付の窓口」)。製品層は **人** が開いてやり取りするためのもの;API 層はプログラムが呼び出す、同じ能力の入口です。

2. **[基礎・誤解問題]** 「自分のシステムに AI を組み込みたいから、ChatGPT の会員になろう。」――どこが間違いか?
   - A. 間違っていない。会員になればよい
   - B. **プログラム** に AI を使わせるには **API** を使う(鍵を申請し、トークン単位で課金);会員になるのは **人** に製品を使わせることで、別の二つのこと
   - C. もっと高い会員を買うべきなのが間違い
   - D. システムは AI を組み込めないのが間違い
   > **答え B。** これはこの章の筆頭の誤解です。「人が画面の中で使う」と「プログラムが裏側で自動的に呼ぶ」は、まったく異なる二本の道です:前者は製品を使い(会員になる)、後者は API を使う(第17章)。まず自分がどちらを必要とするか区別してから、やり方の話をします。

3. **[基礎・概念問題]** API を使うときのあの「鍵(API key)」は何をするものか?
   - A. モデルの計算を速める
   - B. 「呼んでいるのはあなただ」と証明し、これに基づいて利用量をあなたの請求に記録する(トークン単位で課金)
   - C. モデルが間違えないようにする
   - D. あるチャット製品の会員カード
   > **答え B。** 鍵はあなたの「注文の証 + 記帳の根拠」に相当します。それが一方であなたの請求につながっているからこそ、**漏れたら他人があなたのアカウントで叩ける**――だから環境変数に置き、公開リポジトリに提出せず、利用上限を設ける(第37・38章)。

4. **[応用・誤解問題]** コードを書くのを楽にしようと、鍵をコードファイルに直書きし、しかも公開された GitHub リポジトリに push しました。最も起こりやすいことは?
   - A. 大丈夫、誰も見ない
   - B. 公開リポジトリの中の鍵は自動スキャンプログラムにすぐ拾われ盗用され、身に覚えのない多額の請求を叩き出されるかもしれない
   - C. コードがもっと速く走る
   - D. GitHub が自動で暗号化してくれる
   > **答え B。** 公開リポジトリは専門のスキャンプログラムに見張られていて、平文の鍵はたいてい数秒以内に拾われます。正しいやり方は **鍵を環境変数に置き、絶対に提出せず、利用上限で食い止める**(第37・38章)――しかもこれを、最初の一行の機能コードより前にやっておくことです。

5. **[基礎・シナリオ問題]** あなたのプログラムが API を通してモデルに「『高評価』か『低評価』だけで答えて」と言わせていますが、たまに一文の説明を返します。最も適切な対処は?
   - A. 言うことを聞くまで繰り返しリトライする
   - B. プログラムの中で返ってきた結果に検査とフォールバックをする。モデルが毎回おとなしく形式に従うと想定しない
   - C. 鍵を別のものに替える
   - D. API が壊れたと決めつける
   > **答え B。** モデルが返す内容は依然として間違うし、形式に従わないことがあります(ハルシネーションは相変わらず起きます、第9章)。堅牢なプログラムは「言うことを聞かないかもしれない」とあらかじめ想定し、返り値に検査とフォールバックをします。むき出しで「永遠に正しい」と想定はしません。

6. **[基礎・操作/観察問題]** 本当にコードを書く必要はありません――紙の上に「API をつなぐ」データの流れを一度描いてみましょう:「あなたのプログラムが質問を送り出す(鍵を携えて)」から、「モデルが答えを送り返す」、さらに「あなたのプログラムが答えを使う」まで。それから図の上に、お金/安全に関わる三か所に印を付けます:鍵はどこに置くか、どこで課金されるか、どこで上限を設けるか。(具体的な書き方は **公式ドキュメントを参照**。)
   > **観察できるはず:** この線は実はとても短い――送り出し、受け取り、使う、たった三歩です。これが「AI をつなぐ」考え方のすべてです。しかし、あなたが印を付けたあの三点(鍵を環境変数に置く、トークン単位で課金される、管理画面で利用上限を設ける)こそが、本当に事故が起きやすく、初日から管理しなければならないところです(第37・38章)。データの流れとこの三点をはっきり描けば、あなたはエンジニアと話を合わせられるうえに、最低限の一線も守れます。
