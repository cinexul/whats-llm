前章では Claude Code を扱いました。本章は **Codex** の番です。飛ばし読みでここに来た方のために、背景を一言。第24章は「調査 → 計画 → 実行 → 検収」という、プログラミングエージェントとともに仕事を進める **共通ワークフロー** を徹底的に述べ、第28章はそれを Claude Code に当てはめました――**本章は、その同じワークフローを Codex に当てはめる回** です。

## 一、Codex とは何か：OpenAI 発のプログラミングエージェント

まず安定した事実を一つ覚えましょう。**Codex は OpenAI 発です。** Claude Code とは「同類だが別の出どころ」――どちらも、あなたのプロジェクトの中で **自分でファイルを読み、コードを変更し、コマンドを実行する** プログラミングエージェント(coding agent)ですが、提供元とコアモデルが異なります。

Claude Code と同じく、**Codex も特定の Web ページやソフトを指すわけではありません**。次のように、複数の形態で出会う可能性があります。

- **コマンドライン(CLI)**：ターミナルの中で動かしながら作業する。
- **コードエディタ(IDE)プラグイン**：エディタに組み込む。
- **Web 画面（しばしば Codex の Web ／クラウドのワークスペースと呼ばれる）**：ブラウザの中でタスクを振る。
- **クラウドのタスク**：作業をクラウドのバックグラウンドで走らせ、後から結果を受け取る。

> **ポイント：** 第28章と同じ一言です。**あなた自身のマシンで動く形態もあれば、クラウドで動く形態もあり、具体的な形態はそれぞれ異なり、しかも急速に更新されています――公式ドキュメント／製品の公式な形態を参照してください。** 確実に覚えておけるのは二点だけです。OpenAI 発であること、そして手を動かすプログラミングエージェントであること。残りの「どうインストールするか、どんな見た目か、何を打つか」は、一律 公式ドキュメントで今の版を確認してください。

## 二、Claude Code との異同：心法は同じ、手法は各様

ここが本章で最も役立つ一節です。第28章と本章を並べて見れば、「共通のもの」と「各様のもの」がどこにあるか、すぐにつかめます。

| | 同じ（安定、一度学べばどこでも使える） | 違う（**公式ドキュメントを参照**、変わる） |
| --- | --- | --- |
| **位置づけ** | どちらも「あなたのプロジェクトで手を動かす」プログラミングエージェント | 提供元(OpenAI vs Anthropic)、コアモデルが異なる |
| **ワークフロー** | どちらも第24章「調査→計画→実行→検収」が当てはまる | 「計画を出す」「diff を見る」を具体的にどう起動するかが各様 |
| **安全** | どちらも許可の確認 + 巻き戻せる環境を備えるべき | デフォルトの挙動、許可の粒度、確認のしかたが異なる |
| **入門** | どちらもまずやり直しのきく環境で小さく試す | インストール方法、ログイン／認証方法、コマンドが異なる |
| **形態** | どちらも CLI ／ IDE ／ Web ／ クラウドなど複数の形態を持つ | 各形態の具体的な名称、入口、能力が異なる |

これを一言にまとめれば――**第24章で学んだ判断力（どんな仕事を振り、どう計画を承認し、どう diff を見て、どう検収するか）は、Codex に替えても一字も変えずに通用する。改めて調べ直す必要があるのは「Codex でこれらの動作を具体的にどうするか」だけ** です。

ですから Codex に向き合う正しい姿勢は、前章とまったく同じです。

- **すぐに操作に飛びつかない。** 公式ドキュメントで「いま推奨されるのはどの形態か、どうインストールし、どうログインするか」を見て、そのとおりに（**公式ドキュメントを参照**）。
- **第24章のワークフローを持ち込む。** **まず調査させ、計画を出させ、承認してから手を動かさせ、終わったら丁寧に検収する**――このリズムはどのツールでも正しい。
- **最初はやり直しのきく小さなプロジェクトで。** Git で管理し、新しいブランチを切った状態で最小のリクエストを一つ通します（第25、27章）。いきなり大事なものに手を入れさせてはいけません。

> **実際によくある場面：** 少なからぬ人が Claude Code と Codex を併用し、「なぜこのコマンドが向こうでは効かないのか」につまずきます。**一方のツールの具体的なコマンドを、もう一方のものと取り違えないでください。** 本当にツールをまたいで使えるのは、第24章の「間違えうる・手を動かす助手とどう付き合うか」という判断力です。具体的なコマンドは、それぞれの公式ドキュメントを確認してください。これさえ覚えておけば、両者の差異の中で堂々巡りせずに済みます。

## 三、鍵と費用：初日から管理する

この一節は単独で重点的に述べます――Codex のようなツールはしばしば **API キー(API key)** を使う必要があり、しかも **鍵と費用は、初日に最も事故が起きやすいところ** だからです。ここでは「なぜ管理するのか、どう判断するのか、何に気をつけるのか」だけを述べ、第35章・第37章・第38章に呼応します。

### 1. 鍵とは何か、なぜ重く受け止めるのか

**API キー** は、「あなたの身元と請求を代理する一連の鍵」と理解できます。**これを手にした者は、あなたの名義でサービスを呼び出し、あなたのお金を使えます。** ですから安全レベルはパスワードと同等――いや、しばしば直接課金につながるぶん、それ以上です。

ここから、**譲れない規律** がいくつか導かれます（第37章・第35章で改めて展開します）。

- **鍵を絶対に対話に貼らない。** 手間を惜しんで、鍵を AI とのチャットに貼り込まないでください――貼り込んだものは差し出したのと同じです（第3章の「貼れば＝差し出した」）。鍵を設定する必要があるときは、公式ドキュメントが指定する **安全な方法** に従ってください（**具体的な方法は公式ドキュメントを参照**）。
- **鍵を絶対に公開リポジトリに入れない。** 鍵をコードに書き、それを公開の Git リポジトリにプッシュするのは、最も典型的で、最も悲惨な漏洩です――クローラが数分で拾い上げ、誰かがあなたの代わりに請求を膨らませます。設定類の秘密は **コミットされない場所** に置きます（第35章で「環境変数／鍵管理」といったやり方に触れます。**具体的には公式ドキュメントを参照**）。
- **漏洩を疑ったら、ただちに失効させて再発行する。** うっかり貼ったり、コミットしてしまったりしたら、ただちに該当プラットフォームで **古い鍵を失効させ、新しいものに換えて** ください（**具体的な操作は公式ドキュメントを参照**）。たかをくくってはいけません。

> **ポイント：** 一言、頭に刻んでください――**鍵はあなたの財布の鍵。対話に入れない、公開リポジトリに入れない。** これはツールも形態も問わず、鍵を使うあらゆる場面で成り立ちます。呼応するのは第37章（プライバシーとデータ分類）の鉄則です。**最高機密のものは、絶対に渡さない。**

### 2. 費用：トークン単位、知らぬ間にお金を燃やさない

Codex のようなツールは本質的に **トークン単位で課金** されます（第3章・第38章で、トークンが計量単位だと述べました）。これが意味するのは――

- **読ませるファイルが多いほど、会話が長いほど、変更が多いほど、減りが速い。** 一度「プロジェクト全体を読ませてから大改修」とやると、知らぬ間にかなりのトークンを燃やしかねません。
- **クラウド／バックグラウンドのタスクは特に心づもりが要る。** 仕事をクラウドで走らせるのは便利ですが、「自分で長く走り続ける」ことも意味します。開始前に「だいたいどれくらいの仕事量か」の見当をつけ、放りっぱなしにしないでください。
- **タスクを絞ることが、最良の節約法。** タスクを小さく分け、会話をすっきり保てば、トークンを節約でき品質も上がります――これは第38章「燃やしすぎも浪費もしない」と同じことの両面です。

> **ポイント：** 心の中に「予算のチェックポイント」を設けましょう。開始前に自問します。このタスクは、これだけ読ませ、これだけ走らせる **価値があるか**？ タスクを小さく分け、いつでも止められること（第26章・第30章）は、安全であると同時に節約でもあります。具体的な課金方式・単価・上限は **公式ドキュメントを参照**――本書は「トークン単位、話すほど高い」という変わらない仕組みだけを述べます。

## 四、深い使い方と具体操作：公式ドキュメントを参照、付録 C へ

Claude Code と同じく、Codex にも独自の一そろいの上級能力・設定・コマンドがあります――しかしそれらは **まさに更新が最も速く、本に書き込むべきでない部分** です。本書の扱いは依然として明確です。

- **具体的にどうインストールし、どうログインし、何を打ち、どのボタンを押すか**――一律 **公式ドキュメントを参照**。
- **上級の使い方（カスタム設定、外部ツール／データとの接続、自動化など）**――**付録 C** と公式ドキュメントへ向け、本文には書き込みません。
- **ある具体的なことができるか・どんな段階があるか・モデルのバージョン名は何か**――これらはすべて変わりやすい項目で、**公式ドキュメントを最新の拠り所** とします。

> **ポイント：** 「Codex と Claude Code は結局どちらが強いか、どちらが X をできるか」が気になるかもしれません。この種の比較は **バージョンが上がるたびに頻繁に入れ替わる** ので、本書はこうした結論を下しません。本当に身につけるべきは、**具体的なツールに依存しない力** です――第24章のワークフローを理解し、鍵と検収の規律を守ること。どちらが強いか弱いかは変わりますが、この判断力は変わりません。具体的な能力の比較は、それぞれ公式ドキュメントを参照してください。

## 五、安全チェックリスト（前章と一そろいの最低線を共有する）

Claude Code を使おうと Codex を使おうと、これらの最低線は同じです。ここでもう一度くぎを刺します。

- **やり直しのきく環境を優先する。** ローカルで手を動かすときは、まず **Git** で始め、先に **新しいブランチを切る**（第26章）。クラウド／Web の形態はプラットフォーム側の隔離・巻き戻しを使う（**公式ドキュメントを参照**）。
- **許可の確認を何も考えず通さない。** 結果を伴うことをするたびに許可を求めてくる（第26章）。ずっと「同意」を押さない――それはブレーキを外すこと。
- **鍵・パスワード・本物の個人情報は絶対に外に貼らない。** 本章第三節・第37章を参照。
- **検収はあなたの責任。** diff を見る、テストを走らせる、この部分が何をしているか一言で言える――読めないものはマージしない（第25章）。生成物はあなたの責任（第39章）。

## まとめ

- **Codex は OpenAI 発**（安定した事実）で、あなたのプロジェクトで手を動かすプログラミングエージェント。**形態は多様（CLI／IDE／Web／クラウド）で、しかも更新中**――どうインストールし、どう操作するかは **公式ドキュメントを参照**。
- Claude Code と **心法は同じ、手法は各様**：第24章「調査→計画→実行→検収」は共通、**具体的なコマンド／インストール／能力はそれぞれ異なり、更新が速い**。
- **鍵と費用は初日から管理する**：鍵は絶対に対話に入れない、公開リポジトリに入れない（第37章・第35章）。トークン単位で課金され、タスクを小さく分けてこそ節約できる（第38章）。
- **深い使い方と具体操作は付録 C と公式ドキュメントを参照**。本書は書き込まず、「どちらが強い」という結論も下さない。
- 安全の最低線は前章と一そろいを共有する：やり直しのきく環境、許可を何も考えず通さない、鍵を外に貼らない、検収は自分で責任を持つ。

次章は角度を変えます。どのツールを使おうと、**つまずきは起こります**。第30章で、八つのよくある「つまずきの現場」と「行き詰まったときの応急処置」を一度に述べます。

   > **観察できるはず：** これらの最低線が共通するのは、それらが「**間違えうる + 手を動かす**」という、あらゆるプログラミングエージェントに共通の特性（第24章）から来ているからです。ある製品の設計から来ているのではありません。ツールは変わり、コマンドも変わりますが、「間違えうるうえにあなたのものを動かせる」かぎり、この安全カードはずっと役立ちます。具体的なコマンドと深い使い方は、それぞれ **公式ドキュメントを参照**（付録 C を参照）。
