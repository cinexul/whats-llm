〔本章で具体的な製品の形態・インストール・コマンド・画面・能力に触れる箇所は、いずれもバージョン更新とともに変わります。一律 **公式ドキュメントを参照** してください。本章では、一群のツールに **共通する、安定した** メンタルモデルだけを扱います――この部分は古くなりません。〕

先の第19章で、私たちは **Agent(エージェント)** を知りました。ただ「ひと言を答える」のではなく、**自分で手を動かして仕事をする** AI です――目標を分解し、ツールを呼び出し、何ステップも続けて進められます。本章では、Agent 一族の中で最も成熟していて、一般の読者にとっても最も知る価値のある分枝にレンズを向けます:**コーディングエージェント(coding agent)** です。

その代表が、おそらく名前は聞いたことのある **Claude Code** と **Codex** です。

先に耳の痛いことを言っておきます:**本章は、コードを書けることを求めません。一行も要りません。** ここで築くのは一種の「メンタルモデル」です――この道具が **いったい何で、どこまでできて、何を期待してはいけなくて、最終的に誰が責任を負うのか** をはっきりさせることです。これを腑に落とせて初めて、この先の章で扱う「diff を見る、テストを走らせる、巻き戻す、計画を見る」といった具体的なチェックの動作について、**自分が何を守っているのか、なぜ守るのか** が分かるようになります。

## 一、Agent の一種:「コードが書ける」だけでなく「仕事ができる」

多くの人は「プログラミング AI」への第一印象が「コードが書ける」で止まっています――「ソート関数はどう書く?」と聞けば、ひとまとまりのコードをくれて、それをコピーして持っていく。この印象は、**Web チャット** のレイヤーのものです(第17章でモデル/製品/Agent のレイヤー分けを扱いました)。

コーディングエージェントは、これより **一段上のレイヤー** です。「コードが書ける」のではなく、「**コードを書くという行為を使って仕事をする**」のです。違いはどこにあるのでしょうか。

| | 「コードが書ける」(チャットレイヤー) | 「仕事ができる」(エージェントレイヤー) |
| --- | --- | --- |
| 何を与えるか | 具体的な小さい問題ひとつ | ひとつの **目標**(「このサイトに検索ボックスを足して」) |
| 何をくれるか | ひとまとまりの文章。自分で使う | **自分で事を成し遂げる**――ファイルを読み、コードを変更し、コマンドを実行する |
| どれだけ知っているか | 貼り付けた数行しか知らない | **自分でプロジェクト全体を見て回り**、現状を把握する |
| 進め方 | こちらがひと言聞き、ひと言答える | 目標に向かって **何歩も続けて進む**:見てみる→直してみる→試してみる→また直す |

これはまさに、第19章での Agent の定義が「プログラミング」という具体的な領域に落ちてきた姿です:**知覚(プロジェクトを読む)→ 意思決定(案を考える)→ 行動(ファイルを変更する、コマンドを実行する)→ また知覚(結果が合っているか見る)**、これを自分でぐるぐる回します。だから最初の一文を覚えてください――

> **ポイント:** **コーディングエージェント ＝ 第19章の「手を動かす Agent」。ただし手を動かす場所があなたのコードプロジェクトなのです。** 「コードが書ける」は、ひとつのスキルにすぎません。「このスキルを使って一つの事を成し遂げられる」ことこそ、その真の正体です。これを理解すれば、もう「より賢い検索ボックス」として扱うことはなくなります。

## 二、「どんな姿」をしているか:形態は変わりやすい。一つに縛りつけない

こう聞きたくなるかもしれません:では、どこで使うのか? あるサイト? あるアプリ?

答えはこうです:**唯一の答えはなく、しかも変わり続けています。** 普通こうしたツールに触れるのは **ターミナル(コマンドライン)、コードエディタ(IDE)、Web のタスク、あるいはクラウド環境** の中です。**自分のマシンで動くものもあれば、クラウドで動くものもあり、具体的な形態はそれぞれ異なり、しかも更新され続けています――公式ドキュメント、製品の公式な形態を参照してください。**

なぜ本書は「この Web を使え/このソフトを入れろ」と言いたがらないのでしょうか。それは、この種のツールの形態が **特に速く変わる** からです:今日の主流はコマンドライン、明日は IDE プラグインのほうが手に馴染むかもしれず、明後日には Web 版・クラウドのバックエンド版が出てくるかもしれません。**本を一つの形態に固定して書けば、半年で古くなります。** だから本書は **古くならない** 部分だけを扱います――どう仕事をするか、どう率いるか、です。具体的に「どこで開くか、どのボタンを押すか、どのコマンドを打つか」は、第25章と付録 C で展開しますが、最終的にはどれも **公式ドキュメントを参照** してください。

> **ポイント:** 二つのことは **安定** していて、安心して覚えてよいものです:Claude Code は Anthropic 発で、コアは Claude。Codex は OpenAI 発。この「提供元/コア」の帰属は、むやみに変わりません。しかし **具体的にどんな姿か、どうインストールするか、何ができるか** は「変わる」ほうの類です――こうした細部に出くわしたら、公式ドキュメントを確認してください。

## 三、核心となるメンタルモデル:「とても優秀だが、入社初日の新人同僚」と思え

これは本書が繰り返し使い、ここで最もしっくりくるたとえです。どうか本当に一人の **人** として想像してみてください:

**Claude Code / Codex は、手が速く、よく働き、能力もとても高い、けれど今日着任したばかりで、あなたたちの会社のことを何も知らない新人同僚のようなものです。**

この人物像を分解して見ると、一つひとつが、実際の「振る舞い」に対応しています:

- **手が速い:** 任せた力仕事を、速く、たくさんこなします――これが最大の価値です。無駄にしないでください。
- **あなたたちの「流儀」を知らない:** プロジェクトのルール、命名の習慣、どこが触ってはいけない地雷か、何も知りません――**こちらが言わなければ、知りようがありません**。(これらのルールを「覚えさせる」やり方はあります。第28章を参照。)
- **自信たっぷりに間違える:** **まったく存在しない書き方** を、一点の迷いもない顔で使うこともあります――これが第9章で扱った「ハルシネーション」で、コードを書くときも同じように起き、しかもコードは「専門的」に見えるぶん、専門知識がない人をだましやすいのです。
- **最新のことを知らない:** その知識には「カットオフ日付」があり(第9章)、とても新しい技術は、聞いたことがなかったり、旧バージョンとして覚えていたりします――最新の資料を伝えない限り。

では、こんな新人同僚を前にして、頼りになる「率いる側の人」はどうするでしょうか。**放任もしないし、何でも自分で代わりにやってしまうこともしません。** 仕事を振り、肝心な節目を見張り、確認します。これこそ、あなたの立ち位置です――

> **誤解 vs 実際:** 多くの人は、いきなり AI を「全自動の代筆マシン」として扱います:用件を投げ、仕上がりを待ち、そのまま使う。**実際には、鍵を渡して任せきる請負業者というより、面倒を見る実習生に近いのです。** 一行一行を書ける必要はありませんが、**仕事を振り**、**できているか・いないかを判断** できる必要があります。請負業者扱いすると、いずれ「一見問題ないが、実は地雷が埋まっていた」結果のツケを払うことになります。

## 四、最終的な責任は、いつも人の側にある

これは、本章、いや第4部全体で **最も覚えておくべき** ひと言なので、単独で取り出して述べます。

「やり終えた」ことは、「やり遂げた」こととは **等しくありません**。「できました」と言っても、本当に成ったこととは **等しくありません**。理由は先に布石を打ってあります:**AI は間違える(ハルシネーションを起こす)＋ 手を動かせる(本当にあなたのものを変更した)**――この二つが重なると、危険な組み合わせになります:**一点の迷いもない顔で、あなたのプロジェクトを壊し、なお「完成しました」と告げうるのです。**

だから、この道具がどれほど強くとも、未来にどれほど強くなろうとも、動かない一線があります:

> **ポイント:** **署名するのはあなた、責任を負うのもあなた。** AI が生成したものを、あなたのプロジェクトにマージし、あなたの顧客に渡し、あなたの名前を掲げる前に、**見直しと確認はあなたの責任** です(第39章で「AI が生成した内容の責任」を専門に扱います)。ひと言、脳に刻みつけてください――**「分からないもの、検証していないものを、通すな。」** 一行一行を読み解ける必要はありませんが、「この件はやり遂げたのか」を判断できる必要があります。

これはあなたに釘を刺すためではなく、むしろ逆です:**責任がはっきりと人の側に落ちているからこそ、安心して大量の力仕事を任せられるのです。** 「確認」という関所さえ守れば、その前にある「速くてたくさん」の仕事は、すべて丸ごと得になります。

## 五、本章は、後の章への何の道を敷いたか

上の四点をつなげると、もう一枚の「地図」を手にしています。後の数章は、この地図の上に細部を埋めていくものです:

- **手を動かし、しかも間違える** → だから **事後のチェック** を学ぶ必要があります:何を変更したか(diff)、別のものを壊していないか(テスト)、分からなければマージしない――これが **第25章** で、本書で最も練習して身につけるべき一章です。
- **本当にあなたのものを変更する** → だから **やり直しのきく環境** で手を動かさせ、しかも **危険な操作は先に尋ねさせる** 必要があります――これが **第26章** の「許可の仕組みと巻き戻せる環境」です。
- 大事は **一気に突っ走ってはいけない** → だから **先に計画を出させ、承認してから手を動かさせ**、大きなタスクを **小さく分けて小さく確認** する必要があります――これが **第27章** です。

気づきましたか? この三つは、すべて本章の核心となる心構えを実地に落とし込むものです:**優秀だが間違える新人同僚。仕事は振るが、チェックはあなたに。** 続く三章は、「どうチェックするか」を、一手一手教えていきます。

## まとめ

- **コーディングエージェント(Claude Code / Codex)は、第19章の「手を動かす Agent」がプログラミング領域に現れた姿** です:「コードが書ける」のではなく、「コードを書くことで一つの事を成し遂げる」――自分でプロジェクトを読み、ファイルを変更し、コマンドを実行し、何歩も続けて進む。
- **形態は多様で、しかも変わり続けている**(ターミナル/IDE/Web/クラウド、ローカルもクラウドもある)。本書は古くならないメンタルモデルだけを扱う。具体的にどうインストールし、何を打つかは **公式ドキュメントを参照**(第25章、付録 C)。
- 核心の心構え:**とても優秀だが、入社初日の新人同僚** として扱う――手は速く、よく働くが、あなたたちの流儀を知らず、自信たっぷりに間違える。
- **最終的な責任はいつも人にある**:「分からないもの、検証していないものを、通すな。」 チェックがあなたにあるからこそ、力仕事を安心して任せられる。
- 本章は地図、後の三章は細部を埋めるもの:**確認(diff を見る/テストを走らせる、第25章)、許可と巻き戻し(第26章)、まず計画それから実装(第27章)**――すべて「仕事は振り、チェックはあなたに」を実地に落とし込むもの。

次章(第24章)では、この「調査→計画→実行→検収」という共通のワークフローを、まるごと一通り歩いてみます。

1. **[基礎・概念問題]** 「コーディングエージェント」を第19章の「Agent」と照らし合わせたとき、最も正確なのはどれ?
   - A. コーディングエージェントと Agent は無関係で、別々の二つのもの
   - B. コーディングエージェントは Agent の一種――手を動かす AI で、ただ手を動かす場所があなたのコードプロジェクトなだけ
   - C. コーディングエージェントは手を動かさず、助言をくれるだけ
   - D. Agent はコーディングエージェントの一つの機能ボタン
   > **答え B。** 第19章の Agent ＝ 自分で目標を分解し、ツールを呼び出し、何歩も続けて進む AI。コーディングエージェントは、この本領を「コードプロジェクトを変更する」ことに使ったものです。A・D を選ぶ人は「Agent の一種」という関係を築けていません。C を選ぶ人は「チャット AI は文字をくれるだけ」という古い印象に止まっています。

2. **[基礎・概念問題]** 「コードが書ける」と「仕事ができる」、コーディングエージェントにとって違いはどこにある?
   - A. 違いはない、同じこと
   - B. 「コードが書ける」はひとまとまりの文章を渡されあなたが自分で使う。「仕事ができる」は自分でプロジェクトを読み、ファイルを変更し、コマンドを実行して目標を成し遂げる
   - C. 「仕事ができる」とはコーヒーを淹れられること
   - D. 「コードが書ける」のほうが上等
   > **答え B。** これこそチャットレイヤーとエージェントレイヤーの境界です:前者は文字を吐き、あなたが実行する。後者は自分で実行し、何歩も続けて進む。AI を「より賢い検索ボックス」扱いするのは、「コードが書けるだけ」のレイヤーと取り違えているのです。

3. **[基礎・誤解問題]** 「それは AI なのだから、書いたコードは正しいはず。やり終えたらそのまま使う」――どこが間違い?
   - A. 間違っていない、AI が書いたものは正しい
   - B. 自信たっぷりに間違える(ハルシネーションは同じように起きる)。「やり終えた」 ≠ 「やり遂げた」。分からない・検証していないものは通すべきでない
   - C. 書いたコードは永遠に動かない
   - D. 使う前に課金が必要
   > **答え B。** ハルシネーション(第9章)はコードを書くときも同じように起き、しかもコードは「専門的」に見えるぶん専門知識がない人をだましやすい。だからこそ後で「diff を見る、テストを走らせる」を学ぶのです。C は別の極端へ行っています――たいていは動くもので、問題は「動く」 ≠ 「正しい」という点にあります。

4. **[応用・誤解問題]** 「Claude Code / Codex は、ある固定の Web(あるソフト)であって、形態は決まりきっている」――なぜそう考えると問題が起きるのか?
   - A. 問題は起きない、形態は固定だ
   - B. 形態が多様(ターミナル/IDE/Web/クラウド)で、ローカルもクラウドもあり、しかも更新され続けている。一つに縛りつけるとすぐ古くなる――具体は公式ドキュメントを参照
   - C. クラウドの一形態しかない
   - D. そもそも画面がない
   > **答え B。** 形態は「変わる」ほうの類なので、本書は古くならないメンタルモデルだけを扱い、「どこで使うか、どうインストールするか」は公式ドキュメントに委ねます。ただし二点は安定です:Claude Code は Anthropic 発(コア Claude)、Codex は OpenAI 発――この帰属は安心して覚えてよいものです。

5. **[基礎・場面問題]** 初めて使うとき、AI があっという間に「機能を追加できました」と言った。「率いる側の人」として、次にすべきことは?
   - A. 直ちに信じて、マージして公開する
   - B. 入社したての新人同僚として扱う――確認する:いったい何を変更したか、別のものを壊していないか、この部分が何をしているか自分で言えるか
   - C. 二言ほめて終わりにする
   - D. もっと大きな次の機能をさらにやらせる
   > **答え B。** 「やり終えた ≠ やり遂げた」。優秀だが着任したての新人同僚のようなもので、確認は率いる側の人の責任です(具体的な三つの動作は第25章)。A は「鍵を渡して任せきる請負業者」扱い――これこそ最もよくある失敗の姿勢です。

6. **[応用・操作/観察問題]** どんなツールも実際にインストールする必要はありません:身の回りで「仕事をする vs 助言をくれるだけ」の対比を一つ探してください――たとえば「ナビが直接あなたを曲がり角まで連れていく」のと「通りすがりの人が方向を指さす」の対比。考えてみましょう:コーディングエージェントはどちらに近い? また「ひとまとまりのコードを渡してあなたにコピーさせるだけ」のチャット AI とは、どのステップで違う?
