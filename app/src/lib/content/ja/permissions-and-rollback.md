第23章で、私たちは少し恐ろしい組み合わせを見極めました:**それは間違える ＋ それは手を動かせる。** 第25章で、私たちは「事後にどう検収するか」を学びました。本章では、もう半分の問題を解決します:**事前と事中に、どうやってそれを「手は動かすが暴走させない」ようにするか?**

答えは、互いに組み合わさる二つの事で、本章で一つずつ徹底的に扱います:**一つは「許可の仕組み」――危険な操作は手を動かす前に先にあなたに尋ねる。もう一つは「巻き戻せる環境」――万一事が起きても、やり直せる。** 先に一言置いておきます:**AI には「大したことのない」仕事は思い切りやらせ、いやしくも「災いを起こしうる」ものは、永遠に人の頷きに残す。**

## 一、核心となるループ:リクエスト → 提案を見る → 許可 → 検証

手を動かすエージェントを率いるとき、その全過程は実は一つの絶えず繰り返される小さなループです。これを脳に刻めば、あなたはもう「それが私のパソコンの中で勝手に走り回り、私には制御できない」と感じることはなくなります:

```text
   あなたが目標を与える
      │
      ▼
 ①それがリクエスト ──→ ②あなたが提案を見る ──→ ③あなたが許可 ──→ それが手を動かす ──→ ④あなたが検証 ──→(次の一歩でまた①へ)
(「Xを変更するつもり」)  (それが何をしたいか見る)  (頷く/拒む)                  (diffを見る/テストを走らせる)
```

- **① リクエスト**:「結果を伴う」事(ファイルを変更する、コマンドを実行する、ネットにつなぐ……)をするたびに、それは通常 **立ち止まって、まず何をするつもりかをあなたに告げ**、黙って突っ走ったりはしません。
- **② 提案を見る**:それが「何を、どこを動かすか」をはっきり見ます。このステップがあなたの介入の窓口です――おかしいと感じたら、今この場で止めるのが最も代価が小さい。
- **③ 許可**:あなたが頷いて(この一回/この一類に同意して)初めて、それは手を動かします。あなたは拒むことも、別のやり方に変えさせることもできます。
- **④ 検証**:それがやり終えたら、第25章のあのやり方(diff を見る、テストを走らせる、検収の三つの問い)で結果を確かめます。そして次の一歩へ進み、**ループが改めて始まります**。

> **ポイント:** この「リクエスト→提案を見る→許可→検証」のループは、第24章の「調査→計画→実行→検収」が **一つひとつの具体的な操作** の上に縮小して映ったものです。その魂は、「決定権の肝心な節目」を **人の手に握る** ことにあります:AI は速くたくさん仕事をすることを担い、**人は分かれ道で頷くか、止まれと叫ぶことを担う** のです。このループを理解すれば、それへの掌握感が築かれます――それは手綱を離れた馬ではなく、結果を伴う一つひとつの動作は、すべてあなたのこの関門を通らねばなりません(具体的にどのステップで止まり、どこまで細かく止まるかはツールごとに異なります。**公式ドキュメントを参照**)。

## 二、「許可の仕組み」とは何か:それはあなたの代わりに門番をしている

上の①②③を単独で取り出すと、それが **許可の仕組み(permission)** です。

たとえてみましょう:許可の仕組みは、家のリフォーム時に呼んだ職人のようなものです。**壁を一面壊す、水道や電気を一箇所いじる、そのたびに先に「この壁は壊していいですか」と一声かけ**、あなたが頷いて初めて手を動かします。頼りになる職人は、勝手な判断で耐力壁を取り壊したりしません。許可の仕組みとは、あの「手は速いが入社したての新人同僚」(第23章)に取り付けた、この「手を動かす前に一声かける」という規律です。

それは実際どう動くのでしょうか。おおよそこうです:それが **結果を伴う** 操作をしようと身構えたとき、**立ち止まって、「私は何をするのか」をあなたに見せ、あなたの応答を待ちます**。あなたがよく目にする選択肢は、せいぜい数種類です:**この一回は同意 / この類の操作は今後すべて同意 / 不同意 / 別のやり方に変えさせる**。

> **誤解 vs 実際:** 初心者は初めて使うとき、「それがどうしてしょっちゅう立ち止まって私に尋ねるのか」にいらいらして、それが **鈍い、頼りない** のだと思いがちです。**まったく逆です――それが立ち止まって尋ねるのは、まさに「許可の仕組み」があなたを守っているからで、良いことであって、欠陥ではありません。** 本当に警戒すべきは、「何も尋ねず、黙って最後まで突っ走る」設定のほうです。「それが私に尋ねてくる」を「シートベルトが効いている」と理解すれば、もうそれを煩わしく思わなくなります。

ここで一つ、とても重要な「トレードオフ」を補足します:**許可は締めることも、緩めることもできます。** 多くのツールは、ある類の「安全な小さな操作」を「今後はもう尋ねないで」に設定でき(たとえばただファイルを読むだけ)、それで滑らかに仕事をさせられます。また「危険な操作」は常に「毎回必ず尋ねる」に留めておくこともできます。**どう調整するか、どこまで緩めるかは、あなたの権限であり、あなたの責任でもあります**――そして次の節が、この権限の最低ラインです。

## 三、どんな操作が「永遠に人の頷きを要する」か:不可逆の事は、絶対に自動で通さない

すべての操作が同じように危険なわけではありません。**間違えても取り消しが簡単な事もあれば(たとえば一文字直すなら、巻き戻せばよい)、間違えると、こぼれた水が戻らない事もあります。** 後者こそ、必ず「永遠に人の頷きに残す」べき越えてはならない一線です。

どれが「戻らない水」でしょうか。次のいくつかの類の **不可逆/高代価** の操作を覚えてください:

| 危険の類別 | なぜ危険か | 暮らしのたとえ |
| --- | --- | --- |
| **削除/上書き** | 削除したファイル、上書きされた内容は、二度と取り戻せないかもしれない | ファイルをゴミ箱に入れるのでなく、直接シュレッダーにかける |
| **対外送信** | いったん送り出したら(メール送信、メッセージ送信、公開の場への投稿)、**取り戻せず**、外に出してはいけないものを連れ出してしまうこともある | メールは「送信」を押したら追いつけない |
| **お金を使う/有料サービスの呼び出し** | 真のお金の支出、または課金の発生で、事後に返ってくるとは限らない | あなたの代わりにカードを切る |
| **本番/プロダクション環境の変更** | 影響するのは **今まさに本物の人に使われている** システムで、間違えれば事故 | 下書きを直すのでなく、生放送中に直す |

これらの類には、原則は一つだけ、しかも割引は許されません:

> **ポイント:** **削除、上書き、対外送信、お金を使う――この類の「不可逆または高代価」の操作は、必ず「人手による確認を要する」に設定し、絶対に自動で通してはいけません。** 滑らかさのために「ファイルを読む」のような大したことのない事は「あまり尋ねないで」に設定してよいのですが、**戻らない事ほど、それを「毎回必ず自分で頷く」という段にがっちり留めねばなりません**。一言で――**「危険な操作は永遠に人の頷きに残す。」** これはこだわりではなく、最低ラインです。具体的にどの操作が既定で止まり、どこまで厳しく止まるかはツールごとに異なります。**公式ドキュメントを参照**。けれども「危険な操作は人の確認を要する」というこの原則は、どこへ行っても変わりません。

なぜ「絶対に自動でない」と強調するのでしょうか。第23章のあの鉄則がここで最も命取りになるからです:**それは自信たっぷりに間違える。** 「自信たっぷりに間違えた」削除指令が、もし自動で通されてしまえば、あなたが気づいたときには、水はもうこぼれています。**それを立ち止まってあなたに尋ねさせるこの一手こそ、「それが今まさに間違えているかもしれない」場合のために残す、最後のブレーキなのです。**

## 四、巻き戻せる環境:手を動かす前に、先に「後悔薬」を敷いておく

許可の仕組みが管轄するのは「手を動かす前に一度止める」ことです。けれどもすべては止められません――すり抜ける誤りは必ずいくらか出ます。だからもう一重の保険が要ります:**それを「やり直しのきく」環境で手を動かさせること。** これは第25章で巻き戻しを扱ったときにすでに触れましたが、ここでは「なぜこれが前提なのか、どう敷くか」をはっきりさせます。

道理はとても素朴です:**それが間違えるからこそ、あなたはなおさら「間違えても取り消せる」ことを保証せねばなりません。** この二つの事をそろえる――手を動かす前に許可が一度止め、手を動かした後に巻き戻しが一度取り消す――ことで、あなたはようやく本当にそれに「仕事もできるが、暴走もしない」檻をかぶせたことになります。

この「やり直しのきく」環境はどう敷くのでしょうか。やはり二つの形態に分けます(第25章に呼応):

- **それがあなたのローカルのプロジェクトで手を動かすとき**:必ず **先にバージョン管理(Git)でプロジェクトを管理し、しかも先に「新しいブランチ」を切ってから手を動かさせます**。「新しいブランチを切る」 ＝ **それのために下書きスペースを一区画用意し**、下書きの上でこね回させ、あなたの正式稿には触れさせない。万一変更を失敗しても、下書きを捨ててきれいな状態に戻せばよい。**このステップは、それに手を動かさせる前にやっておかねばなりません**――事後に思い出しても、あの安全網はもう張られていません。
- **それが Web/クラウドの形態で手を動かすとき**:それは通常 **プラットフォームが備える隔離環境** の中で作業し、プラットフォームは普通、対応する **取り消し/破棄/リセット** のやり方を提供しています。具体的にどう取り消すか、どこまで戻せるかは、**公式ドキュメントを参照**。

> **ポイント:** 「巻き戻せる環境」を **手を動かす前提条件** として扱い、事が起きてから補う宿題ではないとしてください。ローカルなら先に Git のブランチを切り、クラウドならプラットフォームの隔離を使う――**形は違っても、目的は同じです:それを「ひっくり返しても惜しくない下書きスペース」の中で仕事をさせること。** この下書きスペースがあるからこそ、あなたは大胆に仕事を任せられます。それがなければ、「それに手を動かさせる」たびに、あなたの正式稿の上で運試しをしていることになります。

## 五、二重の保険を、どう一組に組み合わせるか

許可の仕組みと巻き戻せる環境を合わせると、それが本章の全貌です――**前後に一つずつ、二重の保険**:

```text
            ┌──────────────── 手を動かす前 ────────────────┐
後悔薬を敷く: 「やり直しのきく」環境の中で(ローカルは Git のブランチを切る / クラウドはプラットフォームの隔離を使う)
            └──────────────────────────────────────────┘
                              │
                              ▼
   ①それがリクエスト ─→ ②あなたが提案を見る ─→ ③許可(危険な操作は人手の確認が必須!)─→ それが手を動かす
                              │
                              ▼
   ④あなたが検証(diff を見る/テストを走らせる、第25章)
                              │
              ┌───────────────┴───────────────┐
          結果に満足                       結果がおかしい / 直すほど乱れる
              │                                 │
           次の一歩へ                  「後悔薬」で巻き戻し、きれいな会話を開き直してやり直す
```

- **許可の仕組み** は「事前のブレーキ」です:それが手を動かす前に、結果を伴う操作を止めてあなたに見せ、**危険なあのいくつかの類は必ずあなたが自分で頷く**。
- **巻き戻せる環境** は「事後の取り消し」です:万一誤りがすり抜けても、**ワンタッチできれいな状態に戻し**、損失をゼロにします。
- 両者は **どちらも欠かせません**:ブレーキだけで後悔薬がなければ、止まりきれなかったその一回で転覆します。後悔薬だけでブレーキがなければ、あなたは絶え間ない「取り消してやり直す」に疲れ果てます。そろえて初めて、あなたはそれの速さを享受しつつ、びくびくせずに済むのです。

最後のこの総綱を覚えてください:**AI に手を動かさせる、けれど暴走させない――頼りにするのは「危険な操作は永遠に人の頷きに残す」 ＋ 「永遠にやり直しのきく環境で手を動かさせる」、この二つです。**

## まとめ

- 手を動かすエージェントを率いる核心は、一つのループ:**リクエスト → 提案を見る → 許可 → 検証**――AI は仕事を担い、**人は肝心な節目で頷くか止まれと叫ぶことを担う**(第24章「調査→計画→実行→検収」に呼応)。
- **許可の仕組み** ＝ 手を動かす前に一声かける規律で、リフォーム職人の「壁を壊す前に先に尋ねる」のように。それが立ち止まって尋ねるのは **あなたを守っている** のであって、欠陥ではない。
- **危険な操作は永遠に人の頷きに残す**:削除、上書き、対外送信、お金を使うといった **不可逆/高代価** の操作は、**必ず「人手による確認を要する」に設定し、絶対に自動で通さない**――それは自信たっぷりに間違えるから。
- **巻き戻せる環境** は手を動かす **前提条件**:**ローカルは先に Git のブランチを切り**(それのために下書きスペースを用意)、**Web/クラウドはプラットフォーム備え付けの隔離を使う**(公式ドキュメントを参照)、それを「ひっくり返しても惜しくない」場所で仕事をさせる。
- 二重の保険をそろえる――**事前のブレーキ(許可)＋ 事後の後悔薬(巻き戻し)**――ことで「AI に手を動かさせるが暴走させない」を実現できる。各社の既定と粒度は異なる。**公式に従う**。原則は変わらない。

次章(第27章)では、「事前」のもう半分を解決します:大きな変更に手を動かす前、どうそれに **先に計画を出させ、あなたが承認してから書かせ**、タスクを **小さく分けて小さく検収** するか――これが最も手戻りを省く一歩です。

   - A. 間違っていない、しょっちゅう尋ねるのは出来が悪い

   - C. それが立ち止まるのは壊れているから

4. **[応用・誤解問題]** 「手を動かせるなら、いっそ全自動に設定して、何も私に尋ねないのが一番楽」――なぜこうするととても危険なのか?
   - A. 危険ではない、全自動が最も効率的
   - B. 削除/上書き/対外送信/お金を使うといった **不可逆または高代価** の操作が、いったん自動で通されると、あなたが気づいたときには「水はもうこぼれて」、取り戻せない
   - C. 全自動はそれを遅くする
   - D. 全自動には追加課金が必要
   > **答え B。** 「ファイルを読む」のような大したことのないものは「あまり尋ねないで」に設定してよいのですが、戻らない事ほど「毎回人手で確認」に留めねばなりません。それが「自信たっぷりに間違えた」削除指令が、もし自動で実行されれば、その一手が事故です。**危険な操作は永遠に人の頷きに残す。**

5. **[基礎・場面問題]** あなたは **ローカルのプロジェクト** で、それに比較的大きな変更を一度させようとしています。手を動かす **前** に、最も先にやっておくべき一事は?
   - A. パソコンを一度再起動する
   - B. 先に Git でプロジェクトを管理し、新しいブランチを切って、それのために「下書きスペース」を一区画用意し、万一失敗してもワンタッチで巻き戻せるようにする
   - C. まず二言ほめて、真面目にやらせる
   - D. すべての許可を切って、思い切りやらせる
   > **答え B。** 巻き戻せる環境は **手を動かす前提条件** で、事後の補習ではありません。新しいブランチを切る ＝ それを下書きスペースでこね回させ、正式稿には触れさせず、失敗したら下書きを捨てる。D はまさに逆です――大きな変更の前ほど、許可を締め、後悔薬を敷くべきです。クラウドの形態なら、プラットフォーム備え付けの隔離を使い、**公式に従う**。

6. **[応用・操作/観察問題]** 想像してください(実際にやる必要はありません):あなたがそれに「一群の古いファイルを整理して」と頼もうとしている。手を動かさせる前に、どの操作を「必ず私が自分で確認する」に設定しますか? なぜ? さらに考えてみましょう:万一それが削ってはいけないものを誤って削ったら、あなたは何を頼りに損失を取り戻しますか?
   > **観察できるはず:** 「削除」は典型的な不可逆の操作で、「人手による確認」に留めねばなりません――絶対に自動で通さない。そして取り返しのよりどころは **巻き戻せる環境** から来ます:ローカルで先に Git のブランチを切っておけば、削除前のきれいな状態に巻き戻せますし、クラウドならプラットフォーム備え付けの取り消し/リセットに頼ります(**公式ドキュメントを参照**)。二重の保険――許可が前で止め、巻き戻しが後ろで受ける――をそろえて初めて、「それに手を動かさせる」は「運を天に任せる」と等しくなくなるのです。
