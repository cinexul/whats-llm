> 〔本章で具体的な製品のインストール・コマンド・画面・能力に触れる箇所は、いずれもバージョン更新とともに変わります。一律 **公式ドキュメントを参照** してください。本章では、二種類のツールに **共通する、安定した** メンタルモデルとワークフローだけを扱います――この部分は古くなりません。一字一句のインストールとコマンドは第25章と付録 C を参照。〕

第19章で、私たちは Agent――「自分で手を動かす」AI を知りました。本章に登場する **Claude Code** と **Codex** は、まさに「プログラミング」という仕事をする Agent です。あなたが普通の言葉で一つの用件を伝えると、Agent は **自分でプロジェクトのファイルを読み、コードを変更し、コマンドを実行します**。コピー&ペースト用の文章を渡すだけではありません。

「プログラミング」という言葉に身構えないでください。**本章は、あなたがコードを書けることを求めません。** ここで学ぶのは、「こうした手を動かす助手をどう連れて仕事をするか」――どんな仕事を任せ、できているか・いないかをどう見極め、つまずいたときどう収拾するか――です。このワークフローは、パソコンの分かるふつうの人なら誰でも身につけられます。そして身につけたあとには、「AI に自分の代わりに仕事をさせる」ことへの、まったく新しい手応えが生まれます。

## 一、それらはいったい何か

Claude Code と Codex は、ある特定の Web ボタンではなく、**あなたのプロジェクトの上で手を動かすプログラミングエージェント(coding agent)** という一群のものです。あなたが普通それらに触れるのは、**ターミナル(コマンドライン)、コードエディタ(IDE)、Web のタスク、クラウド環境** などの中です――**あなた自身のマシンで動くものもあれば、クラウドで動くものもあり、具体的な形態はそれぞれ異なり、しかも更新され続けています。公式ドキュメント/製品の公式な形態を参照してください**。共通点は次のとおりです。

- **あなたのプロジェクトを見られる:** 自分でフォルダを見て回り、コードを読み、現状を把握します――あなたが貼り付けた数行だけしか知らない、ということがありません。
- **手を動かせる:** ファイルを直接変更し、コマンドを実行し(たとえばテストを走らせる)、必要なときはネットで資料を調べます。
- **何ステップも続けて進められる:** あなたが与えた目標に向かって、自分で「見る→直す→試す→また直す」のループを回します。

> **ポイント:** Claude Code は Anthropic 発で、コアは Claude。Codex は OpenAI 発――この「提供元/コア」の帰属は安定した事実です。しかし **それらの具体的な形態(CLI/IDE プラグイン/Web/クラウド)、インストール方法、コマンド、能力はそれぞれ異なり、しかもどれも急速に更新されています**――だから本章では「どのキーを押す、どのコマンドを打つ、どこで動かす」は書きません(それらは変わりやすく、第25章・付録 C・公式ドキュメントの領分です)。本章で扱うのは、それらが **共有している仕事のやり方** です。一度学べば、両方に通用し、しかも古くなりにくいものです。

## 二、「Web チャット」との根本的な違い

| | Web/アプリのチャット(製品レイヤー) | プログラミングエージェント(Agent レイヤー) |
| --- | --- | --- |
| コードの扱い方 | 文章を渡され、あなたが自分でコピーして使う | **あなたのプロジェクトのファイルを直接変更する** |
| どれだけ知っているか | あなたが貼り付けた内容しか知らない | 自分で **プロジェクト全体を探索** して現状を把握する |
| 進め方 | あなたが一言聞き、一言答える | 目標に向かって **自分で何歩も先まで進む** |
| 間違えたときの結果 | せいぜい答えが違うだけ。使わなければよい | 本当にあなたのファイルを変更してしまう――**だから巻き戻せること** |

最後の行が肝心です。Agent は **本当にあなたのものを動かす** からこそ、「やり直しのきく環境で使う」ことは、こだわりではなく前提なのです(後の第五節で詳しく述べます)。

## 三、メンタルモデル：「優秀だが入社したての新人同僚」と考える

本書はこのたとえを繰り返し使ってきましたが、ここで最もしっくりきます。**Claude Code / Codex は、とても優秀で、手が速い、けれど入社初日の同僚のようなもの** です。

- **動きが機敏:** 任された力仕事を、速く、たくさんこなします。
- **あなたのプロジェクトの流儀を知らない:** あなたたちのルール、命名、地雷を知りません――あなたが教える必要があります(第28章の「入社のしおり」がこれをする道具です)。
- **自信たっぷりに間違える:** まったく存在しない書き方を使うこともあります(これが第9章の「ハルシネーション」で、コードを書くときも同じように起きます)。
- だから――**あなたが、この新人を率いる側** です。仕事は任せられますが、**仕事を振ることと、検収することは、あなたの責任** であり、最終的な署名もあなたのものです。

> **ポイント:** この一文は一生覚えておいてください。**「分からないもの、検証していないものを、通すな。」** 新人がやり終えたことは、正しくできたこととは限りません。あなたは一行一行を書ける必要はありませんが、「この件はうまくいったか」を判断できる必要があります――本章第四節のワークフローは、その判断を実地に落とし込むためのものです。

## 四、核心ワークフロー：調査 → 計画 → 実行 → 検収

うまくいくタスクのほとんどは、同じ四ステップのループをたどります。**中間の二つの「関門」を人が守り**、残りを AI に任せます。

```text
   あなたが目標を与える
        │
        ▼
 ①調査 ──→ ②計画 ──→【人が承認】──→ ③実行 ──→ ④検収 ──→【人が許可】
(プロジェクトを読む)(案を出す)  ↑関門1   (ファイルを変更) (diff確認/テスト実行) ↑関門2
```

**① 調査(Agent がやり、あなたは一目見る)**
いきなり「変更を始めて」と言ってはいけません。まず **調査させます**。「この件をやるには、関連するコードはどのファイルにある? 今はどう実装されている?」 すると Agent はプロジェクトを見て回り、場所付きの説明をくれます。あなたはそれにざっと目を通し、見当違いの場所を探していないか確認します。
> 一言の指示の例:*「まだ手を動かさないで、調査だけして。ページに検索ボックスを追加するには、どのファイルが関わって、今はどう実装されている?」*

**② 計画(Agent がやり、あなたが承認する)――関門 1**
「どう変更するつもりか」を、先に **計画として書かせます**:どのファイルを触るか、新しいものを足すか、何ステップに分けるか。**あなたがこの計画を読み、承認して初めて、Agent は手を動かし始めます。** このステップは数十秒で済みますが、「数十分も本筋からそれていた」という手戻りの大半を防げます。多くのツールは「先に計画を出し、承認してから手を動かす」モードを内蔵しています(第27章で詳述。**具体的な呼び名やスイッチは公式ドキュメントを参照**)。

**③ 実行(Agent がやり、あなたはいつでも止められる)**
承認後、Agent は変更を始めます。その間、「結果を伴う操作」(ファイル変更、コマンド実行)をするたびに、たいてい **立ち止まってあなたに許可を求めます**(第26章の「許可の仕組み」)。本筋からそれたと感じたら、**いつでも中断**し、用件をはっきり伝え直してから続けさせます。

**④ 検収(あなたがやる)――関門 2**
これが **あなたの最も重要なステップ** で、動作は三つだけです(第25章で詳説)。
- **diff を見る:** diff とは「変更前後の対比」のことで、「どの行を足し、どの行を削ったか」をはっきり示してくれます。**実際に何を変更したか**、触ってはいけない箇所を動かしていないかを確認します。
- **テストを走らせる:** テストを実行させ、**もともと正常だった機能を壊していないか** を確かめます。
- **はっきり説明できるか:** 「この部分は何をしているか」を一言で言えるか試します。言えなければ、戻って「この部分は何をしているの?」と聞きましょう――**分からないものはマージしない**。

> **なぜこの二つの関門が必須なのか:** プログラミングエージェントの二つの特性が重なるからです:**間違える + 手を動かせる**。この二つが同時に成り立つ以上、「手を動かす前に計画を見る、手を動かしたあとに検収する」は選択肢ではありません。この二つの関門さえ守れば、残りの力仕事は安心して任せられます。

## 五、初めて動かすとき、何を期待すべきか

「理想の AI」に脅かされたり失望したりしないよう、期待値を少し下げておきましょう。

1. **まず「やり直しのきく」環境を整える。** **Agent があなたのローカルのプロジェクトで手を動かすときは**、必ず **バージョン管理(Git)** で管理されたプロジェクトの中で、しかも **新しいブランチを切って** から手を動かさせてください(第26章で手取り足取り教えます)。こうすれば、万一ぐちゃぐちゃになっても、ワンタッチできれいな状態に戻せます――これがあなたの最大の安全網です。(Web/クラウドの形態を使っている場合は、プラットフォーム側に対応する隔離・巻き戻しの仕組みがあるのが普通です。公式ドキュメントを参照してください。)
2. **まず「しばらく読む」。** Agent はプロジェクトを見て回る時間が必要です。固まったと思わないでください。
3. **頻繁に確認を求めてくる。** これは良いことです。「許可の仕組み」があなたを守っているのであって、Agent が鈍いわけではありません。
4. **間違えるし、本筋からそれることもある。** 正常です。そんなときは――**ぐちゃぐちゃの中で無理に直そうとしない**。最もすっきりした損切りは、**きれいな状態に巻き戻し、新しい会話を開き、「さっき踏んだ落とし穴」を前提として伝え直す** ことです(第13章・第30章)。乱れたコンテキストの中で一言一言追いながら直すと、たいてい直すほど悪化します。
5. **ネットにつながなければ、最新のことは知らない。** とても新しい技術が関わるときは、用件の中で **バージョンをはっきり伝える** か、資料を貼り付けてください(第9章の知識カットオフ)。

> **実際にどう出くわすか:** 初心者が最もよくやらかすのは、「大きな機能を一気に全部作らせる」ことです。変更が大きいと、あなたは一行ずつ検収しようがなく、問題が起きても巻き戻しにくくなります。**正しい姿勢はタスクを小さく分けること** です――小さな一かたまりを作り、小さく検収し、一度コミットして、次の一かたまりへ(第27章)。小さく刻んでゆっくり進むことが、かえって最も速い道です。

## 六、安全と費用、初日から心に留めておく

- **鍵/パスワード/本物の個人情報** を会話に貼ったり、コードに書き込んだりしないでください――それは外部に送信されます(第37章)。
- この種のツールは **トークン単位で課金** されます。読ませるファイルが多いほど、会話が長いほど、減りが速くなります。タスクを絞り、会話をすっきり保てば、お金も節約でき、品質も上がります(第3章・第38章)。
- 生成されたコードは **あなたの責任** です。マージ前に検証し、それに責任を持つ必要があります(第39章)。

## 七、Claude Code と Codex：どこが同じで、どこが違うか

| | 同じ(本章の主眼、安定) | 違う(**公式ドキュメントを参照**、変わる) |
| --- | --- | --- |
| 位置づけ | どちらも「あなたのプロジェクトで手を動かす」プログラミングエージェント | 提供元、コアモデル、**動作形態(ローカル/クラウド/Web)** が異なる |
| ワークフロー | どちらも「調査→計画→実行→検収」ループが当てはまる | 具体的なコマンド、ショートカット、画面はそれぞれ違う |
| 安全 | どちらも許可の仕組み + 巻き戻せる環境を備えるべき | 既定の振る舞い、許可の粒度が異なる |
| 入門 | どちらもまず Git のブランチの中で小さく試す | インストール方法、ログイン方法が異なる |

> **ポイント:** 本章で学んだワークフローは、**どのツールに替えても通用します**――それは「間違えうる Agent をどう率いるか」という不変の道理から来ているのであって、ある製品のボタンから来ているのではないからです。「どうインストールするか、何を打つか」の具体については、第25章と付録 C を見て、**公式ドキュメントを最新の拠り所** にしてください。

## 八、よくある誤解、まとめて訂正

| 誤解 | 実際 |
| --- | --- |
| AI なのだから書いたコードは正しいはず。そのままマージ | Agent は自信たっぷりに間違える。diff は見る、テストは走らせる、分からないものはマージしない |
| 大きな機能を一度に全部作らせるのが一番楽 | 大きな変更は検収しにくく、巻き戻しにくい。小さく分け、小さく検収するのが最も速い |
| Agent はプロジェクト全体と昨日の会話を覚えている | モデル本体は「毎回プロジェクトを読む + 今回のコンテキスト」で動き、既定では会話をまたいだ記憶はない。ただし多くのツールは **製品レイヤー** で「プロジェクト記憶」(CLAUDE.md/プロジェクトコンテキストファイルなど)を提供し、会話をまたいでプロジェクトの流儀を覚えさせられる――それは製品の機能であって、モデル本体の記憶ではない(第3章・第28章) |
| プログラミングができないと使えない | 一行一行を書ける必要はないが、「仕事を振る + 検収の三つの問い」はできる必要がある。敷居は判断にあり、コードを打つことにはない |
| 本筋からそれたら、追いかけてずっと直し続ける | 巻き戻して + 新しい会話でやり直すべき。乱れたコンテキストの中で無理に直すと、たいてい悪化する |

## まとめ

- Claude Code / Codex は、**あなたのプロジェクトの中で手を動かす** プログラミングエージェントの一群で、ターミナル/エディタの中で動く。**具体的な形態はそれぞれ異なり、しかも更新され続けている。公式ドキュメントを参照**。
- Web チャットとの根本的な違い:**ファイルを直接変更し、自分でプロジェクトを探索し、何歩も先まで進む**――だから必ず **巻き戻せる** 環境で使う。
- メンタルモデル:**優秀だが入社したての新人同僚**。仕事は任せられるが、**仕事を振ることと検収することはあなたの責任**。
- 核心ワークフローは **調査 → 計画 →【人が承認】→ 実行 → 検収 →【人が許可】**。人が二つの関門を守る:計画の承認と、変更の検収(diff を見る、テストを走らせる、はっきり説明できる)。
- 初めて使うとき:まず Git のブランチを切り、タスクを小さく分け、それたら巻き戻してやり直し、鍵は絶対に外部に貼らない。
- このワークフローは **具体的なツールに依存しない**。「間違えうる Agent をどう率いるか」から来ているからだ――一度学べば、どこでも通用する。

## 小テスト

> 全 6 問。概念・誤解・場面・操作の 4 種類をカバーしています。まず自分で考えてから、引用ブロックの答えと解説を見てください。

1. **[基礎・概念問題]** プログラミングエージェント(Claude Code / Codex など)と、Web チャット AI の最も根本的な違いは？
   - A. 画面がきれい
   - B. あなたのプロジェクトのファイルを直接読み書きし、コマンドを実行できる。コピー用の文章を渡すだけではない
   - C. 間違えない
   - D. モデルを使っていない
   > **答え B。** チャット AI は「あなたが行き来してコピペする」もの。プログラミングエージェントは「あなたのプロジェクトの中で直接手を動かす」もの。これこそが、効率的でありながら「許可の仕組み」で見張る必要がある理由です。

2. **[基礎・概念問題]** 「調査 → 計画 → 実行 → 検収」のループのうち、人にとって最も肝心な仕事はどのステップ？
   - A. 調査
   - B. 計画の承認 + 検収(diff を見る、テストを走らせる)
   - C. 実行
   - D. 人は関わらなくてよい
   > **答え B。** AI に仕事をさせるとき、人は二つの関門を守ります:**手を動かす前に計画を承認し、手を動かしたあとに変更を検収する**。中間の力仕事は AI に任せます。

3. **[基礎・誤解問題]** 「AI なのだから書いたコードは正しいはず。そのままマージすればよい」――どこが間違い？
   - A. 間違っていない
   - B. Agent は自信たっぷりに、間違ったもの/存在しないものを書く。**分からないもの、検証していない変更はマージすべきでない**
   - C. Agent の書くものは永遠に動かない
   - D. マージには課金が必要
   > **答え B。** ハルシネーション(第9章)はコードを書くときも同じように起きます――存在しない関数をでっち上げたり、バージョン違いの書き方を使ったり。diff は見る、テストは走らせる、分からなければまず聞いてはっきりさせましょう。

4. **[応用・誤解問題]** 「大きな機能を一度に全部作らせるのが一番楽」――なぜしばしば、かえって手間が増えるのか？
   - A. 増えない。大きいほど楽
   - B. 変更が大きいほど本筋からそれやすく、しかも大きすぎて一行ずつ検収しようがなく、問題が起きても巻き戻しにくい
   - C. Agent は一度に一つのファイルしか変更できない
   - D. 大きなタスクは追加課金が必要
   > **答え B。** 大きな変更 = 大きな死角。タスクを小さく分け、先に計画を見て、小さく検収するほうが、かえって最も速いのです(第27章に呼応)。

5. **[基礎・場面問題]** コードを直させたら、Agent が本筋からそれて、直すほど乱れてきた。最善の損切りの動作は？
   - A. 追いかけて一言一言、直し続ける
   - B. バージョン管理で変更をきれいな状態に巻き戻し、新しい会話を開いて、「踏んだ落とし穴」を前提として伝え直す
   - C. プロジェクトを削除して作り直す
   - D. パソコンの電源を切る
   > **答え B。** 乱れたコンテキストの中で無理に直すと、たいてい直すほど歪んでいきます。**巻き戻し + 教訓を携えたきれいな会話でやり直す** ことが、ほぼいつも速いのです(第13章・第30章に呼応)。最初に Git から始め、先にブランチを切るのは、まさにこの瞬間のためです。

6. **[応用・操作/観察問題]** ツールを実際にインストールする必要はありません。あなたが読んで分かるコードの変更(または第25章の diff の例)を一つ見つけ、三つの問いに答える練習をしましょう――**何を変更したか? 別のものを壊さないか? この部分が何をしているか、一言で説明できるか?** 三つ目に答えられなければ、戻って「この部分は何をしているの?」と聞くべきです。
   > **観察できるはず:** 「検収」は神秘的なものではなく、この三つの問いだけです。「分からなければ通さない」という筋肉の記憶を身につけることが、どんなコマンドを覚えるよりも大切です。具体的なツールのインストールとコマンドは **公式ドキュメントを参照**(第25章と付録 C を参照)。
