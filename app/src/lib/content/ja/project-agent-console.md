前の三つのプロジェクト（第31・32・33章）では、AI は基本的に「一言いえば、一つこなす」というやり方で、あなたに付き添って作業してきました。この章では趣向を変えます。**エージェント・コンソール** を一つ作るのです――目標と一連の流れを与えると、AI が **自分で続けて何ステップも走り**、あなたは横でその「調査→実行」を見守り、肝心なところでだけ手を出します。

これこそ、第19章で出てきた「自分で手を動かす AI」が、初めて本当にあなたの手元に降りてくる瞬間です。そして、まさに自分で手を動かすからこそ、この章の主役は「どうすればもっと激しく走らせられるか」ではなく、その逆――**まず「ブレーキ」をきちんと取り付ける** ことです。安心して使える Agent を作るうえで難しいのは、走らせることでは決してなく、**どのステップは絶対に自分の判断で進めさせてはいけないか** を考え抜くことだと、いずれ気づくはずです。

> 〔本章で扱うのは「自分で走る Agent をどう連れて仕事をするか」という汎用のやり方であって、特定のプラットフォームに縛られません。どの製品で、どんな画面を使って組むか、ツールをどう設定するかは各社で異なり、しかも更新が速いので、一律 **公式ドキュメントを参照** してください。〕

## 一、準備

手を動かす前に、まず「自分はいったい何を作ろうとしているのか」をはっきりさせましょう。いきなり機能を盛らないでください。

- **「多ステップだが、どのステップも危なくない」小さな目標を選んで練習する。** 初めて Agent を作るなら、目標は小さく、安全に。良い例：「あるフォルダの中の数件の文書を読み、それぞれ一言の要約を書き、一覧にまとめる」――これには何ステップも必要ですが（一つずつ読む、一つずつ要約する、最後にまとめる）、**どのステップも、何も削らず、何も送らず、お金もかかりません**。悪い例：「メールボックスを自動で片づける」「顧客に自動で返信する」――最初からこんな取り返しのつかない作業に手を出すのは、初心者ドライバーをいきなり高速道路に放り込むようなものです。
- **この作業がどのステップに分かれるかを考え抜く。** まず頭の中で（または紙の上で）、目標を「調査→やる→確認する」のいくつかのステップに分けておく必要があります。自分でも説明できない流れは、AI に任せようがありません（これは第14章の「万能ループ」の実戦版です）。
- **「やり直しのきく」環境を用意する。** Agent は本当に手を動かすので、**触れるものは、できるだけすべて元に戻せるもの** にしておきます。ローカルのファイルが関わるなら、Git で管理された、独立したフォルダの中で試します（第24・27章に呼応）。プラットフォーム付属の Agent を使う場合は、その隔離/サンドボックスのやり方を確認してください――**公式ドキュメントを参照**。
- **最も大事な一つ：「必停リスト（必ず止まるリスト）」を作る。** 走らせる前に、「どのステップは自分が承認して初めて実行できるか」を書き出しておきます。このリストこそこの章の核で、次の第二節で作り方を一つずつ説明します。

> **ポイント：** Agent を作る準備の仕事は、半分は技術（環境、ツール）、もう半分は **判断**（目標は十分に安全か、流れはきちんと分けたか、ブレーキはすべて挙げたか）です。後者の半分こそが、あなたが横転するかどうかを本当に決める部分なのです。

## 二、一歩ずつやる

以下の流れは、特定の製品に依存しません。どのプラットフォームで組むにせよ、この考え方を当てはめればよいだけです（具体的なボタン、コマンドは **公式ドキュメントを参照**）。

### ステップ 1：普通の言葉で、目標 + 一連の流れを与える

目標を一つ放り投げて手を離す、ということをしてはいけません。「どんなステップで進めるか」も一緒にはっきり伝えます。プログラミング未経験者向けのリクエストは、だいたいこんな形になります。

```text
目標:「整理待ち」フォルダの中の数件の文書を、それぞれ一言の要約にし、一覧にまとめる。

次の流れで進めてください:
① まず調査:フォルダの中にどの文書があるか一覧にして、まだ手を動かさず、ひと目見せて。
② 次に一つずつ読み、一つずつ一言の要約を書く。
③ 最後に一覧にまとめて私に渡す。

ルール(とても重要):
- あなたは「読む」ことと「一覧を作る」ことしかできません。削除も、原本の変更も、何かを外に送ることも禁止です。
- 調査が終わったら、まず止まって、一覧が合っているか私に見せて。私が「続けて」と言ってから先へ進んで。
- 判断に迷うところは、私に代わって推測せず、止まって私に聞いて。
```

この文章では、**半分の分量が「何をしてはいけないか」と「どこで止まるか」に使われている** ことに注意してください。これはくどいのではなく、あなたが取り付けるブレーキそのものなのです。

### ステップ 2：まず「必停リスト」を作る――この章の核心

「開始」を押す前に、次のような動作を **あらかじめ** 「人による確認が必須」と印を付けておきます。判断基準は一言だけです：**このステップは元に戻せるか？ 後に残る結果があるか？**

| 自分の判断でやらせてよいもの | 止まって承認を待つべきもの |
| --- | --- |
| ファイルを読む、整理する、分類する、下書きを作る、一覧を作る | **お金を使う**：回数ごとに課金されるサービスの呼び出し、注文、利用枠の消費 |
| どうでもいい小さな選択肢の中から一つ選ぶ | **データ削除/上書き**：あなたの原本を削除・クリア・書き換える |
| 一ステップ終えるごとに、報告してから次へ進む | **外部への送信**：メール送信、提出、公開、外部インターフェースの呼び出し |
| 内容をある形式から別の形式へ変換する | **判断に迷い、あなたの裁定が必要** な場面 |

この表は、第14章の「ブレーキを握る」と第26章の「許可の仕組み」を、プロジェクトに落とし込んだものです。**リストを作るタイミングが肝心です。AI が手を動かす前に作るのであって、問題を起こしてから後付けするのではありません。**

### ステップ 3：走らせる。あなたは「調査→実行」の過程を見張る

開始を押したら、**席を立たないでください**。Agent の価値は自分で前へ走ることですが、初めて使うときは、その一歩一歩を目に焼き付けます。

- **その「調査」が合っているか見る**：正しいファイルを見つけ、目標を正しく理解しているか？ 最初のステップでずれると、後は間違った前提の上で、ますます「きれいに」作り込んでいきます（第14章で触れたこの落とし穴です）。
- **「必停点」に来たとき、本当に止まるか見る**：確認すべきところに来たら、止まってあなたに聞くはずです。もし止まらず、取り返しのつかない動作をそのまま進めたら――あなたのブレーキが設定できていない証拠なので、すぐ止めさせます。
- **ずれたと感じたら、いつでも中断する**：はっきり言い直してから続けさせる。あるいはいっそ巻き戻してやり直す。

### 結果を見る/検収する

一巡走り終えたら、次の三つで検収します（第27章の「検収三問」に呼応）。

1. **結局、何をしたか？** 渡してきた最終的な一覧だけでなく、途中でどんなものに触れたかも見ます――触らせていないところに手を出していないか。
2. **止まるべきところで止まったか？** 振り返って確認します。先ほどの「必停点」のいくつかで、ちゃんと止まって待ったか。これは結果がきれいかどうかより大事です。
3. **結果を信じられるか？** 生成した要約の、肝心な情報をいくつか自分で抜き取り検査します――その自己点検は鵜呑みにできません（第14章）。きれいなことは正しいことと同じではありません。

三つすべてに通って初めて、今回の Agent は「走り通したうえに、暴走もしなかった」と言えます。

## 三、よくある落とし穴

| 落とし穴 | どういうことか/どう避けるか |
| --- | --- |
| **いきなり「取り返しのつかない」大きな仕事を与える** | 最初からメール削除、メッセージ送信、お金を動かす――事故が起きたら本当に事故です。まず「読むだけ・書かない」安全な目標で練習を |
| **目標だけ与えて、ブレーキを与えない** | 「メールボックスを片づけて」とだけ言い、「削除する前に私に聞いて」と言わない――そのまま最後まで削り続けるかもしれません。**ブレーキは目標と一緒に与え、しかも AI が手を動かす前に書く** |
| **開始を押したら席を立つ** | Agent は自分で走りますが、初めて使うときは全工程を見張る必要があり、とりわけ「必停点で本当に止まるか」を見ます。完全な放任こそ、第19章が繰り返し警告する危険な使い方です |
| **「確認したと言っている」のを本当に確認したと思い込む** | その自己点検もやはり「続きを返している」だけで、客観的な目があるわけではありません（第14章）。肝心な情報は自分で再確認を |
| **ずれているのに、混乱の中で無理に直す** | 直すほどずれていきます。いっそ **きれいな状態に巻き戻し、新しい会話を開いて**、「さっき踏んだ落とし穴」を前提として伝え直しましょう（第13・30章） |
| **「ステップ数が多い = 強い」と思い込む** | 一気に十数ステップ走らせると、あなたは一歩ずつ見張りようがありません。**歩幅を小さく、節目を密に** 取るほうが、かえって安定します |

## 四、もう一歩進める

最初の安全な小さな Agent をうまく走らせたら、次のように **慎重に** 広げられます――各ステップで「ブレーキは足りているか」を改めて問いながら。

- **ツールやデータソースを一つ二つつなぐ**（第22章の MCP/Skills の考え方で）。より多くを読み、より多くのことができるようになります。ただし、次章と第36章が繰り返し説くことを忘れずに。**新しい能力を一つつなぐたびに、間違いと攻撃の入口が一つ増える** ので、必要なものだけを、必要最小限だけつなぐこと。
- **よく走らせる流れを固定化する**：ある「調査→やる→確認」の一式を繰り返し使うなら、それを一つの固定された流れにまとめられます（Skills の考え方、第22・36章）。次回は一言で呼び出せます。
- **「必停点」を少しずつ緩める――ただし、とても抑制的に**：あるステップの安全性に完全に確信があり、しかも本当に元に戻せるときに限って、自分でやらせることを検討します。**お金を使う、データを削除する、外部へ送信する、この数項目は、永遠に「必停」に残すことをおすすめします**――どれだけ慣れても。
- **「どう仕事をすべきか」の説明書を書く**：このプロジェクトのルール、踏んではいけない地雷、触れてよいもの・いけないものを書いて渡します（プロジェクトコンテキストの考え方、第28章）。会話をまたいでもルールを守らせられます。

> **ポイント：** エージェント・コンソールを作る本当の力量は、「どれだけ多くのことをやらせられるか」ではなく、「**どのステップを自分の判断で進めさせてはいけないか、あなたが前もって考え抜いたか**」にあります。能力は AI がもたらすもので、安全はあなたが設計するものです。一言、一生覚えておいてください。**まずブレーキを取り付け、それからアクセルを踏む――お金を使う、データを削除する、外部へ送信する、この三種類の動作は、いつでも必ずあなたに先に確認させる。**

## まとめ

- このプロジェクトは、あなたが初めて本当に **自分で続けて何ステップも走る Agent** を連れて仕事をするものです（第19章）。核心で練るのは「激しく走らせる」ことではなく「走り続けさせる」ことです。
- 準備段階で最も肝心なのは技術ではなく **判断** です：「多ステップだが、どのステップも安全」な目標を選び、流れをきちんと分け、「必停リスト」を作り、巻き戻せる環境を備える。
- やるときは：**普通の言葉で、目標 + 流れ + ブレーキを与え**、その「調査→実行」の過程を見張り、とりわけ「必停点で本当に止まるか」を見ます。
- 検収三問：結局何をしたか、止まるべきところで止まったか、結果を信じられるか。**止まるべきところで止まらなかったことは、結果がきれいかどうかより致命的です。**
- **お金を使う、データを削除する、外部へ送信する** この三種類の取り返しのつかない動作は、永遠に「人による確認が必須」に設定することをおすすめします（第14・19章に呼応）。具体的なプラットフォームと組み方は **公式ドキュメントを参照**。

次章では、別の「AI のつなぎ方」に切り替えます：出来合いの画面を使うのではなく、あなた自身のプログラムに **API** を通してモデルを直接呼び出させます。

## 小テスト

> 全 6 問。概念・誤解・シナリオ・操作の 4 種類をカバーしています。答えと解説は各問の下の引用ブロックにあります――まず自分で考えてから、照らし合わせてください。

1. **[基礎・概念問題]** エージェント・コンソールを作ることと、前のプロジェクトの「あなたが一言聞き、一言答える」との最も根本的な違いは？
   - A. 画面がもっと派手
   - B. 一つの目標に向かって **自分で続けて何ステップも走り**（調査→実行→確認）、あなたは肝心なところでだけ見張る
   - C. 使っているモデルがもっと強い
   - D. もう間違えない
   > **答え B。** Agent の要は「自分で前へ一区間進む」こと（第19章）であって、もっと賢いとか、間違えないということではありません。自分で走り、しかも止まって聞かないこともあるからこそ、この章は「ブレーキを取り付ける」ことを最優先に置いたのです。

2. **[基礎・概念問題]** この章が繰り返し強調する「ブレーキ」（必停リスト）の本質は何か？
   - A. Agent を少し遅く走らせること
   - B. **あらかじめ**「お金を使う、データを削除する、外部へ送信する、裁定が必要」といった、取り返しのつかない/後に残る結果のあるステップを、「人による確認が必須」と印を付けること
   - C. Agent をシャットダウンするボタン
   - D. Agent を読むだけにして考えさせないこと
   > **答え B。** ブレーキは速度を落とすためではなく、**安全な範囲を定める** ためのものです。範囲内では手放しで走らせ、取り返しのつかない/後に残る結果のある動作にぶつかったら必ず止まって承認を待つ（第14・19章に呼応）。判断基準は一言。このステップは元に戻せるか？ 後に残る結果があるか？

3. **[基礎・誤解問題]** 「目標をはっきり伝え、開始を押したら、もう他のことをしに行ける」――初めて Agent を作るとき、この考えはどこが間違いか？
   - A. 間違っていない。Agent はまさに手を離すためのもの
   - B. 初めては全工程を見張る必要があり、とりわけ「必停点で本当に止まるか」を見る。完全な放任こそ危険な使い方
   - C. 目標を与えるべきでないのが間違い
   - D. Agent はそもそも自分で走れないのが間違い
   > **答え B。** Agent が自分で走るのは確かですが、「止まるべきところで本当に止まるか」は、あなたが自分の目で確認してこそ安心できます。止まらず取り返しのつかないことをそのまま進めたら、ブレーキが設定できていない証拠なので、すぐ止めさせます（第19章）。完全な丸投げは、本書が繰り返し警告する危険な姿勢です。

4. **[応用・シナリオ問題]** Agent に顧客資料のフォルダを整理してもらっています。次のどのステップが **最も** 「止まって聞く」に設定すべきか？
   - A. 資料を会社名順に並べる
   - B. 表を一つの形式に統一する
   - C. 「重複で余計だ」と判断した顧客レコードを **そのまま削除する**
   - D. 内容を表から一段落の文章説明に整理する
   > **答え C。** 削除は **取り返しがつかず**、重大な結果のある動作で、まさに必停点です。A・B・D はいずれも範囲内で手放してよい低リスクの整理です。区別の基準は常に：このステップは元に戻せるか、後に残る結果があるか（第14章に呼応）。

5. **[応用・シナリオ問題]** Agent が途中で明らかにずれ始め、やるほど乱れてきました。最善の損切りの動作は？
   - A. 混乱の中で一言ずつ追いかけて直す
   - B. きれいな状態に巻き戻し、新しい会話を開いて、「さっき踏んだ落とし穴」を前提として伝え直す
   - C. パソコンをシャットダウンする
   - D. まずこの一巡を終わらせてから、最後に考える
   > **答え B。** 汚染されたコンテキストの中で無理に直すと、たいてい直すほどずれていきます（第13・30章）。だから準備段階で「巻き戻せる環境」を備えるのです――まさにこの瞬間、すっきりとやり直せるように。C は問題を解決しません。D は間違った方向へますます走らせる放任です。

6. **[基礎・操作/観察問題]** 本当にプラットフォームを組む必要はありません――まず、あなたが Agent にやらせそうな多ステップの小さな用件を一つ選び（例：「三件の文書を読み、それぞれ一言の要約をしてまとめる」）、それを「調査→やる→確認」のいくつかのステップに分け、さらに「必停リスト」を作ってください。どのステップはあなたの承認が必須か？（具体的な組み方は **公式ドキュメントを参照**。）
   > **観察できるはず：** 安全な練習用の目標では、「必停点」が一つもないかもしれない（全部、読むだけ・送らない・削らない）――これこそ初めて選ぶべき目標です。いったん目標に「削る・送る・お金を使う」が現れたら、対応するステップは必ず「必停リスト」に入れます。**まずブレーキを考え抜いてから、走らせる話をする**――これが Agent を作る正しい順序です（第14・19章に呼応）。
