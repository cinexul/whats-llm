〔本章涉及具体产品的命令、界面、按钮之处，都会随版本更新而变化，一律**以官方文档为准**。本章只讲**共通、稳定**的把关动作——这部分不会过时。〕

如果整个第四部分你只来得及学会一章，那就是这一章。

第23章我们说定了一条铁律：**它会动手、也会犯错，所以"看不懂、没验证的东西，不要放行"。** 可问题来了——**一个不会写代码的人，怎么"验证"它写的代码？**

好消息是：**验收不等于看懂每一行代码。** 你需要的，是四件再朴素不过的事——**看懂改动（diff）、跑一遍测试、知道怎么一键反悔（回滚）、用三个问题做最终验收**。这四件事，外行都学得会，而且学会之后，你就真正握住了"让 AI 替你干活"的方向盘。

我们一件一件来。

## 一、看懂 diff：一场"找不同"游戏

它改完代码，第一件事不是问它"好了吗"，而是**让它把改动以 diff（差异／差分）的形式给你看**。

**diff，就是"改动前后的对比"。** 别被这个词唬住，它的本质特别简单——就像两张几乎一样的图片让你玩"找不同"，diff 帮你把"不同的地方"**直接圈出来**，你不用拿原稿一行行比对。

它长这样（这是**通用示意**，具体颜色、符号在不同工具里略有差别，**以官方文档为准**）：

```diff
  function greet(name) {
-   return "你好，" + name;
+   return "你好，" + name + "！欢迎回来";
  }
```

读法只有三条，记住就够用：

- **绿色、前面带 `+` 的行 ＝ 新加的**（上面那行"加了欢迎语"）。
- **红色、前面带 `-` 的行 ＝ 删掉的**（上面那行"原来的写法被替换了"）。
- **没颜色、没符号的行 ＝ 没动**，只是放在那里给你看上下文。

就这么简单。一次改动，可能涉及好几个文件、好几处增删，diff 会**一处一处列清楚**："在哪个文件、加了哪几行、删了哪几行。"你的任务不是看懂每行代码的语法，而是回答两个外行也答得上来的问题：

1. **它动的地方，对不对？** 你让它"改首页的标题颜色"，结果 diff 里它动了"用户登录"那段代码——这就**明显不对**，哪怕你看不懂那段代码，也知道它跑偏了。
2. **它动的范围，是不是大得离谱？** 你只让它改一个小标题，diff 却显示它改了十几个文件、几百行——**警铃该响了**。要么它误解了任务，要么它"顺手"动了一堆不该动的地方。

> **要点：** **看 diff，看的是"动了什么、动了多大"，不是"每行代码什么意思"。** 这就像你不是厨师，但你能看出"我点的是一份沙拉，端上来的怎么是一锅红烧肉"——**对不对、量对不对，外行完全判断得了。** 这正是 diff 这个工具最大的价值：它把"它到底改了什么"摊开给你看，让你不必信它一面之词。

## 二、跑测试：确认"没把原来好好的东西弄坏"

看懂了"它改了什么"，下一个要命的问题是：**它在添新功能的时候，会不会把原来好好的功能弄坏了？**

这在编程里太常见了，常见到有个专门的说法叫"按下葫芦浮起瓢"——你让它修 A，它把 A 修好了，却不小心碰坏了 B。**而 B 出问题的地方，可能根本不在你看的那段 diff 里**，光看 diff 发现不了。

这时候就轮到**测试**上场了。

**"测试"是一批预先写好的小检查，专门用来自动确认"这些功能还正常不正常"。** 你可以把它想成产品出厂前的**质检流水线**：每个产品过一遍传送带，机器自动检查"该亮的灯亮不亮、该转的轮转不转"，有一项不合格，**红灯立刻亮起**。

对你这个"带他的人"来说，操作朴素到不可思议：

- 改完之后，**让它"把测试跑一遍"**。
- 看结果：**全绿（全部通过）＝ 大概率没碰坏旧功能**；**出现红色（有测试失败）＝ 它弄坏了什么，别合并，让它去修。**

你完全不需要会写测试、也不需要看懂测试代码——**你只需要看那个"全过 / 没全过"的结果**，就像你不懂质检机的原理，但你看得懂"绿灯还是红灯"。

> **误解 vs 实际：** 很多新手以为"它说改好了，那就是好了"。**实际上，"它觉得好了"和"经得起测试"是两回事。** 它可能真心以为自己改对了（它不是骗你，是它"自信地错了"，第9章的幻觉），但测试这条流水线**不讲情面**——跑一遍，是真是假立刻见分晓。所以养成习惯：**改完必跑测试，红灯亮着不合并。**

一个补充：不是每个项目都有现成的测试。要是没有，至少做**最朴素的人工验证**——把改动影响的那个功能**亲手用一遍**，看它还正常不正常。哪怕只是"点开页面看看有没有崩"，也比什么都不验强。

## 三、回滚：一键反悔，这是你最大的底气

前面两步是"事后检查"。但你心里可能在打鼓：**万一它真把项目改乱了，我一个外行，怎么收拾这个烂摊子？**

答案能让你彻底安心：**回滚——一键反悔，把项目恢复到它动手之前的干净样子。**

为什么这件事能做到？这里要分清两种情况：

**情况一：它在你本地项目上动手。** 这时回滚靠的是一个叫 **版本控制（Git）** 的东西。你可以把 Git 想成一台**给项目不停拍快照的相机**：它在动手之前，项目是"干净的一张快照"；它一通改之后，万一你不满意，**一句话就能让项目"回到那张干净快照"**，刚才的改动全部作废，干干净净，像什么都没发生过。

正因为有这台"快照相机"兜底，你才敢放手让它去试——**反正随时能反悔**。所以有一条前提必须先说清：

> **要点：** **当它在你本地项目上动手时，务必先用 Git 管理项目、并先切一个"新分支"再让它动手。** "切新分支"你可以理解成：**给它单独开一块草稿区，让它在草稿上折腾**，不碰你的"正式稿"。这样万一改砸了，扔掉草稿区就行，正式稿毫发无损。这一步是你**最大的安全网**，具体怎么操作第26章手把手讲。

**情况二：它在网页 / 云端形态里动手。** 这时它通常**不在你本机直接改**，而是在平台自带的隔离环境里干活，**平台一般有它自己的撤销 / 丢弃 / 重置方式**。机制和 Git 不完全一样，但目的相同：**让你能反悔。** 具体怎么撤、撤到哪一步，**以官方文档为准**。

记住这件事的精髓就一句：**让它在一个"能反悔"的环境里动手。** 至于反悔的具体按钮叫什么，本地是 Git、云端看平台——但"必须能反悔"这条，到哪都不变。

## 四、验收三问：最后这一关，看不懂就别合并

看了 diff、跑了测试、确认了能回滚——最后，把这一切收拢成一个**你随时能用、不用任何工具就能做**的动作：**验收三问**。

每次它说"做好了"，你就用这三个问题过一遍。**任何一个答不上来，就先别合并。**

1. **"它改了什么？"**——看着 diff，你能不能说出"它动了哪几处、大概在干嘛"。说不清，就让它解释，或者它**改太多了**（拆小，见第27章）。
2. **"会不会弄坏别的？"**——测试跑了吗？全绿吗？没测试的话，受影响的功能你亲手试过了吗？
3. **"我能不能用一句话讲清这段在干嘛？"**——这是最关键的一问。试着对自己（或对一个外行）用**一句大白话**说"这段改动是干什么的"。**如果你说不出来，那就意味着你没看懂——看不懂的东西，绝不放行。**

第三问最容易被偷懒跳过，但它恰恰是**整套把关的灵魂**。说不上来，不丢人，正确做法是**回去问它**："这段是干嘛的？为什么要这么改？" 让它解释到你能用一句话复述为止。它解释不清、或者越解释越可疑，那这段改动本身可能就有问题。

> **要点：** **"看不懂别合并"——这是这一章、也是整个第四部分的底线。** 它做完不等于做对（第23章）；diff 让你看清它改了什么，测试帮你确认它没弄坏什么，回滚给你"反悔"的底气，而验收三问，是把这三样拧成一根你随手就能拉的安全绳。**记住：把关在你，不在它。**

## 五、把四件事串成一套动作

别把它们当四个孤立知识点，它们其实是一条顺下来的流水线，对应第24章那道"验收"闸门：

```text
它说"做好了"
     │
     ▼
①看 diff ──→ 它改了什么？动对地方没？范围正常吗？
     │
     ▼
②跑测试 ──→ 全绿吗？有红灯就别合并，打回去修
     │
     ▼
③验收三问 ──→ 改了什么 / 会不会弄坏别的 / 能不能一句话讲清
     │
     ├── 三问都过 ──→ 合并，进入下一小步
     │
     └── 有一问卡住 ──→ 回去问清楚 / 让它改 / 不行就【回滚】重来
```

最后那条"回滚重来"特别重要：当它**越改越乱**，最干脆的不是在烂摊子里硬纠，而是**回滚到干净状态、重开一个对话、把刚才踩的坑作为前提重新交代**（第13、30章讲过这个止损思路）。**回滚不是失败，是你手里最好用的一张牌。**

## 六、常见误解，集中纠正

| 误解 | 实际 |
| --- | --- |
| 看 diff 得看懂每行代码，我不会 | 看 diff 看的是"动了什么、动多大、对不对地方"，外行就能判断；绿＝加、红＝删 |
| 它说"改好了"，那就是好了 | "它觉得好"和"经得起测试"是两码事；改完必跑测试，红灯亮着不合并 |
| 测试我看不懂，没法用 | 你不用看懂测试代码，只看那个"全绿／有红"的结果就行 |
| 万一改坏了我就完了 | 在"能反悔"的环境里动手就不怕：本地靠 Git（先切新分支），网页／云端靠平台自带撤销（**以官方为准**） |
| 没全看懂，但它语气很肯定，那就合并吧 | **看不懂就别合并**——验收三问第三问答不上来，回去问清楚，问不清就别放行 |
| 它跑偏了，我得在原地一行行纠正它 | 越改越乱时，**回滚 + 开新对话重来**往往最快（第13、30章） |

## 小结

- **diff（差异）＝ 改动前后的对比**，像玩"找不同"：**绿＝加、红＝删**。看的是"动了什么、动多大、对不对地方"，**不用看懂每行代码**。
- **跑测试 ＝ 确认没把原来好好的功能弄坏**，像出厂质检流水线：你只看"全绿还是有红灯"，红灯亮着别合并。没测试就亲手把受影响的功能试一遍。
- **回滚 ＝ 一键反悔**：让它在"能反悔"的环境里动手——**本地靠 Git（先切新分支）**，网页／云端靠**平台自带的撤销方式（以官方文档为准）**。
- **验收三问**：①它改了什么？②会不会弄坏别的？③我能不能一句话讲清？——**任何一问答不上来，就先别合并。**
- 底线一句话：**"看不懂别合并。"** 把关永远在人这边（第23、39章）；越改越乱时，回滚重来好过原地硬纠（第13、30章）。

下一章（第26章），我们把"在能反悔的环境里动手"这件事讲透——许可机制怎么保护你、危险操作为什么永远要留给人点头。

## 小测验

> 共 6 题，涵盖概念、误区、情景、操作四类。先自己想，再看引用块里的答案解析。

1. **[基础·概念题]** 在一份 diff 里，"前面带 `+` 的绿色行"代表什么？
   - A. 被删掉的行
   - B. 新加上去的行
   - C. 出错的行
   - D. 注释行
   > **答案 B。** diff 是"改动前后对比"：**绿＝加（`+`）、红＝删（`-`）**，没颜色的是没动、给你看上下文的。选 A 把加和删记反了——这是初学者最容易混的一点，记成"加号就是加上"就不会错。

2. **[基础·概念题]** "跑测试"主要是为了确认什么？
   - A. 代码写得好不好看
   - B. 这次改动有没有把原来好好的功能弄坏
   - C. AI 的态度好不好
   - D. 项目能不能上线赚钱
   > **答案 B。** 测试像出厂质检流水线，专门自动检查"原有功能还正不正常"。它能抓到那些**不在你眼前 diff 里**、光看改动发现不了的"按下葫芦浮起瓢"。你只需看"全绿还是有红灯"。

3. **[基础·误区题]** "它说'改好了'，那肯定就是好了，直接合并"——错在哪？
   - A. 没错，它说好就是好
   - B. "它觉得好"和"经得起测试"是两回事；它可能自信地错了，要靠 diff 和测试来验
   - C. 它从不说"改好了"
   - D. 合并要先交钱
   > **答案 B。** 它不是骗你，是会"自信地错"（第9章的幻觉）。测试这条流水线不讲情面，跑一遍真假立现。盲信它的语气，正是新手最常见的翻车方式。

4. **[进阶·误区题]** "看 diff 得能看懂每一行代码，我一个外行根本做不到"——这个顾虑哪里多余了？
   - A. 不多余，看不懂代码就没法看 diff
   - B. 看 diff 看的是"动了什么、动多大、对不对地方"，就像看出"我点的是沙拉怎么端来红烧肉"，外行完全判断得了
   - C. diff 只有程序员能打开
   - D. 外行看 diff 会把项目弄坏
   > **答案 B。** 验收不等于读懂每行语法。"它动对地方没、范围正不正常"，这种判断不需要编程功底。把"看 diff"误当成"读懂全部代码"，会让你白白放弃这个最有用的把关工具。

5. **[基础·情景题]** 它在你**本地项目**上改代码，越改越乱，你想"当作什么都没发生过"。前提是你之前做了什么，才能做到？
   - A. 截了图
   - B. 用 Git 管理项目、并先切了一个新分支让它在"草稿区"动手，这样能一键回滚到干净状态
   - C. 把代码背了下来
   - D. 关掉电脑
   > **答案 B。** "回滚（一键反悔）"靠的是 Git 这台"快照相机"，而**先切新分支**＝给它单开一块草稿区折腾、不碰正式稿。这一步是你最大的安全网（第26章细讲）。若用网页／云端形态，则靠平台自带的撤销方式，**以官方为准**。

6. **[进阶·操作/观察题]** 找一份现成的代码改动（或本章的 diff 示例），不必看懂代码，只练"验收三问"：**它改了什么？会不会弄坏别的？我能不能用一句话讲清这段在干嘛？** 第三问答不上来时，你打算怎么办？
   > **该观察到：** 验收三问不神秘，外行就能做。第三问答不上来时，正确动作是**回去问它**"这段是干嘛的、为什么这么改"，让它解释到你能一句话复述为止——而**不是**因为它语气肯定就放行。养成"看不懂就别合并"的肌肉记忆，比会任何命令都值钱。具体工具的命令、界面**以官方文档为准**。
