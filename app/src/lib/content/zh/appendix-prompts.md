## 怎么用这份模板

每个模板都长一个样，方便你抄：

1. **什么时候用**：一句话告诉你这个模板适合哪种场景。
2. **提示词模板**：放在灰色代码框里，里面用 `<尖括号>` 标出"要你自己填的空"。**直接整段复制**，再把每个 `<...>` 换成你的真实内容。
3. **填空说明**：逐个解释每个空填什么。
4. **小贴士**（有时有）：一句注意事项。

三个使用要点，先记住：

- **尖括号连同里面的字一起替换掉**。比如把 `<你的职业>` 整个换成"小学老师"，不要留下尖括号。
- **模板只是起点，不是圣经**。用顺了就大胆改：删掉用不上的约束、加上你自己的要求。AI 不会因为你"没按格式"就生气。
- **好提示词的五件套**（贯穿本附录，正文第 11 章详解）：**给背景**（我是谁、要干嘛）、**明确任务**（具体做什么）、**给约束**（有哪些限制和偏好）、**指定输出形式**（要表格还是要分点）、**给"成功的标准"**（什么样算做对了）。模板里这五样都帮你预留好了位置，你只管填。

> **一条贯穿全书的安全提醒：** 不要把密码、API 密钥、银行卡号、别人的隐私信息贴进对话框。需要贴一段含敏感信息的材料时，先把敏感部分用 `××` 涂掉再贴。

## 一、日常：问答、生活、决策、解释概念

### 把复杂概念讲到我能听懂

什么时候用：遇到一个看不懂的名词或道理，想要一个真正说人话的解释。

```text
我想搞懂「<你想弄懂的概念>」，但我没有相关背景，请当我是<对象，例如：完全外行的成年人>。
请你：
1. 先用一个生活里的比喻让我抓住大意；
2. 再用三五句话讲清楚它到底是什么、有什么用；
3. 最后举一个我日常生活中可能遇到它的例子。
请全程用大白话，遇到专业词就顺手解释一下。如果这个概念有常见的误解，也帮我点破。
```

填空说明：
- `<你想弄懂的概念>`：你看不懂的那个词或事，比如"通货膨胀""区块链""什么是抗体"。
- `<对象>`：你希望它把你当成谁来讲，比如"完全外行的成年人""一个初中生"。

小贴士：如果第一次解释还是太绕，直接回它一句"还是没懂，请换个更简单的比喻"。

### 帮我把一个决定想清楚

什么时候用：在两三个选项之间纠结，想要一个条理清楚的分析（注意：最后拿主意的是你，不是 AI）。

```text
我正在做一个决定，需要你帮我理清思路，不要直接替我下结论。
背景：<简单说说你的处境、在纠结什么>。
我在考虑的选项：<选项 A>、<选项 B>（如有更多一并列出）。
我最看重的是：<你最在乎的两三点，例如：花钱少、省时间、风险低>。
请用一个表格列出每个选项的好处和坏处，并对照我看重的点逐一打分（高/中/低）。
最后告诉我：要再补充哪些信息，我才能更有把握地决定。
```

填空说明：
- `<背景>`：你的真实处境，越具体 AI 越帮得上忙。
- `<选项 A/B>`：你正在权衡的几个做法。
- `<你最在乎的点>`：你的判断标准，这是它打分的依据。

小贴士：AI 不掌握你的全部情况，它的"建议"只是帮你看清利弊，别当成命运指引。

### 当面对面的问答助手（追问到底）

什么时候用：想就一个话题打破砂锅问到底，又怕自己问得乱。

```text
我想弄明白「<话题>」。我对它的了解程度大概是：<例如：只听过名字 / 知道个大概>。
请你扮演一个有耐心的讲解者，我们一问一答地推进：
- 你每次只讲一个小点，讲完反问我"这点清楚了吗"；
- 我说"清楚"你就讲下一点，我说"不清楚"你就换个说法再讲一遍；
- 如果我问到你不确定或可能已经过时的信息，请如实说"这点我不确定，建议你再核实"，不要硬编。
我们现在开始。
```

填空说明：
- `<话题>`：你想搞懂的主题。
- `<了解程度>`：你的起点，决定它从多基础讲起。

小贴士：这是"一次只学一口"的节奏，适合啃硬主题，不容易被信息淹没。

### 把一件麻烦事变成清单

什么时候用：面对一件头大的事（搬家、办手续、准备一场活动），想要一份可执行的待办清单。

```text
我要做一件事：<这件事是什么>。
我的情况：<时间、预算、人手等关键限制>。
请帮我把它拆成一份"按顺序做"的清单：
- 每一项写清楚"做什么"，必要时注明大概要花多少时间或多少钱；
- 把"必须先做"和"可以最后做"分开排序；
- 标出哪些事容易被忘、哪些有时间限制（比如要提前预约）。
最后用一句话提醒我：这件事最容易出岔子的地方在哪。
```

填空说明：
- `<这件事是什么>`：你要办的事，写具体。
- `<你的情况>`：你的时间、钱、人手等限制，它会据此排优先级。

### 帮我润色一段要发出去的话

什么时候用：微信、短信、朋友圈里有句话拿不准怎么说得体，想要几个版本挑。

```text
我要对<对方是谁>说一件事，想说得<你想要的语气，例如：礼貌又不卑微 / 轻松一点>。
我现在的草稿是：「<你的原话>」
请给我三个改写版本，分别偏<例如：正式、亲切、简短直接>，每个不超过<字数>字。
不要添加我没提到的事实，也别说得太肉麻。
```

填空说明：
- `<对方是谁>`：收信人，比如"房东""不太熟的同事"。
- `<你的原话>`：你想表达的意思，哪怕写得很糙也行。
- `<你想要的语气>` / `<字数>`：你的偏好和长度限制。

小贴士：用肯定句说要求（"说得简洁"）比否定句（"别啰嗦"）更管用——这是全书反复强调的小窍门。

## 二、工作：写作、邮件、表格、整理、翻译、汇报

### 写一封得体的工作邮件

什么时候用：要发一封正经邮件，但不想从零憋开头结尾。

```text
请帮我写一封工作邮件。
发件人身份：<你的角色，例如：项目助理>。
收件人：<对方是谁、什么关系>。
这封邮件想达到的目的：<你要对方做什么 / 知道什么>。
要点（请都写进去）：<列出 2-4 个必须提到的信息>。
语气：<例如：礼貌、专业、不啰嗦>。请控制在<字数>字以内，并给我一个清晰的主题行。
写完后，把其中任何你"猜测填上的信息"单独列出来，提醒我核对。
```

填空说明：
- `<目的>`：这封邮件最想达成什么，是它组织全文的主线。
- `<要点>`：必须出现的事实，防止它漏写或瞎编。
- 最后一句让它"标出自己猜的内容"，能帮你抓住它替你脑补的细节。

### 把杂乱信息整理成表格

什么时候用：手上有一堆零散文字（报名信息、报价、清单），想变成整齐的表格。

```text
下面是一段杂乱的信息，请帮我整理成表格。
表格的列我想要这几项：<列名 1>、<列名 2>、<列名 3>（按需增减）。
要求：
- 只用我下面给的内容来填，不要自己补充或猜测；
- 某项信息缺失就在格子里写"未提供"，不要编一个；
- 整理完在表格下面告诉我：哪些行的信息不全。
材料如下：
<把你的杂乱信息粘贴在这里>
```

填空说明：
- `<列名>`：你想要表格有哪几栏。
- `<材料>`：你的原始信息，整段贴进来。

小贴士："缺了就写未提供、不要编"这条很关键，能防止它给你一张"看着完整、其实编了一半"的表。

### 把长篇内容缩成汇报要点

什么时候用：读完一份长材料，要向上级或同事做口头/书面汇报，需要抓重点。

```text
请把下面的内容整理成一份汇报要点，给<汇报对象，例如：不了解细节的领导>看。
请基于材料本身来写，不要加入材料里没有的信息；拿不准的地方宁可标"待确认"也不要编。
输出结构：
1. 一句话结论（最重要的事先说）；
2. 三到五条关键要点；
3. 需要对方拍板或我接下来要做的事。
全文控制在<字数>字以内，用对方能秒懂的大白话。
材料如下：
<粘贴材料>
```

填空说明：
- `<汇报对象>`：决定了详略和用词，比如"领导"就要"结论先行"。
- `<材料>`：要汇报的原始内容。

### 翻译并解释，而不只是直译

什么时候用：翻译一段外文，希望既准确又地道，还想知道难点。

```text
请把下面这段<源语言>翻译成<目标语言>。
这段文字的用途/场合是：<例如：正式合同 / 朋友间聊天 / 产品说明>，请用相应的语气。
要求：
- 译得自然通顺，符合<目标语言>的表达习惯，不要逐字硬翻；
- 专有名词、人名、缩写如果不确定标准译法，请保留原文并标注，不要乱译；
- 翻完另起一段，挑出 1-2 处翻译时需要权衡的地方，简单说说你为什么这么处理。
原文如下：
<粘贴原文>
```

填空说明：
- `<源语言>` / `<目标语言>`：从哪种语言译到哪种语言。
- `<用途/场合>`：决定语气，合同和聊天的译法完全不同。

小贴士：涉及合同、医疗等高风险内容，AI 译文务必请懂行的人再核一遍。

### 帮我起草一份文档/通知

什么时候用：要写通知、说明、规章这类有固定套路的文档，想要一个能改的初稿。

```text
请帮我起草一份<文档类型，例如：活动通知 / 操作说明>。
背景：<这份文档是给谁看的、要解决什么>。
必须包含的信息：<逐条列出关键内容，如时间、地点、步骤、注意事项>。
风格要求：<例如：正式、条理清楚、读者照着就能操作>。
请用<分点 / 分段 / 表格>的形式呈现，并在开头用一句话点明这份文档是干什么的。
没把握的细节请留空并标注"<待填>"，不要替我编。
```

填空说明：
- `<文档类型>`：你要写的东西的名字。
- `<必须包含的信息>`：核心内容清单，越全它越不会漏。

小贴士：让它把不确定的地方标成"<待填>"，比让它硬凑一个数字安全得多。

### 把会议/讨论记录变成行动项

什么时候用：开完会有一堆零散记录，想理出"谁该做什么"。

```text
下面是一段会议记录，请帮我整理成可执行的行动项。
请只依据记录内容整理，记录里没写的不要补。
输出两部分：
1. 一句话会议结论；
2. 一张待办表，三列：<要做的事>、<负责人>、<截止时间>。负责人或时间没提到的，就写"未指定"。
整理完提醒我：哪些事还没明确负责人或时间，需要我去补。
材料如下：
<粘贴会议记录>
```

填空说明：
- `<材料>`：你的会议记录或速记，整段贴入。

小贴士："没指定就写未指定"能帮你一眼看出会上哪些事其实没落实。

## 三、学习：陪读、出题自测、查证、做笔记、备考

> 本类模板有一条共同底线：**让 AI 基于你给的材料回答，不要自己编；拿不准就明说不知道。** 这是因为大模型会"一本正经地说错话"（正文第 9 章详解），学习场景里尤其要守住这条。

### 当我的陪读老师

什么时候用：读一篇文章/一章书，想要边读边有人帮你解释、追问你。

```text
我在读下面这段材料，请你当我的陪读老师。
请严格基于这段材料来讲解，材料里没有的内容不要补充；如果我问到材料没覆盖的，请直接说"这段材料里没提到"。
请这样做：
1. 先用三五句话概括这段在讲什么；
2. 把其中我可能看不懂的概念挑出来，逐个解释；
3. 最后问我两个问题，检查我是不是真的读懂了。
材料如下：
<粘贴材料>
```

填空说明：
- `<材料>`：你要读的那段内容，整段贴进来。

小贴士：它问你的两个问题别跳过，自己先答一遍，是检验"真懂还是假懂"的好办法。

### 给我出题自测

什么时候用：学完一部分内容，想出几道题考考自己。

```text
请根据下面的材料给我出<数量>道自测题，用来检查我有没有掌握。
要求：
- 题目只考材料里讲过的内容，不要超纲；
- 题型用<例如：选择题 / 简答题>，难度<例如：由易到难>；
- 先只给题目，不要给答案，等我答完我再说"公布答案"，你再给出答案和解析。
材料如下：
<粘贴材料>
```

填空说明：
- `<数量>` / `<题型>` / `<难度>`：你想要几道、什么形式、多难。
- `<材料>`：出题依据。

小贴士：先做后看答案，效果远好于直接看答案——别让它一上来就把答案给你。

### 帮我查证一个说法（带提醒）

什么时候用：你听到一个说法拿不准真假，想让 AI 帮你分析（但要清楚它本身也可能错）。

```text
我听到一个说法：「<那个说法>」。请帮我判断这个说法靠不靠谱。
请你：
1. 说说这个说法哪些部分大概率成立、哪些部分存疑或要看条件；
2. 如果你不确定或这可能涉及较新的信息，请明确说"这点我不确定"，不要硬给结论；
3. 告诉我：要确认这件事，我应该去查什么样的权威来源（不用给具体链接，告诉我类型即可）。
```

填空说明：
- `<那个说法>`：你想核实的那句话。

小贴士：**AI 不是事实来源**。它能帮你理清思路、指出疑点，但最终结论要靠权威资料核实（正文第 9、15 章反复强调）。

### 把材料整理成学习笔记

什么时候用：读完一份资料，想要一份结构清晰、便于复习的笔记。

```text
请把下面的材料整理成一份学习笔记，目的是方便我以后快速复习。
请完全基于材料整理，不要添加材料里没有的内容。
笔记结构：
1. 这份材料的核心结论（一两句）；
2. 关键知识点（分点列出，每点一句话说清）；
3. 容易混淆或需要记住的细节；
4. 一句话总结。
请用简洁的分点形式，别长篇大论。
材料如下：
<粘贴材料>
```

填空说明：
- `<材料>`：你要做笔记的原始资料。

### 帮我做一份备考计划

什么时候用：要准备一场考试，时间有限，想要一份可执行的复习安排。

```text
我要准备<考试名称>，考试时间是<日期>，现在距离考试还有<多少天>。
我每天大概能挤出<可用时间>来复习。
要复习的范围：<列出科目或章节>，其中我比较薄弱的是<薄弱点>。
请帮我排一份复习计划：
- 按周（或按天）安排，把薄弱的部分多留点时间；
- 留出复习巩固和模拟测试的时间，别排得太满；
- 用表格呈现，列出每个时间段做什么。
如果信息不够你排计划，请先问我缺什么。
```

填空说明：
- `<考试名称>` / `<日期>` / `<多少天>` / `<可用时间>`：排期的硬约束。
- `<薄弱点>`：让它给你的弱项多分配时间。

小贴士：计划是死的，人是活的——执行中跟不上就回来让它帮你重排。

## 四、资料总结：长文摘要、多文档对比、提取要点、做大纲

> 本类模板的共同底线同上：**只用我给的材料回答，不要自己编；材料里没有就说没有。** 喂长材料时，如果一次贴不下，可以分几段贴，并在开头说明"材料分几次发，我说'发完了'你再开始总结"。

### 长文摘要（先结论后展开）

什么时候用：一篇又长又啰嗦的文章/报告，想快速抓住它在说什么。

```text
请帮我总结下面这篇长文，给一个没时间读全文的人看。
请只依据原文内容总结，不要加入原文没有的信息或你的评价；原文没说清楚的地方就标"原文未明确"。
输出：
1. 一句话主旨（这篇到底想说什么）；
2. 三到六条关键要点，按重要性排序；
3. 如果原文有明确的结论或建议，单独列出来。
全文控制在<字数>字以内，用大白话。
原文如下：
<粘贴长文>
```

填空说明：
- `<字数>`：你想要的摘要长度。
- `<原文>`：要总结的长文，整段贴入。

小贴士：摘要再好也会丢细节，重要决定别只看摘要、要回看原文对应段落。

### 多份材料横向对比

什么时候用：手上有两三份讲同一件事的材料（几份报价、几篇观点），想看清异同。

```text
下面有<数量>份材料，都和「<共同主题>」有关。请帮我做横向对比。
请只依据这些材料对比，不要引入材料之外的信息。
请用一个表格：每一列是一份材料，每一行是一个对比维度（维度我想看：<维度 1>、<维度 2>……）。
表格下面再补两点：
1. 这几份材料最大的共识是什么；
2. 它们之间最明显的分歧或差异在哪。
某份材料没提到某个维度，就在格子里写"未提及"。
材料如下：
材料一：<粘贴>
材料二：<粘贴>
（按需继续）
```

填空说明：
- `<共同主题>`：这几份材料共同围绕的事。
- `<维度>`：你想从哪些角度比，比如"价格、服务、风险"。

小贴士：把每份材料清楚标上"材料一/材料二"，它对比时不容易张冠李戴。

### 从材料里提取特定要点

什么时候用：一份大材料里，你只想捞出某一类信息（所有日期、所有金额、所有待办）。

```text
下面是一份材料。请帮我把其中所有的「<你要提取的东西>」都找出来。
要求：
- 只提取材料里确实出现的，不要推测或补全；
- 每条都顺便标一下它出现在材料的哪个部分（比如哪一段、哪个小标题下），方便我回去核对；
- 如果材料里其实没有这类信息，就直接告诉我"没有找到"，不要硬凑。
材料如下：
<粘贴材料>
```

填空说明：
- `<你要提取的东西>`：你想捞的那类信息，比如"所有提到的时间点""所有人名"。
- `<材料>`：原始材料。

小贴士：让它"标出处"，你就能快速回原文验证，而不是盲信它的提取结果。

### 把材料梳理成大纲

什么时候用：要基于一份材料写文章/做汇报/讲课，先要一个清晰的骨架。

```text
我要基于下面的材料做一份<用途，例如：演讲 / 文章 / 培训>，请先帮我搭一个大纲。
请基于材料内容来组织，不要添加材料里没有的论点。
大纲要求：
- 分成几个大块，每块一个清晰的小标题；
- 每个小标题下用一两句话说明这块要讲什么、对应材料的哪部分；
- 整体顺序符合<例如：由浅入深 / 时间先后>的逻辑。
搭完大纲，告诉我：材料里有没有哪块内容偏薄、可能需要我再补充。
材料如下：
<粘贴材料>
```

填空说明：
- `<用途>`：决定大纲的组织方式，演讲和文章的骨架不同。
- `<材料>`：搭大纲的依据。

### 多轮长资料的"先理解再总结"

什么时候用：材料特别长、特别专业，怕它一上来就总结跑偏。

```text
我接下来会分几次发给你一份较长的材料，请按下面的步骤配合我：
1. 我发材料时，你先不要总结，只回一句"已收到第 N 部分"；
2. 我说"发完了"，你再开始；
3. 开始时先用三五句话告诉我这份材料整体在讲什么，确认你抓对了方向；
4. 我确认方向没错后，你再按我的要求做详细总结。
全程只依据我发的材料，不要自己补充。明白请回复"明白，请开始发材料"。
```

填空说明：
- 这个模板本身不用填空，直接发给 AI，然后开始一段段贴你的材料。

小贴士：先让它复述"整体在讲什么"，等你确认方向对了再让它细做——这一步能挡掉大部分"总结跑题"。

## 五、Claude Code：调查 → 计划 → 实现 → 验收

> Claude Code 是一种"会动手改代码、跑命令"的 AI 编程工具。下面的模板**都是你用大白话对它说的话，不是产品里的命令或按钮**。它具体怎么安装、有哪些快捷键和功能，会随版本变化，**以官方文档为准**。
> 这一类贯穿一条安全主线（正文第 23–30 章）：**先让它调查、先看它的计划、看它改了什么（diff）、跑测试、最后由人来验收，看不懂的代码不要合并；任何含密钥/密码的内容都不要贴进去。**

### 第一步：只调查，先别动手

什么时候用：要在一个项目里做改动，但你（和 AI）还不清楚相关代码在哪、现在是怎么写的。

```text
先别改任何代码，这一步只做调查。
我想做的事是：<你想实现的改动，用大白话说>。
请帮我调查并列清楚（带上文件路径）：
- 和这件事相关的代码现在在哪几个文件里、大致是怎么实现的；
- 项目里有没有类似的现成做法可以参考或复用；
- 这次改动大概会牵动哪些地方。
调查完只给我一份清单，先不要提出具体改法，等我看完再说下一步。
```

填空说明：
- `<你想实现的改动>`：你的目标，越具体越好。

小贴士：把它列出的两三个文件路径自己点开看一眼，确认它没有"指错地方"，再往下走。

### 第二步：先给我看计划，别直接写

什么时候用：调查清楚后，要它先拿出方案，你批准了再让它动手。

```text
根据刚才的调查，请先给我一份实现计划，**在我批准之前不要修改任何代码**。
我要做的事：<再说一遍你的目标>。
计划里请写清楚：
- 准备改动或新增哪几个文件，分别做什么；
- 会复用项目里已有的哪些现成做法；
- 是否需要引入新的外部工具/库——如果要，说明为什么、有没有更省事的替代。
请按现有项目的写法和风格来设计，不要另起一套自己的风格。
计划给我后，我确认没问题会回复"按这个计划做"。
```

填空说明：
- `<你的目标>`：和上一步保持一致。

小贴士：计划这一步是最省返工的关卡。觉得方案不对就别批准，直接说哪里要改，让它重出计划（正文第 29 章）。

### 第三步：按计划实现，遇到偏差先问

什么时候用：你已经批准了计划，让它正式开始改代码。

```text
按我们确认的计划开始实现。
请严格遵守：
- 完全按现有项目的目录结构、命名和代码风格来写；
- 优先复用我们调查时找到的现成做法，不要重新造一套；
- 如果实现中发现计划有遗漏、或需要做计划外的改动，先停下来告诉我，别自作主张扩大改动范围；
- 每改动一处，简单说一句你改了什么、为什么。
```

填空说明：
- 这个模板直接用，无需填空。

小贴士：感觉它开始改计划外的东西时，及时叫停并指出来——具体怎么中断、怎么回退，**以官方文档为准**。

### 看懂它改了什么（解释 diff）

什么时候用：它改完了，你要看"到底动了哪些地方"，但又看不太懂代码。

```text
请把你这次所有的改动，按文件分别给我讲清楚，讲给一个不太懂代码的人听：
- 每个文件你改了什么、为什么改；
- 哪些改动是核心、哪些只是配套的小修小补；
- 这些改动有没有可能影响到原来正常的功能？如果有风险，指出来。
有哪个地方我看了还是不放心，我会追问"这段具体在做什么"。
```

填空说明：
- 直接使用，无需填空。

小贴士：除了听它解释，最好也自己过一遍它列出的改动清单。**看不懂、解释不通的代码，就不要合并。**

### 让它先写测试，再确认没弄坏

什么时候用：改完代码，想确认"新功能对了、老功能没被弄坏"。

```text
请帮我验证这次改动是否可靠。
1. 先为这次新增/修改的功能补上测试，覆盖正常情况，也要覆盖异常和边界情况（比如空数据、超大数据、格式怪异的输入）；
2. 把项目里现有的测试也全部跑一遍，确认没有把原来好好的功能弄坏；
3. 如果有测试没通过，告诉我是哪条、为什么，并提出修复办法（先别直接大改）。
最后用一句话总结：现在这次改动可以放心合并了吗？
```

填空说明：
- 直接使用，无需填空。

小贴士：让它自己跑测试、根据结果改，是它"自我纠错"的关键（正文第 27 章）。但"能放心合并吗"的最终判断权在你。

### 改 Bug：先复现，再修

什么时候用：有个 Bug 要修，希望它别瞎改，而是先搞清楚问题。

```text
有一个 Bug 需要修。先别急着改，按下面来：
现象：<出问题时发生了什么>。
什么情况下会出现：<复现步骤，或你观察到的规律>。
（如有报错信息可贴在这里，但**贴之前先检查里面有没有密钥、密码、个人信息，有就涂掉**。）
请你：
1. 先调查可能的原因，按可能性从高到低列出来；
2. 先写一个能"重现这个 Bug"的测试（让它现在是失败的）；
3. 再动手修，直到这个测试通过、且原有测试都还正常。
每一步做完跟我说一声，我确认后再继续。
```

填空说明：
- `<现象>`：出 Bug 时的表现。
- `<复现步骤>`：怎样能让 Bug 再次出现，这是定位问题的关键。

小贴士：先写一个"能重现 Bug 的失败测试"，修好后它自然变绿——这样你才有证据 Bug 真的被修好了，而不是"看起来好了"。

## 六、Codex：另一种编程智能体的用法

> Codex 是另一种"会动手"的 AI 编程智能体，用法和上一类（Claude Code）的心智模型很像：**都靠你用大白话给任务，都该走"调查 → 计划 → 实现 → 验收"的安全流程。** 下面给的同样是"对它说的话"，**不是产品命令**。
> 重要：Codex 的具体功能、安装方式、命令、支持的操作会随版本变化，本类涉及的所有具体操作一律**以官方文档为准**。本附录只教"怎么把话说清楚"，不替它的产品功能背书。

### 上手：让它先认识这个项目

什么时候用：第一次在某个项目里用它，想让它先摸清情况再干活。

```text
这是我第一次让你在这个项目里工作，请先帮我熟悉它，**这一步不要改任何东西**。
请告诉我：
- 这个项目大体是做什么的、主要由哪几部分组成；
- 关键的文件/目录分别是干嘛的；
- 如果我想做<你打算做的事>，大概会涉及哪些地方。
用我这种不太懂代码的人能听懂的话讲，带上文件路径方便我对照。
```

填空说明：
- `<你打算做的事>`：你接下来想做的改动，让它顺带圈出相关位置。

小贴士：它能调用哪些工具、能不能跑命令，各版本不同，**以官方文档为准**。

### 给任务前，先要计划

什么时候用：要交给它一个具体任务，但希望它先列方案、你点头再开工。

```text
我有个任务要交给你，但请先给方案、**别直接动手改代码**。
任务：<你要它做的事，说具体>。
背景：<相关情况，比如这个功能现在的问题、你的期望>。
请先给我一份计划：要改哪些地方、怎么改、会不会影响其他功能、要不要装新东西。
请尽量沿用项目现有的写法和风格。计划我确认后，会回复"开始吧"。
```

填空说明：
- `<任务>`：你的具体需求。
- `<背景>`：相关情况，帮它理解上下文。

小贴士：和 Claude Code 那类一样，"先看计划再动手"是省返工的关键一步——这是通用的好习惯，不依赖任何特定产品功能。

### 实现并解释改动

什么时候用：计划批准后，让它动手，同时要它把改动讲清楚。

```text
按确认的计划开始实现。请遵守：
- 沿用项目现有的目录结构、命名和风格；
- 能复用现成做法就别重造；
- 发现要做计划外的改动，先停下来问我；
- 改完后，按文件给我讲清楚"改了什么、为什么"，讲给不太懂代码的人听。
```

填空说明：
- 直接使用，无需填空。

小贴士：怎么查看具体改动、怎么撤销，**以官方文档为准**；但无论用什么工具，原则不变——**改动要能被你看懂，看不懂就别采用**。

### 跑测试、确认没弄坏

什么时候用：改完后想确认质量，让它补测试并验证。

```text
请帮我确认这次改动靠谱：
1. 给新增/改动的功能补测试，正常、异常、边界情况都要覆盖；
2. 把现有测试全跑一遍，确认没弄坏原来的功能；
3. 有不通过的，告诉我是哪条、为什么，给出修复思路，先别大刀阔斧地改。
最后一句话告诉我：这次改动是否可以合并。
```

填空说明：
- 直接使用，无需填空。

小贴士：能不能自动跑测试、怎么跑，依产品而定，**以官方文档为准**。但"测试要覆盖异常和边界、最后由人拍板"这条永远成立。

### 改 Bug（安全流程）

什么时候用：要它修一个 Bug，希望它先查清楚再动手。

```text
有个 Bug 要修，先别急着改。
现象：<问题表现>。
怎么复现：<复现步骤或规律>。
（要贴报错信息的话，**先检查并涂掉里面的密钥、密码、个人信息**。）
请：①先按可能性高低列出可能原因；②先写一个能重现这个 Bug 的失败测试；③再修到测试通过、且原有测试不受影响。每步做完跟我确认。
```

填空说明：
- `<现象>` / `<复现步骤>`：和上一类的改 Bug 模板一样，是定位问题的关键。

小贴士：不管用哪个工具，"先复现、再修、人来验收、不贴密钥"这套安全习惯都一样适用。

## 七、项目管理：任务拆分、计划、进度跟踪、复盘、风险

### 把一个大目标拆成可执行任务

什么时候用：手上有个大目标（办活动、做一个产品、完成一个项目），不知从何下手。

```text
我有一个目标：<你的目标>。
背景和限制：<可用时间、预算、人手、截止日期等>。
请帮我把它拆解成可执行的任务清单：
- 先分成几个大阶段，每个阶段一个清晰的名字；
- 每个阶段下列出具体要做的小任务；
- 标出任务之间的先后依赖（哪些必须先完成才能开始下一个）。
用分层清单或表格呈现。拆完告诉我：哪个阶段最关键、最容易出问题。
```

填空说明：
- `<你的目标>`：你要达成的事。
- `<背景和限制>`：时间、钱、人手等约束，决定任务怎么排。

### 排一份带时间的计划

什么时候用：任务拆好了，要安排"什么时候做什么"。

```text
下面是我要做的任务清单，请帮我排成一份带时间的计划。
开始时间：<起始日>，希望完成时间：<截止日>。
可投入的人/时间：<例如：只有我自己，每天 2 小时 / 三个人全职>。
请用表格排期，列出每个任务的开始和结束时间、负责人（如有）。
注意：
- 给关键任务和容易拖延的任务留一点缓冲，别排得太死；
- 把"必须按顺序做"的任务排在正确的先后位置。
任务清单如下：
<粘贴你的任务清单>
```

填空说明：
- `<起始日>` / `<截止日>`：排期的两个端点。
- `<可投入的人/时间>`：决定能并行多少、每天能推进多少。

小贴士：计划赶不上变化，把它当成第一版草稿，执行中随时回来调。

### 整理进度、看看落后没有

什么时候用：项目进行到一半，想理清"做完了哪些、卡在哪、还差什么"。

```text
我来跟你汇报一下项目目前的进展，请帮我梳理现状。
原计划是：<简述原计划或贴上计划>。
目前实际情况：<已完成什么、正在做什么、遇到什么卡点>。
请基于我说的情况帮我理出：
1. 哪些任务已完成、哪些进行中、哪些还没开始；
2. 对照原计划，现在是超前、正常还是落后；
3. 如果落后，是卡在哪、有什么办法追回来。
只根据我提供的信息分析，信息不足的地方请直接问我，不要替我假设。
```

填空说明：
- `<原计划>`：原本的安排，作为对照基准。
- `<目前实际情况>`：真实进展，越如实越有用。

小贴士："信息不足就问我、别假设"这条很重要——否则它可能基于脑补给你一份好看但失真的进度判断。

### 提前找出项目的风险

什么时候用：项目开始前或进行中，想提前预判"可能会出什么岔子"。

```text
请帮我给这个项目做一次风险排查，提前找出可能出问题的地方。
项目简介：<这个项目是做什么的、目标和关键限制>。
请扮演一个经验丰富、爱挑刺的项目顾问，帮我列出可能的风险：
- 每条风险写清楚：可能发生什么、为什么、影响有多大；
- 按"发生可能性"和"影响大小"分别标高/中/低；
- 对高风险项，给一两条预防或应对的办法。
请重点提醒那些"容易被乐观忽略"的风险。
```

填空说明：
- `<项目简介>`：项目情况，越具体，风险预判越贴合实际。

小贴士：让它"扮演爱挑刺的顾问"，比泛泛地问"有什么风险"更能逼出真问题——给角色是个好用的小技巧（正文第 11 章）。

### 项目复盘（做完之后总结）

什么时候用：一个项目或阶段结束了，想总结经验教训。

```text
请帮我做一次项目复盘。
项目/阶段简介：<做了什么、原定目标是什么>。
实际结果：<最终做成什么样、和目标相比如何>。
过程中的关键事件：<顺利的地方、踩过的坑、临时的变动>。
请基于我提供的信息，帮我整理：
1. 做得好、值得以后保留的做法；
2. 出了问题或可以改进的地方，以及下次怎么避免；
3. 三条最值得记住的经验教训。
只根据我说的来总结，不要编我没提到的情节。
```

填空说明：
- `<项目简介>` / `<实际结果>` / `<关键事件>`：复盘的三块原料，决定总结的质量。

小贴士：复盘重在"下次怎么做得更好"，让它把改进办法落到具体动作，别停在"要更努力"这种空话。

### 给关键人物写进度同步

什么时候用：要向领导、客户或合作方简短同步项目进展。

```text
请帮我写一段项目进度同步，发给<对象，例如：领导 / 客户>。
请基于我提供的情况来写，不要添加我没说的进展。
内容来自：<已完成什么、下一步做什么、有没有需要对方支持或拍板的事>。
要求：
- 结论和重点先说，对方花十几秒就能看明白；
- 分成"已完成 / 进行中 / 需要您支持"三小块；
- 语气<例如：专业、简洁、不报喜不报忧>。
控制在<字数>字以内。
```

填空说明：
- `<对象>`：同步给谁，决定详略和语气。
- `<内容来自>`：真实进展，是它组织内容的原料。

小贴士：让它把"需要对方支持/拍板的事"单列出来，对方才知道该为你做什么决定。

## 通用万能骨架（记不住模板时，照这个填）

> 上面所有模板，骨子里都是这一个结构。任何场景，只要把下面五块填清楚，就是一条好提示词。记不住具体模板时，回到这个骨架即可。

```text
【背景】我是<你的身份/处境>，正在<你要解决的事或所处场景>。
【任务】请你帮我<具体要做的事，越明确越好>。
【约束】请遵守这些要求：
- <限制条件 1，比如：篇幅 / 风格 / 必须包含或避开什么>；
- <限制条件 2>；
- （如果涉及材料或事实）请只根据我给的内容来做，材料里没有的就说没有、拿不准就说不确定，不要编。
【输出形式】请用<表格 / 分点 / 一段话 / 先结论后展开……>的形式给我，篇幅大约<长度>。
【成功标准】对我来说，做到<什么样就算合格，比如：领导一眼能看懂 / 测试能通过 / 涵盖了这几个要点>就算成功。
如果你觉得信息还不够、或者有更好的做法，先告诉我再动手。
```

填空说明：
- **背景**：你是谁、在什么处境——AI 读不懂你的心，这是它理解你的前提。
- **任务**：你到底要它做什么，用动词说清楚（"整理成表格""改写成邮件"）。
- **约束**：你的限制和偏好；涉及材料/事实时，务必加上"只依据材料、不编、拿不准就说不知道"。
- **输出形式**：你想要的样子，先说清楚省得它给错格式。
- **成功标准**：什么样算做对了——这是全书最强调的一条，给了它"自我检查"的标尺。

> 用肯定句说要求（"按现有风格写""控制在 300 字"）比否定句（"别乱写""别太长"）更管用。这是本书一以贯之的小窍门，套用在上面任何一个 `<约束>` 里都成立。
