〔本章涉及具体产品的形态、安装、命令、界面、能力之处，都会随版本更新而变化，一律**以官方文档为准**。本章只讲一类工具**共通、稳定**的心智模型——这部分不会过时。〕

前面第19章里，我们认识了 **Agent（智能体）**：不只是"答一句话"，而是会**自己动手做事**的 AI——它能拆解目标、调用工具、连着走好几步。这一章，我们把镜头对准 Agent 家族里最成熟、也最值得普通人了解的一个分支：**编程智能体（coding agent）**。

它的代表，就是你大概率听过名字的 **Claude Code** 和 **Codex**。

先把丑话说在前头：**这一章不要求你会写代码，一行都不用。** 我们要建立的，是一种"心智模型"——搞清楚这东西**到底是什么、能干到哪、不能指望它什么、最终谁负责**。把这件事想明白，后面几章那些"看 diff、跑测试、回滚、看计划"的具体把关动作，你才知道自己**在守什么、为什么要守**。

## 一、它是 Agent 的一种：会"做事"，不只是"会写代码"

很多人对"编程 AI"的第一印象，停留在"它会写代码"——你问它"怎么写一个排序函数"，它给你一段代码，你复制走。这个印象，是**网页聊天**那一层的（第17章讲过模型/产品/Agent 的分层）。

编程智能体比这**高一个层级**。它不是"会写代码"，而是"**会用写代码这件事去做事**"。差别在哪？

| | 会"写代码"（聊天层） | 会"做事"（智能体层） |
| --- | --- | --- |
| 你给它什么 | 一个具体的小问题 | 一个**目标**（"给这个网站加个搜索框"） |
| 它给你什么 | 一段文字，你自己去用 | 它**自己把事做了**——读文件、改代码、跑命令 |
| 它知道多少 | 只知道你贴进去的几行 | **自己去翻整个项目**，搞清楚现状 |
| 推进方式 | 你问一句、它答一句 | 朝目标**连走好几步**：看一看→改一改→试一试→再改 |

这正是第19章里 Agent 的定义落到"编程"这个具体领域的样子：**感知（读项目）→ 决策（想方案）→ 行动（改文件、跑命令）→ 再感知（看结果对不对）**，自己转这个圈。所以请记住第一句话——

> **要点：** **编程智能体 ＝ 第19章那个"会动手的 Agent"，只不过它动手的地方是你的代码项目。** "会写代码"只是它的一项技能；"会拿这项技能去把一件事办成"，才是它真正的身份。理解这一点，你就不会再把它当成"一个更聪明的搜索框"。

## 二、它"长"什么样：形态多变，别钉死在某一种

你可能会问：那我在哪儿用它？是个网站？是个 App？

答案是：**没有唯一答案，而且一直在变。** 你通常会在**终端（命令行）、代码编辑器（IDE）、网页任务或云端环境**里接触到它们；**有的在你自己的机器上运行，有的在云端运行，具体形态各不相同、而且会更新——以官方文档/产品官方形态为准。**

为什么本书不肯告诉你"就用那个网页 / 就装那个软件"？因为这类工具的形态**变得特别快**：今天主流是命令行，明天可能 IDE 插件更顺手，后天又冒出网页版、云端后台版。**把书写死在某一种形态上，半年就过时了。** 所以本书只讲**不会过时**的那部分——它怎么干活、你怎么带它。具体"在哪打开、点哪个按钮、敲哪条命令"，第25章和附录 C 会展开，但最终都**以官方文档为准**。

> **要点：** 有两件事是**稳定**的、可以放心记：Claude Code 来自 Anthropic，内核是 Claude；Codex 来自 OpenAI。这种"出品方／内核"的归属不会乱变。但**它们具体长什么样、怎么装、能做什么**，属于"会变"的那一类——遇到这些细节，认准官方文档。

## 三、核心心智模型：把它当"很能干、但刚入职第一天的新同事"

这是整本书反复用、到这里最贴切的一个比喻。请你真的把它当成一个**人**来想象：

**Claude Code / Codex，就像一个手脚麻利、肯干、能力很强，但今天才到岗、对你们公司一无所知的新同事。**

把这个形象拆开看，每一条都对应它一个真实脾气：

- **他手脚麻利**：交代的体力活，他做得又快又多——这是它最大的价值，别浪费。
- **他不了解你的"门道"**：你们项目的规矩、命名习惯、哪些地方碰不得，他统统不知道——**你不说，他就不知道**。（让他"记住这些规矩"是有办法的，见第28章。）
- **他会自信地犯错**：他可能一脸笃定地用一个**根本不存在的写法**——这就是第9章讲的"幻觉"，写代码时照样会犯，而且因为代码看起来很"专业"，更容易骗过外行。
- **他不知道最新的事**：他的知识有"截止日期"（第9章），很新的技术他可能没听过、或记成了旧版本——除非你把最新资料告诉他。

那么，面对这样一个新同事，一个靠谱的"带他的人"会怎么做？**不会撒手不管，也不会事事代劳。** 会派活、会盯着关键节点、会验收。这正是你的位置——

> **误解 vs 实际：** 很多人一上来就把它当"全自动代写机器"：丢个需求，等它交活，拿来就用。**实际上，它更像你带的实习生，而不是交钥匙的承包商。** 你不必会写每一行代码，但你得会**派活**、会**判断他做得对不对**。把它当承包商，你迟早会为一个"看起来没问题、其实埋了雷"的结果买单。

## 四、最终责任，永远在人这一边

这是这一章、甚至整个第四部分**最该记住**的一条，单独拿出来讲。

它做完，**不等于**做对。它说"搞定了"，**不等于**真的成了。原因前面已经埋好了：**它会犯错（会幻觉）＋ 它能动手（真的改了你的东西）**——这两条叠在一起，意味着一个危险的组合：**它可能一脸自信地，把你的项目改坏，还告诉你"完成了"。**

所以，无论这工具多强、未来多强，有一条边界不会移动：

> **要点：** **署名的是你，负责的是你。** 它生成的东西，合并进你的项目、交给你的客户、挂上你的名字之前，**复核与验收是你的责任**（第39章会专门讲"AI 生成内容的责任"）。一句话刻进脑子——**"看不懂、没验证的东西，不要放行。"** 你不需要看懂每一行，但你需要能判断"这件事到底成没成"。

这不是给你泼冷水，恰恰相反：**正因为责任清楚地落在人这边，你才能放心地把大量体力活交给它。** 你守住"验收"这道关，前面那些又快又多的活，就都是净赚的。

## 五、这一章为后面铺了什么路

把上面四点连起来，你已经拿到了一张"地图"，后面几章都是在这张图上填细节：

- 它**会动手、也会犯错** → 所以你得学会**事后把关**：看它改了什么（diff）、有没有弄坏别的（测试）、看不懂就别合并——这是**第25章**，本书最该练会的一章。
- 它**真的会改你的东西** → 所以你得让它在一个**能反悔的环境**里动手，并且**危险操作要它先问过你**——这是**第26章**的"许可机制与可回滚环境"。
- 大事**不能一口气莽** → 所以你得让它**先出计划、你点头再动手**，并把大任务**拆小、小步验收**——这是**第27章**。

发现了吗？这三件事，全都是在落实本章那个核心心态：**它是能干但会犯错的新同事，派活给它，但把关在你。** 接下来三章，就是把"怎么把关"这件事，一招一式教给你。

## 六、常见误解，集中纠正

| 误解 | 实际 |
| --- | --- |
| 编程智能体就是"一个更会写代码的聊天 AI" | 它是**会动手做事**的 Agent（第19章）：自己读项目、改文件、跑命令、连走好几步，不只是吐一段文字 |
| 它就是某个网页 / 某个软件，形态固定 | 形态多样（终端／IDE／网页／云端），本地云端都有，而且在更新——**以官方文档为准** |
| 它是 AI，写的代码应该对，拿来就用 | 它会**自信地写错**（幻觉照样发生）；做完不等于做对，**看不懂别放行** |
| 我不会编程，根本用不了 | 你不用会写每一行，要会的是**派活 + 判断成没成**；门槛在判断，不在敲代码 |
| 它说"完成了"，责任就交给它了 | **署名和责任永远在你**：合并、交付前的复核验收是人的活（第39章） |

## 小结

- **编程智能体（Claude Code / Codex）是第19章"会动手的 Agent"在编程领域的样子**：不是"会写代码"，而是"会用写代码去把一件事办成"——自己读项目、改文件、跑命令、连走好几步。
- 它的**形态多样且在变**（终端／IDE／网页／云端，本地与云端都有），本书只讲不会过时的心智模型；具体怎么装、敲什么，**以官方文档为准**（第25章、附录 C）。
- 核心心态：把它当**很能干、但刚入职第一天的新同事**——手快、肯干，但不懂你的规矩、会自信地犯错。
- **最终责任永远在人**："看不懂、没验证的东西，不要放行。"正因为把关在你，才敢把体力活放心交给它。
- 本章是地图，后面三章填细节：**验收（看 diff／跑测试，第25章）、许可与回滚（第26章）、先计划后动手（第27章）**——全是在落实"派活给它、把关在你"。

下一章（第24章），我们就把这套"调查→计划→动手→验收"的共通工作流，完整地走一遍。

## 小测验

> 共 6 题，涵盖概念、误区、情景、操作四类。先自己想，再看引用块里的答案解析。

1. **[基础·概念题]** 把"编程智能体"和第19章的"Agent"对照，下面哪句最准确？
   - A. 编程智能体和 Agent 没关系，是两种东西
   - B. 编程智能体就是 Agent 的一种——会动手的 AI，只不过它动手的地方是你的代码项目
   - C. 编程智能体不会动手，只会给建议
   - D. Agent 是编程智能体的一个功能按钮
   > **答案 B。** 第19章的 Agent ＝ 会自己拆目标、调工具、连走好几步的 AI；编程智能体就是把这套本事用在"改代码项目"上。选 A、D 的人没把"它是 Agent 的一种"这层关系建立起来；选 C 的人还停留在"聊天 AI 只会给文字"的旧印象里。

2. **[基础·概念题]** "会写代码"和"会做事"，对编程智能体来说差别在哪？
   - A. 没差别，一回事
   - B. "会写代码"是给你一段文字你自己用；"会做事"是它自己读项目、改文件、跑命令，把目标办成
   - C. "会做事"指它会泡咖啡
   - D. "会写代码"更高级
   > **答案 B。** 这正是聊天层和智能体层的分界：前者吐文字、你来执行；后者自己执行、连走好几步。把它当"更聪明的搜索框"，就是错把它当成了"只会写代码"那一层。

3. **[基础·误区题]** "它是 AI，写的代码应该是对的，做完直接用"——错在哪？
   - A. 没错，AI 写的就是对的
   - B. 它会自信地犯错（幻觉照样发生），"做完"不等于"做对"，看不懂、没验证的不该放行
   - C. 它写的代码永远跑不起来
   - D. 用之前要先付费
   > **答案 B。** 幻觉（第9章）在写代码时照样发生，还因为代码看起来"专业"更容易骗过外行。这就是为什么后面要学"看 diff、跑测试"。选 C 走到了另一个极端——它常常能跑，问题在于"能跑"不等于"对"。

4. **[进阶·误区题]** "Claude Code / Codex 就是某个固定的网页（或某个软件），形态是定死的"——为什么这么想会出问题？
   - A. 不会出问题，形态就是固定的
   - B. 它们形态多样（终端／IDE／网页／云端）、本地云端都有、而且在更新，钉死在某一种很快就过时——具体以官方文档为准
   - C. 它们只有云端一种形态
   - D. 它们根本没有界面
   > **答案 B。** 形态是"会变"的那一类，所以本书只讲不会过时的心智模型，把"在哪用、怎么装"交给官方文档。不过有两点是稳定的：Claude Code 来自 Anthropic（内核 Claude）、Codex 来自 OpenAI——这种归属可以放心记。

5. **[基础·情景题]** 你第一次用它，它三下五除二就说"功能加好了"。作为"带他的人"，你接下来最该做的是？
   - A. 直接相信，合并上线
   - B. 把它当刚入职的新同事——验收一下：它到底改了什么、有没有弄坏别的、我能不能讲清这段在干嘛
   - C. 夸它两句就结束
   - D. 再让它做下一个更大的功能
   > **答案 B。** "做完不等于做对"。它像能干但刚到岗的新同事，验收是带他的人的责任（具体三个动作见第25章）。选 A 是把它当"交钥匙的承包商"——这正是最常见的翻车姿势。

6. **[进阶·操作/观察题]** 不必真装任何工具：找一个你身边"会做事 vs 只给建议"的对比——比如"导航软件直接带你转弯"对比"路人给你指个方向"。想一想：编程智能体更像哪个？它和"只会给你一段代码让你复制"的聊天 AI，差在哪一步？
   > **该观察到：** 编程智能体像"直接带你转弯"的导航——它自己执行、连走好几步；聊天 AI 像"指个方向"的路人——给你信息，路要你自己走。差就差在"谁来动手、谁连走好几步"。也正因为它自己动手、还可能指错路，"验收"这道关才必须留给你（第25、26章）。具体工具的形态、安装、命令**以官方文档为准**。
