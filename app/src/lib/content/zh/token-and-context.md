很多人用 AI 时都遇到过这些"怪事"：聊着聊着它就忘了开头说的事；贴一篇长文章进去，它说"太长了"；明明问得差不多，这个月的额度却用得飞快。

这些看起来不相干的现象，背后其实是同一件事——**AI 看文字的方式，和我们不一样**。它不是一个字一个字地读，也不是把整段对话"记"在脑子里。搞懂这一章的两个词，**token（令牌）**和**上下文窗口（context window）**，上面那些怪事就全说得通了。这也是整本书里最"值回票价"的一章之一：理解它，你对 AI 的很多操作会突然"开窍"。

## 一、Token：AI 切分文字的"颗粒"

我们读"请帮我写封邮件"，看到的是 7 个字。模型看到的不是 7 个字，而是被切成的几块**token**：

```text
输入：  请帮我写封邮件
token： 请 / 帮 / 我 / 写 / 封 / 邮件      （切分只是示意）
```

token 是模型处理文字的**最小单位**，但它既不等于"一个字"，也不等于"一个词"。它是模型在训练时，根据"哪些片段经常一起出现"自己总结出来的一套"文字颗粒"。

- 大致来说：**英文**常见词往往是 1~2 个 token（`unhappiness` 可能被切成 `un` / `happiness`）；**中文**则大致"一个字 ≈ 1 个 token 上下"；标点、空格、换行也各自占 token。
- 但要记住：**每个模型用的"分词器"（tokenizer）都不一样**，同一句话在不同模型里被切成的 token 数可以差出两三成。上面只是帮你建立"量级感"的**粗略经验，不是固定比例**。

> **为什么模型要这么切？** 因为如果按"字母"切，序列太长、太碎，模型抓不住意义；如果按"整词"切，世界上的词又太多、还层出不穷（新词、拼错的词、外语词）。token 是两者之间的折中：**既不太碎，又能拼出任何文字**，包括它没见过的词。你不用记住具体怎么切——只要知道"模型眼里的长度是按 token 算的，和你数的字数不完全一样"就够了。

> **速算专栏：一段文字大概多少 token？**
> 建立一个**量级感**就够用：一段几百字的短文，通常是几百到一千多个 token。想要更细的经验——常见英文大约"4 个字符 ≈ 1 个 token"，中文常常"1 个字 ≈ 1~2 个 token"。
> 但这些都是**粗略经验，不是铁律**：到底多少 token，取决于用的是**哪个模型的分词器**，还受标点、空格、数字、夹杂的代码影响。过去常说"英文比中文省 token"，但在新一些的分词器上这个差距已经小了很多，不能当定律。**要精确，就用对应产品官方的 token 计数工具（以官方文档为准）。**

## 二、上下文窗口：模型的"工作台"

知道了 token，第二个词就好懂了。

**上下文窗口**，是模型**这一次能同时"摊开来看"的 token 总量上限**。

打个比方：上下文窗口就像一张**办公桌**。你要处理一件事，得把相关的资料都摊在桌面上——你说的话、它之前的回答、你贴进来的文档、给它的指示，**全都要占桌面**。桌子有大有小，但再大也有边。

```text
这次对话桌面上摊着的东西，全都算进上下文窗口：
┌─────────────────────────────────────┐
│  你给的指示（"请用口语风格"）         │
│  你贴进来的资料（那篇 5000 字的文章）  │
│  你历次提的问题                       │
│  它历次给的回答                       │
└─────────────────────────────────────┘   ← 加起来不能超过窗口上限
```

> **要点：** 桌面上的东西**不只是你打的字**。你贴进去的长文、它回复的长篇大论，才是占地方的大头。很多人盯着自己那一行短短的问题，纳闷"我没说几句啊"，却忘了刚才贴进来的 10 页 PDF 和它洋洋洒洒的 2000 字回答，早把桌子铺满了。

不同产品、不同模型的桌子大小不一样，而且一直在变大，所以**本书不写具体数字**（写了也很快过时，以官方文档为准）。你只需要建立"桌面有限"这个直觉。

## 三、最反直觉的一点：模型其实"没有记忆"

这是初学者最容易误会的地方，单独拿出来讲。

你可能以为：我和它聊了半天，它"记住"了我们聊过的内容。**其实没有。** 模型本身不保存任何对话。

真相是：每当你发出新的一句话，软件会在背后把**到目前为止的整段对话，原封不动地重新发给模型一次**。模型每次收到的，都是"这是一段对话记录，请接着往下写"的全新请求。

```text
第 1 轮：       [你的问题A]                         → 回答a
第 2 轮：    [问题A + 回答a + 你的问题B]             → 回答b
第 3 轮： [问题A + 回答a + 问题B + 回答b + 问题C]    → 回答c
            └────────── 每轮都把前面全部重发一遍 ──────────┘
```

所谓"它记得前面说过的话"，其实是**软件每次都把前面的话重新摊到桌面上**给它看而已。一旦这段记录太长、超过桌面，**最早的内容就会被挤下桌**——于是它就真的"忘了"。

> **误解 vs 实际：** "我昨天告诉过它我叫什么，今天它该记得吧？" —— 实际上，新开一个对话，桌面是**全空的**，它对你一无所知。除非某个产品专门提供了"长期记忆"功能（**这类功能各家不同、且在变化，以官方文档为准**），否则**默认它没有昨天**。把这当成默认设定，你就再也不会奇怪"它怎么翻脸不认人"。

这也顺带解释了一个常被忘记的安全点：因为每句话都会被发给模型，所以**你贴进去的东西就等于交了出去**——这一点我们留到第 37 章细讲。

### 那 Claude Projects、ChatGPT 的"记忆"又是什么？

你可能会立刻反问：可我明明见过一些产品能"记住"我的偏好、记住整个项目啊？比如 Claude 的 Projects、ChatGPT 的 Memory（记忆）、写代码工具里的"项目上下文"、账号里的个性化设置、公司内部的知识库……这和"模型没有记忆"矛盾吗？

**不矛盾。** 关键是分清两层：

| | 模型本体 | 产品层的"记忆" |
| --- | --- | --- |
| 是什么 | 那个只会"接话"的内核 | 产品在模型外面加的一层功能 |
| 记不记得 | **不记得**，每次空白开始 | 帮你把"该记的东西"存起来 |
| 怎么做到的 | —— | **每次对话时，自动把这些信息重新塞回这次的上下文里**（又回到上一节那张"桌子"） |

也就是说：所谓"它记住了我"，**不是模型本体长出了记忆，而是产品每次替你把相关信息重新摊回桌面**。掀开盖子看，底下还是"无记忆的模型 + 每次重新喂上下文"。

> **要点：** 以后看到 Claude Projects、ChatGPT Memory、项目上下文文件这类功能，不必觉得和本章冲突——它们都是**产品层**的贴心设计，不是模型本体的记忆。各家具体怎么记、记多久、默认开还是关、能不能删，**以官方文档为准**（这些功能多数默认开启、可在设置里关闭）。理解了"底层无记忆、上层靠重新喂"，你就能预判它们的脾气：它"记得"的，永远是产品决定塞回桌面的那部分；没塞回来的，它照样不知道。

## 四、为什么"越聊越笨"？

就算还没超过上限，**对话越长，质量也往往越往下掉**。原因有三个，都能用"桌面"来理解：

| 原因 | 桌面比喻 | 具体表现 |
| --- | --- | --- |
| 重点被淹没 | 桌上资料太多，真正重要的那张纸被压在底下 | 你早先定的关键要求，它后来"看不见"了 |
| 矛盾在堆积 | 试错过程中产生的旧方案、错信息还摊在桌上 | 它被自己之前的错误带着走，越绕越偏 |
| 摘要丢细节 | 桌子快满时，旧资料被"压缩成便签"，细节没了 | 早期说过的具体数字、约定，被模糊化 |

> **实际会怎么遇到：** 一个典型场景——你和它来回打磨一篇稿子，前 10 轮挺好，越往后它越"轴"，开始无视你的新要求、翻炒早就否决过的写法。这不是它"累了"，是桌面被这一长串试错塞满了，**噪音盖过了信号**。

## 五、token 还决定了"费用"和"能塞多少"

最后一件实用的事：**token 是计量单位**。

- 你用 AI 的**花费**（无论是按量付费，还是套餐额度），本质上都按 **token 数**来算——发进去的、吐出来的，都算。
- 你一次能贴进去多少资料的**上限**，也是用 token 数衡量的（就是上下文窗口）。

所以会出现这种事：你觉得"我就问了个小问题"，但因为前面贴了一大段资料、又让它生成了长长的回答，**token 蹭蹭涨，额度掉得飞快**。第 38 章会专门讲怎么"省着用"，这里先记住一句话——

> **要点：** **桌面铺得越满 = 越慢、越贵、还越容易出错。** 这三件事是连在一起的。所以"把对话保持清爽"不只是为了省钱，也是为了让它答得更准。这正是后面"上下文管理"（第 13 章）那一整套技巧的根。

## 六、常见误解，集中纠正

| 误解 | 实际 |
| --- | --- |
| token 就是一个字 | token 是模型的切分单位，可能是一个词、半个词或一个字，和"字数"不完全对应 |
| 它记得我们以前的对话 | 模型本体默认没有跨对话记忆，每次都是软件把历史重新摊上桌。（产品另提供的"记忆/项目记忆"是产品层把信息重新喂回来，不是模型本体记得，见上节） |
| 对话框是空的，这次没花多少 | 你贴的长文、它回的长答，才是占 token 的大头 |
| 上下文窗口越大越没烦恼 | 桌子越大确实能放更多，但塞太满照样"重点被淹没"，而且更慢更贵 |
| 它"忘事"是出了故障 | 这是上下文有限的正常表现，不是 bug |

## 小结

- **token** 是模型切分文字的"颗粒"，不等于字也不等于词；你的费用和输入上限都按它计。
- **上下文窗口**是模型这次能同时摊在"桌面"上的 token 上限——你说的、它说的、你贴的资料，全都占桌面。
- 模型**没有跨对话记忆**：是软件每次把历史重新发给它，超出窗口的旧内容会被挤掉，于是它"忘了"。
- 对话越长越容易"变笨"：重点被淹没、矛盾在堆积、细节被压缩。
- **桌面越满 = 越慢、越贵、越容易错**。保持对话清爽，是贯穿全书的基本功（第 13、38 章）。

## 小测验

> 共 6 题，涵盖概念、误区、情景、操作四类。答案与解析在每题下方的引用块里——先自己想，再对照。

1. **[基础·概念题]** 下面哪一句最准确？
   - A. token 就是一个汉字或一个英文字母
   - B. token 是模型切分文字的单位，可能是一个词、半个词或一个字
   - C. token 就是一句话
   - D. token 是模型的记忆单位
   > **答案 B。** token 是模型自己的"文字颗粒"，不等于"字"也不等于"词"——英文常常一个词被切成一两个 token，中文则大致一字一个上下。A 是最常见的误解（把 token 当字符）；D 把 token 和"记忆/上下文"混为一谈。

2. **[基础·概念题]** "上下文窗口"指的是？
   - A. 聊天软件的窗口大小
   - B. 模型一次最多能同时"看"的 token 数量上限
   - C. 模型记住你的天数
   - D. 网速上限
   > **答案 B。** 它是模型这一次能摊在"桌面"上同时处理的文字量上限——你说的、它说的、你贴的资料，全都算在内。

3. **[基础·误区题]** "我昨天告诉过它我的名字，今天它应该还记得"——这个想法错在哪？
   - A. 没错，它会永久记住
   - B. 它本就没有跨对话的记忆；每次对话都是从空白桌面开始
   - C. 它记得，但要付费解锁
   - D. 它只记数字不记名字
   > **答案 B。** 模型本体默认没有"昨天"，新开一个对话它对你一无所知。**注意**：有些产品会在模型之外提供"记忆/项目记忆"（如 Claude Projects、ChatGPT Memory、项目上下文文件），那是**产品层**每次帮你把信息重新塞回这次对话，不是模型本体记得了（**具体以官方文档为准**）。把"模型本体无记忆"当默认，你既不会奇怪它"忘事"，也不会被产品的记忆功能搞糊涂。

4. **[进阶·误区题]** "对话框是空的，所以这次没占多少 token"——哪里不对？
   - A. 完全正确
   - B. 你贴进去的长文、它给的长回答，都在悄悄占满窗口
   - C. token 只算你打的字
   - D. 空对话框会扣双倍 token
   > **答案 B。** 很多人只盯着自己打的那行字，忘了**贴进来的 10 页 PDF 和它回的 2000 字**才是吃 token 的大头。

5. **[基础·情景题]** 你和 AI 聊了很久，它开始答非所问、忘记前面说的设定。最该先做什么？
   - A. 重启电脑
   - B. 开一个新对话，把"还需要的信息"重新交代一遍
   - C. 把同一句话再问五遍
   - D. 卸载重装
   > **答案 B。** 这是上下文窗口被塞满、早期内容被挤掉的典型症状。新对话 = 干净桌面，比在一张乱桌子上继续翻找更快更准（详见第 13 章）。

6. **[基础·操作/观察题]** 找一段约 1000 字的中文短文，先估一估"大概多少 token"，再贴进任意带 token 统计的工具看看实际值（**具体以官方文档为准**）。观察：中文和英文，谁更"费 token"？
   > **该观察到：** 一段几百字的文字通常是几百到一千多个 token；具体数字取决于用的是哪个模型的分词器，还受标点、数字、夹杂代码影响——**这是粗略经验，不是固定比例**（"英文一定更省"在新分词器上已不太成立）。要精确就用官方计数工具。亲手比一次，"篇幅—token—费用"的直觉就建立起来了。
