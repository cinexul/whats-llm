〔本章不依赖任何具体工具的命令或界面——它讲的是"翻车的共性"和"急救的通用动作"。涉及某工具具体怎么操作之处，一律**以官方文档为准**（见附录 C）。〕

前面几章把"怎么带编程智能体顺利干活"讲透了：通用工作流（第24章）、Claude Code 的落地（第28章）、Codex 的落地（第29章）。但有一件事必须坦白：**不管你多熟练，翻车都会发生。** 它会犯错、会跑偏、会自信地胡来——这是第24章就说过的，编程智能体"会犯错 + 会动手"的本性决定的。

所以这一章不教你怎么"避免一切翻车"（那不现实），而是教你两件更实在的事：**一，认得出八种最常见的"翻车现场"，知道病根在哪、怎么对治；二，真卡住了、改乱了，有一套"急救清单"能把你捞回干净状态。** 这一章不分工具，Claude Code、Codex 都适用。

## 一、八大翻车现场：症状 → 原因 → 对策

下面每一条都是新手反复踩的坑。建议你**对号入座**——多半你已经中过其中几条。

### 1. 任务太曖昧，它丢三落四

- **症状**：你说"帮我优化一下这个页面"，它改出来的东西完全不是你想要的，或只做了一半。
- **原因**：指令太笼统，它只能**猜**你要什么——而它常常猜错（呼应第11章"把要什么说清楚"）。
- **对策**：把目标说**具体**：要达成什么、动哪一块、什么算"做完"。先让它**复述一遍它的理解**再动手，比事后返工省得多。

### 2. 一次改太大，根本没法验收

- **症状**：你让它"把整个功能全做完"，它改了十几个文件，你看着满屏 diff 一脸懵，出了问题也不知从哪查。
- **原因**：改动越大，**盲区越大**；大到你没法逐行验收，也难回滚（第24章反复强调过）。
- **对策**：**任务拆小**——做一小块、验收一小块、提交一次，再做下一块。小步慢跑反而最快（第27、29章）。

### 3. 不看 diff，连点"同意"

- **症状**：它每次请求许可你都秒点"同意"，回头发现它动了不该动的文件，甚至删了重要东西。
- **原因**：把"许可确认"当成麻烦、当成例行公事——**等于把刹车拆了**（第26章）。
- **对策**：**许可前先看它要做什么、改了什么（diff）**。这一步几秒钟，挡掉的可能是几小时的灾难。看不懂就先问"这步是干嘛的"，**看不懂别放行**（第25章）。

### 4. 一个对话从头聊到尾，不切换

- **症状**：你在同一个对话里聊了几十轮，它越来越"轴"——无视你的新要求、翻炒早被否决的废案、自相矛盾。
- **原因**：这正是第3、13章那张"桌面"——对话越长，**噪音盖过信号**，重点被淹没、矛盾在堆积。它不是变笨了，是桌子被塞满了。
- **对策**：**一个话题一个对话**；觉得它开始"轴"了，果断**开新对话**，把"还需要的信息"重新交代清楚（第13章）。换张干净桌子，比在乱桌上硬翻快得多。

### 5. 轻信幻觉，不复核

- **症状**：它信誓旦旦地用了一个根本不存在的写法、引了个查无此事的依据，你照单全收，结果出错。
- **原因**：它追求的是"像真的"，不是"是真的"（第9章"幻觉"）。**它说得越流畅自信，越不代表越可靠。**
- **对策**：**越要紧、越是它张口就来的事，越要亲手验**。让它给依据、跑测试验证；关键事实自己再查一遍（第9章）。

### 6. 失败方针硬纠，越改越歪

- **症状**：一个方向明显走不通了，你还揪着它在原方案上一句句改，结果越改越乱、越绕越偏。
- **原因**：错误的上下文已经堆满了"桌面"，它被自己之前的错误**带着走**（第13章）。在乱上下文里硬纠，往往是负功。
- **对策**：**承认此路不通，回滚 + 重开**。回到上一个干净状态，开个新对话，把"刚才踩的坑"当成前提重新交代（第13、24章）。这几乎总比硬纠快。

### 7. 没测试就大改

- **症状**：它改完你直接合并、上线，过两天才发现它把原来好好的功能弄坏了。
- **原因**：把"它说做完了"当成了"它做对了"——**没有验证，就没有把握**（第24、27章）。
- **对策**：改完**让它跑测试**，确认没把原来的功能弄坏；自己也试着用一句话说清"这段在干嘛"。**说不清、没测过，别合并**（第25章）。

### 8. 贴了密钥/敏感信息

- **症状**：图省事把 API 密钥、密码、真实个人信息贴进了对话，或写进代码推到了公开仓库。
- **原因**：忘了那条铁律——**你贴进去的东西就等于交了出去**（第3章）；密钥是"钱包钥匙",谁拿到都能花你的钱（第29章）。
- **对策**：**密钥/密码/敏感信息绝不进对话框、不进公开仓库**（第37章）。万一贴出去了，**立刻去对应平台吊销旧密钥、换新的**（**具体操作以官方文档为准**）,别心存侥幸。

> **要点：** 把这八条连起来看，你会发现它们其实只违反了**三条原则**：①没把话说清（1、2）、②没守住"动手前后两道闸"(3、5、7)、③上下文/数据没管好（4、6、8）。记住这三条，比死记八个现场更管用——它们正是第24章那套工作流和全书安全纪律的反面。

## 二、卡住了怎么办：五步急救清单

上面是"防",这一节是"救"。当它已经跑偏、你已经一头雾水、改得一团乱时，**别慌，也别在乱局里继续缠斗**。按下面五步走，几乎总能把你捞回来。

```text
卡住/改乱了，按顺序来：
① 先中断  ──→ ② 说清现状  ──→ ③ 回滚到上个好状态  ──→ ④ 开新对话重来  ──→ ⑤ 诊断
（立刻喊停）  （别急着让它继续改）   （回到干净的地方）      （换张干净桌子）    （想清为什么会偏）
```

**① 先中断——立刻喊停**
觉得不对劲，**第一反应是打断它**,而不是看着它继续往坑里跑。"能随时叫停"是你的基本权利（第26章）,越早停损失越小。停下来，深呼吸，别在慌乱中继续点"同意"。

**② 说清现状——先盘点，别急着让它继续改**
停下后先问自己三句：**它现在改了哪些东西？哪些是我想要的、哪些是它跑偏弄的？我上一个"干净、能用"的状态在哪里？** 把现状盘清楚，你才知道往哪退。这一步是给后面"回滚"打底——不盘清就回滚，容易退过头或退不到位。

**③ 回滚到上个好状态——这就是你之前切分支的意义**
这是急救的核心动作。如果你**听了前几章的话**,一开始就在用 **Git** 管理、并切了**新分支**（第26章）,那么现在你可以**一键回到那个干净状态**,把它跑偏弄的东西一笔勾销。

> **要点：** 第27、28、29章反复念叨"先切 Git 分支再动手",**就是为了第③步这一刻**。回滚不是失败，是你**最大的安全网**。所谓"能反悔的环境",兑现的就是这个能力。（网页/云端形态通常有平台自己的回滚/版本机制，**以官方文档为准**。）具体怎么回滚的命令，**见附录 C 和官方文档**——本章只讲"回到上个好状态"这个动作本身。

**④ 开新对话重来——换一张干净桌子**
回到干净状态后，**别在那个已经乱掉的对话里继续**。开一个**新对话**,把"刚才那条路走不通、踩了什么坑"当成**前提**告诉它，让它带着这个教训重新规划。为什么要开新的？因为旧对话的"桌面"上堆满了错误尝试（第13章）,在那上面继续，它还会被旧错误带着走。

**⑤ 诊断——想清"为什么会偏",别原地再栽一次**
最后花一分钟复盘：这次跑偏，根子在哪？是**任务说得太曖昧**（回第一节第1条）?**一次让它改太多**（第2条）?还是**该早点回滚却硬纠了**（第6条）?把病根找到，下次重来时**从源头堵住**——比如这次先让它出个计划你审一遍（第27章）,或者把任务拆得更小。

> **实际会怎么遇到：** 新手最常见的错，不是"翻车"本身，而是**翻车后死活不肯回滚**——总觉得"再调一下就好了",结果在乱局里越陷越深，半小时后比一开始还糟。**老手的本能恰恰相反：一旦感觉方向不对，宁可回滚重来，也不在烂摊子上缝缝补补。** 回滚 + 开新对话，往往是最快的路，不是认输。

## 三、一句话总纲

把这一章浓缩成一句能随身带的话：

> **防，就守住第24章那套工作流（说清楚、拆小、看 diff、跑测试）;救，就记住"中断→盘点→回滚→重开→诊断"。** 翻车不可怕，可怕的是在乱局里硬撑。你最大的底气，是那个"能反悔的环境"——所以"先切分支再动手"这一步，千万别省。

## 四、常见误解，集中纠正

| 误解 | 实际 |
| --- | --- |
| 我够熟练就能不翻车 | 它"会犯错+会动手",翻车是常态；关键不是不翻，是会救（第24章） |
| 跑偏了就追着它一直纠 | 该回滚+开新对话重来；在乱上下文里硬纠往往越改越糟（第6条、第13章） |
| 许可确认是麻烦，连点同意就行 | 那是拆刹车；许可前先看 diff,看不懂别放行（第3条、第26、27章） |
| 它说做完了就是做对了 | 没测过、说不清就别合并（第7条、第25章） |
| 回滚=失败、=认输 | 回滚是最大的安全网，是老手的本能，往往最快（第②③步） |
| 贴一次密钥应该没事 | 贴进去＝交出去；立刻吊销重发，别侥幸（第8条、第37章） |

## 小结

- 编程智能体**一定会翻车**（会犯错+会动手）;目标不是"绝不翻车",而是**认得出现场、救得回来**。
- **八大翻车现场**:曖昧丢任务、一次改太大、不看 diff 连点许可、会话不切换、轻信幻觉、失败方针硬纠、没测试就大改、贴了密钥。它们其实只违反三条：没说清、没守两道闸、上下文/数据没管好。
- **五步急救清单**:①先中断 ②说清现状 ③回滚到上个好状态 ④开新对话重来 ⑤诊断。核心是第③步回滚——这正是"先切 Git 分支再动手"兑现的价值。
- 最大的误区是**翻车后死不回滚**、在乱局里硬撑；老手的本能是**宁可回滚重来**（呼应第13、24、25章）。
- 具体回滚命令、工具操作**以官方文档为准**（见附录 C）;本章讲的"动作"本身不分工具、不会过时。

到这里，第四部分"AI 编程实战"就告一段落：你已经有了通用工作流、两大工具的上手地图、以及翻车时的急救能力。下一部分，我们就**真正动手做项目**,把这些能力变成看得见的成果。

## 小测验

> 共 6 题，涵盖概念、误区、情景、操作四类。先自己想，再看引用块里的答案解析。

1. **[基础·概念题]** 本章对"翻车"的基本态度是？
   - A. 只要够熟练，就能做到永不翻车
   - B. 编程智能体"会犯错+会动手",翻车是常态；目标不是杜绝翻车，而是认得出、救得回
   - C. 翻车了就说明这个工具不行，该换
   - D. 翻车都是用户太笨
   > **答案 B。** 第24章就讲过，翻车源于编程智能体"会犯错+会动手"的本性。成熟的用法是**会救**——认得出八大现场、用得了五步急救——而不是幻想永不出错。

2. **[基础·概念题]** 五步急救清单的正确顺序是？
   - A. 先让它继续改 → 再说
   - B. 先中断 → 说清现状 → 回滚到上个好状态 → 开新对话重来 → 诊断
   - C. 先删项目 → 重装工具
   - D. 先开新对话 → 再看要不要中断
   > **答案 B。** 顺序很重要：**先停损（中断）、再盘点（说清现状）、回到干净处（回滚）、换张干净桌子（开新对话）、最后想清为什么（诊断）。** 跳过盘点直接回滚容易退错位置，不中断就继续会越陷越深。

3. **[进阶·误区题]** "它跑偏了，我揪着在原方案上一句句纠正就行"——为什么往往越纠越糟？
   - A. 没问题，多纠几次就好
   - B. 错误的尝试已经堆满了上下文这张"桌面",它被自己之前的错误带着走；该做的是回滚+开新对话重来
   - C. 它纠正时会变快
   - D. 纠正不花 token
   > **答案 B。** 这是第八大现场"失败方针硬纠",根子在第13章那张被塞满的"桌面"。**回到干净状态、开个带着教训的新对话**,几乎总比在乱局里硬纠快——这也是急救清单③④步。

4. **[进阶·误区题]** "回滚就等于失败、等于认输"——这个想法错在哪？
   - A. 没错，回滚很丢人
   - B. 回滚是你**最大的安全网**,是老手翻车后的本能；它往往是最快回到正轨的路，不是认输
   - C. 回滚会把整个项目删掉
   - D. 只有付费版才能回滚
   > **答案 B。** "先切 Git 分支再动手"（第26章）铺垫的就是这一刻。新手最常见的错恰恰是**死不回滚、在烂摊子上硬撑**,结果越来越糟。把"宁可回滚重来"练成本能，是这一章最值钱的一句。

5. **[基础·情景题]** 你正盯着它干活，突然发现它要去删一个你觉得很重要的文件。这一刻最该做的是？
   - A. 反正它是 AI，应该有道理，点"同意"
   - B. 立刻**中断/不批准**这步操作，看清它到底想做什么，搞不清就先问"这步是干嘛的"
   - C. 关掉电脑
   - D. 把整个项目重做
   > **答案 B。** 这同时踩中"不看 diff 连点同意"（第3现场）和急救第①步"先中断"。**许可前先看清、看不懂别放行**（第26、27章）——"能随时叫停"是你的基本权利，越早停损失越小。

6. **[基础·操作/观察题]** 不必真装工具：回看本章八大翻车现场，**对号入座**——你过去用 AI（哪怕只是聊天）时，最常踩哪两三条？再把对应的"对策"抄下来，贴在你以后会看到的地方。
   > **该观察到：** 多数人会发现自己反复栽在同样一两条上（很常见的是"会话不切换"和"轻信幻觉"）。八大现场其实只违反三条原则——**没说清、没守两道闸、上下文/数据没管好**;盯住你自己的高频项、从源头堵住，比记全八条更有用。具体工具怎么回滚、怎么操作，**以官方文档为准**（见附录 C）。
