上一章讲了 Claude Code，这一章轮到 **Codex**。如果你跳着读到这里，先补一句背景：第24章把"调查→计划→动手→验收"这套带编程智能体干活的**通用工作流**讲透了，第28章把它落到了 Claude Code 上——**这一章是同一套工作流在 Codex 上的落地**。

所以你会发现，这一章和上一章在"心法"上高度一致，但在"具体怎么操作"上各走各的。这正是要点：**工作流通用，工具各异。** 同样地，这一章**不要求你会写代码**，也**不写死任何具体命令、按钮、快捷键**——那些更新太快，全部交给官方文档和附录 C。

## 一、Codex 是什么：来自 OpenAI 的编程智能体

先记一个稳定事实：**Codex 来自 OpenAI。** 它和 Claude Code 是"同类不同家"——都是会在你项目上**自己读文件、改代码、跑命令**的编程智能体（coding agent），但出品方和内核模型不同。

和 Claude Code 一样，**Codex 也不是某个固定的网页或软件**。你可能会在多种形态里遇到它：

- **命令行（CLI）**：在终端里带它干活；
- **代码编辑器（IDE）插件**：把它嵌进你的编辑器；
- **网页界面（常被叫作 Codex 的网页/云端工作区）**：在浏览器里给它派任务；
- **云端任务**：把活儿交给它在云端后台跑，你回头来收。

> **要点：** 和第28章同一句话：**有的形态跑在你自己的机器上，有的跑在云端，具体形态各不相同、而且在快速更新——以官方文档/产品官方形态为准。** 你能稳稳记住的只有两点：它来自 OpenAI、它是个会动手的编程智能体。剩下"怎么装、长什么样、敲什么",一律去官方文档看当下的版本。

## 二、和 Claude Code 的异同：心法一样，手法各异

这是本章最有用的一节。把第28章和这一章并排看，你会立刻抓住"通用的"和"各异的"分别在哪。

| | 相同（稳定，学一次到处用） | 不同（**以官方文档为准**，会变） |
| --- | --- | --- |
| **定位** | 都是"会在你项目上动手"的编程智能体 | 出品方（OpenAI vs Anthropic）、内核模型不同 |
| **工作流** | 都适用第24章"调查→计划→动手→验收" | 具体怎么触发"出计划""看 diff"各异 |
| **安全** | 都该配许可确认 + 可回滚环境 | 默认行为、许可粒度、确认方式不同 |
| **上手** | 都先在能反悔的环境里小步试 | 安装方式、登录/认证方式、命令不同 |
| **形态** | 都有 CLI / IDE / 网页 / 云端等多形态 | 各形态的具体名称、入口、能力不同 |

把它读成一句话：**你在第24章学到的判断力——派什么活、怎么批准计划、怎么看 diff、怎么验收——换到 Codex 上一字不改地适用；需要重新查的，只有"在 Codex 里这些动作具体怎么做"。**

所以面对 Codex，正确的姿势和上一章如出一辙：

- **先别急着上手操作。** 去官方文档看"当下推荐哪种形态、怎么装、怎么登录"，照着走（**以官方文档为准**）。
- **把第24章那套工作流带过来。** 让它**先调查、再出计划、你批准了再动手、动完认真验收**——这套节奏到哪个工具都对。
- **第一次用能反悔的小项目。** 在用 Git 管理、切了新分支的状态下跑通一个最小请求（第25、27章），别一上来就让它在要紧的东西上动手。

> **实际会怎么遇到：** 不少人同时用 Claude Code 和 Codex,然后被"为什么这个命令在那边不灵"绊住。**别把一个工具的具体命令当成另一个的。** 真正能跨工具搬运的，是第24章那套"怎么带一个会犯错、会动手的助手"的判断力；具体命令请各自看各自的官方文档。记住这条，你就不会在两边的差异里打转。

## 三、密钥与费用：从第一天就管好

这一节单独拎出来重点讲——因为 Codex 这类工具往往要用到 **API 密钥（API key）**，而**密钥和费用是最容易在第一天就出事的地方**。这里只讲"为什么要管、怎么判断、注意什么"，呼应第35、37、38章。

### 1. 密钥是什么、为什么要当回事

你可以把 **API 密钥**理解成一串"代表你身份和账单的钥匙"。**谁拿到它，就能以你的名义调用服务、花你的钱。** 所以它的安全等级，等同于密码——甚至更高，因为它常常直接连着付费。

由此推出几条**不容商量的纪律**（第37、35章会再展开）：

- **绝不把密钥贴进对话框。** 别因为图省事，把密钥粘进和 AI 的聊天里——你贴进去的东西就等于交了出去（第3章那条"贴进去＝交出去"）。需要配置密钥时，按官方文档指定的**安全方式**来（**具体方式以官方文档为准**）。
- **绝不把密钥提交到公开仓库。** 把密钥写进代码、又推到公开的 Git 仓库，是最经典、也最惨烈的泄露——爬虫几分钟就能扫到，然后有人替你刷账单。配置类的秘密应放在**不会被提交**的地方（第35章会讲到"环境变量/密钥管理"这类做法，**具体以官方文档为准**）。
- **怀疑泄露就立刻作废重发。** 万一不小心贴出去、提交出去了，第一时间去对应平台**吊销旧密钥、换新的**（**具体操作以官方文档为准**）,别心存侥幸。

> **要点：** 一句话刻进脑子——**密钥就是你的钱包钥匙，不进对话框、不进公开仓库。** 这条不分工具、不分形态，任何要用密钥的地方都成立。它呼应的是第37章（隐私与数据分级）那条铁律：**最高密级的东西，绝对不喂。**

### 2. 费用：按 token 计，别不知不觉烧钱

Codex 这类工具本质上是**按 token 计费**的（第3、38章讲过 token 是计量单位）。这意味着：

- **你让它读越多文件、聊越长、改越多，花得越快。** 一次"让它把整个项目读一遍再大改",可能悄悄烧掉不少 token。
- **云端/后台任务尤其要心里有数。** 把活儿丢给它在云端跑，方便，但也意味着它可能"自己跑很久";开工前对"大概要做多少"有个预期，别撒手不管。
- **保持任务聚焦，就是最好的省钱法。** 把任务拆小、对话保持清爽，既省 token 又提质量——这和第38章"别烧钱也别浪费"是同一件事的两面。

> **要点：** 设个心理上的"预算闸"。开工前问自己：这个任务**值不值得**让它读这么多、跑这么久？把任务拆小、随时能叫停（第26、30章）,既是安全，也是省钱。具体的计费方式、单价、额度**以官方文档为准**——本书只讲"按 token 计、聊越多越贵"这个不变的机制。

## 四、深度用法与具体操作：以官方文档为准，见附录 C

和 Claude Code 一样，Codex 也有它自己的一整套进阶能力、配置、命令——但这些**恰恰是更新最快、最不该写进书里的部分**。本书的处理依旧明确：

- **具体怎么装、怎么登录、敲什么命令、点哪个按钮**——一律**以官方文档为准**。
- **进阶玩法（自定义配置、与外部工具/数据的连接、自动化等）**——指向**附录 C** 与官方文档，不在正文里写死。
- **它能不能做某件具体的事、有哪些档位、模型版本叫什么**——这些都是易变项，**以官方文档为最新依据**。

> **要点：** 你也许会好奇"Codex 和 Claude Code 到底谁强、谁能做 X"。这类对比**会随版本变化而频繁翻盘**,本书不下这种结论。你真正要带走的，是一种**不依赖具体工具的能力**:看懂第24章那套工作流、守住密钥和验收的纪律。工具谁强谁弱会变，这份判断力不会。具体能力对照，请各自查官方文档。

## 五、安全清单（和上一章共用一套底线）

无论用 Claude Code 还是 Codex，这几条底线都一样，放这儿再钉一遍：

- **能反悔的环境优先。** 它在本地动手时，先用 **Git** 起步、先**切新分支**（第26章）;云端/网页形态用平台自己的隔离与回滚（**以官方文档为准**）。
- **许可确认别无脑放行。** 它每要做有后果的事会请求许可（第26章）,别一路点"同意"——那是拆刹车。
- **密钥/密码/真实个人信息绝不外贴。** 见本章第三节、第37章。
- **验收是你的责任。** 看 diff、跑测试、说得清这段在干嘛——看不懂别合并（第25章）;生成内容算你的责任（第39章）。

## 小结

- **Codex 来自 OpenAI**（稳定事实），是会在你项目上动手的编程智能体；它**形态多样（CLI/IDE/网页/云端）且在更新**——怎么装、怎么操作**以官方文档为准**。
- 与 Claude Code **心法一样、手法各异**：第24章"调查→计划→动手→验收"通用，**具体命令/安装/能力各不相同且更新快**。
- **密钥与费用从第一天就管好**：密钥绝不进对话框、不进公开仓库（第37、35章）；它按 token 计费，任务拆小才省钱（第38章）。
- **深度用法与具体操作见附录 C 和官方文档**，本书不写死、也不下"谁更强"的结论。
- 安全底线和上一章共用一套：能反悔的环境、不无脑放行许可、密钥不外贴、验收自己负责。

下一章我们换个角度：不管用哪个工具，**翻车都会发生**。第30章把八大常见"翻车现场"和"卡住了怎么急救"一次讲清。
