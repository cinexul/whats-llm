到目前为止，你用 AI 的方式都是"打开一个界面、自己在里头聊"。这一章要做的事不一样：让**你自己的程序**去用 AI——不是你坐在对话框前打字，而是你的网站、App 或脚本，在背后**自动**把问题发给模型、把回答拿回来，再嵌进你自己的应用里。

这就是第17章那张"四层地图"里的第③层——**API（接口）**。还记得那个比喻吗：餐厅是给走进来的客人吃的，而别的店想用这位厨师的手艺，就走"接单窗口"下单。API 就是这个窗口，只不过来下单的不是人，是程序。

先说清楚这一章的定位：**这一章只讲"思路与数据流"，不写死任何一种语言的代码。** 因为各家 API 的具体写法、参数、SDK 用法不一样、还经常更新——那些一律**以官方文档为准**。你要带走的，是"接 API 到底是怎么一回事"这个能让你和工程师对上话、也能自己判断的框架。

## 一、准备

哪怕你不亲自写代码，下面几件事也得心里有数——尤其是密钥和费用，**从写第一行起就要管好**。

- **先确认你真的需要 API，而不是开个会员。** 这是最常见的误区，单独拎出来说：如果只是你一个人、偶尔让 AI 帮你改改文章，用现成的聊天**产品**就够了（第17、18章）。只有当你要让**自己的系统自动调用** AI——比如让公司官网的客服自动回话、让你的程序批量处理数据——才需要走 API。**接 API ≠ 开会员，这是两件事。**
- **拿一把"密钥（API key）"。** 用 API 通常要先在对应平台申请一把密钥，用来证明"是你在下单"，并据此**按 token 计费**（第3、38章）。具体怎么申请**以官方文档为准**。
- **从第一天就把密钥和费用当回事。** 这把密钥极其敏感——**泄露了等于别人能拿你的账号点单、记你的账单**。所以下面第二节的第 0 步，专门讲怎么管它。别等出了事才补。
- **想清楚"数据流"长什么样。** 接 API 的本质，就是设计一条"问题出去、回答回来、再用起来"的流水线。先把这条线在脑子里画清楚，写代码只是把它落地。

> **要点：** 接 API 这件事，技术门槛其实没你想的高（最小可跑的版本很短），真正的门槛在两个判断：**一是分清"该用 API 还是该开会员"，二是从一开始就把密钥和费用管好**。这两件事想明白了，后面交给官方文档照着走即可。

## 二、一步步做

下面把"接 API"拆成几步，全部只讲思路和数据流。具体用什么语言、怎么写、调哪个接口，**以官方文档为准**。

### 第 0 步：先把密钥安顿好（在写任何功能之前）

这一步排在所有功能前面，因为它最容易被忽略、又最容易闯大祸。三条铁律（呼应第37章隐私、第38章费用）：

| 该怎么做 | 为什么 |
| --- | --- |
| **把密钥放进"环境变量"，别写进代码里** | 写死在代码里，密钥就跟着代码到处跑、随时可能泄露 |
| **绝不把密钥提交到公开仓库（如公开的 GitHub）** | 公开仓库里的密钥会被自动扫描程序秒级捡走、盗用 |
| **去平台后台给账号设"用量上限 / 预算告警"** | 万一密钥泄露、或程序写出死循环狂刷，上限能帮你兜住损失 |

"环境变量"通俗说，就是把密钥这种敏感信息**放在程序外面**、运行时再读进来，而不是明晃晃写在代码文件里。具体怎么设环境变量、怎么设用量上限，**以官方文档为准**——但"放外面、别提交、设上限"这三条原则是通用的，不会过时。

> 把这一步当成"接 API 的安全带"：车还没开，带子先系好。

### 第 1 步：让程序代你"问一句"

最小可跑的核心，就一件事：**你的程序把一个问题，通过 API 发给模型。** 用大白话描述这条请求：

```text
（你的程序，在背后自动做这件事）
对模型说：「这是一段用户留言，请帮我判断它是好评还是差评，
          只回答'好评'或'差评'。留言内容：……」
```

这和你在聊天框里打字本质一样——都是"把一段话发给模型"。区别只在于：**现在是程序替你发，而且可以一天发一万次，不用你手动打。** 发什么内容（也就是"提示词"），还是第11章那套讲究：把要求说清楚、把格式定死。

### 第 2 步：把模型的回答接回来

模型处理完，会把回答**传回给你的程序**。你的程序拿到的，通常是一段文字（或带点结构的数据，具体格式**以官方文档为准**）。

这里有个新手常踩的认知坑：**模型返回的内容仍然可能出错、可能不按你要的格式来**（幻觉照样发生，第9章）。你让它"只回答好评或差评"，它偶尔可能回一整句解释。所以程序里要**对返回结果做检查和兜底**——这部分属于工程细节，但理念你得有：**别假设模型每次都乖乖听话。**

### 第 3 步：把回答"用起来"，嵌进你的应用

拿到回答后，接下来就和 AI 没关系了——是你**自己应用的逻辑**：把"好评/差评"存进数据库、显示在页面上、触发一封通知邮件……这部分完全由你的程序决定。

把整条数据流连起来看，就是这么一条线：

```text
 用户/你的数据
      │
      ▼
 你的程序  ──(通过 API，带着密钥)──→  模型
      ▲                                │
      │                                ▼
      └────────  模型把回答传回  ───────┘
      │
      ▼
 你的程序把回答用起来（存库 / 显示 / 触发下一步）
```

看明白这条线，你就抓住了"给程序接 AI"的全部精髓：**你的程序把问题发出去、把回答收回来、再嵌进自己的流程。** 中间那个"模型"是租来的能力，你付的是 token 的账。

### 看结果 / 验收

最小版本跑起来后，按这几条验收：

1. **能不能稳定地"发出去、收回来"？** 拿几条真实数据走一遍，确认这条流水线通了。
2. **密钥安全吗？** 回头检查：密钥是不是放在环境变量里、没写进代码、没提交到公开仓库、设了用量上限。**这一条要反复确认。**
3. **返回结果有没有兜底？** 故意喂一条刁钻的输入，看模型不按格式回时，你的程序会不会崩、有没有处理。
4. **花了多少钱？** 去平台后台看一眼这次测试的用量和花费，对"调一次大概多少钱"建立直觉（第38章）。

## 三、常见坑

| 坑 | 怎么回事 / 怎么躲 |
| --- | --- |
| **以为"接 API = 开会员"** | 这是最大的误区。开会员是给**人**用产品；接 API 是给**程序**用接口，要申请密钥、按 token 计费（第17章）。先分清你要哪个 |
| **把密钥写死在代码里 / 提交到公开仓库** | 等于把家门钥匙贴在门上。密钥放环境变量、别提交、设上限——**第一行代码之前就做**（第37、38章） |
| **没设用量上限，被狂刷或死循环烧爆** | 程序一个 bug 可能一秒调几百次。后台设好预算告警和用量上限，给账单兜底（第38章） |
| **假设模型每次都按格式乖乖回** | 它会出错、会跑偏（第9章）。程序里要对返回结果做检查和兜底，别裸奔 |
| **把用户隐私数据原样发给 API** | 记住第37章那句话：**发进去就等于交出去了**。敏感信息要先脱敏，是否用于训练取决于条款与设置（第37章，以官方文档为准） |
| **照着网上某段旧代码硬抄** | 各家 API 写法不同、还常更新。**以官方文档为准**，别迷信过时的代码片段 |

## 四、可以再往前一步

最小流水线跑通后，可以这样扩展（每一步都别忘了密钥和费用这根弦）：

- **把"提示词"打磨好**：你发给模型的那段话，质量直接决定回答质量。把第11、12章的提示词技巧用上，必要时固化成模板。
- **接更结构化的输出**：让模型返回更规整的结构（方便程序直接用），具体怎么要求**以官方文档为准**。
- **省着用 token**：批量调用时，token 就是钱。别把无关的长文一股脑塞进每次请求，保持请求精简（第38章那套"省着用"在这里直接变成省钱）。
- **如果开始让程序"自己做多步、调工具"**：那你其实正在从"调一次 API"走向搭一个 **Agent**（第19、34章）——这时第34章那套"装刹车"的纪律就得跟上：花钱、删数据、对外发送，仍要设成需要人确认。
- **想给它接公司内部资料 / 外部工具**：那是第22、36章 Skills / MCP 的话题——记住每接一个新源都是一个新的风险入口。

> **要点：** "给程序接 AI"就是用 **API**（第17章）：你的程序把问题发给模型、拿回答、再嵌进自己的应用——**思路简单，数据流清晰**。但有两件事必须从第一行代码就盯死：**① 接 API ≠ 开会员**（分清人用产品还是程序调接口）；**② 密钥与费用从一开始就管好**——放环境变量、别写进代码、别提交到公开仓库、设用量上限（第37、38章）。具体写法**以官方文档为准**。

## 小结

- "给现有程序接上 AI" = 用 **API**（第17章第③层）：让你自己的程序在背后自动把问题发给模型、收回答、嵌进应用。
- 最小可跑的数据流就一条线：**你的程序 →（带密钥）→ 模型 → 回答传回 → 程序用起来**。本章只讲这条思路，不写死某语言的 SDK 代码（**以官方文档为准**）。
- **最大的误区是"接 API = 开会员"**：开会员给人用产品，接 API 给程序用接口、按 token 计费。
- **密钥与费用从写第一行就管好**：放环境变量、别写进代码、别提交到公开仓库、设用量上限（呼应第37、38章）。这是第 0 步，排在所有功能前面。
- 模型返回的内容仍会出错、会不按格式来（第9章），程序要做检查和兜底；发进去的数据等于交出去了，敏感信息先脱敏（第37章）。

下一章，我们把第22章那两个概念——**Skills 与 MCP**——真正落地，给 AI 扩展出读数据、用工具的新能力。

## 小测验

> 共 6 题，涵盖概念、误区、情景、操作四类。答案与解析在每题下方的引用块里——先自己想，再对照。

1. **[基础·概念题]** "给现有程序接上 AI"，本质上是在用四层地图（第17章）里的哪一层？
   - A. 产品层（给人用的界面）
   - B. **API 层**（给程序用的接口）
   - C. 模型层（裸内核）
   - D. Agent 层
   > **答案 B。** 让**程序**在背后自动调用模型，走的就是 API（第17章第③层那个"接单窗口"）。产品层是给**人**点开聊的；API 层是给程序调用的同一份能力入口。

2. **[基础·误区题]** "我们要在自己的系统里接入 AI，那就去开个 ChatGPT 会员吧。"——错在哪？
   - A. 没错，开会员就行
   - B. 让**程序**用 AI 要走 **API**（申请密钥、按 token 计费）；开会员是给**人**用产品，两回事
   - C. 错在应该买更贵的会员
   - D. 错在系统不能接入 AI
   > **答案 B。** 这是本章头号误区。"人在界面里用"和"程序在背后自动调用"是两条完全不同的路：前者用产品（开会员），后者用 API（第17章）。先分清你要哪一个，再谈怎么做。

3. **[基础·概念题]** 用 API 时那把"密钥（API key）"是干什么的？
   - A. 加快模型运算
   - B. 证明"是你在调用"，并据此把用量记到你账上（按 token 计费）
   - C. 让模型不会出错
   - D. 一种聊天产品的会员卡
   > **答案 B。** 密钥相当于你的"点单凭证 + 记账依据"。正因为它一头连着你的账单，**泄露了别人就能拿你的账去刷**——所以放环境变量、别提交到公开仓库、设用量上限（第37、38章）。

4. **[进阶·误区题]** 写代码图省事，直接把密钥写在代码文件里、还推到了公开的 GitHub 仓库。最可能发生什么？
   - A. 没事，没人会看
   - B. 公开仓库里的密钥会被自动扫描程序很快捡走盗用，你可能莫名其妙被刷出一大笔账单
   - C. 代码会跑得更快
   - D. GitHub 会自动帮你加密
   > **答案 B。** 公开仓库会被专门的扫描程序盯着，明文密钥往往几秒内就被捡走。正确做法是**密钥放环境变量、绝不提交、并设用量上限兜底**（第37、38章）——而且这要在写第一行功能代码之前就做好。

5. **[基础·情景题]** 你的程序通过 API 让模型"只回答'好评'或'差评'"，但它偶尔回了一整句解释。最该怎么办？
   - A. 反复重试直到它听话
   - B. 在程序里对返回结果做检查和兜底，别假设模型每次都乖乖按格式来
   - C. 换一把密钥
   - D. 认定 API 坏了
   > **答案 B。** 模型返回的内容仍可能出错、不按格式（幻觉照样发生，第9章）。健壮的程序会预设"它可能不听话"，对返回值做检查和兜底，而不是裸奔假设它永远正确。

6. **[基础·操作/观察题]** 不必真写代码——在纸上画一遍"接 API"的数据流：从"你的程序把问题发出去（带着密钥）"，到"模型把回答传回来"，再到"你的程序把回答用起来"。然后在图上标出三处和钱/安全有关的点：密钥放哪、在哪计费、在哪设上限。（具体写法**以官方文档为准**。）
   > **该观察到：** 这条线其实很短——发出去、收回来、用起来，三步而已，这就是"接 AI"的全部思路。但你标的那三个点（密钥放环境变量、按 token 计费、后台设用量上限）才是真正容易出事、必须从第一天就管好的地方（第37、38章）。把数据流和这三个点画清楚，你就既能和工程师对上话，也守住了底线。
