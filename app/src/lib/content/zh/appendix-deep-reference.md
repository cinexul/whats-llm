正文里，我们一再把"具体怎么敲那条命令、那个开关叫什么名字"往后推，说"细节进附录 C"。这就是附录 C。它把 Claude Code 和 Codex 这两个工具**到底有哪些能力、每个能力解决什么问题、什么时候该用、容易在哪里翻车**集中讲清楚——但**不替你背确切的命令**。

为什么不背命令？因为这正是本书反复强调的那条红线：

> **要点（请先读这一段）：** 这两个工具都在飞快迭代。**命令的名字、参数、配置项的字段、文件叫什么、放在哪、事件怎么命名——这些都会随版本变化。** 所以本附录只讲"有哪些能力、各自解决什么问题、怎么判断要不要用、容易踩什么坑"这些**相对稳定**的东西；**确切的命令、参数、配置写法，一律以官方文档为准**。把这份附录当"地图"，把官方文档当"实时路况"。

## 一、先记住去哪查（官方文档）

地图再好，也要配最新路况。遇到"这条命令现在到底怎么写"，去这两处查准没错：

| 工具 | 出品方 | 官方文档去哪找 |
| --- | --- | --- |
| Claude Code | Anthropic | 搜索"Claude Code 官方文档"，或访问 Anthropic 文档站（如 docs.claude.com 下的 Claude Code 部分）〔以官方为准〕 |
| Codex | OpenAI | 搜索"OpenAI Codex 官方文档"，或在 OpenAI 官方帮助中心 / 开发者文档里查 Codex〔以官方为准〕 |

> **要点：** 一个好习惯——**用之前先扫一眼官方文档的"快速开始"或"更新日志"**。版本一变，最省事的就是直接看官方怎么说，而不是凭记忆硬敲。

## 二、它们会"长在哪里"：终端、IDE、网页、云端

在背任何命令之前，先破除一个常见误解。

很多教程会说"Claude Code / Codex 就是装在你电脑上的命令行工具"。这**只说对了一种形态**。实际上，你可能在这些地方遇到它们：

- **终端（命令行）**：在你自己电脑的黑框框里运行。
- **IDE 插件**：装进 VS Code、JetBrains 这类编辑器里，边写代码边用。
- **网页 / 云端任务**：在浏览器里给它派活，它在**远端的云环境**里跑，跑完把结果给你。
- 以及随版本不断出现的新形态。

> **要点（贯穿全书的第三条红线）：** **不要默认"它一定跑在你这台电脑上"。** 它**既可能在本地、也可能在云端**，具体形态随产品更新而变。这件事很重要，因为它直接决定了"它能不能读到你本地的文件""你的代码有没有离开过这台机器"——涉及隐私和安全（见第 37 章）。**具体以官方文档对你所用形态的说明为准。**

## 三、会话内命令（斜杠命令等）

**是什么：** 在和工具对话的过程中，用一个特殊前缀（常见是斜杠 `/`）敲出的"快捷指令"，让它做一些和"写代码"无关的操作——比如查看帮助、清空当前上下文、切换模型或模式、初始化项目规约、管理外部连接等。

**心智模型：** 普通的话是"我要你做的事"；斜杠命令是"我要调一下你这个工具本身"。前者是内容，后者是控制开关。

**怎么知道有哪些、怎么写：** 最稳的办法——在会话里查看它自带的命令列表（很多工具输入 `/` 或"帮助"就会列出来）。**确切有哪些命令、各自的参数，以官方文档和工具内提示为准。**

**常见坑：** 把"对 AI 说的话"和"斜杠命令"搞混。你想让它清空记忆，是敲那个专门的命令，而不是打字说"请你忘掉前面"——后者它不一定真清。

## 四、许可机制与配置文件

**是什么：** 这类工具能动手改文件、跑命令，所以都内置了"**许可（permission）**"机制：危险操作之前先停下来问你，或按你预设的规则自动放行 / 拦截。这些规则通常存在一个**配置文件**里（Claude Code 习惯叫 `settings.json` 一类；Codex 有自己的配置）。

**心智模型：** 把它想成给一个能干的实习生设权限——哪些事可以自己做、哪些必须先举手问你。配置文件就是这张"权限清单"。

**几种典型的许可强度（直觉，不是确切名称）：**

| 大致档位 | 含义 | 适合 |
| --- | --- | --- |
| 每步都问 | 任何改动、任何命令都先征求同意 | 刚上手、或在重要项目里 |
| 自动放行一部分 | 读文件、查信息等"只读"操作自动做，改动仍要确认 | 熟练之后提速 |
| 只做计划不动手 | 只给方案、不真正改东西 | 想先看清楚它要干嘛 |

> **要点：** **确切有几种模式、各叫什么、配置文件里字段怎么写，全部以官方文档为准。** 新手的安全默认是"改动类操作都让它先问你"。

**常见坑：** 一上来就把许可调到"全自动"，图快。等它在不该动的地方动了手，才想起没设防。**越重要的项目，越要保留人工确认那道闸。**

## 五、项目规约文件：给工具一份"本项目的说明书"

**是什么：** 你可以在项目里放一个**规约文件**，把"这个项目用什么技术、目录怎么分、有哪些必须遵守的规矩、哪些事禁止做"写下来。工具每次开工会读它，照着办。Claude Code 习惯用一个叫 `CLAUDE.md` 的文件；Codex 有自己的等价做法。**确切的文件名、位置、生效方式以官方文档为准。**

**这是"产品层记忆"，不是模型长了记性。** 这一点呼应第 3、13 章那条贯穿全书的红线：

> **要点：** 模型本体**没有跨对话的记忆**。它之所以"记得这个项目的规矩"，是因为**产品每次开工都把这份规约文件重新塞进上下文**喂给它——是产品层在"重新喂"，不是模型自己记住了。理解这条，你就明白：**规约写得越清楚、越具体，它表现越稳**；规约里没写的，它就"不记得"。

**怎么写才有用（要点）：**

- **用专有名词，别用形容词。** "代码要干净"没用；"HTTP 请求一律走 `services/apiClient`，禁止直接 `fetch`"才有用。
- **写清楚"必须做"和"禁止做"。** 比如"新增第三方依赖前必须先问""不要绕过现有的公共模块自己造轮子"。
- **把踩过的坑随手补进去。** 它哪次跑偏了，就把对应的规矩加上——这份文件是**越用越好用**的活文档。

详细的"规约文件雏形"和怎么搭整套工作流，见**附录 D（Harness 方法论）**。

## 六、计划模式与"推理强度"

**计划模式（plan mode 一类）：** 让它**先给方案、不直接动手**，你看过、点头了，它再实现。这是本书最看重的习惯之一（第 27 章），很多工具内置了这个开关，不必每次都手动叮嘱。

**推理强度 / 思考力度：** 有些工具允许你调"让它多想一会儿还是快速作答"。难题、要权衡的设计，给高一点；琐碎小事，低一点省时间省钱。

> **要点：** **这两个开关确切叫什么、怎么调，以官方文档为准。** 但背后的道理是稳定的：**重要的事，让它先想清楚、先出计划、再动手。**

## 七、子代理（subagents）

**是什么：** 把某一类活儿，交给一个**专门配置过的"分身"**去干——比如一个专做代码审查的、一个专做资料调研的。它们各自有自己的说明（一段配置，常包含名字、职责描述、能用哪些工具等）。

**心智模型：** 像给团队分工。主对话是"项目经理"，子代理是各有专长的"组员"，把对应的活派出去，互不干扰、各自专注。

**常见坑：** 一上来就搭一堆子代理。其实**大多数日常任务，一个主对话就够了**。等你明确感到"这类活老是重复、需要固定套路"，再考虑专门配一个。**确切的配置字段、定义方式以官方文档为准。**

## 八、钩子（hooks）

**是什么：** 让工具在**某些时刻自动触发**你指定的动作。比如"每次它改完文件，自动跑一遍格式化 / 测试""开始某类操作前，自动做一道检查"。

**心智模型：** 像给家里装的自动化——"开门就亮灯"。你定义"什么事件发生时，自动做什么"，剩下的它替你盯着。

**常见坑：** 这是偏进阶的功能，普通使用**完全可以先不碰**。需要时再了解。**有哪些事件、怎么配置，以官方文档为准。**

## 九、Skills（技能）

**是什么：** 把一套"做某类事的标准流程 + 需要的资料"打包成一个可复用的"**技能包**"，需要时让工具调用。和正文第 22 章讲的是一回事，这里只提醒：它是**让能力可复用**的机制。

**常见坑：** 别为了用而用。先把活儿用大白话讲清楚，**反复出现的固定套路**才值得做成技能包。**打包格式、调用方式以官方文档为准。**

## 十、MCP：给 AI 接上外部工具和数据

**是什么：** MCP（第 22 章详讲）是一个"通用接口"，让 AI 工具能连上**外部的工具和数据源**——比如某个数据库、某个内部系统、某类文件服务。连接通常通过一条"添加 MCP 服务"的命令或一段配置来完成（如 Claude Code 的 `claude mcp add` 一类用法）。

**心智模型：** 给 AI 配"外设"。模型本身只会聊天，接上 MCP，它就能去查、去取、去用那些外部资源。

> **要点（安全）：** 接入外部数据 = 把**外部内容**引进了对话。**外部内容里可能藏着"假指令"（提示注入，第 22、40 章）。** 只接你信任的源，别给它过大的权限。**确切的添加命令、配置写法以官方文档为准。**

## 十一、安装、跑起来、与排错

**最小路径（思路，不是逐字命令）：** 装好工具 → 登录 / 配置好账号或密钥 → 在一个项目目录里跑起来 → 先给它一个安全的小任务（比如"用大白话解释一下这个项目是干嘛的"）跑通，建立信心。

**最常见的几类卡壳：**

- **装不上 / 跑不起来**：多半是运行环境（版本、依赖）或网络问题——把报错原文复制给 AI 让它帮你读，往往最快。
- **登录 / 鉴权失败**：账号、套餐、密钥的问题，对照官方"快速开始"逐项核对。
- **它读不到你的文件**：先确认你用的是哪种形态（本地还是云端，见本附录第二节）——云端形态本来就读不到你本机的文件。

> **要点：** **确切的安装命令、系统要求、登录方式以官方文档为准**，且最容易随版本变。装之前看一眼官方"快速开始"最省事。

## 十二、Git / GitHub / PR 等操作

**是什么：** 在本地代码项目里，这类工具通常能帮你做版本管理相关的事——查看改动、提交、开分支、甚至帮你起草一个 PR（合并请求）。这正是第 25 章"看 diff、回滚、验收"在工具里的落地。

**心智模型：** 它能帮你**记录和回退**改动，所以你才敢放手让它改——不满意就回滚（第 25、26 章）。

**常见坑：** 让它"顺手提交 / 推送"而你没看 diff。**提交什么、推到哪、要不要开 PR，应该由你拍板**；尤其"推送 / 合并"这类对外、难撤回的操作，务必自己确认。**确切的命令以官方文档为准。**

## 小结

- 这份附录是 Claude Code / Codex 的**能力地图**：讲清楚有哪些能力、各解决什么问题、什么时候用、容易踩什么坑。
- **它故意不替你背确切命令**——因为命令、参数、配置、文件名、事件名都会随版本变，**一律以官方文档为准**。用之前先扫一眼官方"快速开始 / 更新日志"。
- 几条贯穿全书的红线在这里再强调一遍：**别默认它跑在你本机**（终端 / IDE / 网页 / 云端都可能）；**项目规约文件是"产品层记忆"**，靠每次重新喂回去生效；**越重要的项目，越要保留"先看计划、改动先问、提交自己拍板"那几道闸**。
