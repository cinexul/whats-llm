〔本章涉及具体产品的默认行为、许可粒度、命令、界面之处，都会随版本更新而变化，一律**以官方文档为准**。本章只讲**共通、稳定**的理念——这部分不会过时。〕

第23章我们认清了一个有点吓人的组合：**它会犯错 ＋ 它能动手。** 第25章我们学会了"事后怎么验收"。这一章解决另一半问题：**事前和事中，怎么让它"动手却不失控"？**

答案是两件互相搭配的事，本章逐一讲透：**一是"许可机制"——危险操作动手前先问过你；二是"可回滚环境"——万一出事，能反悔。** 一句话先撂这儿：**让 AI 放手去干那些"没大不了"的活，但凡"可能闯祸"的，永远留给人点头。**

## 一、核心循环：请求 → 看提案 → 许可 → 验证

带一个会动手的智能体，整个过程其实就是一个不断重复的小循环。把它刻进脑子，你就再不会觉得"它在我电脑里乱跑、我管不住"：

```text
   你给目标
      │
      ▼
 ①它请求 ──→ ②你看提案 ──→ ③你许可 ──→ 它动手 ──→ ④你验证 ──→（下一步又回到①）
（"我打算改X"）（看它要干嘛）（你点头/拒绝）          （看diff/跑测试）
```

- **① 请求**：每要做一件"有后果"的事（改文件、跑命令、联网……），它通常会**停下来，先告诉你它打算干什么**，而不是闷头就做。
- **② 看提案**：你看清楚它"要动什么、动哪里"。这一步是你介入的窗口——觉得不对，现在拦下来代价最小。
- **③ 许可**：你点头（同意这一次／这一类），它才动手；你也可以拒绝、或让它换个做法。
- **④ 验证**：它做完，你用第25章那套（看 diff、跑测试、验收三问）确认结果。然后进入下一步，**循环重新开始**。

> **要点：** 这套"请求→看提案→许可→验证"的循环，就是第24章"调查→计划→动手→验收"在**每一个具体操作**上的缩影。它的灵魂是把"决定权的关键节点"**握在人手里**：AI 负责又快又多地干，**人负责在岔路口点头或喊停**。理解了这个循环，你对它的掌控感就建立起来了——它不是脱缰的，每一步有后果的动作，都得过你这一关（具体在哪一步停、停多细，各工具不同，**以官方文档为准**）。

## 二、"许可机制"是什么：它在替你把门

把上面的①②③单拎出来，就是**许可机制（permission）**。

打个比方：许可机制就像家里装修时请的工人，**每要砸一面墙、改一处水电，都先来问你一声"这面墙能砸吗"**，得你点头才动手。靠谱的工人不会自作主张把承重墙给你拆了。许可机制，就是给那个"手快但刚入职的新同事"（第23章）装上的这道"动手前先问一声"的规矩。

它实际是怎么运作的？大致是：当它准备做一件**有后果**的操作时，会**暂停下来，把"我要做什么"摆给你看，等你回应**。你常见的选择无非几类：**这一次同意 / 这类操作以后都同意 / 不同意 / 让它换个做法**。

> **误解 vs 实际：** 新手第一次用，常被"它怎么老停下来问我"搞得不耐烦，以为是它**笨、不靠谱**。**恰恰相反——它停下来问，正是"许可机制"在保护你，是好事，不是毛病。** 真正该警惕的，是一个"什么都不问、闷头一路干到底"的设置。把"它来问我"理解成"安全带在起作用"，你就不会嫌它烦了。

这里要补一句很重要的"权衡"：**许可可以收紧，也可以放松。** 多数工具允许你把某类"安全的小操作"设成"以后别再问了"（比如只是读读文件），好让它顺畅地干活；也允许你把"危险操作"始终卡在"每次都必须问"。**怎么调、调到多松，是你的权力，也是你的责任**——而下一节就是这条权力的底线。

## 三、哪些操作"永远要人点头"：不可逆的事，绝不自动放行

不是所有操作都一样危险。**有些事做错了，撤回来很容易（比如改一行字，回滚就好）；有些事做错了，泼出去的水收不回。** 后者，就是必须"永远留给人点头"的红线。

哪些算"收不回的水"？记住这几类**不可逆 / 高代价**操作：

| 危险类别 | 为什么危险 | 生活类比 |
| --- | --- | --- |
| **删除 / 覆盖** | 删掉的文件、被盖掉的内容，可能再也找不回来 | 把文件直接粉碎，不是丢进回收站 |
| **对外发送** | 一旦发出去（发邮件、发消息、提交到公共地方），**收不回来**，还可能把不该外传的东西带出去 | 邮件点了"发送"就追不回 |
| **花钱 / 调用付费服务** | 真金白银的支出，或触发计费，事后未必退得了 | 替你刷了卡 |
| **改动线上 / 生产环境** | 影响的是**正在被真人使用**的系统，错了就是事故 | 不是在草稿上改，是在直播时改 |

对这几类，原则只有一句，而且不容打折：

> **要点：** **删除、覆盖、对外发送、花钱——这类"不可逆或高代价"的操作，务必设成"需人工确认"，绝不能自动放行。** 你可以为了顺畅，把"读文件"这种没大不了的事设成"别老问我"；但**越是收不回的事，越要把它牢牢卡在"每次都得我亲自点头"那一档**。一句话——**"危险操作永远留给人点头。"** 这不是讲究，是底线。具体哪些操作默认就卡、卡得多严，各工具不同，**以官方文档为准**；但"危险操作要人确认"这条原则，到哪都不变。

为什么强调"绝不自动"？因为第23章那条铁律在这儿最致命：**它会自信地犯错。** 一个"自信地错了"的删除指令，如果被自动放行，等你发现时，水已经泼出去了。**让它停下来问你这一下，就是给"它可能正在犯错"留的最后一道刹车。**

## 四、可回滚环境：动手之前，先铺好"后悔药"

许可机制管的是"动手前拦一道"。但拦不住一切——总有些错会溜过去。所以还要有第二重保险：**让它在一个"能反悔"的环境里动手。** 这一点第25章讲回滚时已经点过，这里说清"为什么这是前提、怎么铺"。

道理很朴素：**正因为它会犯错，你才更要保证"犯了也能撤"。** 把这两件事配齐——动手前有许可拦一道、动手后能回滚撤一道——你才算真正给它套上了"既能干活、又不失控"的笼子。

怎么铺这个"能反悔"的环境？还是分两种形态（呼应第25章）：

- **它在你本地项目上动手时**：务必**先用版本控制（Git）管理项目、并先切一个"新分支"再让它动手**。"切新分支"＝**单独给它开一块草稿区**，让它在草稿上折腾，不碰你的正式稿；万一改砸了，扔掉草稿、回到干净状态即可。**这一步要在让它动手之前就做好**——事后才想起来，那道安全网就没张开。
- **它在网页 / 云端形态里动手时**：它通常在**平台自带的隔离环境**里干活，平台一般提供对应的**撤销 / 丢弃 / 重置**方式。具体怎么撤、能撤到哪一步，**以官方文档为准**。

> **要点：** 把"可回滚环境"当成**动手的前置条件**，不是出了事才补的功课。本地就先切 Git 分支，云端就用平台的隔离——**形式不同，目的一样：让它在一块"打翻了也不心疼的草稿区"里干活。** 有了这块草稿区，你才敢大胆把活交出去；没有它，每一次"让它动手"都是在你的正式稿上赌运气。

## 五、两道保险，怎么配合成一套

把许可机制和可回滚环境合起来，就是这一章的全貌——**一前一后，两道保险**：

```text
            ┌──────────────── 动手之前 ────────────────┐
铺好后悔药： 在"能反悔"的环境里（本地切Git分支 / 云端用平台隔离）
            └──────────────────────────────────────────┘
                              │
                              ▼
   ①它请求 ─→ ②你看提案 ─→ ③许可（危险操作必须人工确认！）─→ 它动手
                              │
                              ▼
   ④你验证（看diff/跑测试，第25章）
                              │
              ┌───────────────┴───────────────┐
          结果满意                         结果不对 / 越改越乱
              │                                 │
           进入下一步                    用"后悔药"回滚，重开干净对话再来
```

- **许可机制**是"事前刹车"：在它动手前，把有后果的操作拦下来给你看，**危险的那几类必须你亲自点头**。
- **可回滚环境**是"事后撤销"：万一错溜过去了，**一键回到干净状态**，损失归零。
- 两者**缺一不可**：只有刹车没有后悔药，刹不住的那次就翻车；只有后悔药没有刹车，你会疲于不停地"撤销重来"。配齐了，你才既能享受它的快，又不用提心吊胆。

记住最后这句总纲：**让 AI 动手，但不失控——靠的就是"危险操作永远留给人点头" ＋ "永远在能反悔的环境里动手"。**

## 六、常见误解，集中纠正

| 误解 | 实际 |
| --- | --- |
| 它老停下来问我，是它笨 / 不靠谱 | 那是**许可机制在保护你**，是好事；该担心的是"什么都不问、闷头干到底" |
| 既然它会动手，索性全自动、别问我，最省事 | 危险操作（删除／覆盖／对外发送／花钱）**绝不能自动放行**，必须人工确认——它会自信地犯错 |
| 删除、发邮件这种事，让它顺手做了也行 | 这些是"泼出去收不回的水"，是必须卡在"每次人工确认"的红线 |
| 回滚是出了事才需要去研究的东西 | 可回滚环境是**动手的前置条件**：本地先切 Git 分支、云端用平台隔离，要在动手前就铺好 |
| 有许可机制就够了，不用管能不能回滚 | 两道保险缺一不可：许可是事前刹车，回滚是事后后悔药——总有错会溜过刹车 |
| 各家工具默认行为都一样 | 默认卡哪些、卡多严、许可粒度各不相同，**以官方文档为准**；不变的是"危险操作要人点头"这条原则 |

## 小结

- 带会动手的智能体，核心是一个循环：**请求 → 看提案 → 许可 → 验证**——AI 负责干，**人负责在关键节点点头或喊停**（呼应第24章"调查→计划→动手→验收"）。
- **许可机制**＝动手前先问你一声的规矩，像装修工人"砸墙前先问"。它停下来问**是在保护你**，不是毛病。
- **危险操作永远留给人点头**：删除、覆盖、对外发送、花钱这类**不可逆／高代价**操作，**务必设成"需人工确认"，绝不自动放行**——因为它会自信地犯错。
- **可回滚环境**是动手的**前置条件**：**本地先切 Git 分支**（给它单开草稿区）、**网页／云端用平台自带隔离**（以官方文档为准），让它在"打翻也不心疼"的地方干活。
- 两道保险配齐——**事前刹车（许可）＋ 事后后悔药（回滚）**——才能"让 AI 动手却不失控"。各家默认与粒度不同，**以官方为准**；原则不变。

下一章（第27章），我们解决"事前"的另一半：大改动动手前，怎么让它**先出计划、你批准再写**，并把任务**拆小、小步验收**——这是最省返工的一步。

## 小测验

> 共 6 题，涵盖概念、误区、情景、操作四类。先自己想，再看引用块里的答案解析。

1. **[基础·概念题]** 带一个会动手的智能体，核心循环是哪一个？
   - A. 它自己干完，你最后看一眼
   - B. 请求 → 看提案 → 许可 → 验证，关键节点由人点头或喊停
   - C. 你下指令，它绝不停顿一路干到底
   - D. 它问一句、你答一句，仅此而已
   > **答案 B。** 这正是第24章"调查→计划→动手→验收"落到每个具体操作上的样子：AI 干活，人在岔路口把关。选 A、C 都把"人点头"这个关键节点丢了——而那恰恰是不失控的核心。

2. **[基础·概念题]** "许可机制"最贴切的类比是？
   - A. 一个永远不说话、闷头干活的机器人
   - B. 装修工人"砸墙、改水电前先问你一声"，你点头才动手
   - C. 一个只会聊天、不动手的助手
   - D. 一个自动把所有操作都执行的脚本
   > **答案 B。** 许可机制＝给"手快但刚入职的新同事"装上"动手前先问一声"的规矩。它实际就是：做有后果的操作前暂停、把要做的事摆给你看、等你点头。选 D 恰恰是它要避免的"什么都自动放行"。

3. **[基础·误区题]** "它老是停下来问我能不能动手，真烦，是不是它不太行？"——错在哪？
   - A. 没错，老问就是不行
   - B. 它停下来问正是许可机制在保护你，是好事；真正该担心的是"什么都不问、闷头干到底"
   - C. 它停下来是因为坏了
   - D. 停下来问会多扣费
   > **答案 B。** 把"它来问我"理解成"安全带在起作用"。一个会动手的 AI 若一路不问、自动放行危险操作，那才危险——尤其因为它会"自信地犯错"（第23章）。

4. **[进阶·误区题]** "既然它会动手，干脆设成全自动、什么都别问我，最省事"——为什么这么做很危险？
   - A. 不危险，全自动最高效
   - B. 删除／覆盖／对外发送／花钱这类**不可逆或高代价**操作一旦被自动放行，等你发现时"水已泼出去"，收不回了
   - C. 全自动会让它变慢
   - D. 全自动需要额外付费
   > **答案 B。** "读文件"这种没大不了的可以设成别老问；但越是收不回的事，越要卡在"每次人工确认"。它"自信地错了"的一条删除指令若被自动执行，那一下就是事故。**危险操作永远留给人点头。**

5. **[基础·情景题]** 你准备让它在**本地项目**上做一次较大的改动。动手**之前**，最该先做的一件事是？
   - A. 把电脑重启一遍
   - B. 先用 Git 管理项目、并切一个新分支，给它单开一块"草稿区"，确保万一改砸能一键回滚
   - C. 先夸它两句让它认真点
   - D. 把所有许可都关掉让它放手干
   > **答案 B。** 可回滚环境是**动手的前置条件**，不是事后补课。切新分支＝让它在草稿区折腾、不碰正式稿，砸了就扔掉草稿。选 D 正好反了——大改动前更该收紧许可、铺好后悔药。云端形态则用平台自带隔离，**以官方为准**。

6. **[进阶·操作/观察题]** 想象（不必真做）你要让它帮你"清理一批旧文件"。在让它动手前，你会把哪些操作设成"必须我亲自确认"？为什么？再想想：万一它误删了不该删的，你靠什么把损失补回来？
   > **该观察到：** "删除"是典型的不可逆操作，必须卡在"人工确认"——绝不自动放行。而补救的底气来自**可回滚环境**：本地先切了 Git 分支就能回滚到删除前的干净状态，云端则靠平台自带的撤销／重置（**以官方文档为准**）。两道保险——许可拦在前、回滚兜在后——配齐了，"让它动手"才不等于"听天由命"。
