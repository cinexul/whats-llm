〔本章涉及 Claude Code 的安装、命令、界面、能力之处，都会随版本更新而变化，一律**以官方文档为准**。本章只讲"是什么、何时用、怎么判断、要注意什么"；逐字的命令与深度功能见**附录 C** 和官方文档。〕

第24章我们把"调查→计划→动手→验收"这套通用工作流讲透了。这一章和下一章，是把它**落到具体工具上**——这一章讲 **Claude Code**，下一章讲 **Codex**。两章会有意"分家"：因为它们虽然共享同一套干活方式，但**怎么装、怎么操作、各有什么脾气**并不一样。

先说一句定心的话：**这一章依然不要求你会写代码。** 我们要建立的，是"第一次摸 Claude Code，心里该有的那张地图"——它长什么样、CLAUDE.md 这个东西到底在干嘛、常用操作的直觉是什么。具体到"敲哪条命令、按哪个键"，本章一概不写死，全部交给官方文档和附录 C。原因第24章说过：这些东西更新太快，写进书里只会过时、甚至误导你。

## 一、最小上手心态：先认清它"有好几副面孔"

很多人卡在第一步，是因为脑子里把 Claude Code 想成"一个固定的网页"或"一个固定的软件窗口"。**它不是。**

Claude Code 来自 **Anthropic**，内核是 **Claude**——这两点是稳定事实，可以记牢。但它的**形态不止一种**：你可能会在**终端（命令行）、代码编辑器（IDE）插件、桌面或网页界面、乃至云端任务**里遇到它；**有的跑在你自己的机器上，有的跑在云端，具体形态各不相同、而且还在更新——以官方文档/产品官方形态为准。**

所以正确的上手心态是这样三句：

- **别纠结"它到底是个啥软件"。** 先去官方文档看"当下推荐哪种形态、怎么装"，照着走一遍就行（**以官方文档为准**）。
- **认准它的内核不变。** 无论哪种形态，背后都是会"自己读项目、改文件、跑命令"的 Claude——第24章那套带它干活的方法，到哪种形态都适用。
- **第一次别贪。** 装好之后，先用一个**能反悔的小项目**跑通一个最小请求（详见第25章），别一上来就让它在你重要的东西上动手。

> **要点：** 把"Claude Code 是什么形态"这个问题，**外包给官方文档**。你要带进来的，不是"它是个网页还是命令行"，而是第24章那套判断力——派什么活、怎么看它做得对不对。形态会变，判断力不变。

## 二、CLAUDE.md ＝ 给项目的"入职须知"

这是 Claude Code 最值得普通人理解的一个设计，也是本章的重头。

回想第24章那个比喻：Claude Code 像个**很能干、但刚入职第一天的新同事**。新同事最大的问题是什么？**他不知道你们这个项目的规矩。** 你们习惯用什么风格、哪些地方是雷区不能碰、跑测试该敲什么、命名有什么讲究——这些他统统不知道，你不说他就会按自己的想当然来，于是踩坑。

**CLAUDE.md 就是你写给这位新同事的那份"入职须知"。** 它通常是放在你项目里的一个文本文件（**具体文件名、放哪、怎么生效以官方文档为准**），里面用大白话写清这个项目的"规矩"，比如：

- 这个项目大致是干嘛的、主要文件在哪；
- 我们的代码风格、命名习惯；
- 哪些地方碰不得（"别动这个配置""这个目录是自动生成的，不要手改"）；
- 跑测试、跑检查该用什么方式。

有了这份须知，Claude Code 每次开工时就能"先读一遍规矩"再动手，少踩很多坑。你也不必每次对话都把这些话重新交代一遍——写一次，长期生效。

### 关键澄清：这是"产品层的记忆"，不是模型本体记得了

这里**必须澄清一个最容易误会的点**，它直接呼应第13章（也别忘了第3、9章）。

你可能会想："哦，原来它能**记住**我这个项目了。"——这话对一半，错一半。

> **模型本体确实没有跨对话记忆**，每次都是从空白开始（第3、9章）。但 **Claude Code 在模型之外，加了一层"产品层记忆"**：CLAUDE.md 这份须知，是**每次开工时由产品自动把它重新塞回这次的上下文里**给模型看的。所以不是模型"长出了记性"，而是**产品每次替你把项目规矩重新摊回那张"桌面"**（回到第3章那张桌子）。掀开盖子看，底下还是"无记忆的模型 + 每次重新喂上下文"。

为什么非要把这层窗户纸捅破？因为想通了这一点，你就能**预判它的脾气**，而不是迷信它：

- **没写进须知的规矩，它就不知道。** 你以为"它应该懂"的潜规则，只要没落到 CLAUDE.md（或这次对话）里，它照样不知道——别怪它"忘了"，是你没"喂"。
- **须知越长，占的"桌面"越多。** 因为它每次都要被塞回上下文（第3章），写得又臭又长会挤占空间、还可能让重点被淹没。**好的须知是"精炼的规矩清单"，不是项目说明书全文。**
- **它也会"看走眼"。** 须知塞回去了不等于它每条都照办——该验收还得验收（第24、27章）。

> **要点：** 看到"项目记忆""项目上下文文件"这类说法（CLAUDE.md 是其中一种），不要觉得和第3章"模型没记忆"矛盾。它们都是**产品层**的贴心设计：**每次自动把该记的东西重新放回上下文**。这正是第3章那张"桌面"的延伸，不是模型本体的记忆。具体怎么写、放哪、优先级如何，**以官方文档为准**。

## 三、常用操作的直觉：只讲"做什么"，不写"按哪个键"

下面这些是你几乎一定会用到的几类操作。**注意：这里只讲"它是用来干嘛的、什么时候用",具体怎么触发（命令、快捷键、按钮）一律以官方文档为准。** 这正是本书的安全做法——把"概念"教给你，把"易变的具体键位"交给官方。

| 你想做的事 | 它大概是什么 | 心里要有的数 |
| --- | --- | --- |
| **让它先别动手、只调查** | 用一句话指示它"先只看不改"（第24章的"调查"步） | 这是省返工的第一招；先确认它没找错地方 |
| **让它先出计划再动手** | 很多形态内置了"先给计划、你批准再改"的模式 | 这是第24章的"闸门1"；几十秒挡掉几十分钟返工（详见第27章） |
| **许可确认** | 它每要做"有后果的事"（改文件、跑命令）通常会停下来问你 | 别无脑连点"同意"——这是第26章要细讲的安全阀 |
| **看 diff（改动前后对比）** | 它会标出"加了哪行、删了哪行" | 这是验收的核心动作；看不懂别放行（第25章） |
| **用 @ 之类的方式引用某个文件** | 把"我说的是这个文件"精确指给它 | 比让它满项目乱猜高效得多；**具体符号以官方为准** |
| **中断 / 叫停** | 觉得它走歪了，随时打断，说清楚再让它继续 | "能随时喊停"是你的基本权利，不是打扰它 |

这几样合起来，就是你日常带 Claude Code 干活的"方向盘和刹车"。你会发现，它们**全都对应第24章那套工作流**——调查、批准计划、看 diff、随时叫停——只是换成了在 Claude Code 里"具体怎么做"。**而"具体怎么做"的那一层，永远以官方文档为准。**

> **实际会怎么遇到：** 新手最常见的两个坑——一是**许可确认时无脑连点"同意"**，等于把刹车拆了（第26、30章）;二是**懒得看 diff 就合并**,把"它做完"当成"它做对"（第25章）。这两件事 Claude Code 都给你留了关卡，坑不在工具，在于你有没有真的去看。

## 四、想深入？这些都在"附录 C"，别在这儿找

Claude Code 还有一大堆"进阶玩法"，能让它更听话、更强大。但这些**恰恰是最容易随版本变化的部分**，所以本书的处理很明确：**只在这里点名它们存在，具体用法一律指向附录 C 和官方文档。**

你将来可能会听到这些名词，先混个脸熟，知道"它们是用来扩展 Claude Code 的"就够了：

- **斜杠命令（slash commands）**：在对话里以斜杠开头的一类快捷指令，用来快速触发常用功能。**全表与用法见附录 C / 官方文档。**
- **settings.json 之类的配置**：用一个配置文件统一设定它的行为（比如默认许可的尺度）。**具体字段以官方为准。**
- **子代理（subagents）**：让它派生出"分身"去并行处理子任务。
- **MCP（外部工具连接）**：给它接上外部工具和数据源的一种标准接口（第22章讲过概念）。
- **Hooks（钩子）/ Skills（技能）**：在特定时机自动触发你设定的动作、或给它装上一套可复用的能力。

> **要点：** 这些进阶功能很强，但**对一个刚上手的非程序员，它们都不是第一步**。你的第一步永远是：在能反悔的环境里（第25、27章），用第24章那套工作流，把一个小任务**安全地**跑通、验收掉。等你对"带它干活"有了手感，再翻**附录 C** 去解锁这些进阶玩法不迟。**所有这些的具体命令、字段、开关，都以官方文档为最新依据。**

## 五、安全与花费，第一天就放心上

这几条第24章提过，放到 Claude Code 上再钉一遍——因为它真的会动你的东西、真的按 token 花钱：

- **先布置"能反悔"的环境。** 当它在你**本地项目**上动手时，务必在用**版本控制（Git）**管理、并**切了新分支**的状态下让它动（手把手见第26章）。改乱了一键回到干净状态，这是你最大的安全网。（网页/云端形态通常有平台自己的隔离与回滚方式，**以官方文档为准**。）
- **密钥、密码、真实个人信息绝不外贴。** 别把它们贴进对话或写进代码——会被发出去（第37章）。
- **它按 token 计费。** 让它读越多文件、聊越长，花得越快；保持任务聚焦、对话清爽，既省钱又提质量（第38章）。
- **它生成的代码算你的责任。** 合并前复核、为它负责（第39章）。

## 小结

- Claude Code 来自 Anthropic、内核是 Claude（稳定事实）；但它**形态多样、且在更新**——怎么装、怎么操作**以官方文档为准**。
- **CLAUDE.md ＝ 给项目的"入职须知"**：把项目规矩写一次，它每次开工自动读。但这是**产品层记忆**——每次把须知重新塞回上下文，**不是模型本体记得了**（第3、13章）。
- 常用操作（调查、出计划、许可确认、看 diff、@ 引用、随时叫停）都对应第24章那套工作流；**具体怎么触发以官方文档为准**。
- 斜杠命令、settings.json、子代理、MCP、Hooks、Skills 等**深度功能一律见附录 C**，不是上手第一步。
- 第一天就放心上：切 Git 分支再动手、密钥绝不外贴、按 token 省着用、生成内容自己负责。

## 小测验

> 共 6 题，涵盖概念、误区、情景、操作四类。先自己想，再看引用块里的答案解析。

1. **[基础·概念题]** 关于 Claude Code，下面哪句是**稳定、可以记牢**的事实？
   - A. 它是一个固定的网页，网址永远不变
   - B. 它来自 Anthropic、内核是 Claude；但具体形态（终端/IDE/网页/云端）多样且在更新
   - C. 它只能跑在你自己的电脑上
   - D. 它一定要先学会所有斜杠命令才能用
   > **答案 B。** "出品方是 Anthropic、内核是 Claude"是稳定事实；而"它是什么形态、怎么装、敲什么命令"是易变项，**以官方文档为准**。A、C 把易变的形态写死了；D 把门槛搞反了——上手只需第24章那套工作流。

2. **[基础·概念题]** CLAUDE.md 最贴切的比喻是？
   - A. 模型的大脑
   - B. 给项目的"入职须知"——把这个项目的规矩、雷区、习惯写给那位"刚入职的新同事"
   - C. 一个付费才能解锁的功能
   - D. 模型的永久记忆芯片
   > **答案 B。** 它是写给"能干但刚入职的新同事"（第24章）的项目规矩，让它每次开工先读一遍、少踩坑。D 错得最离谱——模型本体没有记忆，见下题。

3. **[进阶·误区题]** "我写了 CLAUDE.md，所以模型现在**记住**我这个项目了"——这话哪里不准确？
   - A. 完全准确，模型从此永久记得
   - B. 模型本体没有跨对话记忆；是**产品层**每次开工时自动把这份须知重新塞回这次的上下文，才造成"它记得"的效果
   - C. CLAUDE.md 会改变模型的参数
   - D. 须知只对付费用户生效
   > **答案 B。** 这正是第3、13章反复强调的"**模型本体 vs 产品层**"。CLAUDE.md 是产品层记忆——**每次把规矩重新摊回"桌面"**，不是模型长了记性。想通这点你才能预判它：没写进须知、又没在这次说的，它照样不知道。

4. **[进阶·误区题]** "CLAUDE.md 写得越详细越长越好，把项目说明书全文塞进去最保险"——为什么往往适得其反？
   - A. 不会，越长越好
   - B. 它每次都要被重新塞回上下文（占"桌面"）,又臭又长会挤占空间、淹没重点，反而更容易出错
   - C. 文件太长会让电脑死机
   - D. 长文件要额外付费
   > **答案 B。** 回到第3章那张"桌面":CLAUDE.md 每次开工都占地方。**精炼的规矩清单**比"说明书全文"管用得多——把空间留给真正重要的规矩。

5. **[基础·情景题]** 你刚装好 Claude Code，想马上让它给你一个**重要项目**做个大改动。更稳妥的第一步是？
   - A. 直接在重要项目上让它一口气做完大改动
   - B. 先找个能反悔的小项目、在 Git 新分支里跑通一个最小请求，熟悉"调查→批准计划→看 diff"的节奏再说
   - C. 先把所有斜杠命令背下来
   - D. 把 API 密钥贴进对话框让它"验证身份"
   > **答案 B。** 第一次别贪：在**能反悔**的环境里（第25、27章）用第24章那套工作流跑通小任务。A 风险太大、难回滚；C 把顺序搞反；D 是绝对禁忌——密钥绝不外贴（第37章）。

6. **[基础·操作/观察题]** 不必真装工具：试着为一个你熟悉的"项目/事务"（哪怕是"管理家庭账本"）写一份 5～8 条的"入职须知"——把规矩、雷区、习惯各写两三条。再问自己：哪几条是"潜规则",我以前默认"对方应该懂"、但其实不写就没人知道？
   > **该观察到：** 这正是 CLAUDE.md 的精髓——**把"想当然的潜规则"显式写出来**。你也会体会到"该精炼"的难处：写太多没人看得过来，写太少又漏了关键。具体到 Claude Code 里这份须知叫什么、放哪、怎么生效，**以官方文档为准**（深入见附录 C）。
