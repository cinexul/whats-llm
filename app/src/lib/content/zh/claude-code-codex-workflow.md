> 〔本章涉及具体产品的安装、命令、界面、能力之处，都会随版本更新而变化，一律**以官方文档为准**。本章只讲两类工具**共通、稳定**的心智模型和工作流——这部分不会过时。逐字的安装与命令见第25章和附录 C。〕

第19章里我们认识了 Agent：会"自己动手"的 AI。这一章登场的 **Claude Code** 和 **Codex**，就是专门干"编程"这件事的 Agent——你用大白话交代一件事，它会**自己去读项目里的文件、改代码、运行命令**，而不只是给你一段文字让你复制粘贴。

先别因为"编程"两个字紧张。**这一章不要求你会写代码。** 我们要学的，是"怎么带着这样一个会动手的助手干活"——派什么活、怎么看它做得对不对、出了岔子怎么收场。这套工作流，懂电脑的普通人都学得会，而且学会之后，你对"让 AI 替你做事"会有一种全新的掌控感。

## 一、它们到底是什么

Claude Code 和 Codex 不是某个固定的网页按钮，而是一类**会在你的项目上动手的编程智能体（coding agent）**。你通常会在**终端（命令行）、代码编辑器（IDE）、网页任务或云端环境**里接触到它们——**有的在你自己的机器上运行，有的在云端运行，具体形态各不相同、而且在更新，以官方文档/产品官方形态为准**。它们的共同点是：

- **能看见你的项目**：自己浏览文件夹、读代码，搞清楚现状——而不是只知道你贴给它的那几行。
- **能动手**：直接修改文件、运行命令（比如跑测试）、必要时联网查资料。
- **能连续干好几步**：朝着你给的目标，自己跑"看一看→改一改→试一试→再改"的循环。

> **要点：** Claude Code 来自 Anthropic、内核是 Claude；Codex 来自 OpenAI——这两个"出品方／内核"的归属是稳定事实。但**它们的具体形态（CLI／IDE 插件／网页／云端）、安装方式、命令、能力各不相同，而且都在快速更新**——所以本章不写"按哪个键、敲哪条命令、在哪运行"（那些易变，属第25章、附录 C 与官方文档的事）。本章讲的是它们**共享的那套干活方式**，学一次，两边通用、也不易过时。

## 二、和"网页聊天"的根本区别

| | 网页/App 聊天（产品层） | 编程智能体（Agent 层） |
| --- | --- | --- |
| 它怎么对待代码 | 给你一段文字，你自己复制去用 | **直接改你项目里的文件** |
| 它知道多少 | 只知道你贴进去的内容 | 自己**探索整个项目**来了解现状 |
| 推进方式 | 你问一句、它答一句 | 朝目标**自己连走好几步** |
| 出错的后果 | 顶多答得不对，你不用就行 | 它真的动了你的文件——**所以要能回滚** |

最后一行是关键：因为它**真的会动你的东西**，所以"在一个能反悔的环境里用它"不是讲究，是前提（下面第五节细说）。

## 三、心智模型：把它当"能干但刚入职的新同事"

整本书反复用这个比喻，到这里最贴切：**Claude Code / Codex 就像一个很能干、手很快、但刚入职第一天的同事。**

- 他**手脚麻利**：交代的体力活做得又快又多。
- 他**不了解你的项目习惯**：不知道你们的规矩、命名、雷区——你得告诉他（第28章的"项目入职须知"就是干这个的）。
- 他**会自信地犯错**：可能用一个根本不存在的写法（这就是第9章的"幻觉"，写代码时照样发生）。
- 所以——**你是带他的人**。活儿可以交给他，但**派活和验收是你的责任，最终署名也是你的**。

> **要点：** 这句话请记一辈子：**"看不懂、没验证的东西，不要放行。"** 他做完不等于做对。你不必会写每一行，但你得能判断"这事儿成没成"——本章第四节的工作流，就是帮你把这份判断落到实处。

## 四、核心工作流：调查 → 计划 → 动手 → 验收

几乎所有顺利的任务，都走同一个四步循环。**中间两道"闸门"由人把守**，其余交给 AI：

```text
   你给目标
      │
      ▼
 ①调查 ──→ ②计划 ──→【人批准】──→ ③动手 ──→ ④验收 ──→【人放行】
（它读项目） （它出方案）   ↑闸门1      （它改文件） （看diff/跑测试） ↑闸门2
```

**① 调查（它做，你看一眼）**
先别急着让它"开始改"。让它**先调查**："要做这件事，相关的代码在哪几个文件？现在是怎么实现的？" 它会去翻项目、给你一份带位置的说明。你扫一眼，确认它没找错地方。
> 一句话指令示例：*"先别动手，只调查：要给页面加搜索框，涉及哪些文件、现在是怎么做的？"*

**② 计划（它做，你批准）——闸门 1**
让它把"打算怎么改"先**写成计划**：动哪些文件、加不加新东西、分几步。**你读这份计划、批准了，它才开始动手。** 这一步花几十秒，却能挡掉绝大多数"跑偏了几十分钟"的返工。很多工具内置了"先出计划、批准再动手"的模式（第27章细讲；**具体叫法和开关以官方文档为准**）。

**③ 动手（它做，你随时可叫停）**
批准后它开始改。期间它每要做"有后果的操作"（改文件、跑命令），通常会**停下来请求你许可**（第26章的"许可机制"）。觉得它走歪了，**随时打断**，把话说清楚再让它继续。

**④ 验收（你做）——闸门 2**
这是**你最重要的一步**，就三个动作（第25章详解）：
- **看 diff**：diff 就是"改动前后的对比"，它会清楚标出"加了哪行、删了哪行"。你看它**到底改了什么**、有没有动到不该动的地方。
- **跑测试**：让它跑测试，确认**没把原来好好的功能弄坏**。
- **能不能讲清**：试着用一句话说"这段在干嘛"。说不上来，就回去问它"这段是干嘛的"——**看不懂别合并**。

> **为什么非得有这两道闸？** 因为编程智能体的两个特性叠在一起：**它会犯错 + 它能动手**。只要这两条同时成立，"动手前看计划、动手后验收"就不是可选项。把这两道闸守住，剩下的体力活尽管放心交给它。

## 五、第一次跑起来，该期待什么

降低一下预期，免得你被"理想中的 AI"吓到或失望：

1. **先布置好"能反悔"的环境。** **当它在你本地项目上动手时**，务必在用**版本控制（Git）**管理的项目里、并**切一个新分支**再让它动手（第26章手把手教）。这样万一改乱了，一键就能回到干净状态——这是你最大的安全网。（若你用的是网页/云端形态，平台通常有对应的隔离与回滚方式，以官方文档为准。）
2. **它会先"读一会儿"。** 它需要时间浏览项目，别以为卡住了。
3. **它会频繁找你确认。** 这是好事，是"许可机制"在保护你，不是它笨。
4. **它可能会犯错、会跑偏。** 正常。这时候——**不要在一团乱里硬纠**。最干脆的止损是：**回滚到干净状态，开一个新对话，把"刚才踩的坑"作为前提重新交代**（第13、30章）。在乱掉的上下文里追着它一句句改，往往越改越糟。
5. **它不联网就不知道最新的东西。** 涉及很新的技术，记得在话里**说清版本**，或把资料贴给它（第9章的知识截止）。

> **实际会怎么遇到：** 新手最常见的翻车，是"让它一口气把一个大功能全做完"。改动一大，你根本没法逐行验收，出了问题也难回滚。**正确姿势是把任务拆小**——做一小块、验收一小块、提交一次，再做下一块（第27章）。小步慢跑，反而是最快的路。

## 六、安全与花费，从第一天就放心上

- **别把密钥/密码/真实个人信息**贴进对话或写进代码——它会被发出去（第37章）。
- 这类工具是**按 token 计费**的；你让它读越多文件、聊越长，花得越快。保持任务聚焦、对话清爽，既省钱又提质量（第3、38章）。
- 它生成的代码**算你的责任**：合并前要复核、要为它负责（第39章）。

## 七、Claude Code 与 Codex：同在哪、不同在哪

| | 相同（本章重点，稳定） | 不同（**以官方文档为准**，会变） |
| --- | --- | --- |
| 定位 | 都是"会在你项目上动手"的编程智能体 | 出品方、内核模型、**运行形态（本地／云端／网页）**不同 |
| 工作流 | 都适用"调查→计划→动手→验收"循环 | 具体命令、快捷键、界面各异 |
| 安全 | 都该配许可机制 + 可回滚环境 | 默认行为、许可粒度不同 |
| 上手 | 都先在 Git 分支里小步试 | 安装方式、登录方式不同 |

> **要点：** 你在本章学到的工作流，**换哪个工具都管用**——因为它来自"会犯错的 Agent 该怎么带"这个不变的道理，而不是某个产品的按钮。具体到"怎么装、敲什么"，请看第25章和附录 C，并**以官方文档为最新依据**。

## 八、常见误解，集中纠正

| 误解 | 实际 |
| --- | --- |
| 它是 AI，写的代码应该对，直接合并 | 它会自信地写错；diff 要看、测试要跑、看不懂别合并 |
| 让它一次把大功能全做完最省事 | 大改动难验收、难回滚；拆小、小步验收才最快 |
| 它记得我整个项目和昨天的对话 | 模型本体靠"每次读项目 + 这次的上下文"工作，默认没有跨对话记忆；不过很多工具在**产品层**提供"项目记忆"（如 CLAUDE.md／项目上下文文件）让它跨会话记住项目规矩——那是产品功能、不是模型本体记忆（第3、28章） |
| 我不会编程就没法用 | 你不用会写每一行，但要会"派活 + 验收三问"；门槛在判断，不在敲代码 |
| 它跑偏了就追着它一直改 | 该回滚 + 开新对话重来；在乱上下文里硬纠往往更糟 |

## 小结

- Claude Code / Codex 是一类**会在你项目里动手**的编程智能体，跑在终端/编辑器里；**具体形态各异、且在更新，以官方文档为准**。
- 和网页聊天的根本区别：它**直接改文件、自己探索项目、连走好几步**——所以必须在**能回滚**的环境里用。
- 心智模型：**能干但刚入职的新同事**；活能交给他，**派活与验收是你的责任**。
- 核心工作流是**调查 → 计划 →【人批准】→ 动手 → 验收 →【人放行】**，人把守两道闸：批准计划、验收变更（看 diff、跑测试、讲得清）。
- 第一次用：先切 Git 分支、任务拆小、跑偏就回滚重来、密钥绝不外贴。
- 这套工作流**与具体工具无关**，因为它来自"会犯错的 Agent 该怎么带"——学一次，到哪都管用。

## 小测验

> 共 6 题，涵盖概念、误区、情景、操作四类。先自己想，再看引用块里的答案解析。

1. **[基础·概念题]** 编程智能体（如 Claude Code / Codex）和网页聊天 AI 最根本的区别是？
   - A. 界面更好看
   - B. 它能直接读写你的项目文件、运行命令，而不只是给你一段文字让你复制
   - C. 它不会犯错
   - D. 它不用模型
   > **答案 B。** 聊天 AI 是"你来回复制粘贴"；编程智能体是"它直接在你的项目里动手"。这正是它高效、也需要"许可机制"看着的原因。

2. **[基础·概念题]** "调查 → 计划 → 动手 → 验收"这个循环里，人最关键的工作在哪一步？
   - A. 调查
   - B. 计划批准 + 验收（看 diff、跑测试）
   - C. 动手
   - D. 人不用参与
   > **答案 B。** 让 AI 干活，人守两道闸：**动手前批准计划、动手后验收变更**。中间的体力活交给它。

3. **[基础·误区题]** "它是 AI，写的代码应该是对的，直接合并就行"——错在哪？
   - A. 没错
   - B. 它会自信地写出错的/不存在的东西；**看不懂、没验证的变更不该合并**
   - C. 它写的永远跑不了
   - D. 合并需要付费
   > **答案 B。** 幻觉（第9章）在写代码时照样发生——编造不存在的函数、用错版本的写法。diff 要看、测试要跑、看不懂就先问清楚。

4. **[进阶·误区题]** "让它一次把整个大功能全做完，最省事"——为什么往往更费事？
   - A. 不会，越大越省
   - B. 改动越大越容易跑偏，且大到你根本没法逐行验收，出了问题也难回滚
   - C. 它一次只能改一个文件
   - D. 大任务要额外付费
   > **答案 B。** 大改动 = 大盲区。把任务拆小、先看计划、小步验收，反而最快（呼应第27章）。

5. **[基础·情景题]** 你让它改代码，它跑偏了，越改越乱。最佳止损动作是？
   - A. 继续追着它一句句纠正
   - B. 用版本控制把改动回滚到干净状态，再开新对话、把"踩过的坑"作为前提重新交代
   - C. 删项目重做
   - D. 关电脑
   > **答案 B。** 在乱掉的上下文里硬纠，往往越纠越歪。**回滚 + 重开一个带着教训的干净对话**，几乎总是更快（呼应第13、30章）。先在 Git 里起步、先切分支，就是为了这一刻。

6. **[进阶·操作/观察题]** 不必真装工具：找一段你看得懂的代码改动（或第25章的 diff 示例），练习只回答三个问题——**它改了什么？会不会弄坏别的？我能不能用一句话说清这段在干嘛？** 第三个答不上来，就该回去问"这段是干嘛的"。
   > **该观察到：** "验收"不神秘，就是这三问。养成"看不懂就别放行"的肌肉记忆，比会任何命令都重要。具体工具的安装与命令**以官方文档为准**（见第25章与附录 C）。
