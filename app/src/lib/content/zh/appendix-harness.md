如果说附录 C 是"工具有哪些按钮"，那这份附录 D 讲的是"**怎么把这些按钮串成一套能信赖的干活方式**"。它来自一线团队反复打磨出的方法论，名字叫 **Harness（套上缰绳）**——核心就一句话：**别让 AI 撒开了自由发挥，而是给它套上"先理解、再立规矩、先出计划、动手、最后验证"的框架。** 无论你用 Claude Code 还是 Codex，这套方法都一样管用。

> **要点：** AI 写代码的最大风险不是"写不出来"，而是"**自作主张、另起炉灶**"——它可能无视你项目里现成的写法，硬塞进一套自己的流派，把整体设计搞乱。Harness 就是用来防这件事的：让它**顺着你已有的东西**干活，而不是推倒重来。这份方法论与第 26、27、28、29 章互为表里，建议连起来看。

## 一、Harness 是什么：四个动作

把 AI 当成一个**很能干、但第一天来、还不熟悉你项目**的合作者。你不会让这种人直接往主干上写代码，对吧？你会让他：

1. **先理解**——把现有代码、现有规矩摸清楚；
2. **立规矩**——把"本项目怎么做事"白纸黑字写下来；
3. **先出计划**——动手前先说要改哪些文件、复用哪些现成的东西，你点头他再做；
4. **再验证**——做完对照规矩自检，你也亲自看 diff（第 27 章）。

这四个动作，就是 Harness。它不依赖任何一款工具的特定按钮，是一套"心法"。

## 二、五步工作流（照着做）

把上面四个动作落到具体对话里，是这五步。每一步给一句"对 AI 说的话"的范例——这些是**大白话提示词，不是产品命令**，照着改就能用。

| 步骤 | 这一步要它干嘛 | 提示词范例（大白话） |
| --- | --- | --- |
| ① 探索 | 先摸清现状，别动手 | "动手前，先调查这个项目：整体结构、和这次需求类似的现有做法、可以复用的公共模块（网络请求 / 错误处理 / 日志 / 表单等）、命名和代码风格。**带上文件路径**列给我，先别写代码。" |
| ② 规约化 | 把调查结果固化成规矩 | "把上面的调查结果，整理成一份给 AI 看的项目规约：哪些机制必须复用、哪些事禁止做。" |
| ③ 计划闸门 | 出方案，等你批准 | "任务是〈……〉。动手前先给计划：要改 / 加哪些文件、复用哪些现成机制、要不要加新依赖（要的话说明理由）。**我批准前不要写代码。**" |
| ④ 实现 | 顺着现有的来 | "批准。严格按现有的目录结构、命名和风格，复用刚才找到的公共模块来实现。**不要自己引入新的抽象。**" |
| ⑤ 验证 | 自检 + 你把关 | "对照刚才参考的现有写法自检：有没有自造轮子、有没有破坏整体设计、有没有绕开公共模块。" 然后**你自己看 diff、跑测试**（第 27 章）。 |

> **要点：** 整套流程最关键的，是**第 ③ 步那道"计划闸门"**——动手前先看它要干嘛。很多工具内置了"计划模式"（第 27 章、附录 C）可以帮你强制这一步，**确切开关以官方文档为准**。养成"先看计划再放行"的习惯，比记住任何命令都重要。

## 三、项目规约文件：可复制的雏形

第 ② 步产出的"规约"，最好落地成项目里的一个**规约文件**，让工具每次开工自动读取（Claude Code 习惯用 `CLAUDE.md`，Codex 有自己的等价机制；**确切文件名和位置以官方文档为准**，见附录 C）。

> **要点（产品层记忆，呼应第 3、13 章）：** 这份文件之所以"让它记住项目规矩"，**不是模型本体长了记性**，而是**产品每次开工都把它重新塞进上下文**喂给模型。所以：**写进文件里的，它才"记得"；没写的，它就"不记得"。** 规约越具体，它越稳。

下面是一份通用雏形，把 `<...>` 换成你项目的真实情况：

```markdown
# 项目规约 / AI 协作规则

## 最重要原则（必须遵守）
- 顺着现有的实现方式和设计走，不要带入自己的流派。
- 引入任何新机制（库 / 抽象 / 分层）前，先查现有有没有同类的。
  - 有就复用；没有或不够用，动手前先说明理由和替代方案，等我批准。
- 禁止破坏整体架构（分层结构、依赖方向、命名规则）的改动。

## 实现流程（必须）
1. 先调查相关的现有实现、公共模块、类似功能，列出参考位置（路径）。
2. 给出实现计划（改 / 加哪些文件、复用什么、要不要加新依赖及理由）。
3. 我批准后再实现。批准前不要写代码。
4. 跑通现有的检查 / 格式化 / 测试。

## 技术栈与版本
- 语言 / 框架 / 主要库：<例：TypeScript 5 / React 18 / Vite>

## 目录结构与职责
- <例> components/：界面　hooks/：逻辑　services/：接口通信　utils/：公共函数
- 依赖方向：<例> 界面 → hooks → services 单向，禁止反向或跨层。

## 必须复用的公共机制（禁止重造）
- 网络请求：用 <例 services/apiClient>（禁止到处直接写 fetch）
- 错误处理：用 <例 AppError / errorHandler>
- 日志：用 <例 logger>（禁止直接 console.log）

## 禁止事项
- 擅自新增第三方依赖
- 绕开现有公共机制自己造轮子
- 跨层的违规依赖（如界面层直接碰数据库 / 接口细节）
```

## 四、让它真正起作用的几条诀窍

| 诀窍 | 怎么做 |
| --- | --- |
| 规约用专有名词 | "写干净点"没用；把**要复用的模块名、文件路径、禁止事项**用专有名词写死才有用。 |
| 探索、计划、实现分开 | 别想一句话搞定全部。**调查和"计划批准"那道闸门**单独走一轮，是最大的诀窍。 |
| 小步快跑 | 一次让它改一大坨，最容易跑偏。**拆小、每小步都 review**。 |
| 规约要养 | 发现它跑偏，就把对应的规矩补进规约文件，团队共享、持续更新——**越用越好用**。 |
| 最后一定人来看 | 有 Harness 也可能跑偏。**务必亲自按"是否符合现有写法"的角度审 diff**（第 27 章）。 |
| 别过度抽象 | "复用"过头会催生没必要的封装。**贴着现有代码的颗粒度走**就好。 |

## 五、上线前检查清单（最该照着过一遍的部分）

把这两张清单贴在手边。**喂进去之前**过第一张，**把它的产出真正用 / 提交之前**过第二张。

### 把内容喂给 AI 之前

- 要喂的内容里，有没有**机密代码、生产数据、个人隐私**？（有就先脱敏，或干脆别喂——见第 37 章）
- 有没有混进 **API 密钥、密码、连接信息**？（密钥绝不能喂——见第 40 章）
- 你用的**服务和套餐**，是不是公司 / 场景规定允许的？
- 有没有把**前提**讲清楚（语言、框架、版本）？
- 有没有给够**上下文**（相关代码、日志、期望的行为）？

### 提交 / 采用它的产出之前

- **diff 有没有一行行读懂再 review**？（第 27 章）——**读不懂的代码不要用**。
- 自动 / 手动**测试是否通过**，有没有把原来好好的功能搞坏（回归）？
- 有没有从**安全角度**看一眼（输入校验、权限、注入风险）？
- 新加的**依赖库**，许可证和来源可信吗？
- 它给的**函数名 / API / 配置项是不是真的存在**？（拿官方文档核对——它会一本正经编造，见第 9 章）
- **重要的事实、数字、引文**，有没有自己复核？（第 9、15、39 章）

## 六、团队要立规矩时（给带队的人）

一个人用，靠习惯；一群人用，要靠**明文规矩**。带团队的话，至少把这六件事定下来：

1. **划线**：哪些数据可以喂给 AI、哪些绝对不行，白纸黑字写清楚。
2. **限定工具**：允许用哪些服务、哪些套餐、能连哪些外部源（呼应附录 C 的 MCP 安全）。
3. **定审批流**：AI 生成的代码，review → 合并 → 部署的批准流程是什么。
4. **管密钥**：密钥怎么存、谁能碰、**泄露了怎么轮换作废**（第 40 章）。
5. **建共享资产**：把项目规约文件、常用提示词模板（附录 B）整理出来，团队共用。
6. **建机制**：定期做分享、设个答疑窗口、规矩定期复审——AI 在变，规矩也要跟着更新。

## 小结

- **Harness = 给 AI 套上"理解 → 规约 → 计划 → 实现 → 验证"的缰绳**，让它顺着你现有的东西干活，而不是另起炉灶。Claude Code 和 Codex 都适用。
- 落地是**五步工作流**，其中**"计划闸门"（动手前先看计划）是命门**。
- 把规约写进**项目规约文件**——记住那是**产品层记忆**（每次重新喂回去），写得越具体越稳。
- 两张**检查清单**护住两端：喂之前别泄密、用之前先读懂；**读不懂的代码不要用，重要的事实自己复核**。
- 产品的规格、价格、数据条款都会变——**最新信息以官方文档和你自己（公司）的 AI 使用规程为准**。
